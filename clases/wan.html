🌍 ¿Qué es una WAN y por qué existe?

Una WAN (Wide Area Network) conecta redes LAN geográficamente separadas — entre ciudades, países o continentes. A diferencia de la LAN (propiedad y gestión completa de la empresa), la WAN generalmente utiliza la infraestructura de un proveedor de servicios (ISP o carrier) para transportar el tráfico entre sedes remotas.

La pregunta fundamental que define el diseño WAN de cualquier empresa es: ¿cómo conecto mis oficinas de Bogotá, Medellín y Miami de forma segura, confiable y económica? La respuesta ha evolucionado dramáticamente en las últimas décadas.

🏢 Conectividad sede-sede

Interconectar sucursales, oficinas y datacenters de una empresa distribuida geográficamente. El tráfico entre empleados de diferentes ciudades usa la WAN.

☁️ Acceso a Internet / Cloud

Conectar la red corporativa con Internet, servicios cloud (AWS, Azure, GCP), SaaS (Office 365, Salesforce) y aplicaciones críticas del negocio.

🔐 Seguridad perimetral

La frontera entre la red interna y la WAN es donde se aplican firewalls, VPNs, filtrado de contenido y políticas de seguridad corporativas.

📊 QoS y SLA

Los proveedores WAN ofrecen acuerdos de nivel de servicio con garantías de latencia, jitter y disponibilidad. Crítico para VoIP y videoconferencia.

📌 La evolución WAN: de circuitos dedicados a Internet+VPN

Históricamente, las empresas usaban circuitos dedicados muy costosos (Frame Relay, ATM, T1) para conectar sedes. Hoy, la tendencia dominante es usar Internet de banda ancha económico + VPN cifrada para la mayoría de los casos, reservando MPLS solo para tráfico crítico con requisitos de QoS. SD-WAN lleva este concepto al siguiente nivel automatizando la selección del enlace en tiempo real.

Las tecnologías WAN han evolucionado dramáticamente desde los años 90. Entender esta evolución es clave para el CCNA y para diseñar redes modernas.

📌 Frame Relay — Conceptos clave para el CCNA

  • DLCI (Data Link Connection Identifier): número local que identifica el circuito virtual. Local — el mismo PVC puede tener DLCIs diferentes en cada extremo.
  • PVC (Permanent Virtual Circuit): circuito virtual permanentemente configurado por el ISP entre dos puntos.
  • NBMA (Non-Broadcast Multi-Access): múltiples destinos pero sin capacidad de broadcast nativa. Complica OSPF.
  • LMI (Local Management Interface): protocolo de señalización entre el router y el switch Frame Relay del ISP. Tipos: Cisco, ANSI, ITU-T.

📌 SD-WAN — La revolución actual

SD-WAN permite que una empresa use simultáneamente MPLS + Internet broadband + 4G LTE, y el controlador central decide en tiempo real qué tráfico va por cuál enlace basándose en la calidad del enlace, el tipo de aplicación y las políticas del negocio. Si el MPLS falla, el tráfico de voz migra automáticamente a Internet en milisegundos.

⚠️ DSL vs Cable — Diferencia fundamental

DSL: cada cliente tiene su propia línea dedicada al DSLAM del ISP. El ancho de banda no se comparte con vecinos.
Cable: el medio físico (coaxial) se comparte entre todos los usuarios del mismo segmento. La velocidad real varía según cuántos vecinos estén activos simultáneamente.

🔵 HDLC — High-Level Data Link Control

HDLC (ISO 3309) es el protocolo de encapsulación por defecto en todos los enlaces seriales de routers Cisco. Es sencillo, eficiente y de bajo overhead — pero con una trampa crítica: la versión Cisco de HDLC incluye un campo propietario de 2 bytes que identifica el protocolo de Capa 3, haciendo que sea incompatible con implementaciones HDLC de otros fabricantes.

⚡ Lightweight

Overhead mínimo. Sin autenticación, sin negociación de parámetros. Solo envía y recibe. Ideal para enlace punto a punto entre dos routers Cisco.

🚫 Sin autenticación

HDLC no tiene mecanismo de autenticación. Cualquier dispositivo que conecte al enlace puede participar. En PPP se resuelve con PAP/CHAP.

🔒 Solo Cisco ↔ Cisco

La variante Cisco de HDLC es propietaria. Para conectar con dispositivos de otros fabricantes, usar PPP (estándar abierto).

📦 Estructura del frame HDLC

CampoTamañoValor/Descripción
Flag1 byte0x7E — delimita inicio y fin del frame
Address1 byte0x0F (unicast) o 0xFF (broadcast) en HDLC Cisco
Control1 byte0x00 — tipo de frame (Information frame)
Protocol (Cisco)2 bytesCAMPO PROPIETARIO CISCO: identifica el protocolo L3 (0x0800=IP, 0x86DD=IPv6)
DataVariableEl paquete IP (o de otro protocolo L3)
FCS2-4 bytesFrame Check Sequence — detección de errores (CRC)
Flag1 byte0x7E — cierre del frame

⚠️ El campo "Protocol" de 2 bytes = HDLC propietario Cisco

El estándar HDLC ISO no incluye este campo. Cisco lo agregó para soportar múltiples protocolos de Capa 3. Por eso, un router Cisco con HDLC en una serial no puede comunicarse con un router Juniper o Huawei que use HDLC estándar ISO — se necesitaría PPP.

🟠 PPP — Point-to-Point Protocol

PPP (RFC 1661) es el estándar abierto para encapsulación en enlaces punto a punto. A diferencia de HDLC, PPP incluye autenticación (PAP y CHAP), negociación de parámetros de enlace, compresión y detección de errores. Es multi-vendor e interopera con cualquier fabricante.

🏗️ Arquitectura de PPP — Los tres componentes

  1. Framing: el formato del frame PPP (basado en HDLC). Encapsula los datagramas para transmisión.
  2. LCP (Link Control Protocol): establece, configura, prueba y termina el enlace PPP. Negocia parámetros: MTU, autenticación, compresión, multilink.
  3. NCP (Network Control Protocol): negocia los parámetros de cada protocolo de red separadamente. IPCP para IP, IPv6CP para IPv6, IPXCP para IPX (obsoleto).

🤝 Proceso de establecimiento PPP — 3 fases LCP

Fase 1 — Link Establishment (LCP)

LCP intercambia paquetes de configuración para negociar los parámetros del enlace:

  • Configure-Request: propone opciones (autenticación CHAP, MRU, compresión)
  • Configure-Ack: acepta todas las opciones propuestas
  • Configure-Nak: acepta parcialmente — sugiere valores alternativos
  • Configure-Reject: rechaza opciones desconocidas o no soportadas

Fase 2 — Authentication (opcional)

Si se negoció autenticación en la Fase 1, se ejecuta PAP o CHAP. Si falla → el enlace se cierra.

Fase 3 — Network Layer Protocol (NCP/IPCP)

IPCP negocia los parámetros de IP: dirección IP, servidor DNS, servidor WINS. Solo después de esta fase el enlace puede transportar tráfico IP real.

🔑 CHAP — El proceso detallado (3-way handshake)

  1. El autenticador (servidor) envía un Challenge: número aleatorio + nombre del autenticador
  2. El autenticado (cliente) calcula MD5(id + contraseña + Challenge) y envía el resultado (Response)
  3. El autenticador calcula el mismo MD5 localmente con la contraseña que tiene en su base de datos. Si coincide → Success. Si no → Failure y el enlace se cierra.

La contraseña nunca viaja por el enlace. El Challenge cambia en cada autenticación, por lo que una respuesta capturada no puede reutilizarse (anti-replay).

🚇 GRE — Generic Routing Encapsulation

GRE (RFC 2784) es un protocolo de tunneling que encapsula paquetes de cualquier protocolo dentro de paquetes IP. Crea una interfaz virtual punto a punto (Tunnel) entre dos routers a través de una red IP intermedia (generalmente Internet). El tráfico encapsulado "viaja" transparentemente por la red pública.

📦 Cómo funciona la encapsulación GRE

Cuando el router R1 necesita enviar un paquete al router R2 a través de Internet:

  1. El paquete original (IP destino: 192.168.2.10) llega al router R1
  2. R1 agrega una cabecera GRE (4 bytes) al paquete original
  3. R1 agrega una cabecera IP exterior: src=IP pública R1, dst=IP pública R2
  4. El paquete encapsulado viaja por Internet como tráfico IP normal
  5. R2 recibe el paquete, elimina las cabeceras GRE e IP exterior
  6. R2 enruta el paquete original hacia 192.168.2.10
Característica GREValor
Protocolo IP número47 (en la cabecera IP exterior)
Overhead24 bytes (20 IP exterior + 4 GRE)
MTU efectivo1476 bytes (1500 - 24 bytes de overhead)
Cifrado❌ No — GRE solo encapsula, NO cifra
Multicast/Broadcast✅ Sí — puede transportar multicast
Protocolos transportadosIP, IPv6, IPX, cualquier protocolo
AutenticaciónOpcional — clave simple (no CHAP)

⚠️ GRE no cifra el tráfico

GRE crea el túnel pero no proporciona confidencialidad. El contenido del paquete original es visible para quien capture el tráfico en Internet. Para tráfico seguro sobre Internet se debe combinar GRE con IPsec, o usar directamente IPsec sin GRE. GRE solo es apropiado en redes privadas o cuando el cifrado no es un requisito.

⚙️ Configuración completa GRE Tunnel

Escenario: Conectar LAN 192.168.1.0/24 con LAN 192.168.2.0/24 via GRE sobre Internet

! ─── TOPOLOGÍA ──────────────────────────────────────
! R1: LAN 192.168.1.0/24, IP pública: 203.0.113.1
! R2: LAN 192.168.2.0/24, IP pública: 198.51.100.1
! Tunnel: 172.16.0.0/30 (IPs lógicas del túnel)

! ─── R1 ─────────────────────────────────────────────
R1(config)# interface Tunnel0
R1(config-if)# description GRE-TUNNEL-A-R2
R1(config-if)# ip address 172.16.0.1 255.255.255.252
! IP lógica del extremo R1 del túnel
R1(config-if)# tunnel source GigabitEthernet0/0
! O: tunnel source 203.0.113.1 (IP pública de R1)
R1(config-if)# tunnel destination 198.51.100.1
! IP pública del otro extremo (R2)
R1(config-if)# tunnel mode gre ip
! Default: GRE sobre IP. Explícito para claridad.
R1(config-if)# no shutdown

! Ruta hacia la red remota via el túnel
R1(config)# ip route 192.168.2.0 255.255.255.0 172.16.0.2
! O usar un protocolo de routing sobre el túnel (OSPF, EIGRP)

! ─── R2 ─────────────────────────────────────────────
R2(config)# interface Tunnel0
R2(config-if)# description GRE-TUNNEL-A-R1
R2(config-if)# ip address 172.16.0.2 255.255.255.252
R2(config-if)# tunnel source GigabitEthernet0/0
R2(config-if)# tunnel destination 203.0.113.1
! Destino = IP pública de R1
R2(config-if)# no shutdown
R2(config)# ip route 192.168.1.0 255.255.255.0 172.16.0.1

🔐 VPN IPsec — Conexión segura sobre redes públicas

IPsec (IP Security) es una suite de protocolos que proporciona confidencialidad, integridad y autenticación para el tráfico IP. A diferencia de GRE que solo encapsula, IPsec cifra el contenido. Es el estándar de facto para VPNs empresariales site-to-site y acceso remoto seguro.

🔒 Confidencialidad

Cifrado del tráfico con AES (128/256 bits), 3DES. Solo quien tiene la clave puede descifrar.

🔏 Integridad

Hash HMAC-SHA256/SHA512 garantiza que el paquete no fue alterado en tránsito.

✅ Autenticación

Pre-shared key (PSK) o certificados digitales (PKI) verifican la identidad del otro extremo.

🔄 Anti-replay

Números de secuencia previenen que un atacante replaye paquetes capturados.

🏗️ Arquitectura IPsec — Los componentes clave

AH vs ESP — Los dos protocolos IPsec

ProtocoloNúmero IPConfidencialidadIntegridadAutenticaciónUso

ESP (Encapsulating Security Payload)50✅ Sí✅ Sí✅ Sí✅ Estándar — cifra y autentica

Modos IPsec — Transport vs Tunnel

Modo¿Qué cifra?Cabecera IPUso

Tunnel ModeTodo el paquete IP originalNueva cabecera IP exterior✅ Site-to-site VPN (lo normal)

IKE (Internet Key Exchange) — La negociación en 2 fases

📋 IKEv1 — Fases de negociación

Fase 1 (ISAKMP SA): establece un canal seguro bidireccional para la negociación. Negocia: algoritmo de cifrado (AES/3DES), hash (SHA/MD5), método de autenticación (PSK/RSA), grupo Diffie-Hellman (DH1/2/5/14). Resultado: ISAKMP SA.

Fase 2 (IPsec SA): usa el canal seguro de Fase 1 para negociar los parámetros del túnel IPsec real. Negocia: protocolo (ESP/AH), algoritmo de cifrado, hash, tráfico interesante (ACL). Resultado: 2 IPsec SAs unidireccionales.

⚙️ Configuración VPN IPsec Site-to-Site en Cisco IOS

Configuración completa en 5 pasos

! ─── PASO 1: ISAKMP Policy (Fase 1) ────────────────
R1(config)# crypto isakmp policy 10
R1(config-isakmp)# encryption aes 256
! Cifrado Fase 1: AES-256
R1(config-isakmp)# hash sha256
! Hash para integridad: SHA-256
R1(config-isakmp)# authentication pre-share
! Autenticación: PSK (pre-shared key)
R1(config-isakmp)# group 14
! Diffie-Hellman group 14 (2048 bits) — más seguro que group 2
R1(config-isakmp)# lifetime 86400
! SA válida 24 horas antes de renegociar

! ─── PASO 2: Pre-Shared Key ─────────────────────────
R1(config)# crypto isakmp key VPNsecret2024! address 198.51.100.1
! La PSK debe ser IDÉNTICA en ambos extremos
! 198.51.100.1 = IP pública del otro extremo (R2)

! ─── PASO 3: IPsec Transform Set (Fase 2) ───────────
R1(config)# crypto ipsec transform-set TS-AES256 esp-aes 256 esp-sha256-hmac
! esp-aes 256: cifrado con AES-256
! esp-sha256-hmac: hash SHA-256 para integridad

! ─── PASO 4: ACL de tráfico interesante ─────────────
R1(config)# ip access-list extended ACL-VPN-INTERESANTE
R1(config-ext-nacl)# permit ip 192.168.1.0 0.0.0.255 192.168.2.0 0.0.0.255
! Solo el tráfico entre LAN1 y LAN2 se cifra y envía por el túnel

! ─── PASO 5: Crypto Map ─────────────────────────────
R1(config)# crypto map CMAP-VPN 10 ipsec-isakmp
R1(config-crypto-map)# set peer 198.51.100.1
! IP pública del peer (R2)
R1(config-crypto-map)# set transform-set TS-AES256
R1(config-crypto-map)# match address ACL-VPN-INTERESANTE
R1(config-crypto-map)# set pfs group14
! Perfect Forward Secrecy: nuevas claves en cada renegociación

! ─── APLICAR EN LA INTERFAZ WAN ─────────────────────
R1(config)# interface GigabitEthernet0/0
R1(config-if)# crypto map CMAP-VPN
! El crypto map se aplica en la interfaz hacia el peer

🏷️ MPLS — Multiprotocol Label Switching

MPLS es una tecnología de red que acelera el reenvío de paquetes usando etiquetas cortas en lugar de buscar la dirección IP en cada salto. El enrutador ya no necesita consultar la tabla de routing completa para cada paquete — simplemente lee la etiqueta y reenvía. Esto reduce la latencia y permite ingeniería de tráfico avanzada.

📌 MPLS es la red privada del ISP — El cliente solo ve la "nube"

Desde la perspectiva del cliente, MPLS es transparente. El cliente conecta sus sedes con interfaces Ethernet o Serial hacia el proveedor MPLS (PE router). El proveedor gestiona toda la infraestructura interna de etiquetas. Para el cliente, parece que todas sus sedes están directamente conectadas en una red privada.

🏗️ Componentes de la red MPLS

ComponenteFunciónUbicación
CE (Customer Edge)Router del cliente que conecta a la red MPLS del ISP. Habla routing (OSPF/BGP) con el PE.En las instalaciones del cliente
PE (Provider Edge)Router de borde del proveedor. Asigna y elimina etiquetas MPLS. Mantiene las tablas VRF por cliente.En el POP del ISP
P (Provider)Routers del núcleo del ISP. Solo reenvían basándose en etiquetas MPLS — no ven rutas del cliente.Core del ISP
LSP (Label Switched Path)El camino que sigue un flujo de tráfico a través de la red MPLS, determinado por las etiquetas.Virtual — a través de la red
VRF (Virtual Routing & Forwarding)Tabla de routing separada por cliente/VPN en el PE. Permite aislar el tráfico entre clientes.En los PEs del ISP

🔖 La etiqueta MPLS — Estructura técnica

La etiqueta MPLS se inserta entre la cabecera de Capa 2 (Ethernet) y la cabecera IP. Por eso se dice que MPLS opera en la "Capa 2.5".

CampoBitsDescripción
Label Value20 bitsEl valor de la etiqueta (0 a 1.048.575). Identifica el LSP.
TC (Traffic Class)3 bitsClase de servicio (QoS) — equivale al DSCP de IP. Permite QoS en la red MPLS.
S (Bottom of Stack)1 bit1 = última etiqueta del stack. MPLS soporta etiquetas apiladas.
TTL8 bitsTime To Live — decrementado en cada LSR para evitar bucles.

⚙️ Operaciones MPLS — Push, Swap, Pop

OperaciónDescripciónQuién la realiza
PUSH (Imposición)Agrega una etiqueta MPLS al paquete entrantePE de entrada (Ingress PE)
SWAP (Conmutación)Reemplaza la etiqueta entrante por una nueva etiquetaRouters P del núcleo
POP (Disposición)Elimina la etiqueta MPLS, dejando el paquete IP originalPE de salida (Egress PE) o penúltimo salto (PHP)

✅ Ventajas de MPLS para empresas

📊 QoS garantizado

Los ISP MPLS ofrecen SLAs con latencia máxima garantizada. Crítico para VoIP y videoconferencia. El campo TC de la etiqueta MPLS permite marcar el tráfico.

🔒 Privacidad

El tráfico viaja en la red privada del ISP, segregado de otros clientes mediante VRFs. No pasa por Internet público.

⚡ Routing dinámico CE-PE

El cliente puede ejecutar OSPF, EIGRP o BGP con el router PE del ISP para intercambio dinámico de rutas entre sedes.

🌐 Any-to-Any

Todas las sedes pueden comunicarse entre sí directamente (full-mesh lógico) sin necesidad de un hub central. Simplifica la topología.

📊 MPLS vs Internet+VPN — ¿Cuándo usar cuál?

📋 Resumen Maestro — WAN para el examen CCNA

✅ Los 28 puntos más críticos del examen

  1. WAN conecta redes LAN geográficamente separadas usando infraestructura de proveedores.
  2. Componentes WAN: CPE (cliente), demarcación, local loop, POP del ISP, backbone.
  3. HDLC es la encapsulación por defecto en interfaces seriales de Cisco IOS.
  4. HDLC Cisco es propietario — tiene campo Protocol de 2 bytes que lo hace incompatible con HDLC ISO estándar.
  5. Para conectar Cisco con otros fabricantes en enlace serial → usar PPP (estándar abierto).
  6. PPP (RFC 1661): encapsulación estándar con autenticación, multiprotocolo, compresión.
  7. PPP tiene 3 componentes: Framing + LCP (establece enlace) + NCP (negocia protocolos L3).
  8. PPP establece en 3 fases: Link Establishment (LCP) → Authentication → Network Layer (NCP/IPCP).
  9. PAP: 2-way handshake. Envía contraseña en texto plano. Inseguro.
  10. CHAP: 3-way handshake. Usa MD5(Challenge+Password). Nunca envía la contraseña. Seguro.
  11. CHAP: el username configurado debe ser el hostname del vecino. La contraseña debe ser idéntica en ambos extremos.
  12. GRE (RFC 2784): protocolo IP número 47. Encapsula cualquier protocolo. No cifra.
  13. GRE overhead: 24 bytes (20 IP exterior + 4 GRE). MTU efectivo: 1476 bytes.
  14. GRE no cifra → para seguridad combinar con IPsec (GRE over IPsec) o usar IPsec directamente.
  15. Configurar GRE: tunnel source (IP/interfaz local) + tunnel destination (IP extremo remoto).
  16. IPsec: suite de protocolos para confidencialidad, integridad y autenticación IP.
  17. IPsec protocolos: AH (IP#51, no cifra) y ESP (IP#50, cifra). ESP es el estándar.
  18. IPsec modos: Transport (solo payload) y Tunnel (todo el paquete original). Tunnel para site-to-site.
  19. IKE Fase 1 = ISAKMP SA (canal seguro para negociar). Fase 2 = IPsec SA (túnel real).
  20. IPsec config en Cisco: ISAKMP Policy → PSK → Transform Set → ACL interesante → Crypto Map → aplicar en interfaz.
  21. MPLS: etiquetas de 32 bits (20 label + 3 TC + 1 S + 8 TTL) entre L2 y L3 (capa 2.5).
  22. MPLS componentes: CE (cliente) ↔ PE (borde ISP) ↔ P (core ISP).
  23. MPLS operaciones: PUSH (PE entrada) → SWAP (routers P) → POP (PE salida).
  24. VRF en MPLS: tabla de routing separada por cliente — aísla el tráfico entre clientes del ISP.
  25. DSL velocidad inversamente proporcional a la distancia del cliente al DSLAM.
  26. SD-WAN: abstrae múltiples enlaces WAN (MPLS+Internet+LTE) con selección inteligente automática.
  27. show interfaces Serial: verificar encapsulación y estado. show ppp all: estado LCP/NCP.
  28. debug ppp authentication: ver en tiempo real el handshake PAP/CHAP (en laboratorio).

⚠️ Los 12 errores más comunes en el examen CCNA

  1. Poner clock rate en el lado DTE del enlace Serial (solo el DCE necesita clock rate)
  2. En CHAP: username del vecino mal escrito o contraseñas diferentes en cada router
  3. Confundir encapsulaciones en extremos del Serial (R1=HDLC, R2=PPP) → line protocol down
  4. Creer que GRE cifra el tráfico — solo encapsula, sin confidencialidad
  5. No configurar ruta estática (o protocolo de routing) apuntando al túnel GRE
  6. Tunnel source e destination invertidos (source del extremo remoto en lugar del local)
  7. Confundir AH (no cifra) con ESP (cifra) en IPsec
  8. ACL de tráfico interesante IPsec errónea → no se establece el túnel
  9. No aplicar el crypto map en la interfaz WAN → IPsec nunca funciona
  10. Confundir IKE Fase 1 (ISAKMP SA) con Fase 2 (IPsec SA) en show crypto
  11. Creer que HDLC funciona entre Cisco y otros fabricantes — es propietario
  12. En PAP olvidar el comando ppp pap sent-username en el lado del cliente
