¿Qué son los protocolos de descubrimiento de vecinos?
Los protocolos de descubrimiento de capa 2 permiten que los dispositivos de red descubran automáticamente información sobre sus vecinos directamente conectados — sin configuración manual ni acceso físico al dispositivo remoto. Operan en la Capa 2 del modelo OSI y son fundamentales para documentación, troubleshooting y gestión. Los frames van a una dirección multicast L2 — no se reenvían más allá del enlace directo.
🗺️
Mapeo de topología
Descubren dispositivos, puerto de conexión y capacidades. Esencial para documentar redes desconocidas en minutos.
🔧
Troubleshooting
Verifican conexiones físicas, detectan misconfiguraciones y confirman la topología real vs la documentada.
📊
Inventario automático
Obtienen modelo, versión IOS, IP y capacidades del vecino sin acceso directo al dispositivo.
⚡
Sin IP requerida
Funcionan aunque el vecino no tenga IP configurada. Solo requieren conectividad a nivel de enlace L2.
🔄
Anuncios periódicos
Envían helllos en intervalos configurables. Si no llega anuncio en el holdtime, el vecino expira.
🎯
Solo vecinos directos
Solo descubren el primer salto. Los frames multicast L2 no se propagan más allá del enlace inmediato.
⚖️

Principio clave: Los frames CDP/LLDP van a una dirección multicast L2 específica y NO son reenviados por switches ni routers. Cada dispositivo anuncia su propia información y recibe la del vecino de forma independiente. Alcance: solo el enlace directo.
🔵
CDP — Cisco Discovery Protocol en profundidad
CDP es un protocolo propietario Cisco encapsulado en frames LLC/SNAP sobre capa 2. Cada dispositivo envía anuncios CDP cada 60 segundos. Si no se recibe ningún anuncio en 180 segundos (holdtime), el vecino se elimina de la tabla CDP. Funciona en routers, switches, teléfonos IP y APs Cisco.

Casos de uso prácticos
  1. Documentar topología desconocida
    show cdp neighbors revela todos los dispositivos conectados directamente. En minutos mapeas una red sin documentación previa.
  2. Verificar conexiones físicas
    Confirmar que el cable está en el puerto correcto del switch remoto sin moverse al rack. El Port-ID indica exactamente dónde conecta.
  3. Detectar duplex mismatch
    CDP informa el duplex del vecino. Si hay mismatch (uno full, otro half) se generan colisiones, errores y pérdida de performance.
  4. Soporte VoIP — Voice VLAN
    Teléfonos IP Cisco usan CDP para negociar la Voice VLAN con el switch y recibir configuración PoE automáticamente.

LLDP — IEEE 802.1AB en profundidad
LLDP (IEEE 802.1AB) es el estándar abierto multi-vendor equivalente a CDP. Funciona entre Cisco, HP/Aruba, Juniper, Huawei y cualquier fabricante compatible. En Cisco IOS está deshabilitado por defecto y debe activarse con lldp run. Característica única: el Tx y Rx son independientes — se pueden controlar por separado por interfaz.
TLVs de LLDP — Obligatorios y Opcionales

Los mensajes LLDP se componen de bloques TLV (Type/Length/Value). Cuatro son mandatorios en cualquier implementación:

Obligatorio
Chassis ID
Identificador único del dispositivo (MAC, hostname o IP)
Obligatorio
Port ID
Identificador del puerto desde donde se envía el frame
Obligatorio
Time to Live
Holdtime en segundos (TTL = Timer × Multiplier)
Obligatorio
End of LLDPDU
Marca el fin del frame LLDP (TLV tipo 0)
Opcional
System Name
Hostname del dispositivo
Opcional
System Description
SO, versión firmware y hardware
Opcional
System Capabilities
Router, Switch, Bridge, Phone, WLAN AP
Opcional
Management Address
IP de gestión del dispositivo
Opcional
Port Description
Descripción del puerto (interface description)
Opcional
VLAN Name
Nombre de VLAN — extensión IEEE 802.1
Opcional
MAC/PHY Config
Duplex, velocidad y capacidades físicas del puerto
Opcional
LLDP-MED
Extensión VoIP, PoE y localización
📞
LLDP-MED — Media Endpoint Discovery
LLDP-MED (ANSI/TIA-1057) extiende LLDP para entornos de comunicaciones unificadas. Proporciona capacidades avanzadas para teléfonos IP, sistemas de conferencia y dispositivos PoE. Compatible entre cualquier fabricante que implemente el estándar.
📞
Network Policy
Comunica VLAN de voz, prioridad CoS/DSCP y política QoS al teléfono IP automáticamente vía switch.
⚡
Extended Power
Negociación PoE detallada — el endpoint informa la potencia exacta requerida al switch PoE.
📍
Location
Información de ubicación física para servicios E-911. El switch comunica la localización al endpoint.
🔎
Inventory
Firmware, modelo y serial number del endpoint para inventario automático de activos.
🏷️
Device Class
Class I (genérico), Class II (media endpoint), Class III (teléfono IP VoIP con llamadas de voz).
🔄
Multi-vendor
Switch Cisco con teléfonos Polycom, Avaya, Yealink o cualquier endpoint que implemente 802.1AB.
💡
CDP vs LLDP-MED en VoIP: Teléfonos IP Cisco usan CDP para negociar Voice VLAN. En redes multi-vendor LLDP-MED es la solución estándar. En entornos empresariales modernos se habilitan ambos simultáneamente para soportar cualquier fabricante.

Riesgos de seguridad — CDP/LLDP como vectores de ataque
Aunque CDP y LLDP son herramientas de gestión esenciales, también representan riesgos de seguridad críticos si están habilitados en puertos orientados a usuarios o redes no confiables. Un atacante con un simple Wireshark puede capturar frames CDP sin ninguna autenticación y obtener información detallada de la infraestructura.
🔍
Reconocimiento de red
CDP revela modelo exacto, versión IOS y topología completa. Un atacante identifica vulnerabilidades específicas de la plataforma para lanzar exploits dirigidos.
🎭
Spoofing de vecinos
Un atacante puede enviar frames CDP/LLDP falsos para llenar la tabla de vecinos o impersonar un switch legítimo de la red.
💥
DoS — Memory exhaustion
Enviar miles de frames CDP con Device-IDs distintos agota la memoria del dispositivo manteniendo entradas de vecinos falsas (Cisco Security Advisory).
📡
Exfiltración de datos
VTP domain, Native VLAN, versión IOS — información que permite planificar ataques VLAN hopping y seleccionar exploits específicos del SO.
☠️
CVE-2020-3110: Vulnerabilidad en el procesamiento de paquetes CDP en cámaras IP Cisco que permitía Remote Code Execution (RCE) sin autenticación. Cisco publicó un advisory recomendando deshabilitar CDP en todos los dispositivos finales. Prueba de que CDP no es solo una herramienta de gestión inocua.

Tarea Práctica — Lab CDP/LLDP en Packet Tracer
🔬 Escenario: Red corporativa sin documentación

Red: R1 → SW1 → (SW2, SW3) → PCs y 2 Teléfonos IP Cisco
Solo tienes acceso inicial a R1. Objetivo: mapear toda la topología usando CDP y LLDP, luego aplicar hardening de seguridad en puertos de acceso.

  1. Desde R1: show cdp neighbors detail
    Documentar SW1: hostname, IP, modelo, interface local y remota. Esta será la puerta de entrada a toda la topología.
  2. Acceder a SW1 y repetir el descubrimiento
    show cdp neighbors detail desde SW1. Descubrir SW2, SW3 y los teléfonos IP. Construir el mapa de topología completo.
  3. Activar LLDP en todos los dispositivos
    lldp run en R1, SW1, SW2, SW3. show lldp neighbors — comparar con CDP. Observar TLVs adicionales que LLDP proporciona.
  4. Hardening: deshabilitar en puertos de usuario
    interface range Fa0/1-24 en cada switch → no cdp enable, no lldp transmit, no lldp receive. Los PCs NO deben aparecer como vecinos.
  5. Verificación final de seguridad
    show cdp neighbors y show lldp neighbors deben mostrar SOLO la infraestructura (switches, router). Los PCs y endpoints deben desaparecer de la tabla.
