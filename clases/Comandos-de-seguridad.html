M2.5 · COMANDOS SEGURIDAD · HARDENING CISCO IOS

Comandos de Seguridad

Domina la seguridad de dispositivos Cisco IOS desde la autenticación local hasta ACLs, Port Security, SSH, NTP y políticas de bloqueo de acceso. Protege switches y routers contra accesos no autorizados, ataques de capa 2 y vulnerabilidades de administración. Esencial para CCNA 200-301 Exam Topics 6.1, 6.2, 6.3.

📋 IOS 15.x / IOS XE 🎯 CCNA 200-301 Exam Topic 6.1, 6.2, 6.3 ⏱ ~100 min lectura 🧪 Lab: 2 Routers + 2 Switches + 2 PCs

Fundamentos de Seguridad en Cisco IOS

La seguridad en redes Cisco se divide en múltiples capas: protección del plano de administración (acceso al dispositivo), plano de control (protocolos de routing), plano de datos (tráfico de usuarios) y seguridad física. Esta clase cubre los comandos esenciales para hardening de dispositivos Cisco.

Pilares de Seguridad Cisco IOS

Principios de Hardening Cisco

🔐

Cifrado de Contraseñas

Todas las contraseñas en running-config deben estar cifradas. Usa service password-encryption para tipo 7 y enable secret para tipo 5/8/9.

service password-encryption
🔒

SSH sobre Telnet

Nunca uses Telnet (texto plano). Configura siempre SSH v2 con dominio, RSA key y solo transport input ssh.

transport input ssh
🛡️

Control de Acceso

Usa ACLs para restringir quién puede administrar el dispositivo y qué tráfico puede atravesarlo.

access-class 10 in
🔌

Seguridad de Capa 2

Port Security, DHCP Snooping y DAI protegen contra MAC flooding, rogue DHCP y ARP spoofing.

switchport port-security
⚠️ Regla de Oro: Nunca uses "enable password" en producción

El comando enable password almacena la contraseña en texto plano o con cifrado reversible tipo 7 (Vigenère). Siempre usa enable secret que aplica hash MD5 (tipo 5) o SHA-256 (tipo 8/9 en IOS moderno). Si configuras ambos, enable secret tiene prioridad.

Contraseñas, Usuarios y Cifrado

La primera línea de defensa es controlar quién puede acceder al dispositivo. Cisco IOS ofrece múltiples mecanismos de autenticación local que deben configurarse correctamente para evitar accesos no autorizados.

Comandos de Contraseña Básicos

Configuración de contraseñas de acceso
! === CONTRASEÑA DE CONSOLA ===
R1(config)# line console 0
R1(config-line)# password CiscoConsole123!
R1(config-line)# login
! "login" fuerza autenticación con la contraseña configurada
R1(config-line)# exit

! === CONTRASEÑA DE LÍNEAS VTY (Telnet/SSH) ===
R1(config)# line vty 0 15
R1(config-line)# password CiscoVTY123!
R1(config-line)# login
R1(config-line)# exit

! === ENABLE PASSWORD (NO RECOMENDADO) ===
R1(config)# enable password CiscoEnable123!
! Almacenado en texto plano o cifrado tipo 7 (débil)
! Fácilmente reversible con herramientas online

! === ENABLE SECRET (RECOMENDADO) ===
R1(config)# enable secret CiscoSecret123!
! Hash MD5 (tipo 5) - NO reversible
! Si existe enable secret, enable password se ignora

! === CIFRAR TODAS LAS CONTRASEÑAS EN RUNNING-CONFIG ===
R1(config)# service password-encryption
! Cifra contraseñas tipo 7 (Vigenère) - reversible pero oculta
! NO cifra enable secret (ya está hasheado)
! NO cifra username secret (ya está hasheado)

! === VERIFICAR EN RUNNING-CONFIG ===
R1# show running-config | section enable
enable secret 5 $1$mERr$H7PDMLdMZ8n.ZGfZ8n.ZGf0
enable password 7 0822455D0A16
! 5 = MD5 hash (enable secret)
! 7 = Cifrado tipo 7 reversible (enable password con service password-encryption)

Autenticación Local con Usuarios

Creación de usuarios locales seguros
! === USUARIO CON CONTRASEÑA TIPO 0 (TEXTO PLANO) ===
R1(config)# username admin password AdminPass123!
! ⚠️ Se ve en texto plano en running-config

! === USUARIO CON CONTRASEÑA TIPO 7 (CIFRADO) ===
R1(config)# username admin password 7 0822455D0A16
! Cifrado reversible tipo 7

! === USUARIO CON SECRET (HASH - RECOMENDADO) ===
R1(config)# username admin secret AdminSecret123!
! Hash SHA-256 (tipo 8 en IOS 15+) o MD5 (tipo 5)
! MUCHO más seguro que password

! === USUARIO CON PRIVILEGIOS ESPECÍFICOS ===
R1(config)# username soporte secret Soporte123!
R1(config)# username soporte privilege 5
! Nivel de privilegio 5 (por defecto 1 = usuario, 15 = admin)

! === AUTENTICACIÓN LOCAL EN LÍNEAS ===
R1(config)# line console 0
R1(config-line)# login local
! Usa base de datos local de usuarios (username)
! En lugar de password de línea genérica

R1(config)# line vty 0 15
R1(config-line)# login local
R1(config-line)# transport input ssh
! Solo SSH, autenticación por usuario local

! === LONGITUD MÍNIMA DE CONTRASEÑAS ===
R1(config)# security passwords min-length 10
! Obliga a que TODAS las contraseñas nuevas tengan mínimo 10 caracteres
! Aplica a enable secret, username, line passwords

Mensajes de advertencia de acceso
! === BANNER DEL DÍA (MOTD) ===
R1(config)# banner motd #

💡 Jerarquía de Seguridad de Contraseñas

De MENOS a MÁS seguro:

  1. password → Texto plano
  2. password 7 → Cifrado tipo 7 (reversible)
  3. secret → Hash MD5 tipo 5
  4. secret 5 → Hash MD5 explícito
  5. secret 8 → Hash SHA-256 con salt (IOS 15.3+)
  6. secret 9 → Hash scrypt (IOS 15.3+, más seguro)

Usa siempre username [nombre] secret [pass] y enable secret [pass].

Configuración de SSH Seguro

SSH (Secure Shell) es el estándar para administración remota segura de dispositivos Cisco. Reemplaza completamente a Telnet al cifrar todo el tráfico de administración. La configuración requiere un nombre de dominio, generación de clave RSA y ajuste de líneas VTY.

Pasos para habilitar SSH v2

Configuración completa de SSH en router/switch
! === PASO 1: NOMBRE DE HOST ===
R1(config)# hostname R1-HQ
! El hostname forma parte del prompt SSH

! === PASO 2: DOMINIO IP (OBLIGATORIO PARA RSA) ===
R1(config)# ip domain-name ciscoacademy.local
! El dominio se usa para generar el par de claves RSA
! Sin esto, crypto key generate rsa falla

! === PASO 3: GENERAR CLAVE RSA ===
R1(config)# crypto key generate rsa
The name for the keys will be: R1-HQ.ciscoacademy.local
Choose the size of the key modulus in the range of 360 to 4096
for your General Purpose Keys. Choosing a key modulus greater than
512 may take a few minutes.

How many bits in the modulus [512]: 2048
% Generating 2048 bit RSA keys, keys will be non-exportable...
[OK] (elapsed time was 3 seconds)

! 2048 bits = mínimo recomendado (1024 es inseguro hoy)
! 4096 bits = máxima seguridad (tarda más en generarse)

! === PASO 4: FORZAR SSH V2 ===
R1(config)# ip ssh version 2
! SSHv1 tiene vulnerabilidades conocidas
! SSHv2 es el único protocolo seguro

! === PASO 5: CONFIGURAR LÍNEAS VTY ===
R1(config)# line vty 0 15
R1(config-line)# transport input ssh
! ⬆️ CRÍTICO: Solo permite SSH, bloquea Telnet
! transport input telnet = solo Telnet (INSEGURO)
! transport input all = Telnet + SSH (NO recomendado)
! transport input none = bloquea todo acceso remoto

R1(config-line)# login local
! Usa base de datos local de usuarios

! === PASO 6: CREAR USUARIO PARA SSH ===
R1(config)# username admin secret CiscoAdmin123!
R1(config)# username admin privilege 15
! Privilege 15 = acceso completo sin necesidad de "enable"

! === PASO 7: TIEMPOS DE SESIÓN SSH ===
R1(config)# ip ssh time-out 60
! Desconecta sesión inactiva después de 60 segundos

R1(config)# ip ssh authentication-retries 3
! Máximo 3 intentos fallidos de contraseña por conexión

Eliminar y Regenerar Clave RSA

Gestión de claves criptográficas
! === VER CLAVE RSA ACTUAL ===
R1# show crypto key mypubkey rsa
% Key pair was generated at: 10:32:15 UTC Apr 20 2026
Key name: R1-HQ.ciscoacademy.local
Usage: General Purpose Key
Key is not exportable.
Key size: 2048
...

! === ELIMINAR CLAVE RSA ===
R1(config)# crypto key zeroize rsa
! ⚠️ Esto deshabilita SSH completamente
! Requiere regenerar la clave para volver a usar SSH

! === REGENERAR CON MÁS BITS ===
R1(config)# crypto key generate rsa modulus 4096
! Genera clave de 4096 bits (máxima seguridad)

Conexión SSH desde otro dispositivo Cisco

Cliente SSH integrado en IOS
! === CONECTAR POR SSH DESDE OTRO ROUTER ===
R2# ssh -l admin 192.168.1.1
! -l = especifica usuario
! Se solicitará contraseña interactivamente

! === CONECTAR CON VERSIÓN ESPECÍFICA ===
R2# ssh -v 2 -l admin 192.168.1.1
! Fuerza SSH versión 2

! === CONECTAR A PUERTO ESPECÍFICO ===
R2# ssh -p 2222 -l admin 192.168.1.1
! Si el servidor SSH escucha en puerto alternativo

! === VER SESIONES SSH ACTIVAS ===
R1# show ssh
Connection Version Mode Encryption  Hmac     State                 Username
0          2.0     IN   aes256-ctr sha1     Session started       admin

! === VER USUARIOS CONECTADOS ===
R1# show users
Line       User       Host(s)              Idle       Location
* 0 con 0   admin      idle                 00:00:00
  2 vty 0   admin      192.168.1.2          00:02:15
⚠️ ATAQUE: Intercepción Telnet vs Protección SSH
1 Telnet expone todo: Un atacante con acceso al segmento de red (sniffer) puede capturar usuarios, contraseñas y comandos en texto plano. Wireshark muestra todo legible.
2 SSH cifra la sesión: Todo el tráfico entre cliente y servidor va cifrado con algoritmos simétricos (AES) y asimétricos (RSA/ECDSA). Incluso si se captura, es irrompible sin la clave privada.
3 Best Practice: Deshabilitar Telnet completamente (transport input ssh), usar solo usuarios locales con secret, y mantener claves RSA de 2048+ bits.

Access Control Lists (ACLs)

Las ACLs son listas secuenciales de condiciones permit/deny que filtran tráfico basado en direcciones IP, protocolos, puertos y otros criterios. Son fundamentales para seguridad de red, NAT, QoS y control de routing.

ACL Standard vs Extended

CaracterísticaStandardExtended
Números1-99, 1300-1999100-199, 2000-2699
Filtra porSolo IP origenIP origen/destino, protocolo, puertos
Ubicación recomendadaCerca del destinoCerca del origen
Uso típicoControl de acceso básico, VTYFiltrado de tráfico específico, NAT

ACL Standard — Configuración y Aplicación

Filtrado por dirección IP origen
! === CREAR ACL STANDARD NUMERADA ===
R1(config)# access-list 10 permit 192.168.10.0 0.0.0.255
! Permite red 192.168.10.0/24
! Wildcard mask: 0.0.0.255 = /24 (bits 0 = deben coincidir)

R1(config)# access-list 10 deny 192.168.20.0 0.0.0.255
! Niega red 192.168.20.0/24

R1(config)# access-list 10 permit any
! ⬆️ IMPLÍCITO AL FINAL: deny any
! Pero es buena práctica agregar permit any explícito para logging

! === CREAR ACL STANDARD NOMBRADA ===
R1(config)# ip access-list standard ADMIN-ACCESS
R1(config-std-nacl)# permit 192.168.99.0 0.0.0.255
R1(config-std-nacl)# permit host 10.1.1.5
! "host" = wildcard 0.0.0.0 (IP exacta)
R1(config-std-nacl)# deny any log
! "log" genera mensaje syslog cuando se activa esta regla

! === APLICAR ACL A INTERFAZ ===
R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip access-group 10 in
! "in" = filtra tráfico ENTRANTE a la interfaz
! "out" = filtra tráfico SALIENTE de la interfaz

! === APLICAR ACL A LÍNEAS VTY (CONTROL REMOTO) ===
R1(config)# line vty 0 15
R1(config-line)# access-class 10 in
! "access-class" en lugar de "access-group" para líneas
! Solo permite acceso SSH/Telnet desde IPs de la ACL 10

ACL Extended — Filtrado Avanzado

Filtrado por protocolo, puertos y direcciones
! === ACL EXTENDED NUMERADA ===
R1(config)# access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 80
! Permite HTTP desde VLAN 10 a cualquier destino

R1(config)# access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 443
! Permite HTTPS desde VLAN 10

R1(config)# access-list 100 deny ip 192.168.10.0 0.0.0.255 192.168.20.0 0.0.0.255
! Bloquea TODO tráfico entre VLAN 10 y VLAN 20

R1(config)# access-list 100 permit ip any any
! Permite todo lo demás

! === ACL EXTENDED NOMBRADA ===
R1(config)# ip access-list extended FILTRO-WEB
R1(config-ext-nacl)# permit tcp any any eq 80
R1(config-ext-nacl)# permit tcp any any eq 443
R1(config-ext-nacl)# deny tcp any any eq 23
! Bloquea Telnet explícitamente
R1(config-ext-nacl)# deny icmp any any echo
! Bloquea ping (ICMP echo request)
R1(config-ext-nacl)# permit ip any any

! === OPERADORES DE PUERTO ===
! eq     = equal (igual a)
! neq    = not equal (distinto de)
! gt     = greater than (mayor que)
! lt     = less than (menor que)
! range  = rango de puertos (ej: range 1000 2000)

R1(config-ext-nacl)# permit tcp any any range 20 21
! Permite FTP (puertos 20 y 21)

! === APLICAR ACL EXTENDED ===
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip access-group 100 in

Wildcard Masks — Cálculo Rápido

Fórmula y ejemplos comunes
! === FÓRMULA ===
! Wildcard = 255.255.255.255 - Máscara de subred

! /24 (255.255.255.0)    → 0.0.0.255
! /16 (255.255.0.0)      → 0.0.255.255
! /30 (255.255.255.252)  → 0.0.0.3
! /32 (host)             → 0.0.0.0

! === ACL PARA HOST ESPECÍFICO ===
R1(config)# access-list 10 permit 192.168.1.5 0.0.0.0
! o equivalente:
R1(config)# access-list 10 permit host 192.168.1.5

! === ACL PARA RANGO IMPAR (ej: IPs .1, .3, .5, .7) ===
R1(config)# access-list 10 permit 192.168.1.1 0.0.0.6
! Wildcard 0.0.0.6 = 00000110 en binario
! Último octeto: 1, 3, 5, 7 (bits variables en posiciones 1 y 2)

Modificar y Eliminar ACLs

Gestión de ACLs existentes
! === ACLs NUMERADAS: NO se pueden editar línea por línea ===
! Debes borrarla completa y reescribirla
R1(config)# no access-list 10
! Elimina TODA la ACL 10

! === ACLs NOMBRADAS: SÍ se pueden editar ===
R1(config)# ip access-list standard ADMIN-ACCESS
R1(config-std-nacl)# no permit 192.168.99.0 0.0.0.255
! Elimina línea específica
R1(config-std-nacl)# 15 permit 192.168.88.0 0.0.0.255
! Inserta línea con número de secuencia 15

! === VER ACLs CON NÚMEROS DE SECUENCIA ===
R1# show ip access-lists
Standard IP access list ADMIN-ACCESS
    10 permit 192.168.10.0, wildcard bits 0.0.0.255
    15 permit 192.168.88.0, wildcard bits 0.0.0.255
    20 deny   any log

! === REMOVER ACL DE INTERFAZ ===
R1(config)# interface GigabitEthernet0/1
R1(config-if)# no ip access-group 10 in
⚠️ Regla Crítica: Orden de las ACLs y deny any implícito

Las ACLs se evalúan secuencialmente de arriba hacia abajo. La primera coincidencia determina la acción. Si ninguna regla coincide, se aplica el deny any implícito al final.

Error común: Poner permit any al principio — todo el tráfico pasa y las reglas siguientes nunca se evalúan. permit any debe ir al final (o no ponerse, dependiendo de la política).

Port Security en Switches Cisco

Port Security es una función de capa 2 que restringe el tráfico de entrada en un puerto basándose en la dirección MAC del dispositivo conectado. Protege contra MAC flooding, conexiones no autorizadas y ataques de suplantación.

Configuración Básica de Port Security

Protección de puertos de acceso
! === HABILITAR PORT SECURITY ===
SW1(config)# interface FastEthernet0/1
SW1(config-if)# switchport mode access
SW1(config-if)# switchport port-security
! Activa port security en el puerto
! Por defecto: máximo 1 MAC, violation shutdown

! === CONFIGURAR MAC ESTÁTICA ===
SW1(config-if)# switchport port-security mac-address 0000.0c12.3456
! Solo permite esta MAC específica

! === CONFIGURAR MAC STICKY (APRENDIZAJE DINÁMICO PERMANENTE) ===
SW1(config-if)# switchport port-security mac-address sticky
! Aprende la primera MAC que se conecta y la guarda en running-config
! Combina flexibilidad de dinámico con persistencia de estático

! === CONFIGURAR MÁXIMO DE MACs ===
SW1(config-if)# switchport port-security maximum 3
! Permite hasta 3 MACs (útil para IP Phone + PC + laptop)

! === CONFIGURAR ACCIÓN ANTE VIOLACIÓN ===
SW1(config-if)# switchport port-security violation ?
  protect   ! Descarta tráfico desconocido, NO alerta, NO desactiva puerto
  restrict  ! Descarta + envía SNMP trap + syslog (recomendado)
  shutdown  ! Pone puerto en err-disabled (default, más drástico)

SW1(config-if)# switchport port-security violation restrict

! === CONFIGURACIÓN COMPLETA RECOMENDADA ===
SW1(config)# interface FastEthernet0/1
SW1(config-if)# description PC-VENTAS-01
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 10
SW1(config-if)# switchport port-security
SW1(config-if)# switchport port-security maximum 2
SW1(config-if)# switchport port-security mac-address sticky
SW1(config-if)# switchport port-security violation restrict
SW1(config-if)# spanning-tree portfast
SW1(config-if)# no shutdown

💡 Buena Práctica: Port Security + Sticky en Puertos de Acceso

En todos los puertos de acceso donde conectas PCs fijas:

  1. Configura switchport port-security
  2. Usa switchport port-security mac-address sticky
  3. Establece maximum 2 si hay teléfonos IP
  4. Usa violation restrict para no desactivar puertos críticos
  5. Guarda running-config para persistir las MACs sticky

Sincronización de Tiempo con NTP

La sincronización horaria es crítica para la seguridad: logs precisos, certificados digitales válidos, troubleshooting correlacionado y prevención de ataques de replay. Cisco IOS soporta NTP como cliente y servidor.

Configuración de Cliente NTP

Sincronización con servidor NTP
! === CONFIGURAR SERVIDOR NTP ===
R1(config)# ntp server 192.168.99.5
! Servidor NTP interno o público (pool.ntp.org)

R1(config)# ntp server 192.168.99.5 prefer
! "prefer" = este servidor tiene prioridad si hay múltiples

! === CONFIGURAR ZONA HORARIA ===
R1(config)# clock timezone EST -5
! EST = Eastern Standard Time, -5 = UTC-5

R1(config)# clock summer-time EDT recurring
! Horario de verano automático

! === CONFIGURAR FECHA/HORA MANUAL (si no hay NTP) ===
R1# clock set 14:30:00 20 April 2026
! Formato: HH:MM:SS Day Month Year

! === VERIFICAR SINCRONIZACIÓN ===
R1# show ntp status
Clock is synchronized, stratum 3, reference is 192.168.99.5
nominal freq is 250.0000 Hz, actual freq is 249.9987 Hz, precision is 2**18
ntp uptime is 2345000 (1/100 of seconds), resolution is 4000

Login Block-for — Protección contra Fuerza Bruta

El comando login block-for protege las líneas VTY y Console contra ataques de fuerza bruta al bloquear temporalmente los intentos de inicio de sesión después de un número definido de fallos en un período de tiempo.

Configuración de Bloqueo de Login

Mitigación de ataques de fuerza bruta
! === BLOQUEO DE ACCESO TRAS INTENTOS FALLIDOS ===
R1(config)# login block-for 300 attempts 5 within 120
! Sintaxis: login block-for [segundos-bloqueo] attempts [N] within [segundos-ventana]
! Si hay 5 intentos fallidos en 120 segundos (2 min), bloquea durante 300 seg (5 min)

! === CONFIGURAR MENSAJE DURANTE BLOQUEO ===
R1(config)# login quiet-mode access-class 20
! Durante el bloqueo, SOLO permite conexiones desde IPs de la ACL 20
! Útil para crear una "lista blanca" de administradores

! === CREAR ACL PARA ACCESO DURANTE BLOQUEO ===
R1(config)# access-list 20 permit 192.168.99.10
R1(config)# access-list 20 permit 10.1.1.0 0.0.0.255
! Solo estas IPs pueden acceder cuando el sistema está en quiet mode

! === DELAY ENTRE INTENTOS (Ralentizar fuerza bruta) ===
R1(config)# login delay 3
! Espera 3 segundos entre cada prompt de login
! Ralentiza ataques automatizados significativamente

! === VERIFICAR ESTADO DE BLOQUEO ===
R1# show login
A default login delay of 3 seconds is applied.
No Quiet-Mode access list has been configured.

All successful login is logged and generate SNMP traps.
All failed login is logged and generate SNMP traps.

Login block-for 300 attempts 5 within 120 is configured.

! === VER SI ESTÁ ACTIVAMENTE BLOQUEADO ===
R1# show login failures
Information about login failures with the device.
Username    SourceIPAddr    lPort     TimeStamp           Count
admin       203.0.113.45    vty 0     14:30:15 UTC Mon Apr 20 2026   3

Combinación con ACLs de VTY

Defensa en profundidad para acceso remoto
! === ACL PARA ADMINISTRACIÓN REMOTA ===
R1(config)# access-list 20 permit 192.168.99.0 0.0.0.255
R1(config)# access-list 20 permit host 10.1.1.100
R1(config)# access-list 20 deny any log

! === APLICAR A LÍNEAS VTY ===
R1(config)# line vty 0 15
R1(config-line)# access-class 20 in
R1(config-line)# transport input ssh
R1(config-line)# login local
R1(config-line)# exec-timeout 5 0
! Desconecta sesión inactiva después de 5 minutos

! === CONFIGURAR TIEMPO DE SESIÓN EN CONSOLA TAMBIÉN ===
R1(config)# line console 0
R1(config-line)# exec-timeout 3 0
! 3 minutos de inactividad en consola

! === LOGGING DE ACCESOS ===
R1(config)# login on-success log
R1(config)# login on-failure log
! Genera syslog cada vez que alguien inicia sesión o falla
⚠️ Importante: Quiet Mode vs ACL Regular

login block-for entra en "quiet mode" cuando se detectan intentos masivos. Durante quiet mode, todas las conexiones son rechazadas excepto las de la ACL configurada con login quiet-mode access-class.

Esto es diferente de las ACLs regulares de VTY: las ACLs de VTY filtran siempre, mientras que quiet mode solo se activa durante el bloqueo temporal tras intentos fallidos.

Troubleshooting de Seguridad en Cisco IOS

Los problemas de seguridad pueden bloquear acceso legítimo o dejar vulnerabilidades abiertas. Esta guía cubre los escenarios más frecuentes en exámenes y producción.

Problema 1: No puedo acceder por SSH

1
Verificar clave RSA
show crypto key mypubkey rsa
Si no hay clave, generar con crypto key generate rsa.
2
Verificar dominio IP
show running-config | include domain
Debe existir ip domain-name para generar clave RSA.
3
Verificar líneas VTY
show running-config | section line vty
Debe tener transport input ssh y login local.
4
Verificar usuario local
show running-config | section username
Debe existir al menos un usuario con secret.

Problema 2: ACL bloquea tráfico legítimo

Diagnóstico de filtrado incorrecto
! === SÍNTOMA: Ping funciona pero HTTP no ===

! Paso 1: Verificar ACL aplicada
R1# show ip interface GigabitEthernet0/1
! ¿Hay "Inbound access list" o "Outgoing access list"?

! Paso 2: Ver contenido de ACL
R1# show ip access-lists 100
! ¿Está permitido el puerto 80/tcp?
! ¿Hay un "deny ip any any" antes del permit?

! Paso 3: Ver contadores
R1# show ip access-lists 100
! ¿La regla deny tiene matches incrementando?
! Eso confirma que el tráfico está siendo bloqueado por esa regla

! Paso 4: Corregir orden de reglas
R1(config)# ip access-list extended 100
R1(config-ext-nacl)# no 20
! Eliminar regla problemática
R1(config-ext-nacl)# 15 permit tcp any any eq 80
! Agregar antes del deny

! === ACL STANDARD BLOQUEA ADMINISTRADOR ===
! Si aplicaste ACL standard a interfaz y te quedaste fuera:
! Debes acceder por consola física (no afectada por ACL de interfaz)
! o reiniciar el dispositivo si no hay acceso físico

Problema 3: Puerto en err-disabled por Port Security

Recuperación y prevención
! === DIAGNÓSTICO ===
SW1# show interface status err-disabled
Port      Name               Status       Reason
Fa0/5     PC-RH-03           err-disabled psecure-violation

! === VER DETALLE ===
SW1# show port-security interface FastEthernet0/5
Security Violation Count   : 1
Last Source Address:Vlan   : 0000.0caa.bbcc:10

! Causa: Se conectó dispositivo con MAC diferente a la configurada

! === SOLUCIÓN 1: Recuperar puerto ===
SW1(config)# interface FastEthernet0/5
SW1(config-if)# shutdown
SW1(config-if)# no shutdown

! === SOLUCIÓN 2: Ajustar configuración ===
SW1(config-if)# switchport port-security maximum 3
! Aumentar límite si hay teléfono + PC

SW1(config-if)# switchport port-security mac-address sticky
! Aprender nueva MAC

! === SOLUCIÓN 3: Cambiar a restrict (no shutdown) ===
SW1(config-if)# switchport port-security violation restrict
! Evita que el puerto se desactive, solo descarta tráfico

Problema 4: Login block-for bloquea al administrador

Acceso durante quiet mode
! === VERIFICAR SI ESTÁ EN QUIET MODE ===
R1# show login
Quiet-Mode access list 20 is configured and is applied.
Router is enabled to watch for login Attacks.
If more than 5 login failures occur in 120 seconds or less,
logins will be disabled for 300 seconds.

Router is currently in QUIET mode.
Only following hosts are allowed to log in:
  [192.168.99.10]

! SOLUCIÓN: Acceder desde IP de la ACL quiet-mode
! o esperar 300 segundos a que expire el bloqueo

! === PREVENCIÓN: Agregar tu IP a quiet-mode ACL ===
R1(config)# access-list 20 permit host 192.168.99.100
! Tu estación de trabajo siempre podrá acceder incluso en quiet mode
💡 Checklist de Troubleshooting Seguridad
  1. ¿Hay enable secret configurado? (show run | sec enable)
  2. ¿Las líneas VTY tienen transport input ssh?
  3. ¿Existe ip domain-name y crypto key?
  4. ¿Hay usuarios locales con secret?
  5. ¿Las ACLs tienen el orden correcto?
  6. ¿La ACL está aplicada en la dirección correcta (in/out)?
  7. ¿Port Security tiene violation shutdown bloqueando puertos?
  8. ¿NTP está sincronizado para logs correctos?
  9. ¿login block-for está en quiet mode?
  10. ¿Hay exec-timeout en líneas para prevenir sesiones abandonadas?
