ANSIBLE — Arquitectura y Conceptos

La herramienta de automatización definitiva para infraestructura de red. Entiende su filosofía agentless, domina la arquitectura de Control Node y Managed Nodes, y aprende por qué redes Cisco eligen Ansible frente a Chef o Puppet para orquestar cientos de dispositivos sin instalar un solo agente.

⚙️ Agentless 🎯 CCNA Domain 6.6 ⏱ ~90 min lectura 🔐 SSH / WinRM 📋 YAML

¿Qué es Ansible y por qué revolucionó la automatización?

Ansible es una plataforma de automatización de código abierto creada por Michael DeHaan en 2012 y adquirida por Red Hat en 2015. Su propósito es gestionar la configuración de sistemas, desplegar aplicaciones y orquestar tareas complejas de infraestructura usando un lenguaje declarativo basado en YAML.

A diferencia de los scripts Python tradicionales que describen cómo hacer algo paso a paso (imperativo), Ansible usa playbooks que declaran el estado deseado del sistema (declarativo). Tú dices: "Quiero que el switch tenga la VLAN 10 configurada"; Ansible se encarga de decidir si debe crearla, modificarla o dejarla tal cual.

En el contexto de redes Cisco, Ansible se ha convertido en el estándar de facto porque permite gestionar simultáneamente routers, switches, firewalls y controladores Wi-Fi usando los mismos playbooks, sin importar si el dispositivo ejecuta IOS, IOS-XE, NX-OS o incluso sistemas de otros fabricantes.

💡 Concepto clave del examen CCNA

El dominio 6.6 del examen CCNA 200-301 evalúa tu capacidad para distinguir herramientas de gestión de configuración. La pregunta más frecuente sobre Ansible es: \"¿Cuál es la característica diferenciadora de Ansible frente a Chef o Puppet?\" La respuesta correcta siempre es: es agentless (no requiere instalar software en los nodos gestionados).

Infrastructure as Code (IaC) en redes

Antes de Ansible, configurar 50 switches era un proceso manual: conectarte por SSH a cada uno, pegar comandos, validar, documentar en Excel. Con Ansible, ese mismo proceso se convierte en:

  1. Un inventario que lista tus 50 switches (IPs o nombres).
  2. Un playbook YAML que declara la configuración deseada (VLANs, trunks, SSH, NTP).
  3. Una ejecución única que aplica los cambios en paralelo a todos los dispositivos.
  4. Un log idempotente: si vuelves a ejecutar el playbook, Ansible detecta que ya está configurado y no hace cambios innecesarios.

Esto es Infraestructura como Código (IaC): tu red se versiona en Git, se audita con pull requests y se despliega con pipelines CI/CD, exactamente igual que el software.

¿Por qué Ansible en redes Cisco?

🔌

Agentless

No instalarás nada en tus routers o switches. Solo necesitas IP reachable y credenciales SSH. Ideal para equipos con poco almacenamiento o CPU.

📋

YAML Legible

Sintaxis limpia y legible incluso para ingenieros de red sin background de desarrollo. Un playbook es más fácil de entender que un script Python de 200 líneas.

🎯

Idempotencia

Ejecuta el playbook 1 vez o 100 veces: el resultado final es idéntico. Ansible solo aplica cambios cuando detecta desviación del estado deseado.

🧩

Módulos Oficiales Cisco

La colección cisco.ios provee módulos certificados: ios_config, ios_command, ios_vlans, etc. Soportados directamente por Cisco.

⚡

Ejecución Paralela

Por defecto, Ansible se conecta a 5 dispositivos simultáneamente (forks=5). Configurar 100 switches toma minutos, no horas.

🔒

Seguridad Integrada

Soporta Ansible Vault para cifrar contraseñas y claves SSH. Integración con LDAP/AD y privilegios sudo escalables.

El paradigma Agentless: cero software en el destino

La característica más importante de Ansible —y la que más preguntas genera en el examen CCNA— es que es agentless, es decir, sin agente. Esto significa que no necesitas instalar ningún demonio, servicio o binario permanente en los dispositivos gestionados (Managed Nodes).

¿Cómo es posible? Ansible utiliza protocolos de gestión remotos que ya existen en prácticamente cualquier sistema operativo o dispositivo de red:

  • SSH (predeterminado): Para routers, switches y servidores Linux/Unix.
  • WinRM: Para servidores Windows (PowerShell remoto).
  • HTTP/HTTPS APIs: Para controladores cloud como Meraki, DNA Center o ACI.
  • NETCONF / RESTCONF: Para equipos Cisco modernos con modelos YANG.

→ SSH / API →

Implicaciones de ser Agentless

✅

Bajo consumo de recursos

Los routers y switches Cisco tienen CPU y memoria limitadas. No desperdiciarás recursos en un agente permanente como ocurre con Puppet o Chef.

✅

Menor superficie de ataque

Menos software instalado = menos vulnerabilidades. No hay demonio escuchando en un puerto adicional que pueda ser explotado.

✅

Despliegue inmediato

Añade un nuevo switch a tu inventario y ejecútalo. No necesitas preparar el dispositivo ni instalar nada previamente. Zero-touch automation.

⚠️

Control Node centralizado

Toda la lógica y las credenciales residen en el nodo de control. Debes protegerlo como el activo más crítico de tu red.

¿Qué transporta Ansible durante la conexión?

Cuando Ansible se conecta por SSH a un switch Cisco, el flujo es:

  1. El Control Node abre una conexión SSH hacia el switch usando las credenciales del inventario.
  2. Ansible copia temporalmente un módulo Python pequeño (o un script shell si el objetivo no tiene Python) al dispositivo, usualmente en /tmp o en la memoria flash.
  3. Ejecuta el módulo, recoge el resultado en formato JSON y lo transfiere de vuelta al Control Node.
  4. Limpia los archivos temporales y cierra la sesión SSH.

En dispositivos de red Cisco que no permiten escribir en filesystem de forma estándar, Ansible utiliza conexiones persistentes de tipo network_cli que ejecutan comandos directamente vía shell interactiva sin copiar archivos.

⚡ Trampa frecuente en el examen

\"Ansible requiere Python instalado en los nodos gestionados\" → Falso. Aunque algunos módulos prefieren Python en el target (especialmente en servidores Linux), para dispositivos de red Cisco Ansible usa conexiones CLI/SSH nativas o APIs HTTP que no requieren intérprete Python en el switch.

Anatomía de Ansible: Control Node y Managed Nodes

La arquitectura de Ansible es asimétrica y deliberadamente simple. Se compone de dos actores principales y varios componentes internos que orquestan la magia.

CONTROL NODE (1)
Ansible Instalado
Inventario · Playbooks · Módulos · Plugins
→ SSH / API →
MANAGED NODES (N)
Routers, Switches
IOS · IOS-XE · NX-OS · Linux

Componentes del Control Node

📄

Inventory

Archivo INI o YAML que define los managed nodes. Puedes agrupar por función: [cisco_routers], [cisco_switches], asignar variables por host y por grupo.

📋

Playbooks

Archivos YAML que contienen uno o más "plays". Cada play mapea un grupo de hosts a un conjunto de tareas (tasks) ejecutadas por módulos.

🧩

Módulos

Unidades de trabajo reutilizables. Ejemplos de red: ios_config, ios_command, nxos_vlans, vyos_config. Cada módulo es idempotente.

⚡

Plugins

Extienden la funcionalidad core: inventarios dinámicos (de AWS, Azure, NetBox), callbacks para logging, estrategias de ejecución (linear, free, debug).

📁

Roles

Estructura de directorios estandarizada para organizar playbooks complejos. Un role contiene tasks, variables, templates, files y handlers en carpetas separadas.

🔒

Ansible Vault

Cifrado AES-256 para proteger datos sensibles (passwords, keys) dentro de playbooks e inventarios. Integrado nativamente.

El Inventario: el mapa de tu infraestructura

El inventario es el punto de partida. Sin él, Ansible no sabe a dónde conectarse. Puede ser un archivo estático o un script/plugin dinámico.

inventory.ini — Ejemplo práctico para red Cisco
# Grupos lógicos de dispositivos
[cisco_routers]
R1 ansible_host=192.168.1.1
R2 ansible_host=192.168.1.2

[cisco_switches]
SW1 ansible_host=192.168.1.10
SW2 ansible_host=192.168.1.11
SW3 ansible_host=192.168.1.12

[cisco:children]
cisco_routers
cisco_switches

# Variables por grupo (todas las conexiones usarán SSH)
[cisco:vars]
ansible_network_os=ios
ansible_connection=network_cli
ansible_user=admin
ansible_ssh_pass="{{ vault_cisco_pass }}"
ansible_become=yes
ansible_become_method=enable

Flujo de ejecución de un Playbook

  1. Parseo: Ansible lee el playbook YAML y el inventario.
  2. Determinación de hosts: Identifica qué managed nodes están incluidos en el play.
  3. Forking: Crea procesos paralelos (hijos) para conectarse a múltiples hosts simultáneamente.
  4. Gathering Facts (opcional): Recopila información del sistema (en servidores Linux), o usa ios_facts en redes.
  5. Ejecución de Tasks: Por cada tarea, ejecuta el módulo correspondiente y espera el resultado JSON.
  6. Idempotencia: Si el estado actual coincide con el deseado, el módulo reporta ok sin hacer cambios. Si difiere, aplica el cambio y reporta changed.
  7. Handlers: Si alguna task generó un cambio, se disparan handlers (ej. guardar configuración con write memory).
💡 Tip de arquitectura para el examen

El examen CCNA puede preguntarte quién inicia la comunicación. En Ansible, el Control Node inicia siempre la conexión hacia los Managed Nodes (modelo push). Nunca al revés. Esto es distinto a Puppet, donde el agente en el nodo inicia la conexión hacia el master (modelo pull).

Conectividad: SSH, NETCONF, RESTCONF y HTTPAPI

Ansible no impone un único método de conexión. Para redes Cisco, dispones de varios plugins de conexión que se adaptan a la capacidad del dispositivo y a tus requisitos de seguridad.

🔐

network_cli (SSH)

El método más compatible. Usa SSH para enviar comandos CLI directamente. Funciona en casi cualquier Cisco con IOS/IOS-XE/NX-OS. No requiere APIs ni modelos YANG.

📡

netconf

Protocolo basado en SSH (puerto 830) que usa XML para configurar y monitorear dispositivos con modelos YANG. Idempotencia nativa y transacciones atómicas.

🌐

httpapi / RESTCONF

Usa HTTPS (puerto 443/8443) con payloads JSON/XML. Requiere que el dispositivo exponga una API REST (IOS-XE 16.x+, NX-OS 9K). Más eficiente para JSON parsing.

💻

local

Ansible ejecuta el módulo localmente en el Control Node y este interactúa con la API del dispositivo (útil para Meraki, DNA Center, controladores cloud).

⚡ Seguridad en producción

Nunca uses ansible_ssh_pass en texto plano en producción. Las mejores prácticas son: (1) SSH keys para Linux, (2) ansible-vault para cifrar contraseñas de red, o (3) un Credential Vault externo como HashiCorp Vault o CyberArk integrado con AWX/Tower.

Ansible vs. Chef vs. Puppet: la batalla de la automatización

El examen CCNA 200-301 (Domain 6.6) espera que conozcas las diferencias fundamentales entre las tres grandes herramientas de gestión de configuración. No necesitas ser experto en Chef o Puppet, pero sí distinguir sus arquitecturas.

Desglose por arquitectura

▸ ANSIBLE — El push agentless

Ansible opera desde un único punto (Control Node). Tú ejecutas ansible-playbook y el sistema contacta a cada nodo vía SSH para imponer el estado deseado. No hay demonios corriendo en los routers; no hay "latido" o heartbeat constante. Esto lo hace ideal para redes donde no quieres ni puedes instalar software en el endpoint.

▸ CHEF — La receta del chef

Chef usa una metáfora culinaria: tienes recipes (recetas) que se agrupan en cookbooks (libros de cocina). El nodo gestionado debe tener instalado el Chef Client, que periódicamente contacta al Chef Server para descargar su configuración y aplicarla. Todo se escribe en Ruby DSL. Es poderoso para servidores cloud, pero excesivo para un switch Cisco.

▸ PUPPET — El titiritero

Puppet también requiere un agente en cada nodo. El agente se conecta al Puppet Master, envía un reporte de su estado actual (facts) y recibe un catálogo (catalog) con los recursos que debe aplicar. Usa su propio lenguaje declarativo basado en Ruby. Al igual que Chef, el modelo pull implica que si el master cae, los nodos no pueden recibir configuraciones nuevas hasta que vuelva.

💡 Analogía para recordar en el examen
  • Ansible es como un profesor que va de pupitre en pupitre repartiendo hojas (push, agentless).
  • Puppet es como un sistema de mochilas inteligentes que cada alumno debe tener y que periodicamente se sincronizan con la oficina (pull, agente obligatorio).
  • Chef es similar a Puppet pero con recetas de cocina más que con catálogos (pull, agente obligatorio, Ruby).
⚡ Pregunta tipo examen CCNA

\"Una empresa necesita gestionar configuraciones de red sin instalar software adicional en routers y switches. ¿Qué herramienta recomendarías?\"

Respuesta: Ansible. Chef y Puppet requieren agentes que, en muchos dispositivos de red Cisco, ni siquiera pueden instalarse por falta de soporte de Ruby o permisos de sistema.

Instalación práctica del Control Node

Ansible está escrito en Python y se instala típicamente en una máquina Linux (Ubuntu, RHEL, CentOS). También funciona en macOS y Windows (vía WSL). Para el examen CCNA no necesitas memorizar los comandos de instalación, pero sí entender los requisitos y la verificación.

Ubuntu / Debian — Instalación oficial
# Actualizar repositorios
sudo apt update && sudo apt upgrade -y

# Instalar dependencias
sudo apt install -y software-properties-common

# Añadir repositorio oficial de Ansible (recomendado)
sudo add-apt-repository --yes --update ppa:ansible/ansible

# Instalar Ansible
sudo apt install -y ansible

# Verificar instalación
ansible --version
RHEL / CentOS / Fedora
# RHEL 8+ usa dnf
sudo dnf install -y ansible

# O vía pip (universal pero no recomendado para producción)
pip3 install ansible

# Verificar que el binario esté en PATH
which ansible
ansible --version | head -n 1

Instalación de colecciones Cisco

Ansible separa el core de las colecciones específicas de plataformas. Para gestionar Cisco IOS, debes instalar la colección cisco.ios desde Ansible Galaxy:

Galaxy — Instalar colección oficial Cisco
# Instalar la colección cisco.ios (routers y switches IOS/IOS-XE)
ansible-galaxy collection install cisco.ios

# Instalar colección NX-OS para switches de datacenter
ansible-galaxy collection install cisco.nxos

# Ver colecciones instaladas
ansible-galaxy collection list | grep cisco

Primer comando: ansible ad-hoc

Antes de escribir playbooks, puedes probar conectividad con comandos ad-hoc (de un solo uso). El módulo ping verifica que Ansible pueda conectarse; el módulo ios_command ejecuta show commands en switches.

Prueba de conectividad ad-hoc
# Ping a todos los hosts del inventario (módulo ping, no ICMP)
ansible all -i inventory.ini -m ping

# Ejecutar 'show version' en todos los switches Cisco
ansible cisco_switches -i inventory.ini -m ios_command -a "commands='show version'"

# Ver facts de un router Cisco
ansible R1 -i inventory.ini -m ios_facts -a "gather_subset=hardware"
