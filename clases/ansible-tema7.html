PROYECTO — Despliegue Masivo

Playbook completo que configura VLANs, IPs, OSPF, SSH y políticas de seguridad en una red de 10 dispositivos Cisco desde cero, en menos de 2 minutos. El proyecto integrador definitivo que une Layer 2, Layer 3, WAN, Seguridad e IaC en una sola ejecución idempotente.

🚀 Proyecto Integrador 🎯 CCNA Domain 6.6 ⏱ ~120 min lectura 📋 10 Dispositivos ⚡ Idempotente

El proyecto integrador: red enterprise desde cero

Este es el proyecto final del módulo Ansible. No es una lección teórica más: es un despliegue real de infraestructura de red usando Infrastructure as Code (IaC). En los próximos 120 minutos construirás un playbook que levanta una red de 10 dispositivos Cisco con arquitectura de tres capas: Core, Distribution, Access y WAN Edge.

El objetivo no es memorizar comandos, sino entender cómo orquestar múltiples tecnologías (VLANs, trunking, OSPF, SSH, ACLs, NAT, port-security) en un único pipeline automatizado, idempotente y reproducible. Si tu playbook funciona correctamente, podrás destruir toda la configuración de los 10 dispositivos, volver a ejecutarlo, y obtener exactamente la misma red funcional en menos de 120 segundos.

💡 Filosofía del proyecto

Este playbook es un blueprint vivo. Cada línea es documentación ejecutable. Cuando el auditor pregunte "¿Cómo sabes que todos los switches tienen SSH v2?", tu respuesta no será un PDF: será ansible-playbook site.yml --tags security.

Stack tecnológico del despliegue

🏷️

Layer 2 — Switching

VLANs 10, 20, 30, 99. Trunking 802.1Q. Rapid-PVST. PortFast BPDU Guard. Port-security en cada puerto access.

🌐

Layer 3 — Routing

OSPFv2 Area 0 en todos los routers y switches distribution. Interfaces punto a punto en /30. Passive-interface default.

🔒

Seguridad

SSH v2. Disable Telnet. Enable secret. Service password-encryption. ACLs en WAN. NAT overload. Banner MOTD.

⚙️

Management

NTP sincronizado. Syslog centralizado. Hostname estandarizado. Domain-lookup deshabilitado. DNS local.

📋

IaC — Idempotencia

Si ejecutas el playbook una o cien veces, el resultado final es idéntico. Solo se aplican cambios cuando hay drift.

🚀

Velocidad

Con forks: 10, el despliegue completo de 10 dispositivos toma menos de 2 minutos en hardware real.

Fases del proyecto

  1. Diseño: Inventario YAML con jerarquía de grupos (core, dist, access, wan). Variables centralizadas en group_vars.
  2. Common Baseline: Playbook que aplica SSH, NTP, banners y hardening básico a todos los dispositivos.
  3. Layer 2: Configuración masiva de VLANs, trunks y port-security en los 4 switches de acceso.
  4. Layer 3: Activación de OSPF, asignación de IPs en interfaces y loopbacks en Core, Dist y WAN.
  5. Seguridad Perimetral: ACLs stateless, NAT overload y políticas de control de tráfico en routers WAN.
  6. Verificación: Tests automáticos con ios_facts y assert para validar que la red cumple el diseño.

ℹ️ Diseño de red para el proyecto

Los Access Switches no tienen Loopback porque operan puramente en Layer 2. Los Distribution usan Switched Virtual Interfaces (SVI) para el default gateway de cada VLAN. Los Core y WAN Edge usan loopbacks estables como Router-ID de OSPF.

Play 1: Baseline común (todos los dispositivos)

site.yml — Play 1: Common Hardening
---
- name: "PLAY 1: Baseline de seguridad y management"
  hosts: cisco_all
  gather_facts: no

  tasks:
    - name: "Configurar hostname"
      cisco.ios.ios_config:
        lines:
          - "hostname {{ inventory_hostname }}"
        save_when: "changed"

    - name: "Deshabilitar domain-lookup"
      cisco.ios.ios_config:
        lines: "no ip domain-lookup"

    - name: "Configurar domain name"
      cisco.ios.ios_config:
        lines: "ip domain name {{ domain_name }}"

    - name: "Hardening básico"
      cisco.ios.ios_config:
        lines:
          - "service password-encryption"
          - "service timestamps debug datetime msec"
          - "service timestamps log datetime msec"
          - "no service pad"

    - name: "Banner MOTD"
      cisco.ios.ios_config:
        lines: 'banner motd ^ACCeso autorizado unicamente. Monitoreado por Ansible.^'

    - name: "Usuario local admin"
      cisco.ios.ios_config:
        lines: "username {{ local_username }} privilege 15 secret {{ local_password }}"

    - name: "SSH v2 y líneas VTY"
      cisco.ios.ios_config:
        lines:
          - "ip ssh version 2"
          - "ip ssh time-out 60"
          - "line vty 0 15"
          - " transport input {{ vty_transport }}"
          - " exec-timeout 5 0"
          - " logging synchronous"
        parents: "line vty 0 15"

    - name: "Console timeout"
      cisco.ios.ios_config:
        lines:
          - "line con 0"
          - " exec-timeout {{ console_timeout }}"
          - " logging synchronous"

    - name: "NTP server"
      cisco.ios.ios_config:
        lines: "ntp server {{ ntp_server }}"

    - name: "Syslog server"
      cisco.ios.ios_config:
        lines:
          - "logging {{ syslog_server }}"
          - "logging trap informational"

Play 2: Layer 2 — Access Switches

site.yml — Play 2: VLANs, Trunks y Port-Security
- name: "PLAY 2: Layer 2 Access Switching"
  hosts: access_switches
  gather_facts: no

  tasks:
    - name: "Crear VLANs"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: "{{ item.id }}"
            name: "{{ item.name }}"
        state: merged
      loop: "{{ vlans }}"

    - name: "Configurar uplinks como Trunk"
      cisco.ios.ios_l2_interfaces:
        config:
          - name: "GigabitEthernet1/0/1"
            mode: trunk
            trunk:
              allowed_vlans: "10,20,30,99"
              native_vlan: 99
        state: merged

    - name: "Interfaces access con VLAN datos"
      cisco.ios.ios_l2_interfaces:
        config:
          - name: "GigabitEthernet1/0/2"
            mode: access
            access:
              vlan: 10
          - name: "GigabitEthernet1/0/3"
            mode: access
            access:
              vlan: 20
        state: merged

    - name: "Port-security en puertos access"
      cisco.ios.ios_config:
        lines:
          - "switchport port-security"
          - "switchport port-security maximum {{ port_security_max }}"
          - "switchport port-security violation {{ port_security_violation }}"
          - "switchport port-security mac-address sticky"
        parents: "interface {{ item }}"
      loop:
        - "GigabitEthernet1/0/2"
        - "GigabitEthernet1/0/3"

    - name: "Rapid-PVST y PortFast"
      cisco.ios.ios_config:
        lines:
          - "spanning-tree mode rapid-pvst"
          - "spanning-tree portfast default"
          - "spanning-tree portfast bpduguard default"

Play 3: Layer 3 — Core, Dist y OSPF

site.yml — Play 3: Routing OSPF e IPs
- name: "PLAY 3: Layer 3 Routing y OSPF"
  hosts: core_routers:dist_switches:wan_routers
  gather_facts: no

  tasks:
    - name: "Loopback 0 como Router-ID"
      cisco.ios.ios_config:
        lines:
          - "interface Loopback0"
          - " ip address {{ ospf_router_id }} 255.255.255.255"
          - " ip ospf {{ ospf_process_id }} area {{ ospf_area }}"

    - name: "Activar OSPF y passive-interface default"
      cisco.ios.ios_config:
        lines:
          - "router ospf {{ ospf_process_id }}"
          - " router-id {{ ospf_router_id }}"
          - " passive-interface default"
          - " network 10.0.0.0 0.255.255.255 area {{ ospf_area }}"

    - name: "No passive en enlaces punto a punto"
      cisco.ios.ios_config:
        lines: "no passive-interface {{ item }}"
      loop:
        - "GigabitEthernet0/0/0"
        - "GigabitEthernet0/0/1"
      parents: "router ospf {{ ospf_process_id }}"

    - name: "SVI 99 en Distribution para gateway"
      cisco.ios.ios_config:
        lines:
          - "interface Vlan99"
          - " ip address 10.255.1.{{ ansible_host.split('.')[-1] }} 255.255.255.0"
          - " no shutdown"
      when: "'dist_switches' in group_names"

Play 4: WAN Edge — NAT y ACLs

site.yml — Play 4: Seguridad Perimetral
- name: "PLAY 4: WAN Edge — NAT y ACLs"
  hosts: wan_routers
  gather_facts: no

  vars:
    inside_interfaces: ["GigabitEthernet0/0/0","GigabitEthernet0/0/1"]
    outside_interface: "GigabitEthernet0/0/2"

  tasks:
    - name: "ACL: Permitir redes internas"
      cisco.ios.ios_config:
        lines:
          - "ip access-list extended NAT_ACL"
          - " permit ip 10.0.0.0 0.255.255.255 any"
          - " permit ip 192.168.0.0 0.0.255.255 any"
          - " deny ip any any log"

    - name: "NAT Inside Source Overload"
      cisco.ios.ios_config:
        lines:
          - "interface {{ outside_interface }}"
          - " ip nat outside"
      parents: "interface {{ outside_interface }}"

    - name: "Marcar interfaces inside"
      cisco.ios.ios_config:
        lines:
          - "interface {{ item }}"
          - " ip nat inside"
        parents: "interface {{ item }}"
      loop: "{{ inside_interfaces }}"

    - name: "PAT Overload"
      cisco.ios.ios_config:
        lines:
          - "ip nat inside source list NAT_ACL interface {{ outside_interface }} overload"

    - name: "ACL WAN: Bloquear Telnet y ICMP externo"
      cisco.ios.ios_config:
        lines:
          - "ip access-list extended WAN_INBOUND"
          - " remark Bloquear Telnet entrante"
          - " deny tcp any any eq 23 log"
          - " remark Permitir SSH desde management"
          - " permit tcp 10.255.1.0 0.0.0.255 any eq 22"
          - " deny icmp any any echo log"
          - " permit ip any any"

    - name: "Aplicar ACL en interfaz WAN"
      cisco.ios.ios_config:
        lines: "ip access-group WAN_INBOUND in"
        parents: "interface {{ outside_interface }}"

    - name: "Ruta por defecto hacia ISP"
      cisco.ios.ios_config:
        lines: "ip route 0.0.0.0 0.0.0.0 {{ outside_interface }} dhcp"

Versión profesional con Roles Ansible

En producción, un playbook monolítico de 400 líneas es difícil de mantener. Los roles permiten dividir el playbook en componentes reutilizables, testeables y versionables independientemente. Un role es simplemente una convención de directorios que Ansible reconoce automáticamente.

proyecto_ccna/ ├── site.yml # Orquestador principal ├── ansible.cfg # forks=10, host_key_checking=False ├── inventory.yml # Los 10 dispositivos ├── group_vars/ │ └── all.yml # Variables globales ├── host_vars/ # Variables específicas por nodo (opcional) └── roles/ ├── common/ # SSH, NTP, banners, usuarios │ ├── tasks/ │ │ └── main.yml │ └── handlers/ │ └── main.yml ├── layer2/ # VLANs, trunks, port-security, STP │ ├── tasks/ │ │ └── main.yml │ └── vars/ │ └── main.yml ├── layer3/ # IPs, OSPF, SVIs │ ├── tasks/ │ │ └── main.yml │ └── templates/ │ └── ospf_config.j2 ├── wan/ # NAT, ACLs, default route │ └── tasks/ │ └── main.yml └── verify/ # Tests post-despliegue └── tasks/ └── main.yml

Role: wan (seguridad perimetral)

roles/wan/tasks/main.yml
---
- name: "ACL para NAT"
  cisco.ios.ios_config:
    lines:
      - "ip access-list extended NAT_ACL"
      - " permit ip 10.0.0.0 0.255.255.255 any"
      - " permit ip 192.168.0.0 0.0.255.255 any"

- name: "NAT Overload en interfaz outside"
  cisco.ios.ios_config:
    lines:
      - "interface {{ wan_outside_intf | default('GigabitEthernet0/0/2') }}"
      - " ip nat outside"
    parents: "interface {{ wan_outside_intf | default('GigabitEthernet0/0/2') }}"

- name: "NAT Inside en interfaces internas"
  cisco.ios.ios_config:
    lines:
      - "interface {{ item }}"
      - " ip nat inside"
    parents: "interface {{ item }}"
  loop: "{{ wan_inside_intfs | default(['GigabitEthernet0/0/0','GigabitEthernet0/0/1']) }}"

- name: "PAT overload rule"
  cisco.ios.ios_config:
    lines:
      - "ip nat inside source list NAT_ACL interface {{ wan_outside_intf | default('GigabitEthernet0/0/2') }} overload"

- name: "ACL WAN inbound y default route"
  cisco.ios.ios_config:
    lines:
      - "ip access-list extended WAN_INBOUND"
      - " deny tcp any any eq 23 log"
      - " permit tcp 10.255.1.0 0.0.0.255 any eq 22"
      - " deny icmp any any echo log"
      - " permit ip any any"
      - "ip route 0.0.0.0 0.0.0.0 {{ wan_outside_intf | default('GigabitEthernet0/0/2') }} dhcp"
💡 Ventaja de los roles

Con roles, puedes testear layer2 en un laboratorio de 2 switches sin tocar el resto de la red. También puedes compartir el role common con otros proyectos vía Ansible Galaxy o un repositorio Git privado. La separación de responsabilidades (SoC) es una práctica fundamental de ingeniería de software aplicada a redes.

Ejecutando el despliegue masivo

Ansible ejecuta tareas en paralelo por defecto (forks=5). Para 10 dispositivos, aumentar los forks a 10 reduce el tiempo total a menos de 2 minutos. Antes de tocar producción, siempre valida con --check (dry run) y --diff.

Comandos de ejecución y validación
# 1. Verificar conectividad a los 10 dispositivos
ansible cisco_all -i inventory.yml -m ping

# 2. Dry-run: simular ejecución sin aplicar cambios
ansible-playbook site.yml -i inventory.yml --check --diff

# 3. Ejecutar solo el baseline de seguridad
ansible-playbook site.yml -i inventory.yml --tags common

# 4. Ejecutar solo en los access switches
ansible-playbook site.yml -i inventory.yml --limit access_switches

# 5. Despliegue completo con 10 forks paralelos
ansible-playbook site.yml -i inventory.yml --forks 10

# 6. Ejecutar con vault si las contraseñas están cifradas
ansible-playbook site.yml -i inventory.yml --ask-vault-pass

# 7. Modo verbose para debug de una task específica
ansible-playbook site.yml -i inventory.yml -vvv --start-at-task "Crear VLANs"

Estrategias de ejecución y manejo de errores

🎯

--check (Dry Run)

Predice cambios sin modificarlos. Usa el atributo check_mode de los módulos. No todos los módulos de red lo soportan al 100%, pero ios_config con --diff muestra el delta.

🏷️

Tags y Limits

Usa --tags layer2 para ejecutar solo tareas etiquetadas, o --limit CR1 para probar en un solo dispositivo antes del despliegue masivo.

🛡️

any_errors_fatal

Añade any_errors_fatal: true al play si un fallo en un dispositivo debe detener todo el pipeline (útil en migraciones críticas).

⏱️

Serial y Throttle

serial: 2 ejecuta en grupos de 2 (rolling update). throttle: 1 en una task evita sobrecargar un servidor TFTP o syslog.

⚡ Advertencia de producción

Nunca ejecutes un playbook por primera vez en producción sin --check --diff. Los módulos ios_config con replace: line pueden sobrescribir configuraciones críticas. En redes live, usa save_when: never durante pruebas y programa ventanas de mantenimiento.

Validación post-despliegue con assert

Un despliegue sin verificación es un riesgo operacional. El proyecto incluye un play final de verificación que usa ios_facts y assert para confirmar que la red cumple el diseño antes de dar por terminada la ventana de mantenimiento.

roles/verify/tasks/main.yml — Tests automáticos
---
- name: "RECOLECTAR: Facts de todos los dispositivos"
  cisco.ios.ios_facts:
    gather_subset:
      - "hardware"
      - "interfaces"
      - "config"
  register: "device_facts"

# ── TEST 1: SSH v2 está activo ──
- name: "ASSERT: SSH version 2 habilitado"
  ansible.builtin.assert:
    that:
      - "'ip ssh version 2' in device_facts.ansible_facts.ansible_net_config"
    fail_msg: "CRITICO: {{ inventory_hostname }} no tiene SSH v2"
    success_msg: "OK: SSH v2 verificado"
  ignore_errors: yes

# ── TEST 2: VLANs existen en access switches ──
- name: "ASSERT: VLANs configuradas"
  ansible.builtin.assert:
    that:
      - "device_facts.ansible_facts.ansible_net_vlan is defined"
      - "device_facts.ansible_facts.ansible_net_vlan | length >= 4"
    fail_msg: "FALLO: Faltan VLANs en {{ inventory_hostname }}"
    success_msg: "OK: VLANs presentes"
  when: "'access_switches' in group_names"
  ignore_errors: yes

# ── TEST 3: OSPF neighbors en routers ──
- name: "ASSERT: OSPF tiene vecinos establecidos"
  cisco.ios.ios_command:
    commands: "show ip ospf neighbor"
  register: "ospf_nbr"
  when: "'core_routers' in group_names or 'dist_switches' in group_names or 'wan_routers' in group_names"

- name: "VALIDAR: Al menos 1 vecino OSPF FULL"
  ansible.builtin.assert:
    that:
      - "'FULL' in ospf_nbr.stdout[0]"
    fail_msg: "ALERTA: {{ inventory_hostname }} sin vecinos OSPF FULL"
    success_msg: "OK: OSPF FULL detectado"
  when: "ospf_nbr is defined"
  ignore_errors: yes

# ── TEST 4: Interfaces críticas están UP ──
- name: "ASSERT: Uplinks operacionales"
  ansible.builtin.assert:
    that:
      - "device_facts.ansible_facts.ansible_net_interfaces[item].operstatus == 'up'"
    fail_msg: "CRITICO: {{ item }} está DOWN en {{ inventory_hostname }}"
  loop:
    - "GigabitEthernet0/0/0"
    - "GigabitEthernet0/0/1"
  ignore_errors: yes

# ── TEST 5: Port-security en access ──
- name: "ASSERT: Port-security configurado"
  ansible.builtin.assert:
    that:
      - "'switchport port-security' in device_facts.ansible_facts.ansible_net_config"
    fail_msg: "FALLO: Port-security no activo"
  when: "'access_switches' in group_names"
  ignore_errors: yes

Reporte de verificación ejecutivo

Al finalizar el play de verify, puedes generar un resumen consolidado usando delegate_to: localhost y run_once:

reporte_verificacion.yml — Resumen final
---
- name: "REPORTE: Estado del despliegue"
  hosts: cisco_all
  gather_facts: no

  tasks:
    - name: "Recopolar estado"
      cisco.ios.ios_facts:
        gather_subset: "hardware"
      register: "st"

    - name: "Agregar a reporte"
      ansible.builtin.set_fact:
        report_line: "{{ inventory_hostname }} | {{ st.ansible_facts.ansible_net_version }} | {{ st.ansible_facts.ansible_net_model }} | OK"

  post_tasks:
    - name: "Mostrar resumen"
      ansible.builtin.debug:
        msg: "DESPLIEGUE COMPLETADO: {{ ansible_play_hosts | length }} dispositivos configurados."
      delegate_to: localhost
      run_once: yes
ℹ️ Golden Image vs. Golden Config

Este proyecto implementa una Golden Configuration: un template de configuración declarativa que define el estado deseado. No confundir con Golden Image (usado en servidores virtualizados). En redes Cisco, la configuración es texto; por eso Ansible con ios_config es la herramienta ideal.
