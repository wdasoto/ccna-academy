ROLES, Templates Jinja2 y Ansible Vault

Organiza playbooks complejos con roles reutilizables, genera configuraciones dinámicas con templates Jinja2, y protege credenciales sensibles con Ansible Vault. La tríada que transforma un script de prueba en una plataforma de automatización enterprise para redes Cisco.

📦 Roles 🎯 CCNA Domain 6.6 ⏱ ~90 min lectura 📝 Jinja2 🔐 Ansible Vault

Más allá del playbook monolítico

Cuando comienzas con Ansible, todo cabe en un solo archivo site.yml de 50 líneas. Pero cuando tu red crece a cientos de dispositivos, con múltiples sitios, diferentes modelos de hardware, políticas de seguridad cambiantes y equipos de trabajo colaborativos, necesitas estructura, reutilización y seguridad.

Este tema cubre las tres herramientas que elevan Ansible de "script útil" a "plataforma de automatización industrial":

  • Roles: Empaquetan tareas, variables, templates y handlers en unidades reutilizables y versionables. Un rol base_cisco puede aplicarse a cualquier switch o router sin modificar el playbook principal.
  • Templates Jinja2: Generan configuración dinámica basada en variables. En lugar de hardcodear IPs y nombres de VLAN, un template crea el running-config perfecto para cada dispositivo individualmente.
  • Ansible Vault: Cifra contraseñas, claves SSH y tokens de API con AES-256. Permite guardar secretos junto al código en repositorios Git sin comprometer la seguridad.
💡 Concepto clave del examen CCNA

El dominio 6.6 evalúa si entiendes la organización de Ansible. La pregunta frecuente es: "¿Cómo se protegen las credenciales en un playbook de Ansible?" La respuesta correcta es Ansible Vault. Para reutilizar configuración, la respuesta es Roles.

Los tres pilares de la automatización profesional

📦

Roles

Estructura de directorios estandarizada. Separa tareas, variables, templates, archivos y handlers. Reutilizable entre playbooks y compartible via Ansible Galaxy.

📝

Jinja2

Motor de templates que genera configuración Cisco a partir de variables. Soporta condicionales, bucles, filtros y macros para crear configs únicas por dispositivo.

🔐

Ansible Vault

Cifrado AES-256 para variables sensibles. Integrado nativamente. Permite ansible-playbook --ask-vault-pass o archivos de contraseña para CI/CD.

🧩

Galaxy

Repositorio comunitario y oficial de roles. Descarga roles certificados con ansible-galaxy install. Cisco publica roles oficiales para IOS, NXOS y ASA.

🔄

Idempotencia

La combinación de roles + templates + vault garantiza que cada ejecución sea predecible, segura y repetible, sin importar quién ejecute el playbook.

📂

Estructura Limpia

Un proyecto bien organizado escala de 10 a 10,000 dispositivos sin convertirse en un monstruo inmantenible. La estructura es tan importante como el código.

Roles: empaquetando la reutilización

Un rol es una estructura de directorios estándar que agrupa todo lo necesario para realizar una función específica: tareas, variables, templates, archivos estáticos, handlers y módulos personalizados. En lugar de tener un playbook de 500 líneas, tienes un playbook de 20 líneas que llama a 5 roles bien definidos.

Para redes Cisco, puedes crear roles como base_cisco (configuración inicial), cisco_vlans (gestión de VLANs), cisco_backup (backup programado) o cisco_hardening (seguridad). Cada rol es independiente, testeable y versionable.

Estructura de directorios de un rol

Ansible espera que cada rol siga una convención de nombres de carpetas. No es opcional: si pones un archivo en el lugar correcto, Ansible lo carga automáticamente.

roles/ └── base_cisco/ ├── defaults/ # Variables por defecto (menor precedencia) │ └── main.yml ├── vars/ # Variables del rol (alta precedencia) │ └── main.yml ├── tasks/ # Tasks principales del rol │ └── main.yml ├── handlers/ # Handlers del rol │ └── main.yml ├── templates/ # Plantillas Jinja2 (.j2) │ └── base_config.j2 ├── files/ # Archivos estáticos para copiar │ └── banner.txt ├── meta/ # Dependencias de otros roles │ └── main.yml └── tests/ # Playbooks de prueba ├── inventory └── test.yml

Creando un rol: comando y estructura

Ansible proporciona el comando ansible-galaxy init para generar la estructura de directorios automáticamente:

Crear un rol con Galaxy
# Generar estructura de directorios del rol
ansible-galaxy init base_cisco

# Ver estructura generada
tree base_cisco

# Instalar un rol de Ansible Galaxy (comunidad)
ansible-galaxy install cisco.ios_router_config

# Instalar desde un repositorio Git
ansible-galaxy install git+https://github.com/usuario/rol-cisco.git

Llamando roles desde un playbook

Existen dos formas de invocar roles: la clásica roles: (ejecución antes de las tasks) y la moderna import_role/include_role (ejecución dentro de las tasks, permite condicionales y bucles).

playbook_roles.yml — Invocación de roles
---
# MÉTODO 1: Clásico (roles se ejecutan antes que tasks)
- name: "Configurar switches con roles"
  hosts: cisco_switches
  gather_facts: no

  roles:
    - role: base_cisco
      vars:
        ntp_server: "10.0.0.254"
    - role: cisco_vlans
    - role: cisco_backup

  tasks:
    - name: "Verificación post-rol"
      cisco.ios.ios_command:
        commands: "show running-config | include ntp"
import_role.yml — Método moderno con condicionales
---
# MÉTODO 2: Moderno (roles dentro de tasks, permite when/loop)
- name: "Configuración condicional con roles"
  hosts: cisco_devices
  gather_facts: no

  tasks:
    - name: "Aplicar baseline en todos"
      ansible.builtin.import_role:
        name: base_cisco

    - name: "Configurar VLANs solo en switches"
      ansible.builtin.import_role:
        name: cisco_vlans
      when: "'cisco_switches' in group_names"

    - name: "Aplicar hardening solo en producción"
      ansible.builtin.include_role:
        name: cisco_hardening
      when: "env == 'produccion'"

Precedencia de variables en roles

Los roles introducen nuevos niveles de precedencia. Es crucial entender dónde definir cada variable:

UbicaciónPrecedenciaUso recomendado
defaults/main.ymlMuy bajaValores por defecto que el usuario puede sobrescribir fácilmente
vars/main.ymlAltaVariables internas del rol que el usuario NO debe cambiar
host_vars/Muy altaVariables específicas de un dispositivo
group_vars/AltaVariables por grupo de dispositivos
Extra vars -eMáximaOverride de emergencia en línea de comandos
⚡ Diferencia: import_role vs include_role

import_role se procesa al parsear el playbook (estático). No permite bucles y las condicionales se aplican a todas las tasks del rol. include_role se procesa en tiempo de ejecución (dinámico). Permite bucles, condicionales por host y carga lazy. Para redes Cisco, import_role es más común por su predictibilidad.

Dependencias de roles (meta/main.yml)

Un rol puede depender de otros roles. Ansible las ejecuta automáticamente antes del rol principal:

meta/main.yml — Dependencias
dependencies:
  - role: base_cisco
    vars:
      ntp_server: "10.0.0.254"
  - role: cisco_logging

Jinja2: el motor de configuración dinámica

Jinja2 es el motor de templates por defecto de Ansible. Permite generar archivos de configuración dinámicamente interpolando variables, aplicando filtros, ejecutando condicionales y bucles. En redes Cisco, es la herramienta que convierte un playbook genérico en una configuración única y perfecta para cada router o switch.

Los templates Jinja2 usan la extensión .j2 y se almacenan en la carpeta templates/ de un rol o junto al playbook. Ansible procesa el template en el Control Node y envía el resultado (texto plano) al módulo ios_config o lo guarda como archivo.

Sintaxis básica de Jinja2
{# Comentario: no aparece en la salida #}

{{ variable }}           {# Mostrar valor de variable #}
{{ variable | filter }}  {# Aplicar filtro a variable #}

{% if condicion %}       {# Condicional #}
  config aqui
{% endif %}

{% for item in lista %}  {# Bucle #}
  {{ item }}
{% endfor %}

Variables y filtros esenciales

FiltroDescripciónEjemplo
default()Valor por defecto si variable es undefined{{ ntp_server | default('10.0.0.1') }}
upperMayúsculas{{ hostname | upper }}
lowerMinúsculas{{ sitio | lower }}
replaceReemplazar substring{{ model | replace('-','_') }}
regex_replaceReemplazar con regex{{ ip | regex_replace('\\.','_') }}
joinUnir lista con separador{{ vlans | join(',') }}
splitDividir string en lista{{ ips | split('\n') }}
ipaddrManipulación de IPs (requiere ansible.utils){{ '192.168.1.1/24' | ipaddr('network') }}
to_yamlConvertir a YAML{{ config | to_yaml }}
to_jsonConvertir a JSON{{ facts | to_json }}

Template completo para configuración Cisco

Este template genera una configuración base completa para un switch Cisco, adaptándose dinámicamente a variables por host y por grupo:

templates/base_switch.j2 — Configuración dinámica
hostname {{ inventory_hostname | upper }}

! Banners
banner motd #
ACCESO AUTORIZADO UNICAMENTE.
Dispositivo: {{ inventory_hostname }}
Sitio: {{ site | default('DESCONOCIDO') }}
#

! Configuración de dominio
{% if domain_name is defined %}
ip domain-name {{ domain_name }}
{% else %}
ip domain-name lab.local
{% endif %}

! Servidores NTP
{% for ntp in ntp_servers | default(['10.0.0.254']) %}
ntp server {{ ntp }}
{% endfor %}

! Syslog
{% if syslog_servers is defined %}
{% for syslog in syslog_servers %}
logging {{ syslog }}
{% endfor %}
logging trap informational
{% endif %}

! SNMP
{% if snmp_community is defined %}
snmp-server community {{ snmp_community }} RO
snmp-server location {{ site | default('Data Center') }}
{% endif %}

! VLANs
{% for vlan in vlans | default([]) %}
vlan {{ vlan.id }}
 name {{ vlan.name }}
{% endfor %}

! Interfaces de acceso
{% for port in access_ports | default([]) %}
interface {{ port.name }}
 description {{ port.description | default('Puerto de acceso') }}
 switchport mode access
 switchport access vlan {{ port.vlan }}
 spanning-tree portfast
 no shutdown
{% endfor %}

! Uplinks
{% for uplink in uplinks | default([]) %}
interface {{ uplink.name }}
 description {{ uplink.description | default('Uplink') }}
 switchport mode trunk
 switchport trunk allowed vlan {{ uplink.allowed_vlans | default('all') }}
 switchport trunk native vlan {{ native_vlan | default(99) }}
 no shutdown
{% endfor %}

Usando el template en un playbook

playbook_template.yml — Aplicar template
---
- name: "Aplicar configuración base desde template"
  hosts: cisco_switches
  gather_facts: no

  vars:
    site: "Madrid"
    ntp_servers:
      - "10.0.0.254"
      - "10.0.0.253"
    vlans:
      - { id: 10, name: "VENTAS" }
      - { id: 20, name: "IT" }
    access_ports:
      - { name: "GigabitEthernet1/0/1", vlan: 10 }
      - { name: "GigabitEthernet1/0/2", vlan: 20 }

  tasks:
    - name: "Aplicar template de configuración base"
      cisco.ios.ios_config:
        src: "base_switch.j2"
      notify: "Guardar config"

  handlers:
    - name: "Guardar config"
      cisco.ios.ios_command:
        commands: "write memory"

💡 Tip de depuración de templates

Usa el módulo template con delegate_to: localhost para generar el archivo en el Control Node y revisarlo antes de enviarlo al dispositivo: ansible.builtin.template src=base.j2 dest=/tmp/{{ inventory_hostname }}.cfg. Esto permite auditar la configuración sin tocar la red.

Ansible Vault: seguridad para credenciales

Ansible Vault es una herramienta integrada que cifra variables y archivos sensibles usando el algoritmo AES-256. Permite almacenar contraseñas, claves SSH, tokens de API y comunidades SNMP junto a tu código en repositorios Git sin exponer datos confidenciales.

En entornos de red Cisco, Vault es esencial porque los playbooks contienen credenciales de enable, contraseñas SNMP, claves para VPNs y hasta comunidades de lectura/escritura. Sin Vault, cualquier persona con acceso al repositorio tiene acceso total a tu infraestructura.

Operaciones básicas de Vault

Comandos esenciales de Ansible Vault
# Crear un nuevo archivo cifrado
ansible-vault create group_vars/all/vault.yml

# Editar un archivo cifrado existente
ansible-vault edit group_vars/all/vault.yml

# Ver contenido de un archivo cifrado (sin editar)
ansible-vault view group_vars/all/vault.yml

# Cifrar un archivo existente en texto plano
ansible-vault encrypt secrets.yml

# Descifrar un archivo para edición externa
ansible-vault decrypt secrets.yml

# Cambiar la contraseña de cifrado (rekey)
ansible-vault rekey secrets.yml

# Verificar si un archivo está cifrado
head -n 1 secrets.yml
# Debe mostrar: $ANSIBLE_VAULT;1.1;AES256

Estructura recomendada: vault + variables normales

La mejor práctica es separar las variables cifradas de las no cifradas. Las variables cifradas usan el prefijo vault_ y se referencian desde archivos no cifrados:

group_vars/all/vault.yml — Archivo cifrado
# Este archivo está cifrado con ansible-vault
vault_cisco_password: "SuperSecreta123!"
vault_snmp_community: "C0mmun1ty_Str0ng"
vault_enable_secret: "Enable$3cr3t"
vault_bgp_neighbor_pass: "BGP_P@ssw0rd"
group_vars/all/main.yml — Archivo NO cifrado
# Referencia a variables cifradas (sin el prefijo vault_)
ansible_user: admin
ansible_ssh_pass: "{{ vault_cisco_password }}"
ansible_become_password: "{{ vault_enable_secret }}"
snmp_community: "{{ vault_snmp_community }}"

Ejecutar playbooks con Vault

Métodos de autenticación Vault
# Método 1: Prompt interactivo (desarrollo)
ansible-playbook site.yml --ask-vault-pass

# Método 2: Archivo de contraseña (automatización)
ansible-playbook site.yml --vault-password-file ~/.vault_pass.txt

# Método 3: Script ejecutable (integración con gestores de secretos)
ansible-playbook site.yml --vault-password-file ./get_vault_pass.sh

# Método 4: Variable de entorno
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass.txt
ansible-playbook site.yml

# Método 5: Vault IDs (múltiples contraseñas)
ansible-vault create --vault-id prod@prompt group_vars/prod/vault.yml
ansible-playbook site.yml --vault-id prod@prompt --vault-id dev@~/.dev_pass.txt

Vault IDs: múltiples entornos, múltiples claves

Los Vault IDs permiten usar diferentes contraseñas para diferentes entornos o archivos. Esto es crítico cuando gestionas producción y laboratorio en el mismo repositorio:

vault_ids.yml — Múltiples entornos
# Crear vault para producción
ansible-vault create --vault-id prod@prompt group_vars/production/vault.yml

# Crear vault para desarrollo
ansible-vault create --vault-id dev@prompt group_vars/development/vault.yml

# Ejecutar playbook accediendo a ambos vaults
ansible-playbook site.yml \
  --vault-id prod@~/.prod_vault_pass \
  --vault-id dev@~/.dev_vault_pass
⚡ Seguridad del archivo de contraseña

Si usas --vault-password-file, asegúrate de que el archivo tenga permisos 600 (chmod 600 ~/.vault_pass.txt). Nunca lo commitees en Git. En CI/CD, usa variables de entorno o integraciones con HashiCorp Vault, AWS Secrets Manager o Azure Key Vault.

Cifrado de strings inline (ansible-vault encrypt_string)

Para cifrar una única variable sin crear un archivo completo, usa encrypt_string. Esto es útil para añadir rápidamente una contraseña a un archivo group_vars existente:

Cifrar string individual
# Cifrar una contraseña y obtener el string para pegar en YAML
ansible-vault encrypt_string 'MiPasswordSeguro' --name 'ansible_ssh_pass'

# Salida (copiar y pegar en el archivo YAML):
ansible_ssh_pass: !vault |
  $ANSIBLE_VAULT;1.1;AES256
  6638613465376538623238013616530626362396433633633064303138303535
  ...

# También puedes redirigir a un archivo
ansible-vault encrypt_string 'MiPassword' --name 'vault_pass' >> group_vars/all.yml

Proyecto integrado: Cisco Enterprise

Esta sección une Roles, Jinja2 y Vault en un proyecto realista. La estructura representa un entorno enterprise donde se gestionan switches de acceso en múltiples sitios con configuraciones dinámicas y credenciales protegidas.

proyecto_cisco_enterprise/ ├── ansible.cfg ├── inventory/ │ ├── production/ │ │ ├── inventory.yml │ │ ├── group_vars/ │ │ │ ├── all.yml │ │ │ └── cisco_switches.yml │ │ └── host_vars/ │ └── development/ ├── roles/ │ ├── common/ │ │ ├── tasks/ │ │ ├── templates/ │ │ └── defaults/ │ ├── cisco_vlans/ │ │ ├── tasks/ │ │ └── vars/ │ └── cisco_backup/ │ ├── tasks/ │ └── handlers/ ├── playbooks/ │ └── site.yml └── vault/ ├── .gitignore # Ignorar archivos de password └── vault_pass_prod.txt # chmod 600, NO commitear

Playbook maestro: site.yml

playbooks/site.yml — Orquestación con roles
---
- name: "BASELINE CISCO: Configuración enterprise completa"
  hosts: cisco_switches
  gather_facts: no
  connection: network_cli

  vars_files:
    - ../vault/group_vars/all/vault.yml

  roles:
    - role: common
      tags: ["baseline", "always"]
    - role: cisco_vlans
      tags: ["vlan"]
      when: "configure_vlans | default(true)"
    - role: cisco_backup
      tags: ["backup"]

  tasks:
    - name: "Validación final: verificar VLANs"
      cisco.ios.ios_command:
        commands: "show vlan brief"
      tags: ["validate"]

Rol common: template Jinja2 + Vault

roles/common/templates/base_config.j2

Ejecución segura en producción

Comandos de ejecución con Vault y validación
# 1. Verificar sintaxis del playbook
ansible-playbook playbooks/site.yml --syntax-check

# 2. Simulación en seco (dry run) con Vault
ansible-playbook playbooks/site.yml -i inventory/production/ \
  --check --diff \
  --vault-password-file vault/vault_pass_prod.txt

# 3. Ejecutar solo baseline (sin VLANs ni backup)
ansible-playbook playbooks/site.yml -i inventory/production/ \
  --tags "baseline" \
  --vault-password-file vault/vault_pass_prod.txt

# 4. Ejecutar completo con límite de hosts
ansible-playbook playbooks/site.yml -i inventory/production/ \
  --limit "SW-Acc-MAD-*" \
  --vault-password-file vault/vault_pass_prod.txt

# 5. Backup forzado sin aplicar cambios
ansible-playbook playbooks/site.yml -i inventory/production/ \
  --tags "backup" \
  --vault-password-file vault/vault_pass_prod.txt
ℹ️ Checklist antes de tocar producción
  • Backup: Ejecuta el rol de backup antes de cualquier cambio.
  • Vault: Verifica que el archivo de contraseña tenga permisos 600.
  • Check: Usa --check --diff para previsualizar cambios.
  • Limit: Prueba en un solo switch antes de aplicar al grupo completo.
  • Serial: Considera serial: 1 en el play para rolling update.
