NETCONF — Gestión con XML y SSH

El protocolo estándar RFC 6241 que revolucionó la gestión de redes. Sesiones SSH persistentes, operaciones transaccionales sobre datastores, y modelos YANG estructurados. Domina get-config, edit-config, commit y ncclient para automatización robusta de Cisco IOS-XE.

📡 RFC 6241 🎯 CCNA Domain 6.5 ⏱ ~90 min lectura 💻 Labs con IOS-XE 🧪 Proyecto ncclient

¿Qué es NETCONF y por qué es el estándar de oro?

NETCONF (Network Configuration Protocol), definido en la RFC 6241, es el protocolo de gestión de red creado por la IETF para superar las limitaciones del modelo CLI/SSH tradicional. Mientras que RESTCONF (Tema 2) es stateless y basado en HTTP, NETCONF es stateful, orientado a conexión y transaccional — características críticas para redes de producción donde un cambio mal aplicado puede causar una interrupción masiva.

NETCONF se comunica a través de SSH en el puerto 830 (netconf-ssh) utilizando XML como formato nativo de mensajería. Cada sesión comienza con un intercambio de capabilities, permite múltiples operaciones sobre datastores (running, startup, candidate), y garantiza atomicidad: un <commit> aplica todos los cambios de golpe, o ninguno si falla algo.

💡 NETCONF vs RESTCONF — ¿Cuándo usar cada uno?
  • NETCONF (RFC 6241): Ideal para configuración masiva, transacciones complejas y entornos donde se requiere candidate config + commit. Es el preferido para orquestadores y herramientas de cumplimiento.
  • RESTCONF (RFC 8040): Ideal para aplicaciones web modernas, dashboards y microservicios que necesitan HTTP stateless con JSON. Más fácil de integrar con herramientas DevOps.
  • La realidad: Muchos ingenieros usan ambos. NETCONF para el "día 0" de configuración masiva, RESTCONF para el "día 2" de monitoreo y ajustes rápidos.

Arquitectura de capas RFC 6241

NETCONF se divide en cuatro capas bien definidas, lo que lo hace modular y extensible:

🔒

Transporte Seguro

SSH (puerto 830) es el estándar. También soporta TLS (RFC 7589), SOCKS5 o cualquier sesión confiable. Provee autenticación, integridad y cifrado.

📨

Mensajes (Messages)

Marco XML para codificar RPCs. Un cliente envía <rpc>; el servidor responde <rpc-reply>. Incluye mecanismos de notificación y eventos.

⚙️

Operaciones

Conjunto base de operaciones: <get>, <get-config>, <edit-config>, <copy-config>, <delete-config>, <commit>, <lock>, <unlock>.

🌳

Contenido (Content)

Modelos de datos YANG definen la sintaxis y semántica del XML intercambiado. Los mismos modelos YANG que usa RESTCONF son la base del contenido NETCONF.

⚡ Diferencia clave con RESTCONF

RESTCONF se habilita con restconf y usa HTTPS (443). NETCONF se habilita con netconf-yang y usa SSH (830). Son independientes: puedes tener uno, ambos o ninguno. Para ncclient Python, necesitas exclusivamente netconf-yang y acceso al puerto 830.

RFC 6241 — Especificación completa

Publicada en 2011 y actualizada por RFC 6242 (usar SSH con NETCONF), la RFC 6241 establece un framework robusto para gestionar configuración de dispositivos de red. Su diseño se inspiró en los problemas del modelo SNMP de solo-lectura y la inestabilidad del scripting CLI.

El intercambio de capabilities (<hello>)

Toda sesión NETCONF comienza con un intercambio de mensajes <hello>. Tanto cliente como servidor anuncian qué capacidades soportan. Esto determina qué operaciones y modelos YANG están disponibles durante la sesión.

Ejemplo de mensaje <hello> del servidor (IOS-XE)
<?xml version="1.0" encoding="UTF-8"?>
<hello xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <capabilities>
    <capability>urn:ietf:params:netconf:base:1.0</capability>
    <capability>urn:ietf:params:netconf:base:1.1</capability>
    <capability>urn:ietf:params:netconf:capability:writable-running:1.0</capability>
    <capability>urn:ietf:params:netconf:capability:candidate:1.0</capability>
    <capability>urn:ietf:params:netconf:capability:rollback-on-error:1.0</capability>
    <capability>urn:ietf:params:netconf:capability:validate:1.1</capability>
    <capability>urn:ietf:params:netconf:capability:confirmed-commit:1.1</capability>
    <capability>urn:ietf:params:netconf:capability:with-defaults:1.0?basic-mode=explicit</capability>
    <!-- Capabilities YANG de Cisco -->
    <capability>http://cisco.com/ns/yang/Cisco-IOS-XE-native?module=Cisco-IOS-XE-native&revision=2021-03-01</capability>
  </capabilities>
  <session-id>42</session-id>
</hello>

Capabilities fundamentales

CapabilityURI (urn:ietf:params:netconf:capability:...)Significado
:writable-runningwritable-running:1.0Permite editar directamente el datastore running con <edit-config>.
:candidatecandidate:1.0Soporta datastore candidate. Los cambios no aplican hasta <commit>.
:confirmed-commitconfirmed-commit:1.1Commit con timeout de confirmación. Si no se confirma, hace rollback automático.
:rollback-on-errorrollback-on-error:1.0Si <edit-config> falla en algún punto, revierte TODOS los cambios de esa operación.
:validatevalidate:1.1Permite validar una configuración antes de aplicarla (<validate>).
:with-defaultswith-defaults:1.0Controla si los valores YANG por defecto se incluyen en las respuestas.
:xpathxpath:1.0Permite filtros XPath en <get> y <get-config> (muy potente).
💡 Trampa frecuente del examen

El examen CCNA pregunta cuál capability permite modificar la configuración sin afectar running inmediatamente. La respuesta es :candidate. También pregunta qué capability protege contra commits fallidos: :rollback-on-error. Memoriza estos URIs por su nombre corto, no por el URI completo.

XML — El lenguaje nativo de NETCONF

A diferencia de RESTCONF que puede usar JSON, NETCONF está rigidamente ligado a XML. Cada mensaje es un documento XML bien formado con namespace obligatorio urn:ietf:params:xml:ns:netconf:base:1.0. Entender la estructura XML es entender NETCONF.

Elementos raíz de NETCONF

  • <rpc>: Envuelve cada solicitud del cliente. Atributo obligatorio message-id para correlacionar respuestas.
  • <rpc-reply>: Respuesta del servidor. Incluye el mismo message-id o datos (<data>) o un <ok/>.
  • <rpc-error>: Cuando falla una operación, contiene error-type, error-tag, error-severity, error-message.
Ejemplo completo: get-config de running (mensaje XML crudo)
<!-- CLIENTE envía -->
<rpc message-id="101"
     xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source>
      <running/>
    </source>
  </get-config>
</rpc>

<!-- SERVIDOR responde (fragmento) -->
<rpc-reply message-id="101"
           xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <data>
    <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
      <version>16.12</version>
      <hostname>R1-CORE</hostname>
      <interface>
        <GigabitEthernet>
          <name>0/0</name>
          <ip>
            <address>
              <primary>
                <address>192.168.1.1</address>
                <mask>255.255.255.0</mask>
              </primary>
            </address>
          </ip>
        </GigabitEthernet>
      </interface>
    </native>
  </data>
</rpc-reply>

SSH como transporte (RFC 6242)

NETCONF sobre SSH no usa simplemente una shell interactiva. Usa un subsystem SSH llamado netconf. Esto significa que después del handshake SSH, el cliente invoca ssh -s router netconf y obtiene un canal dedicado para XML.

Conexión manual vía línea de comandos (debug)
# Conectar al subsystem netconf manualmente (útil para troubleshooting)
ssh -p 830 -s admin@192.168.1.1 netconf

# El servidor responderá inmediatamente con su <hello>:
<?xml version="1.0" encoding="UTF-8"?>
<hello xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <capabilities>...</capabilities>
  <session-id>1</session-id>
</hello>]]>]]>

# Luego puedes escribir manualmente un <rpc>:
<rpc message-id="1" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get><filter type="subtree"><native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native"/></filter></get>
</rpc>]]>]]>
⚡ Delimitador especial: ]]>]]>

NETCONF 1.0 usa el delimitador ]]>]]> para indicar fin de mensaje XML sobre el stream SSH. NETCONF 1.1 (chunked framing) usa un encabezado de longitud. ncclient maneja esto automáticamente, pero si debugueas con ssh -s, debes enviar el delimitador manualmente.

Filtrado con Subtree vs XPath

Al obtener configuración, puedes filtrar para no recibir el XML enorme de toda la configuración:

Filtro Subtree (soportado universalmente)
<get-config>
  <source><running/></source>
  <filter type="subtree">
    <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
      <hostname/>
    </native>
  </filter>
</get-config>
Filtro XPath (requiere capability :xpath)
<get-config>
  <source><running/></source>
  <filter type="xpath" select="/native/interface/GigabitEthernet[name='0/0']"/>
</get-config>

Operaciones base de NETCONF

La RFC 6241 define 9 operaciones base. Cada una es un elemento XML dentro de <rpc>. Comprender cuándo usar cada operación es clave para la automatización segura.

📖

<get>

Recupera datos de estado operacional (no configuración). Útil para métricas, contadores, tabla ARP, vecinos OSPF.

📂

<get-config>

Recupera configuración de un datastore específico: running, startup o candidate. El pilar de los backups automatizados.

✏️

<edit-config>

Modifica un datastore target. Soporta merge (default), replace, create y delete. La operación más poderosa y peligrosa.

📋

<copy-config>

Copia contenido completo de un datastore a otro. Equivalente a copy running-config startup-config pero atómico.

<edit-config> en profundidad

Es la operación más compleja. El atributo operation dentro del elemento config determina el comportamiento:

operationComportamientoAnalogía CLI
merge (default)Combina la config enviada con la existente. No borra lo que no menciones.Comandos en modo configuración sueltos.
replaceReemplaza el nodo identificado por el filtro. Si el nodo es la raíz, reemplaza TODO el datastore.config replace o copy-paste de sección.
createCrea el recurso solo si NO existe. Falla con data-exists si ya existe.interface Loopback1 (primera vez).
deleteElimina el recurso. Falla con data-missing si no existe.no interface Loopback1.
removeElimina si existe; si no existe, no hace nada (idempotente).no seguro en scripts.
<edit-config> merge — Añadir descripción a Gi0/0
<rpc message-id="200" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target>
      <running/>
    </target>
    <config>
      <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
        <interface>
          <GigabitEthernet>
            <name>0/0</name>
            <description>NETCONF-MANAGED-WAN</description>
          </GigabitEthernet>
        </interface>
      </native>
    </config>
  </edit-config>
</rpc>
<edit-config> replace — Reemplazar toda la sección de interfaces
<rpc message-id="201" xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target><running/></target>
    <config>
      <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
        <interface xmlns:nc="urn:ietf:params:xml:ns:netconf:base:1.0"
                  nc:operation="replace">
          <!-- TODA la configuración de interfaces se reemplaza -->
        </interface>
      </native>
    </config>
  </edit-config>
</rpc>

Operaciones de control transaccional

<lock> y <unlock> — Evitar corridas concurrentes
<!-- Bloquear running para que nadie más edite -->
<rpc message-id="300">
  <lock>
    <target><running/></target>
  </lock>
</rpc>

<!-- ... realizar múltiples operaciones ... -->

<rpc message-id="301">
  <unlock>
    <target><running/></target>
  </unlock>
</rpc>
<commit> — Aplicar candidate a running (transacción atómica)
<rpc message-id="400">
  <commit/>
</rpc>

<!-- Respuesta si todo OK -->
<rpc-reply message-id="400">
  <ok/>
</rpc-reply>
<discard-changes> — Abortar candidate
<rpc message-id="401">
  <discard-changes/>
</rpc>
💡 Patrón seguro: candidate + validate + commit

El flujo más seguro en producción es: lock candidate → edit-config (target=candidate) → validate → commit → unlock. Si algo falla en validate, haces discard-changes y unlock. Así nunca tocas running hasta estar 100% seguro.

Datastores — El corazón transaccional de NETCONF

Los datastores son la característica distintiva de NETCONF frente a RESTCONF. Mientras RESTCONF opera principalmente sobre running con modificaciones directas, NETCONF permite separar el trabajo en progreso de la configuración activa.

⚡

running

Configuración activa en memoria. Obligatorio en todos los dispositivos. Equivalente a show running-config. Si editas aquí, el cambio es inmediato.

💾

startup

Configuración persistente de arranque. Equivalente a show startup-config. No se modifica directamente por NETCONF base; se usa <copy-config> desde running.

📝

candidate

Copia de trabajo editable sin impacto. Requiere :candidate capability. Permite múltiples ediciones y un <commit> atómico. El más seguro para cambios masivos.

Matriz de operaciones por datastore

OperaciónrunningstartupcandidateNotas
<get-config>✅✅✅Siempre permitido.
<edit-config>✅*❌✅*Requiere :writable-running.
<copy-config>src/tgtsrc/tgtsrc/tgtCopia completa entre datastores.
<delete-config>❌✅✅Nunca borres running.
<commit>N/AN/A✅Copia candidate → running.
<lock>✅✅✅Bloqueo exclusivo.

copy-config: El "write memory" de NETCONF

Persistir running en startup vía NETCONF
<rpc message-id="500">
  <copy-config>
    <target>
      <startup/>
    </target>
    <source>
      <running/>
    </source>
  </copy-config>
</rpc>
⚡ Error crítico común

Intentar hacer <edit-config> sobre running en un dispositivo que NO anunció :writable-running produce error-tag: operation-not-supported. Muchos switches solo soportan candidate. Siempre lee las capabilities del <hello> antes de asumir que puedes escribir en running.

ncclient — El SDK Python para NETCONF

ncclient es la librería Python de facto para NETCONF. Creada originalmente por Cisco, es ahora un proyecto open-source mantenido por la comunidad. Abstrae todo el XML, el framing SSH, los delimitadores ]]>]]> y el manejo de sesiones, permitiéndote enfocarte en la lógica de red.

Instalación y conexión base
import logging
from ncclient import manager

# Opcional: ver XML crudo en consola para debug
logging.basicConfig(level=logging.DEBUG)

# ══════════════════════════════════════════════════════
# CONEXIÓN A IOS-XE
# ══════════════════════════════════════════════════════
with manager.connect(
    host="192.168.1.1",
    port=830,
    username="admin",
    password="Cisco123",
    hostkey_verify=False,          # Solo laboratorio
    device_params={"name": "iosxe"},  # Parámetros específicos Cisco

    look_for_keys=False,
    timeout=30

    # Imprimir capabilities del servidor
    for capability in m.server_capabilities:
        print(capability)

get_config() — Backup y lectura

Obtener running-config completo y guardar a archivo
from ncclient import manager
from datetime import datetime

def backup_router(host, user, pwd, hostname):
    with manager.connect(host=host, port=830, username=user, password=pwd,
                         hostkey_verify=False, device_params={"name": "iosxe"}) as m:

        # Recuperar TODO el running-config
        reply = m.get_config(source="running")

        # reply.data_xml contiene el XML crudo
        filename = f"backup_{hostname}_{datetime.now():%Y%m%d_%H%M%S}.xml"
        with open(filename, "w") as f:
            f.write(reply.data_xml)

        print(f"✅ Backup guardado: {filename} ({len(reply.data_xml)} chars)")
        return filename

backup_router("192.168.1.1", "admin", "Cisco123", "R1-CORE")

⚡ El atributo xmlns en <config> es obligatorio

Si olvidas el namespace xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native" en el nodo raíz de tu payload XML, IOS-XE responde con unknown-namespace o aplica el cambio en el namespace incorrecto. ncclient no valida esto por ti; es tu responsabilidad construir el XML correcto.

Seguridad en NETCONF

NETCONF asume un transporte seguro. La RFC 6241 no define autenticación propia; delega esto en la capa de transporte. Esto significa que SSH (o TLS) es no opcional, sino fundamental.

SSH como transporte mandatorio

🔑

Password Authentication

Usuario/contraseña vía SSH. El método más común. Requiere hostkey_verify=False en ncclient para labs con certificados autofirmados.

🛡️

Public Key Authentication

Clave SSH privada en el cliente, pública configurada en ip ssh pubkey-chain del router. Más seguro; elimina contraseñas en scripts.

🎫

TLS (RFC 7589)

NETCONF over TLS usa certificados X.509 vía puerto 6513. Soportado en IOS-XE moderno. Alternativa cuando SSH no está disponible.

Configuración segura de SSH para NETCONF en IOS-XE
! Asegurar solo SSH v2 (obligatorio para NETCONF seguro)
ip ssh version 2
ip ssh time-out 60
ip ssh authentication-retries 3

! Deshabilitar algoritmos débiles (IOS-XE 17.x+)
ip ssh server algorithm encryption aes256-gcm aes256-ctr
ip ssh server algorithm mac hmac-sha2-256

! Usuario con privilegio 15 (NETCONF requiere privilegios altos)
username admin privilege 15 algorithm-type sha256 secret Cisco123

! Configurar clave pública SSH (alternativa a password)
ip ssh pubkey-chain
  username admin
   key-hash ssh-rsa AABBCC... fingerprint

! Restringir acceso vía ACL al puerto 830
ip access-list extended NETCONF-MGMT
 permit tcp 10.0.0.0 0.0.0.255 any eq 830 log
 deny ip any any log
!
control-plane host
  management-interface GigabitEthernet0/0 allow ssh netconf

Host key verification en producción

En laboratorio usamos hostkey_verify=False. En producción esto es inaceptable:

Verificación correcta de host keys con ncclient
import os
from ncclient import manager

# Opción A: Usar known_hosts del sistema
# ncclient lee ~/.ssh/known_hosts automáticamente si hostkey_verify=True

# Opción B: Especificar archivo de host keys manualmente
with manager.connect(
    host="10.0.0.1",
    port=830,
    username="admin",
    key_filename="/home/netops/.ssh/id_rsa",  # Auth por clave
    hostkey_verify=True,
    look_for_keys=True,
    allow_agent=True
) as m:
    print(m.session_id)
⚡ NETCONF no cifra el contenido XML por sí mismo

NETCONF es "secure transport" porque asume que SSH/TLS ya cifraron el canal. El XML viaja en texto plano dentro del túnel SSH cifrado. Si intentas usar NETCONF sobre TCP plano (sin SSH), estarás violando el estándar y exponiendo credenciales y configuración.

Despliega configuración estándar en múltiples routers con cero downtime y cero riesgo de dejar un router en estado inconsistente.

Debes actualizar el SNMP community y añadir un usuario de backup en 20 routers de sucursal. El cambio debe ser atómico: si un router falla, ningún otro se ve afectado. Además, necesitas backup XML de la configuración previa por auditoría.

proyecto_netconf_orquestador.py — Script completo de producción
#!/usr/bin/env python3
"""
proyecto_netconf_orquestador.py
CCNA Academy M8 T3 — Despliegue atómico multi-router vía NETCONF RFC 6241
─────────────────────────────────────────────────────────────────────────
- Backup previo de running-config en XML
- Edición sobre candidate datastore
- Lock para evitar modificaciones concurrentes
- Commit confirmado (rollback automático si falla confirmación)
- Manejo de errores con discard_changes + unlock
"""

import sys
import logging
from datetime import datetime
from ncclient import manager
from ncclient.operations import RPCError

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%H:%M:%S"
)

# ══════════════════════════════════════════════════════
# INVENTARIO Y PLANTILLA DE CONFIGURACIÓN
# ══════════════════════════════════════════════════════
ROUTERS = [
    {"hostname": "R1-CORE",  "ip": "10.0.0.1", "snmp": "BOGOTA_RW"},
    {"hostname": "R2-DIST",  "ip": "10.0.0.2", "snmp": "MEDELLIN_RW"},
    {"hostname": "R3-ACC",   "ip": "10.0.0.3", "snmp": "CALI_RW"},
]

USER = "admin"
PASS = "Cisco123"

def build_config_xml(snmp_community):
    """Genera payload XML para edit-config"""
    return f"""
<config xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
    <snmp-server>
      <community>
        <name>{snmp_community}</name>
        <RO/>
      </community>
    </snmp-server>
    <username>
      <name>backupuser</name>
      <privilege>15</privilege>
      <secret>
        <encryption>0</encryption>
        <secret>BackupPass123</secret>
      </secret>
    </username>
  </native>
</config>
""".strip()

# ══════════════════════════════════════════════════════
# FUNCIONES ATÓMICAS
# ══════════════════════════════════════════════════════
def backup_running(conn, hostname):
    """Guarda running-config como XML"""
    reply = conn.get_config(source="running")
    ts = datetime.now().strftime("%Y%m%d_%H%M%S")
    fname = f"backup_{hostname}_{ts}.xml"
    with open(fname, "w") as f:
        f.write(reply.data_xml)
    logging.info(f"[{hostname}] Backup guardado: {fname}")
    return fname

def apply_changes_safe(conn, hostname, snmp):
    """
    Patrón seguro: lock → edit candidate → validate → commit → unlock.
    Si falla cualquier paso: discard_changes → unlock → raise.
    """
    config_xml = build_config_xml(snmp)

    try:
        # 1. Bloquear candidate
        conn.lock(target="candidate")
        logging.info(f"[{hostname}] Candidate bloqueado")

        # 2. Aplicar cambios en candidate
        conn.edit_config(target="candidate", config=config_xml)
        logging.info(f"[{hostname}] edit-config aplicado en candidate")

        # 3. Validar (si el dispositivo soporta :validate)
        try:
            conn.validate(source="candidate")
            logging.info(f"[{hostname}] Validación OK")
        except RPCError:
            logging.warning(f"[{hostname}] Validación no soportada, saltando")

        # 4. Commit confirmado (30 seg de ventana para confirmar)
        conn.commit(confirmed=True, timeout="30")
        logging.info(f"[{hostname}] Commit confirmado (30s)")

        # 5. Commit definitivo (confirma el anterior)
        conn.commit()
        logging.info(f"[{hostname}] Commit confirmado definitivamente")

        return True

    except RPCError as e:
        logging.error(f"[{hostname}] Error NETCONF: {e}")
        return False

    finally:
        # 6. Siempre liberar candidate, incluso si falló
        try:
            conn.discard_changes()
        except Exception:
            pass
        try:
            conn.unlock(target="candidate")
            logging.info(f"[{hostname}] Candidate desbloqueado")
        except Exception:
            pass

def process_router(router):
    """Orquesta un router completo"""
    hostname = router["hostname"]
    ip       = router["ip"]

    logging.info(f"{'='*60}")
    logging.info(f"🚀 Procesando {hostname} ({ip})")

    try:
        with manager.connect(
            host=ip, port=830, username=USER, password=PASS,
            hostkey_verify=False, device_params={"name": "iosxe"},
            timeout=30
        ) as conn:

            # Pre-check: ¿soporta candidate?
            caps = list(conn.server_capabilities)
            has_candidate = any("candidate" in c for c in caps)
            if not has_candidate:
                logging.warning(f"[{hostname}] No soporta candidate. Usando running.")

            # Backup previo
            backup_running(conn, hostname)

            # Aplicar cambios
            ok = apply_changes_safe(conn, hostname, router["snmp"])

            if ok:
                logging.info(f"[{hostname}] ✅ DESPLIEGUE EXITOSO")
                return True
            else:
                logging.error(f"[{hostname}] ❌ DESPLIEGUE FALLIDO")
                return False

    except Exception as e:
        logging.error(f"[{hostname}] ❌ Error de conexión: {e}")
        return False

# ══════════════════════════════════════════════════════
# EJECUCIÓN PRINCIPAL
# ══════════════════════════════════════════════════════
def main():
    exitosos = 0
    for r in ROUTERS:
        if process_router(r):
            exitosos += 1

    logging.info(f"
{'='*60}")
    logging.info(f"🎯 RESUMEN: {exitosos}/{len(ROUTERS)} routers exitosos")
    sys.exit(0 if exitosos == len(ROUTERS) else 1)

if __name__ == "__main__":
    main()
▶️ Cómo ejecutar y verificar
  1. Asegúrate de tener al menos un router IOS-XE con netconf-yang habilitado y usuario privilegio 15.
  2. Instala: pip install ncclient
  3. Actualiza ROUTERS con las IPs reales de tu laboratorio.
  4. Ejecuta: python proyecto_netconf_orquestador.py
  5. Observa el log: backup XML, lock, candidate edit, validate, commit confirmado, unlock.
  6. Verifica en el router: show running-config | include snmp-server y show users.

Este patrón lock → candidate → validate → commit → unlock es el estándar de oro en producción. Nunca edites running directamente sin candidata si tienes usuarios concurrentes.
