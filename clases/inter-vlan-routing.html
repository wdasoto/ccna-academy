Inter-VLAN Routing

Legacy · Router-on-a-Stick · Switch L3 SVIs · Routed Ports · Comparativa · Troubleshooting

CCNA 200-301 Módulo 3 · Switching 🔁 L2 + L3 Integration

🔁 ¿Por qué las VLANs no se comunican solas?

Una VLAN es, por definición, un dominio de broadcast separado. Dos dispositivos en VLANs distintas son como si estuvieran en redes físicamente diferentes: no pueden comunicarse directamente aunque estén conectados al mismo switch físico. Para que puedan hacerlo, el tráfico debe pasar obligatoriamente por un dispositivo de Capa 3 (router o switch L3).

📌 Analogía del mundo real

Imagina que cada VLAN es un edificio cerrado. Los dispositivos dentro del mismo edificio (misma VLAN) pueden hablar directamente entre ellos. Pero para que alguien del edificio A (VLAN 10) se comunique con alguien del edificio B (VLAN 20), el mensaje debe pasar por la recepción central (el router o switch L3). La recepción conoce la dirección de ambos edificios y enruta el mensaje correctamente.

🔒 Aislamiento por diseño

El aislamiento entre VLANs es una característica de seguridad, no un bug. Evita que el departamento de RRHH acceda a los servidores de Producción directamente a nivel Capa 2.

🔁 Routing necesario

Para comunicación inter-VLAN se necesita un dispositivo con capacidad de routing (Capa 3). Solo un router o switch L3 puede tomar decisiones basadas en IPs y reenviar entre VLANs.

📋 Tabla de routing

El dispositivo L3 necesita una ruta hacia cada subred de cada VLAN. Estas rutas pueden ser conectadas directamente (lo más común), estáticas o dinámicas (OSPF, EIGRP).

⚡ Gateway obligatorio

Cada host en cada VLAN debe tener configurado un default gateway que sea la IP de la interfaz (física, subinterfaz o SVI) del dispositivo L3 en esa VLAN.

🔄 El flujo de un paquete inter-VLAN — Paso a paso

Entender exactamente lo que ocurre a nivel de frames y paquetes cuando dos hosts de VLANs distintas se comunican es fundamental para el CCNA y para el troubleshooting.

Escenario: PC-A (VLAN 10 · 192.168.10.10) → PC-B (VLAN 20 · 192.168.20.20)

📦 Viaje completo del paquete

  1. PC-A detecta: destino 192.168.20.20 no está en su red (/24). Necesita usar el gateway.
  2. PC-A hace ARP: "¿Quién tiene 192.168.10.1?" → El router responde con su MAC.
  3. PC-A construye el frame: MAC destino = MAC del router (gateway), IP destino = 192.168.20.20. Envía el frame al switch.
  4. Switch (VLAN 10): recibe el frame en el puerto access de PC-A (VLAN 10). Lo reenvía por el trunk hacia el router (con etiqueta 802.1Q VLAN 10).
  5. Router recibe: frame llega en la subinterfaz/SVI de VLAN 10. Quita la cabecera Ethernet, examina la IP destino (192.168.20.20).
  6. Router consulta tabla routing: encuentra que 192.168.20.0/24 está conectada directamente a la subinterfaz/SVI de VLAN 20.
  7. Router hace ARP (VLAN 20): "¿Quién tiene 192.168.20.20?" → PC-B responde.
  8. Router construye nuevo frame: MAC destino = MAC de PC-B, MAC origen = MAC de la interfaz VLAN 20 del router. Envía por trunk con etiqueta VLAN 20.
  9. Switch (VLAN 20): recibe el frame con etiqueta VLAN 20, quita la etiqueta, entrega al puerto access de PC-B.
  10. PC-B recibe el paquete original de PC-A. ¡Comunicación inter-VLAN completada!

⚠️ Lo que cambia y lo que no cambia

  • La IP origen e IP destino nunca cambian en todo el camino (192.168.10.10 → 192.168.20.20)
  • La MAC destino cambia en cada salto (primero MAC del router, luego MAC de PC-B)
  • La MAC origen cambia en cada salto (primero MAC de PC-A, luego MAC del router en VLAN 20)
  • La etiqueta 802.1Q cambia según la VLAN (10 de entrada → 20 de salida)

🏛️ Método Legacy — Router con interfaces físicas múltiples

El método más antiguo y más intuitivo: se conecta una interfaz física del router por cada VLAN. Cada interfaz tiene una IP que actúa como gateway de esa VLAN. El switch tiene puertos access en cada VLAN conectados directamente al router.

⚠️ ¿Por qué es obsoleto?

  • Con 10 VLANs se necesitan 10 interfaces físicas del router y 10 cables
  • La mayoría de routers tienen 2-4 interfaces GigabitEthernet — se quedan sin puertos rápidamente
  • El costo de hardware escala linealmente con el número de VLANs
  • No hay forma de "escalar" sin comprar hardware adicional

Sin embargo, entenderlo es importante para el CCNA porque ayuda a comprender el concepto de inter-VLAN routing antes de ver los métodos modernos.

📋 Topología Legacy

🗺️ Escenario: 2 VLANs, 2 interfaces físicas del router

R1 Gi0/0 → 192.168.10.1/24 → SW1 Fa0/1 (access VLAN 10) → PC-A

R1 Gi0/1 → 192.168.20.1/24 → SW1 Fa0/2 (access VLAN 20) → PC-B

Los puertos del switch hacia el router son access (no trunk) — cada uno en su VLAN respectiva.

Configuración Legacy — Router

! Interfaz hacia VLAN 10
R1(config)# interface GigabitEthernet0/0
R1(config-if)# description GATEWAY-VLAN10
R1(config-if)# ip address 192.168.10.1 255.255.255.0
R1(config-if)# no shutdown

! Interfaz hacia VLAN 20
R1(config)# interface GigabitEthernet0/1
R1(config-if)# description GATEWAY-VLAN20
R1(config-if)# ip address 192.168.20.1 255.255.255.0
R1(config-if)# no shutdown

! Verificar — ambas redes aparecen como Connected
R1# show ip route
C   192.168.10.0/24 directly connected, GigabitEthernet0/0
C   192.168.20.0/24 directly connected, GigabitEthernet0/1

Configuración Legacy — Switch

! Crear VLANs
SW1(config)# vlan 10
SW1(config-vlan)# name VENTAS
SW1(config)# vlan 20
SW1(config-vlan)# name IT

! Puerto access hacia R1 Gi0/0 — VLAN 10
SW1(config)# interface FastEthernet0/23
SW1(config-if)# description UPLINK-R1-Gi0/0-VLAN10
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 10

! Puerto access hacia R1 Gi0/1 — VLAN 20
SW1(config)# interface FastEthernet0/24
SW1(config-if)# description UPLINK-R1-Gi0/1-VLAN20
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 20

! Puertos de hosts
SW1(config)# interface FastEthernet0/1
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 10
SW1(config)# interface FastEthernet0/2
SW1(config-if)# switchport mode access
SW1(config-if)# switchport access vlan 20

🥩 Router-on-a-Stick — Un cable, todas las VLANs

Router-on-a-Stick (RoaS) soluciona el problema del método legacy con una idea elegante: usar un único enlace trunk entre el router y el switch, y dividir ese enlace en múltiples subinterfaces virtuales en el router — una por cada VLAN. El nombre viene de la imagen de un router colgando de un "palo" (el único cable) conectado al switch.

💡 Principio

La interfaz física del router recibe frames 802.1Q etiquetados con diferentes VLAN IDs. Cada subinterfaz está configurada para procesar frames de una VLAN específica.

⚡ Ventaja

Solo se necesita 1 cable físico y 1 puerto del switch sin importar cuántas VLANs haya. Mucho más económico y escalable que el método legacy.

⚠️ Limitación

Todo el tráfico inter-VLAN pasa por el mismo enlace físico — puede convertirse en un cuello de botella en redes con mucho tráfico entre VLANs.

🔬 Subinterfaces — Anatomía completa

Una subinterfaz es una interfaz virtual creada sobre una interfaz física. Se nombra con el formato INTERFAZ.NÚMERO (ej: Gi0/0.10). El número no tiene que coincidir con el VLAN ID, pero por convención y claridad, se usa el mismo número.

📌 Comandos clave de una subinterfaz RoaS

  1. interface Gi0/0.10 — crea la subinterfaz
  2. encapsulation dot1q 10 — asocia la subinterfaz a la VLAN 10 (OBLIGATORIO)
  3. ip address 192.168.10.1 255.255.255.0 — IP gateway de la VLAN
  4. La interfaz física Gi0/0 debe tener no shutdown pero SIN IP

⚠️ Los 3 errores más comunes en RoaS

  1. Olvidar encapsulation dot1q X en la subinterfaz — sin esto, el router no sabe a qué VLAN pertenece
  2. Poner IP en la interfaz física en lugar de dejarse sin IP — esto causa routing incorrecto
  3. No hacer no shutdown en la interfaz física — las subinterfaces heredan el estado de la física

Configuración RoaS completa — Router

! ── INTERFAZ FÍSICA — sin IP, solo activa ───────────
R1(config)# interface GigabitEthernet0/0
R1(config-if)# description TRUNK-A-SW1-Gi0/1
R1(config-if)# no ip address
R1(config-if)# no shutdown
! La interfaz física SIN IP — solo debe estar up
! Si se le pone IP, el routing se vuelve impredecible

! ── SUBINTERFAZ VLAN 10 ──────────────────────────────
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# description GATEWAY-VLAN10-VENTAS
R1(config-subif)# encapsulation dot1q 10
! encapsulation dot1q 10: esta subinterfaz maneja frames VLAN 10
! OBLIGATORIO — sin este comando la subinterfaz no funciona
R1(config-subif)# ip address 192.168.10.1 255.255.255.0

! ── SUBINTERFAZ VLAN 20 ──────────────────────────────
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# description GATEWAY-VLAN20-IT
R1(config-subif)# encapsulation dot1q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0

! ── SUBINTERFAZ VLAN 30 ──────────────────────────────
R1(config)# interface GigabitEthernet0/0.30
R1(config-subif)# description GATEWAY-VLAN30-RRHH
R1(config-subif)# encapsulation dot1q 30
R1(config-subif)# ip address 192.168.30.1 255.255.255.0

! ── SUBINTERFAZ VLAN NATIVA ──────────────────────────
R1(config)# interface GigabitEthernet0/0.99
R1(config-subif)# description VLAN-NATIVA-GESTION
R1(config-subif)# encapsulation dot1q 99 native
! "native" keyword: esta subinterfaz recibe frames SIN etiqueta
R1(config-subif)# ip address 192.168.99.1 255.255.255.0

Configuración RoaS — Switch (trunk hacia el router)

! Crear VLANs en el switch
SW1(config)# vlan 10
SW1(config-vlan)# name VENTAS
SW1(config)# vlan 20
SW1(config-vlan)# name IT
SW1(config)# vlan 30
SW1(config-vlan)# name RRHH
SW1(config)# vlan 99
SW1(config-vlan)# name GESTION

! Puerto trunk hacia el router — el "palo" del RoaS
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# description TRUNK-A-R1-Gi0/0-ROAS
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk encapsulation dot1q
! (en 2960 no es necesario — solo soporta dot1q)
SW1(config-if)# switchport nonegotiate
SW1(config-if)# switchport trunk native vlan 99
! VLAN nativa debe coincidir con "encapsulation dot1q 99 native" en R1
SW1(config-if)# switchport trunk allowed vlan 10,20,30,99

! Puertos de acceso para hosts
SW1(config)# interface range FastEthernet0/1-8
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 10
SW1(config)# interface range FastEthernet0/9-16
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 20

🔀 Switch L3 con SVIs — El estándar moderno

Un switch de Capa 3 (como el Catalyst 3560, 3750 o 9300) combina switching de Capa 2 y routing de Capa 3 en el mismo dispositivo. Con el comando ip routing habilitado, cada SVI (Switch Virtual Interface) actúa como el gateway de su VLAN, y el routing ocurre en hardware a wire speed — sin cuello de botella.

📌 ¿Cómo funciona internamente?

A diferencia de RoaS (donde el tráfico debe salir físicamente del switch, entrar al router, y volver), en un switch L3 el routing ocurre internamente en los ASICs. Un paquete de VLAN 10 a VLAN 20 nunca abandona el switch — se enruta internamente de un dominio de broadcast al otro. Esto elimina el cuello de botella del enlace físico.

Configuración completa Switch L3 con SVIs

! ── PASO 1: Habilitar routing (CRÍTICO) ─────────────
SW3(config)# ip routing
! Sin ip routing: SVIs tienen IP pero NO enrutan — error #1

! ── PASO 2: Crear VLANs ─────────────────────────────
SW3(config)# vlan 10
SW3(config-vlan)# name VENTAS
SW3(config)# vlan 20
SW3(config-vlan)# name IT
SW3(config)# vlan 30
SW3(config-vlan)# name RRHH
SW3(config)# vlan 40
SW3(config-vlan)# name GERENCIA
SW3(config)# vlan 99
SW3(config-vlan)# name GESTION

! ── PASO 3: Crear SVIs (gateways de cada VLAN) ───────
SW3(config)# interface vlan 10
SW3(config-if)# description GATEWAY-VLAN10-VENTAS
SW3(config-if)# ip address 192.168.10.1 255.255.255.0
SW3(config-if)# no shutdown
! SVIs necesitan "no shutdown" — no están activas por defecto

SW3(config)# interface vlan 20
SW3(config-if)# description GATEWAY-VLAN20-IT
SW3(config-if)# ip address 192.168.20.1 255.255.255.0
SW3(config-if)# no shutdown

SW3(config)# interface vlan 30
SW3(config-if)# description GATEWAY-VLAN30-RRHH
SW3(config-if)# ip address 192.168.30.1 255.255.255.0
SW3(config-if)# no shutdown

SW3(config)# interface vlan 40
SW3(config-if)# description GATEWAY-VLAN40-GERENCIA
SW3(config-if)# ip address 192.168.40.1 255.255.255.0
SW3(config-if)# no shutdown

SW3(config)# interface vlan 99
SW3(config-if)# description GESTION-SW
SW3(config-if)# ip address 192.168.99.1 255.255.255.0
SW3(config-if)# no shutdown

! ── PASO 4: Puertos access para hosts ───────────────
SW3(config)# interface range FastEthernet0/1-8
SW3(config-if-range)# switchport mode access
SW3(config-if-range)# switchport access vlan 10

SW3(config)# interface range FastEthernet0/9-16
SW3(config-if-range)# switchport mode access
SW3(config-if-range)# switchport access vlan 20

! ── PASO 5: Trunks hacia switches de acceso ─────────
SW3(config)# interface GigabitEthernet0/1
SW3(config-if)# description TRUNK-SW-ACCESO-1
SW3(config-if)# switchport mode trunk
SW3(config-if)# switchport trunk native vlan 999
SW3(config-if)# switchport trunk allowed vlan 10,20,30,40,99

! ── PASO 6: Uplink hacia router/Internet ────────────
SW3(config)# interface GigabitEthernet0/2
SW3(config-if)# description UPLINK-ROUTER-INTERNET
SW3(config-if)# no switchport
! "no switchport" = Routed Port (Capa 3 directamente)
SW3(config-if)# ip address 10.0.0.2 255.255.255.252
SW3(config-if)# no shutdown

! ── PASO 7: Ruta por defecto hacia Internet ─────────
SW3(config)# ip route 0.0.0.0 0.0.0.0 10.0.0.1

🔌 Routed Ports — Interfaces L3 puras en un switch

Un Routed Port es un puerto físico de un switch L3 que se configura para operar como interfaz de Capa 3 directamente — sin switching de Capa 2. Se deshabilita la funcionalidad de switching con no switchport, y el puerto se comporta exactamente igual que una interfaz de un router.

🔌 Cuándo usar Routed Ports

Enlace punto a punto entre el switch L3 y un router. Enlace entre switches L3 en la capa de distribución/core. Enlace hacia el ISP o firewall.

🚫 Cuándo NO usar Routed Ports

Para conectar hosts usuarios — usar SVIs. Para conectar switches de acceso con múltiples VLANs — usar trunks hacia SVIs.

Configurar y verificar un Routed Port

! Convertir un puerto físico en routed port
SW3(config)# interface GigabitEthernet0/2
SW3(config-if)# no switchport
! "no switchport" elimina toda la funcionalidad L2 del puerto
SW3(config-if)# description UPLINK-HACIA-R1-WAN
SW3(config-if)# ip address 10.0.0.2 255.255.255.252
SW3(config-if)# no shutdown

! Verificar que es routed port
SW3# show interfaces GigabitEthernet0/2 switchport
Name: Gi0/2
Switchport: Disabled  ← routed port confirmado

SW3# show interfaces GigabitEthernet0/2
GigabitEthernet0/2 is up, line protocol is up
  Internet address is 10.0.0.2/30
  (sin información de switchport)

! Restaurar a switchport si es necesario
SW3(config-if)# switchport
! Quita la IP y restaura funcionalidad L2

🎯 ¿Cuándo usar cada método?

🏛️ Legacy

Solo en contexto histórico o teórico. No usar en ningún entorno real nuevo. Útil para entender el concepto base de inter-VLAN routing.

Equipos: cualquier router + switch L2

🥩 Router-on-a-Stick

Redes pequeñas y medianas con presupuesto limitado. Laboratorios CCNA. Cuando ya tienes un router y no quieres comprar switch L3.

Equipos: Cisco 1900/2900/4000 + Catalyst 2960

🔀 Switch L3 SVIs

Entornos empresariales, data centers, campus networks. Cuando el rendimiento y la escalabilidad son prioritarios. Redes con más de 5-10 VLANs.

Equipos: Catalyst 3560/3750/9200/9300

📌 Regla práctica para el examen CCNA

  • Si el examen dice "switch 2960" o "switch L2": usar Router-on-a-Stick
  • Si el examen dice "switch 3560" o "switch L3" o "ip routing": usar Switch L3 SVIs
  • Si el examen pregunta cuál es más escalable o de mejor rendimiento: Switch L3
  • Si el examen pregunta cuál usa menos hardware: Router-on-a-Stick

🏗️ Arquitectura de red típica con Switch L3

🌐 Modelo de 3 capas de Cisco (Hierarchical Network Design)

El switch de distribución (L3) maneja todo el inter-VLAN routing. Los switches de acceso (L2) solo conectan usuarios y envían el tráfico por trunks hacia distribución.

🔧 Troubleshooting Inter-VLAN Routing — Metodología

Cuando el inter-VLAN routing no funciona, hay que seguir una metodología sistemática de arriba hacia abajo. Los problemas más comunes tienen causas específicas y diagnósticos bien definidos.

📋 Metodología de diagnóstico — Del host hacia arriba

  1. Verificar el host: ¿tiene IP, máscara y gateway correctos? ¿puede hacer ping a su propio gateway?
  2. Verificar el switch: ¿el puerto del host está en la VLAN correcta? ¿el trunk está activo y con las VLANs correctas?
  3. Verificar el router/switch L3: ¿las subinterfaces/SVIs están up/up? ¿tienen IP correctas? ¿ip routing habilitado?
  4. Verificar la tabla de routing: ¿aparecen todas las redes destino?
  5. Probar conectividad: ping desde el gateway de origen → gateway destino → host destino

📋 Resumen Maestro — Inter-VLAN Routing para el CCNA

✅ Los 25 puntos críticos del examen

  1. Las VLANs son dominios de broadcast separados — no se comunican sin un dispositivo de Capa 3.
  2. En inter-VLAN routing: las IPs origen/destino nunca cambian; las MACs sí cambian en cada salto.
  3. Método Legacy: 1 interfaz física del router por VLAN. Obsoleto — no escala.
  4. Router-on-a-Stick: 1 enlace trunk al router + subinterfaces. Económico, para redes pequeñas.
  5. Switch L3 SVIs: ip routing + SVIs como gateways. Wire speed, estándar empresarial.
  6. RoaS: interfaz física del router sin IP + no shutdown. Las IPs van en las subinterfaces.
  7. RoaS: encapsulation dot1q X es obligatorio en cada subinterfaz. Sin esto no funciona.
  8. RoaS: subinterfaz con VLAN nativa usa encapsulation dot1q X native.
  9. El número de la subinterfaz (Gi0/0.10) no tiene que ser igual al VLAN ID, pero por convención sí.
  10. Switch L3: ip routing es obligatorio. Sin él, las SVIs tienen IP pero no enrutan.
  11. Switch L3: no switchport convierte un puerto en Routed Port (igual que una interfaz de router).
  12. SVI UP/UP requiere: VLAN existe + puerto activo en esa VLAN + no shutdown en la SVI.
  13. Switch L2 con SVI: usar ip default-gateway. Switch L3 con ip routing: NO usar default-gateway.
  14. El switch de distribución L3 hace el routing; los switches de acceso L2 solo transportan.
  15. El trunk entre switch y router (RoaS) debe tener las mismas VLANs permitidas que las subinterfaces.
  16. VLAN nativa del trunk DEBE coincidir en ambos extremos. Si no, hay VLAN hopping.
  17. El host necesita el gateway correcto configurado. Si el gateway es incorrecto, no hay inter-VLAN routing.
  18. TTL en inter-VLAN routing: el paquete pierde 1 TTL al cruzar el dispositivo L3.
  19. Diagnóstico: primero ping al gateway propio, luego ping entre gateways, luego ping al host destino.
  20. show ip interface brief: verifica estado up/up de subinterfaces/SVIs.
  21. show interfaces Gi0/0.10: muestra "Encapsulation 802.1Q Virtual LAN, Vlan ID 10".
  22. show interfaces trunk: verifica VLANs permitidas — si falta una VLAN, no pasa tráfico.
  23. show ip route: todas las redes VLAN deben aparecer como "C" (connected).
  24. RoaS cuello de botella: todo el tráfico inter-VLAN usa el mismo enlace físico — 1 Gbps máximo total.
  25. Switch L3 sin cuello de botella: el routing ocurre en ASICs internos sin salir del switch.

⚠️ Los 10 errores más comunes del examen

  1. Olvidar encapsulation dot1q X en la subinterfaz RoaS — la más frecuente
  2. Poner IP en la interfaz física Gi0/0 en RoaS en lugar de dejarla sin IP
  3. Olvidar ip routing en switch L3 — las SVIs tienen IP pero no enrutan
  4. Olvidar no shutdown en las SVIs del switch L3
  5. VLAN nativa diferente en el trunk — genera VLAN hopping y tráfico incorrecto
  6. VLAN no permitida en el trunk hacia el router/switch L3 — esa VLAN no puede rutear
  7. SVI en VLAN que no existe o sin puerto activo — permanece up/down
  8. Host con gateway incorrecto — el paquete nunca llega al dispositivo L3
  9. Usar ip default-gateway en switch L3 con ip routing activo — comportamiento inesperado
  10. Confundir cuándo usar RoaS (switch L2) vs SVIs (switch L3)
