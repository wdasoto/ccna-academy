OSPF — Open Shortest Path First

Link-State · Router ID · Adyacencias · LSAs · Áreas · DR/BDR · Costo · SPF · OSPFv3 · Troubleshooting

CCNA 200-301 Módulo 4 · Routing ⭕ Protocolo Link-State

📖 OSPF — El protocolo de routing estándar empresarial

OSPF (Open Shortest Path First) es un protocolo de routing link-state definido en el RFC 2328. Es el protocolo IGP (Interior Gateway Protocol) más utilizado en redes empresariales. "Open" significa que es un estándar abierto — no propietario — interoperable entre todos los fabricantes.

📡 Link-State

Cada router construye un mapa completo de la topología (LSDB). Usa el algoritmo SPF de Dijkstra para calcular la ruta más corta. Converge más rápido que distance-vector.

⭕ Single Area vs Multi-Area

En redes pequeñas: single-area (área 0). En redes grandes: multi-area con un backbone (área 0) y áreas secundarias para reducir el procesamiento SPF y el tamaño de la LSDB.

💰 Métrica: Costo

Basada en el ancho de banda del enlace. Costo = 100,000,000 / BW(bps). Acumulativo por el camino. El router calcula el camino de menor costo total.

🔢 AD = 110

Distancia Administrativa 110. Si OSPF y otro protocolo anuncian el mismo prefijo, gana el de menor AD. OSPF pierde ante EIGRP (AD=90) pero gana ante RIP (AD=120).

🆚 Link-State vs Distance-Vector — La diferencia fundamental

🆔 Router ID — La identidad única de cada router OSPF

El Router ID (RID) es un número de 32 bits en formato IPv4 que identifica unívocamente a cada router dentro del dominio OSPF. Aunque tiene formato de IP, no es necesariamente una IP enrutable — es solo un identificador. Aparece en los LSAs, en la base de datos LSDB y en las adyacencias.

⚠️ Si dos routers tienen el mismo Router ID, OSPF NO formará adyacencia

Los Router IDs deben ser únicos en todo el dominio OSPF. Si hay duplicados, el proceso OSPF detecta el conflicto y no establece la adyacencia. El error aparece en los logs: "OSPF: Detected duplicate router-id".

🏆 Proceso de selección del Router ID — Orden de prioridad

📌 El router selecciona el Router ID en este orden (de mayor a menor prioridad)

  1. Configuración manual (router-id X.X.X.X) — siempre la opción preferida y recomendada.
  2. IP más alta de una interfaz Loopback activa — las loopbacks son siempre up/up mientras el proceso OSPF esté activo.
  3. IP más alta de cualquier interfaz física activa — el router usa la IP más alta en el momento de arrancar OSPF.

⚠️ El Router ID se selecciona UNA vez al arrancar el proceso OSPF

Si el Router ID cambia (ej: nueva loopback con IP más alta), no se aplica inmediatamente. Requiere clear ip ospf process (reinicia las adyacencias — impacto en producción) o reiniciar el router. Por eso siempre se recomienda configurar el Router ID manualmente desde el principio.

⚙️ Configuración del Router ID

! ─── MÉTODO 1: Router ID manual (RECOMENDADO) ────────
R1(config)# router ospf 1
R1(config-router)# router-id 1.1.1.1
! Convención: usar 1.1.1.1 para R1, 2.2.2.2 para R2, etc.
! El cambio no es inmediato — necesita reload o clear ospf process

! ─── MÉTODO 2: Loopback (buena práctica secundaria) ──
R1(config)# interface Loopback0
R1(config-if)# ip address 1.1.1.1 255.255.255.255

🤝 El proceso de adyacencia OSPF — 7 estados

OSPF no forma adyacencias instantáneamente. Sigue un proceso ordenado de 7 estados para asegurar que ambos routers tienen la misma LSDB antes de intercambiar rutas. Cada estado tiene una función específica.

EstadoNombre¿Qué ocurre?
1DownNo se han recibido Hello packets del vecino. Estado inicial.
2AttemptSolo en NBMA. Se envía Hello unicast. Si no hay respuesta → vuelve a Down.
3InitSe recibió un Hello del vecino, pero el Hello recibido NO incluye nuestro Router ID en la lista de vecinos. El vecino nos ha visto, pero nosotros no lo hemos confirmado aún.
42-WayComunicación bidireccional establecida. Nuestro RID aparece en el Hello del vecino. Aquí ocurre la elección DR/BDR en redes broadcast. En enlaces punto a punto se salta este estado rápidamente.
5ExStartSe negocia el Master/Slave y el número de secuencia inicial para el intercambio de DBD. El router con mayor RID es Master.
6ExchangeIntercambio de DBD (Database Description) packets — resúmenes de la LSDB de cada router.
7LoadingSe solicitan (LSR) y reciben (LSU) los LSAs que faltan en la LSDB local. Se confirman con LSAck.
8Full✅ La adyacencia está completamente establecida. Ambas LSDBs son idénticas. Las rutas se calculan con SPF.

📨 Tipos de paquetes OSPF

TipoNombreFunción
1HelloDescubre vecinos, mantiene adyacencias, anuncia parámetros (Hello/Dead timers, área, autenticación). Enviado periódicamente a 224.0.0.5.
2DBD (Database Description)Resumen de la LSDB. Contiene cabeceras de LSAs. Permite al vecino identificar qué LSAs le faltan.
3LSR (Link-State Request)Solicita LSAs específicos que faltan en la LSDB local (identificados durante el Exchange).
4LSU (Link-State Update)Contiene uno o más LSAs completos. La respuesta a un LSR. También se usa para flooding de nuevos LSAs.
5LSAck (Link-State Acknowledgment)Confirma la recepción de LSUs. OSPF requiere confirmación explícita de cada LSA recibido.

📦 LSA — Link-State Advertisements

Los LSAs son los "bloques de información" que OSPF inunda por toda el área. Cada router genera LSAs describiendo sus enlaces y vecinos. El conjunto de todos los LSAs forma la LSDB (Link-State Database) — el mapa completo de la topología. Todos los routers del área deben tener LSDBs idénticas.

🔑 Flooding de LSAs — Cómo se propagan

  1. Cuando cambia la topología, el router afectado genera un nuevo LSA con un número de secuencia incrementado.
  2. Envía el LSA en un LSU a todos sus vecinos OSPF.
  3. Los vecinos instalan el LSA en su LSDB, lo retransmiten a sus propios vecinos, y confirman con LSAck.
  4. El proceso continúa hasta que todos los routers del área tienen el LSA actualizado.
  5. Con la LSDB actualizada, cada router ejecuta el algoritmo SPF para recalcular las rutas.

🗺️ Áreas OSPF — División jerárquica de la red

Las áreas OSPF son agrupaciones lógicas de routers y enlaces que comparten la misma LSDB. La jerarquía de áreas es la solución de OSPF para escalar a redes grandes: cada área mantiene su propia LSDB (más pequeña), y el cálculo SPF solo se ejecuta dentro del área.

📌 Regla fundamental: toda red OSPF multi-área debe tener el Área 0 (backbone)

  • El Área 0 (también llamada backbone area) es obligatoria en cualquier diseño multi-área.
  • Todas las demás áreas deben conectarse directamente al Área 0.
  • Todo el tráfico entre áreas pasa a través del Área 0 (no hay routing directo área-a-área sin pasar por el backbone).

👤 Roles de los routers en OSPF multi-área

Tipo de RouterPosiciónFunción
Internal Router (IR)Todas sus interfaces en una sola áreaSolo genera LSAs tipo 1. LSDB solo de su área. Sin rol especial.
Backbone Router (BR)Al menos una interfaz en el Área 0Participa en el backbone. Puede ser también ABR.
ABR (Area Border Router)Interfaces en múltiples áreas (incluyendo el Área 0)Conecta áreas al backbone. Genera LSAs tipo 3 (resúmenes inter-área). Mantiene LSDB de cada área conectada.
ASBR (AS Boundary Router)Conectado a red externa (no-OSPF)Redistribuye rutas externas en OSPF. Genera LSAs tipo 5. Puede estar en cualquier área excepto stub.

👑 DR y BDR — Optimizando las adyacencias en redes broadcast

En un segmento de red broadcast (Ethernet) con N routers, sin DR/BDR cada par de routers formaría una adyacencia Full. Eso resultaría en N×(N-1)/2 adyacencias — con 5 routers serían 10 adyacencias, con 10 routers serían 45. El DR/BDR reduce esto a N-1 adyacencias Full (todas con el DR/BDR).

📌 Roles DR, BDR y DROther

  • DR (Designated Router): el "líder" del segmento. Todos los demás forman adyacencia Full con el DR. Recibe actualizaciones LSA de todos y las reenvía al resto. Envía updates a 224.0.0.6 (AllDRouters).
  • BDR (Backup Designated Router): el respaldo del DR. Forma adyacencia Full con el DR y con todos los DROthers. Si el DR cae, el BDR asume el rol inmediatamente sin nueva elección.
  • DROther: el resto de los routers. Solo forman adyacencia Full con el DR y el BDR. Entre DROthers la adyacencia queda en estado 2-Way (no Full). Envían updates al DR a 224.0.0.6.

⚠️ Sin DR/BDR: N×(N-1)/2 adyacencias. Con DR/BDR: solo N-1 adyacencias Full

5 routers sin DR/BDR → 10 adyacencias Full → 10 LSDBs sincronizadas simultáneamente.
5 routers con DR/BDR → 4 adyacencias Full → mucho menos overhead de sincronización.

🗳️ Proceso de elección DR/BDR

📌 Reglas de elección (de mayor a menor prioridad)

  1. Mayor prioridad OSPF (0-255). Default=1. Mayor prioridad → DR. Segundo mayor → BDR.
  2. Mayor Router ID — desempata si las prioridades son iguales.
  3. Prioridad 0 = no participa en la elección. Garantiza que ese router nunca sea DR ni BDR.

⚠️ La elección DR/BDR NO es apropiativa (non-preemptive)

Si el DR ya fue elegido y llega un nuevo router con mayor prioridad o mayor RID, el nuevo router NO desplaza al DR actual. El DR cambia solo si el actual DR cae — en ese caso el BDR asciende a DR y se elige un nuevo BDR. Para forzar una re-elección: clear ip ospf process.

⚙️ Configurar prioridad OSPF y verificar DR/BDR

💰 Costo OSPF — Cálculo y ajuste

El costo OSPF es la métrica que determina el camino óptimo. Se calcula como 100,000,000 / BW(bps). El costo de una ruta es la suma acumulada de los costos de todos los enlaces en el camino desde el router hasta la red destino.

⚠️ El problema con FastEthernet, GigabitEthernet y 10GE

InterfazBWCosto (ref=100M)Costo (ref=10G)
Serial T11.544 Mbps646476
FastEthernet100 Mbps1100
GigabitEthernet1000 Mbps1 ← igual a FE10
10GigabitEthernet10000 Mbps1 ← igual a FE/GE1 ← igual a GE
100GigabitEthernet100000 Mbps11

Con el reference bandwidth por defecto (100 Mbps), GigabitEthernet y FastEthernet tienen el mismo costo (1). OSPF no distingue entre ellos y puede elegir el camino incorrecto. Solución: ajustar el reference bandwidth.

⚙️ Ajustar el costo OSPF

Método 1 — Cambiar el reference bandwidth (recomendado)

! ⚠️ Debe configurarse en TODOS los routers del área
! Si no es consistente, SPF calcula rutas incorrectas
R1(config)# router ospf 1
R1(config-router)# auto-cost reference-bandwidth 10000
! 10000 Mbps = 10 Gbps como nuevo reference bandwidth
! Nuevos costos: GE=10, 10GE=1, 100GE=1
R1(config-router)# auto-cost reference-bandwidth 100000
! 100 Gbps: GE=100, 10GE=10, 100GE=1 — mejor granularidad
% OSPF: Reference bandwidth is changed.
  Please ensure reference bandwidth is consistent across all routers.

Método 2 — Costo manual por interfaz

! Asignar costo específico a una interfaz (override del cálculo automático)
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip ospf cost 10
! El costo manual siempre tiene prioridad sobre el calculado
! Útil para influir en la selección de rutas sin cambiar el BW físico

🧮 Algoritmo SPF — Cómo OSPF elige el mejor camino

El algoritmo de Dijkstra (SPF — Shortest Path First) calcula el árbol de caminos más cortos desde el router local hacia todos los destinos. El router con el menor costo acumulado al destino gana.

📌 Ejemplo de cálculo SPF

R1 → R2 (costo 10) → RED_B (costo 1) → Total = 11
R1 → R3 (costo 5) → R4 (costo 5) → RED_B (costo 1) → Total = 11
Ambos caminos tienen el mismo costo → OSPF instala las dos rutas (ECMP) → balanceo de carga.

! Ver rutas OSPF en la tabla de routing
R1# show ip route ospf
O    10.0.23.0/30 [110/128] via 10.0.12.2, 00:05:12, Se0/0/0
O    192.168.3.0/24 [110/129] via 10.0.12.2, 00:05:12, Se0/0/0
O IA 172.16.0.0/16 [110/65] via 10.0.12.2, 00:02:34, Se0/0/0
! O    = ruta OSPF intra-área
! O IA = ruta OSPF inter-área (de otra área)
! [110/129] = [AD=110 / Costo=129]

! ─── passive-interface ───────────────────────────────
! Evitar que OSPF envíe Hellos hacia redes de usuarios
R1(config)# router ospf 1
R1(config-router)# passive-interface default
! Pone TODAS las interfaces en modo pasivo
R1(config-router)# no passive-interface Serial0/0/0
! Luego activa OSPF solo en las interfaces WAN que sí tienen vecinos
! Mejor práctica: deshabilitar Hellos en interfaces LAN de usuarios

🧪 Laboratorio Magistral — OSPF Multi-Área Empresarial

📋 Topología: 3 routers, 2 áreas, ABR central

R1 — Área 0 (backbone): Gi0/0 → 192.168.1.1/24 (LAN-SEDE), Se0/0/0 → 10.0.12.1/30 (hacia R2), Loopback0 → 1.1.1.1/32
R2 — ABR (Área 0 + Área 1): Se0/0/0 → 10.0.12.2/30 (hacia R1), Se0/0/1 → 10.0.23.1/30 (hacia R3), Loopback0 → 2.2.2.2/32
R3 — Área 1: Se0/0/1 → 10.0.23.2/30 (hacia R2), Gi0/0 → 192.168.3.1/24 (LAN-SUCURSAL), Loopback0 → 3.3.3.3/32

Fase 2 — Configuración R2 (ABR Área 0 + Área 1)

📋 Resumen Maestro — OSPF CCNA 200-301

✅ Los 28 puntos más críticos del examen

  1. OSPF es un protocolo link-state, estándar abierto (RFC 2328). AD = 110. IP protocolo número 89.
  2. Cada router construye un mapa completo (LSDB). Usa el algoritmo de Dijkstra (SPF) para calcular rutas.
  3. Multicast 224.0.0.5 = AllSPFRouters. 224.0.0.6 = AllDRouters (solo DR y BDR).
  4. Hello interval Ethernet = 10s. Dead interval = 40s (4× Hello). Serial = 30s/120s.
  5. Router ID: 1º manual (router-id), 2º IP más alta de Loopback, 3º IP más alta de interfaz activa.
  6. Dos Router IDs iguales → adyacencia NO se forma. Siempre configurar manualmente.
  7. Cambio de Router ID requiere clear ip ospf process para aplicarse (reinicia adyacencias).
  8. 7 estados de adyacencia: Down → Init → 2-Way → ExStart → Exchange → Loading → Full.
  9. 5 tipos de paquetes: Hello, DBD, LSR, LSU, LSAck.
  10. Requisitos para adyacencia: misma área, mismos Hello/Dead timers, misma subred, misma autenticación, RIDs únicos.
  11. Process ID es local al router — no necesita coincidir entre vecinos.
  12. LSA Tipo 1 = Router LSA (todos los routers, solo su área). Tipo 2 = Network LSA (DR). Tipo 3 = Summary (ABR, inter-área). Tipo 5 = External (ASBR).
  13. Flooding: LSAs se propagan a todo el área. Todos los routers del área tienen la misma LSDB.
  14. Área 0 = backbone obligatorio en multi-área. Todas las demás áreas se conectan al Área 0.
  15. ABR: conecta múltiples áreas. Genera LSAs tipo 3. Mantiene LSDB de cada área conectada.
  16. ASBR: redistribuye rutas externas. Genera LSAs tipo 5. No puede estar en área stub.
  17. Área Stub: bloquea LSA tipo 5. Usa default route para rutas externas.
  18. Área Totally Stub: bloquea LSA tipo 3, 4 y 5. Solo default route. Solo el ABR configura "no-summary".
  19. DR/BDR: solo en redes broadcast. Reduce adyacencias de N×(N-1)/2 a N-1.
  20. Elección DR: mayor prioridad → mayor Router ID. Prioridad 0 = nunca DR/BDR.
  21. DR/BDR es NO apropiativo: un nuevo router con mayor prioridad no desplaza al DR actual.
  22. Costo OSPF = 100,000,000 / BW(bps). FastEthernet y GigabitEthernet tienen el mismo costo con ref=100M.
  23. Solución: auto-cost reference-bandwidth 10000 en TODOS los routers del área.
  24. passive-interface: la interfaz sigue anunciando la red pero no envía/recibe Hellos.
  25. O = ruta intra-área. O IA = inter-área. O E1/E2 = externa redistribuida.
  26. show ip ospf neighbor: ver estado de adyacencias. FULL = OK.
  27. show ip ospf database: ver la LSDB completa con todos los LSAs.
  28. show ip ospf interface X: ver el rol del router (DR/BDR/DROther), timers y costo en esa interfaz.

⚠️ Los 10 errores más comunes en el examen

  1. Creer que el Process ID debe coincidir entre vecinos — es local al router
  2. Confundir Hello/Dead timers de Ethernet (10/40) con los de Serial (30/120)
  3. No ajustar reference-bandwidth en todos los routers del área — costos inconsistentes
  4. Creer que DR/BDR se elige en enlaces punto a punto — solo en redes broadcast
  5. Pensar que la elección DR es apropiativa — el nuevo router no desplaza al DR actual
  6. Olvidar que passive-interface aún anuncia la red (solo detiene los Hellos)
  7. Configurar área stub solo en algunos routers del área — deben configurarse todos
  8. Poner "no-summary" en routers internos del área totally stub — solo va en el ABR
  9. Creer que el Router ID siempre es una IP enrutable — es solo un identificador de 32 bits
  10. No hacer clear ip ospf process después de cambiar el Router ID — el cambio no aplica
