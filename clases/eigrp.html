EIGRP

Enhanced Interior Gateway Routing Protocol · Algoritmo DUAL · Vecinos · Métricas · Sumarización · Troubleshooting

⚡ ¿Qué es EIGRP?

EIGRP (Enhanced Interior Gateway Routing Protocol) es un protocolo de enrutamiento propietario de Cisco (abierto desde 2013 como RFC 7868) clasificado como protocolo híbrido — combina las ventajas de los protocolos de vector de distancia (como RIP) con características de los protocolos de estado de enlace (como OSPF). Utiliza el algoritmo DUAL para garantizar rutas libres de bucles y convergencia ultrarrápida.
✅
EIGRP es considerado el protocolo IGP de convergencia más rápida disponible en Cisco IOS. En escenarios ideales, la conmutación de ruta puede ser instantánea si existe una ruta de respaldo precalculada (Feasible Successor).
🎯
CCNA Examen: EIGRP es clasificado como protocolo Advanced Distance Vector (o híbrido). Usa multicast 224.0.0.10 para comunicarse con vecinos y el número de protocolo IP 88. AD interna = 90, AD externa = 170.

🏗️ Las 3 tablas de EIGRP

TABLAS INTERNAS EIGRP   1. Tabla de Vecinos (Neighbor Table) Almacena todos los routers EIGRP adyacentes confirmados → show ip eigrp neighbors   2. Tabla de Topología (Topology Table) Contiene TODAS las rutas conocidas hacia cada destino Incluye Successor y Feasible Successors (rutas backup) → show ip eigrp topology   3. Tabla de Enrutamiento (Routing Table) Solo contiene las mejores rutas (Successors) Marcadas con D en la tabla de rutas → show ip route eigrp
💡
La clave de la velocidad de EIGRP es la Tabla de Topología — ya tiene las rutas backup calculadas. Si la ruta principal falla, la ruta backup (Feasible Successor) se instala en la tabla de enrutamiento de forma instantánea, sin recalcular.

🧮 Algoritmo DUAL

DUAL (Diffusing Update Algorithm) es el motor matemático de EIGRP. Garantiza que las rutas calculadas sean libres de bucles en todo momento y permite una convergencia ultrarrápida usando rutas de respaldo precalculadas. Fue desarrollado por J.J. Garcia-Luna-Aceves en la Universidad de California.

📏 Conceptos clave de DUAL

TérminoDefiniciónAnalogía
Reported Distance (RD) Métrica que un vecino anuncia para llegar a una red destino. También llamada Advertised Distance (AD). Lo que el vecino dice que le cuesta llegar al destino
Feasible Distance (FD) Métrica total desde el router local hasta el destino, pasando por ese vecino. FD = RD del vecino + costo del enlace al vecino. Lo que me cuesta a MÍ llegar al destino por ese camino
Successor (S) Vecino con la menor FD hacia un destino. Es la ruta principal instalada en la tabla de enrutamiento. El camino más corto actual
Feasible Successor (FS) Vecino de respaldo que cumple la Feasibility Condition. Se mantiene en la tabla de topología listo para usar. El camino alternativo ya listo
Feasibility Condition (FC) Un vecino es FS si su RD < FD del Successor actual. Garantiza que la ruta alternativa no cause bucles. El vecino promete que su camino es más corto que el mío total

🗺️ Ejemplo DUAL — Successor y Feasible Successor

TOPOLOGÍA — CÁLCULO DUAL   FD=1000 ← costo R1→R2 R1 ──────────────► R2 ──────────► Red 10.0.0.0 │ RD=500 ↑ │ FD=1500 └──────────────► R3 ──────────► Red 10.0.0.0 RD=300 ↑   Desde R1 hacia 10.0.0.0: ┌─────────┬──────────┬──────────┬─────────────────────────┐ │ Vecino │ RD │ FD │ Rol │ ├─────────┼──────────┼──────────┼─────────────────────────┤ │ R2 │ 500 │ 1000 │ SUCCESSOR (menor FD) │ │ R3 │ 300 │ 1500 │ FEASIBLE SUCCESSOR │ │ │ │ │ RD(300) < FD_S(1000) ✅ │ └─────────┴──────────┴──────────┴─────────────────────────┘   Si R2 falla → R3 se instala INSTANTÁNEAMENTE como nueva ruta Sin consultar vecinos, sin recalcular — convergencia = 0ms
⚠️
Sin Feasible Successor: Si ningún vecino cumple la FC, EIGRP pone la ruta en estado Active y envía mensajes QUERY a todos sus vecinos preguntando por rutas alternativas. Esto se llama diffusing computation y puede tardar varios segundos — evitarlo es crítico en producción.
🎯
CCNA Examen — Truco mnemónico: Para que un vecino sea Feasible Successor, su RD debe ser MENOR que la FD actual del Successor. Si RD ≥ FD del Successor → NO es FS → no garantiza ausencia de bucles → DUAL lo rechaza como backup.

🔄 Estados de ruta EIGRP

EstadoDescripciónAcción DUAL
Passive (P)Estado normal — ruta estable, Successor conocidoNada — la ruta funciona correctamente
Active (A)Ruta en recálculo — Successor perdido, sin FS disponibleEnvía QUERYs a vecinos, espera RESPLYs

🤝 Establecimiento de Vecindad EIGRP

EIGRP usa mensajes Hello para descubrir y mantener vecinos. Los Hellos se envían por multicast 224.0.0.10 periódicamente. Cuando dos routers se descubren mutuamente, intercambian sus tablas de topología completas mediante mensajes Update y forman una adyacencia.
ParámetroEnlace rápido (≥1Gbps / T1)Enlace lento (<1.544 Mbps)
Hello Interval5 segundos60 segundos
Hold Time15 segundos (3× Hello)180 segundos (3× Hello)
🎯
CCNA Examen: Los vecinos EIGRP se pueden establecer aunque los Hello/Hold timers sean diferentes entre sí (a diferencia de OSPF). Sin embargo, para formar adyacencia deben coincidir: AS number, K-values y autenticación.

📋 Requisitos para formar vecindad EIGRP

Requisito¿Debe coincidir?Detalle

K-values (métricas)SÍ — obligatorioLos pesos de la fórmula de métrica deben ser idénticos
AutenticaciónSÍ — si está configuradaMisma contraseña MD5 o SHA-256
Subnet de la interfazSÍ — deben estar en la misma redLas IPs de las interfaces deben pertenecer a la misma subred
Hello/Hold TimersNO — pueden ser distintosLa adyacencia funciona aunque sean diferentes
Passive InterfaceBloquea vecindadUna interfaz pasiva no envía ni recibe Hellos

📨 Tipos de paquetes EIGRP

PaqueteTipoPropósitoEnvío
Hello1Descubrir y mantener vecinos. Heartbeat periódico.Multicast 224.0.0.10
Update4Enviar información de rutas. Solo cuando cambia algo.Multicast / Unicast
Query3Preguntar a vecinos por rutas alternativas cuando una ruta queda Active.Multicast / Unicast
Reply4Responder a un Query con información de ruta o indicando que no se conoce.Unicast
Acknowledgment5Confirmar recepción de paquetes confiables (Update, Query, Reply).Unicast
💡
EIGRP usa su propio mecanismo de entrega confiable (RTP — Reliable Transport Protocol). Los paquetes Update, Query y Reply son confiables (requieren ACK). Los Hello y ACK son no confiables (no requieren confirmación).

📐 La Métrica Compuesta de EIGRP

EIGRP usa una métrica compuesta calculada con hasta 5 factores denominados K-values (K1–K5). Por defecto solo se usan K1 (Bandwidth) y K3 (Delay). Los demás (K2=Load, K4=Reliability, K5=MTU) están desactivados por defecto.

🔢 Fórmula de Métrica EIGRP

FÓRMULA COMPLETA   Métrica = 256 × [ (K1 × BW) + (K2 × BW / (256 – Load)) + (K3 × Delay) ] × [ K5 / (Reliability + K4) ]   Donde: BW = 10^7 / Bandwidth_mínimo_en_kbps (peor enlace en el path) Delay = Suma de delays en decenas de microsegundos   ── Con valores por defecto (K1=1, K2=0, K3=1, K4=0, K5=0): ──   Métrica = 256 × (BW_min + Delay_total)   Ejemplo — path con FastEthernet (100Mbps, delay 100µs): BW = 10^7 / 100000 = 100 Delay = 100 / 10 = 10 Métrica = 256 × (100 + 10) = 28160
⚠️
Importante: Para BW, EIGRP usa el mínimo del camino (el enlace más lento — bottleneck). Para Delay, suma todos los delays del camino. Este es un error clásico en el examen: BW = mínimo, Delay = suma.

⚖️ Balanceo de Carga — Equal y Unequal Cost

EIGRP soporta dos tipos de balanceo: Equal-Cost Load Balancing (igual que OSPF, hasta 4 rutas por defecto) y la característica exclusiva Unequal-Cost Load Balancing mediante el comando variance — única en EIGRP entre todos los protocolos IGP.
VARIANCE — BALANCEO DESIGUAL   FD del Successor: 10000 variance 2 → incluye rutas con FD ≤ (10000 × 2) = 20000   Rutas disponibles: R2 → FD = 10000 → Successor ✅ incluida R3 → FD = 15000 → Feasible Succ ✅ 15000 < 20000 → incluida R4 → FD = 22000 → No es FS ❌ 22000 > 20000 → excluida   ! Solo los Feasible Successors pueden participar en variance ! El tráfico se distribuye proporcional al inverso de la FD
🎯
CCNA Examen: variance 1 (defecto) = solo equal-cost LB. variance 2 = incluye rutas hasta 2× la FD del Successor. Solo Feasible Successors pueden entrar al balanceo desigual — nunca rutas en estado Active.

⌨️ Configuración básica EIGRP Clásico

EIGRP tiene dos modos de configuración: el modo clásico (más antiguo, compatible con todos los IOS) y el modo Named EIGRP (IOS 15+, más organizado y con soporte IPv4/IPv6 unificado). El examen CCNA puede evaluar ambos.
EIGRP CLÁSICO — CONFIGURACIÓN COMPLETA R1(config)# router eigrp 100 ! 100 = número de AS (Autonomous System) — debe coincidir en todos los routers   R1(config-router)# network 192.168.1.0 ! Activa EIGRP en interfaces que coincidan con esa red (classful por defecto)   R1(config-router)# network 10.0.0.0 0.0.0.255 ! Con wildcard mask — más preciso, activa EIGRP solo en 10.0.0.x   R1(config-router)# no auto-summary ! Desactivar sumarización automática — OBLIGATORIO en redes discontiguas ! En IOS 15+ está desactivado por defecto   R1(config-router)# eigrp router-id 1.1.1.1 ! Router-ID: IP más alta de loopback, o IP más alta de interfaz activa ! Recomendado configurarlo manualmente para evitar cambios inesperados   R1(config-router)# passive-interface gigabitEthernet 0/0 ! No envía Hellos por Gi0/0 — anuncia la red pero no busca vecinos ahí ! Usar en interfaces LAN donde no hay otros routers EIGRP   R1(config-router)# passive-interface default R1(config-router)# no passive-interface serial 0/0/0 ! Alternativa: todas pasivas por defecto, activar solo en las WAN

📝 Named EIGRP (IOS 15+)

EIGRP NAMED MODE R1(config)# router eigrp EMPRESA ! Nombre descriptivo en lugar de número   R1(config-router)# address-family ipv4 unicast autonomous-system 100 R1(config-router-af)# network 192.168.1.0 R1(config-router-af)# network 10.0.0.0 0.0.0.255 R1(config-router-af)# eigrp router-id 1.1.1.1   R1(config-router-af)# af-interface gigabitEthernet 0/0 R1(config-router-af-interface)# passive-interface R1(config-router-af-interface)# exit

🔍 Verificación y Troubleshooting EIGRP

El diagnóstico de EIGRP sigue un orden lógico: primero verificar vecinos, luego topología y finalmente la tabla de rutas. Si no hay vecinos, no hay rutas — ese es el punto de partida siempre.

🛠️ Comandos Show esenciales

VERIFICACIÓN COMPLETA EIGRP ── Vecinos ───────────────────────────────────────────────── R1# show ip eigrp neighbors ! Ver: IP del vecino, interfaz, Uptime, Q Cnt (0=normal), Seq Num   ── Tabla de topología ────────────────────────────────────── R1# show ip eigrp topology ! Ver: FD, RD, Successor, Feasible Successors, estado P/A   R1# show ip eigrp topology all-links ! Muestra TODAS las rutas, incluso las que no cumplen FC   ── Tabla de rutas EIGRP ──────────────────────────────────── R1# show ip route eigrp ! D = EIGRP interna (AD 90) | D EX = EIGRP externa (AD 170)   ── Interfaces activas ────────────────────────────────────── R1# show ip eigrp interfaces ! Interfaces con EIGRP activo, peers, Hello interval   ── Contadores de tráfico ─────────────────────────────────── R1# show ip eigrp traffic ! Hellos, Updates, Queries, Replies enviados/recibidos   ── Debug (usar con cuidado en producción) ────────────────── R1# debug eigrp neighbors R1# debug eigrp packets R1# undebug all
SALIDA TÍPICA — show ip eigrp neighbors EIGRP-IPv4 Neighbors for AS(100) H Address Interface Hold Uptime SRTT RTO Q Seq (sec) (ms) Cnt Num 0 10.0.12.2 Se0/0/0 12 01:23:45 28 200 0 47 1 10.0.13.3 Gi0/1 14 00:45:12 5 100 0 23   Q Cnt = 0 → Normal. Si Q Cnt > 0 por mucho tiempo → problema de congestión

🚨 Problemas comunes y soluciones

SíntomaCausaVerificación / Solución
No se forman vecinos AS number diferente, K-values distintos, subnet diferente, autenticación show ip eigrp neighbors — sin output. Verificar AS y K-values con show ip protocols
Vecinos fluctúan (flapping) Hello/Hold timers muy ajustados, problema de conectividad en el enlace debug eigrp neighbors — ajustar timers o revisar la capa física
Ruta faltante en tabla auto-summary activado con redes discontiguas, passive-interface mal show ip eigrp topology — verificar si existe en topología pero no en rutas
Ruta en estado Active Successor perdido sin Feasible Successor disponible show ip eigrp topology active — revisar si hay Query loops o SIA
SIA (Stuck in Active) Vecino no responde el Query en 3 minutos EIGRP resetea la adyacencia. Verificar conectividad con el vecino que no respondió
Métrica subóptima Bandwidth de interfaz mal configurado show interface serial 0/0/0 — ajustar con bandwidth (solo afecta métricas, no BW real)
💡
Truco CCNA: El comando bandwidth en una interfaz NO cambia el ancho de banda real del enlace — solo cambia el valor que EIGRP (y OSPF) usa para calcular la métrica. Siempre verifica que el bandwidth configurado refleje la velocidad real del enlace.
