Comandos para DHCP

Domina la configuración, administración y troubleshooting de DHCP en Cisco IOS. Desde la asignación dinámica de direcciones IP hasta el relay entre VLANs, pools, exclusiones, opciones avanzadas y seguridad. Esencial para el examen CCNA 200-301 y redes empresariales.

📋 IOS 15.x / IOS XE 🎯 CCNA 200-301 Exam Topic 5.2, 5.3 ⏱ ~90 min lectura 🧪 Lab: 1 Router + 2 Switches + 4 PCs

Fundamentos de DHCP en Cisco IOS

DHCP (Dynamic Host Configuration Protocol) es un protocolo de capa de aplicación (UDP 67/68) que automatiza la asignación de direcciones IP y parámetros de red a los dispositivos. En Cisco IOS, un router puede actuar como servidor DHCP, cliente DHCP, o agente de relay.

Proceso DORA — Descubrimiento, Oferta, Solicitud, Reconocimiento

Roles DHCP en Cisco IOS

🖥️

Servidor DHCP

El router/switch Cisco mantiene pools de direcciones IP y responde solicitudes DHCP de clientes en redes locales conectadas directamente o vía relay.

ip dhcp pool NOMBRE
💻

Cliente DHCP

La interfaz del router/switch solicita una IP automáticamente desde otro servidor DHCP. Útil para interfaces WAN o conexiones a ISP.

ip address dhcp
🌐

DHCP Relay Agent

Reenvía solicitudes DHCP broadcast desde una subred hacia un servidor DHCP ubicado en otra subred usando unicast dirigido.

ip helper-address IP
📡

DHCP Snooping

Seguridad de capa 2 que filtra mensajes DHCP no confiables y construye una tabla de bindings IP-MAC-VLAN-Port.

ip dhcp snooping
💡 Concepto Clave: Lease Time y Renovación

El lease time es el tiempo que un cliente puede usar una IP. En Cisco, el valor por defecto es 1 día (86400 segundos). El cliente intenta renovar al 50% (T1) del lease. Si no recibe respuesta, al 87.5% (T2) intenta rebind (broadcast a cualquier servidor). Puedes configurarlo con lease {días} [horas] [minutos].

Configuración del Servidor DHCP en Cisco IOS

Un router Cisco puede funcionar como servidor DHCP completo, gestionando múltiples pools para diferentes subredes. Es fundamental para redes pequeñas/medias y para el laboratorio CCNA.

Comandos Esenciales del Servidor DHCP

Configuración completa de un pool DHCP
! === PASO 1: EXCLUIR DIRECCIONES ESTÁTICAS ===
! Siempre excluye primero las IPs que no deben asignarse
R1(config)# ip dhcp excluded-address 192.168.10.1
! Gateway de la red

R1(config)# ip dhcp excluded-address 192.168.10.2 192.168.10.10
! Rango completo: del .2 al .10 (servidores, impresoras, APs)

! === PASO 2: CREAR EL POOL DHCP ===
R1(config)# ip dhcp pool VLAN10-VENTAS
! Entra al modo de configuración DHCP (config-dhcp)#
! El nombre del pool es local al router, descriptivo

R1(dhcp-config)# network 192.168.10.0 255.255.255.0
! Define la red y máscara del pool
! El router asignará IPs del .11 al .254 (por exclusiones)

R1(dhcp-config)# default-router 192.168.10.1
! Gateway por defecto (puedes especificar hasta 8)
! default-router 192.168.10.1 192.168.10.2

R1(dhcp-config)# domain-name empresa.local
! Nombre de dominio DNS para los clientes

R1(dhcp-config)# dns-server 8.8.8.8 8.8.4.4
! Servidores DNS primario y secundario

R1(dhcp-config)# lease 2 12 30
! 2 días, 12 horas, 30 minutos
! lease infinite = sin expiración (NO recomendado)
! lease 0 8 0 = 8 horas

R1(dhcp-config)# netbios-name-server 192.168.10.5
! Servidor WINS (legacy, rara vez usado hoy)

R1(dhcp-config)# exit

! === CONFIGURAR SEGUNDO POOL (VLAN 20 - IT) ===
R1(config)# ip dhcp excluded-address 192.168.20.1 192.168.20.10
R1(config)# ip dhcp pool VLAN20-IT
R1(dhcp-config)# network 192.168.20.0 255.255.255.0
R1(dhcp-config)# default-router 192.168.20.1
R1(dhcp-config)# domain-name empresa.local
R1(dhcp-config)# dns-server 8.8.8.8
R1(dhcp-config)# lease 1 0 0
R1(dhcp-config)# exit

Pool para múltiples subredes (Router-on-a-Stick)

Servidor DHCP sirviendo múltiples VLANs
! === TOPOLOGÍA: R1 con subinterfaces para VLAN 10 y 20 ===

R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# encapsulation dot1Q 10
R1(config-subif)# ip address 192.168.10.1 255.255.255.0
R1(config-subif)# exit

R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# encapsulation dot1Q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0
R1(config-subif)# exit

! === POOL PARA VLAN 10 ===
R1(config)# ip dhcp excluded-address 192.168.10.1
R1(config)# ip dhcp pool POOL-VLAN10
R1(dhcp-config)# network 192.168.10.0 255.255.255.0
R1(dhcp-config)# default-router 192.168.10.1
R1(dhcp-config)# domain-name ventas.empresa.local
R1(dhcp-config)# dns-server 8.8.8.8
R1(dhcp-config)# exit

! === POOL PARA VLAN 20 ===
R1(config)# ip dhcp excluded-address 192.168.20.1
R1(config)# ip dhcp pool POOL-VLAN20
R1(dhcp-config)# network 192.168.20.0 255.255.255.0
R1(dhcp-config)# default-router 192.168.20.1
R1(dhcp-config)# domain-name it.empresa.local
R1(dhcp-config)# dns-server 8.8.8.8 1.1.1.1
R1(dhcp-config)# lease 2
R1(dhcp-config)# exit

! El router automáticamente sabe qué pool usar según la interfaz
! que recibe la solicitud (basado en la red conectada)
⚠️ Error Común en el Examen

Situación: Configuras un pool DHCP pero los clientes no reciben IP.

Causa probable: Olvidaste excluir la IP del gateway (ip dhcp excluded-address) y el servidor DHCP asignó la IP del router a un cliente, causando conflicto de IP.

Solución: Siempre excluye la IP de la interfaz del router (gateway) y cualquier otra IP estática antes de crear el pool.

Configuración de Cliente DHCP en Cisco IOS

Un router Cisco puede comportarse como cliente DHCP, solicitando una dirección IP dinámica para una de sus interfaces. Esto es común en conexiones WAN hacia un ISP o en escenarios de laboratorio donde un router obtiene IP de otro router.

Obtener IP automáticamente en una interfaz

Interfaz como cliente DHCP
! === INTERFAZ COMO CLIENTE DHCP ===
R2(config)# interface GigabitEthernet0/1
R2(config-if)# description WAN-ISP-DHCP
R2(config-if)# ip address dhcp
! La interfaz enviará DHCP Discover y obtendrá IP automáticamente

R2(config-if)# no shutdown
R2(config-if)# exit

! === VERIFICAR IP OBTENIDA ===
R2# show ip interface brief

Liberar y Renovar IP DHCP

Comandos de control del cliente
! === LIBERAR IP (RELEASE) ===
R2# release dhcp GigabitEthernet0/1
! Envía DHCP Release al servidor y libera la IP

! === RENOVAR IP (RENEW) ===
R2# renew dhcp GigabitEthernet0/1
! Fuerza nuevo ciclo DORA para obtener IP (o renovar lease)

! === VER TODOS LOS BINDINGS DHCP (desde cliente) ===
R2# show dhcp lease

! === VER SERVIDORES DHCP CONOCIDOS ===
R2# show dhcp server
! Muestra estadísticas de servidores DHCP contactados

Cliente DHCP con hostname específico

Opciones de identificación del cliente
! === CONFIGURAR CLIENT-ID PERSONALIZADO ===
R2(config)# interface GigabitEthernet0/1
R2(config-if)# ip address dhcp client-id GigabitEthernet0/1
! Usa la MAC de la interfaz como client identifier

! === CONFIGURAR HOSTNAME QUE ENVÍA EL CLIENTE ===
R2(config)# hostname R2-BRANCH-01
! El hostname se envía en la opción 12 del DHCP
! Algunos ISPs lo usan para identificar al cliente

! === DESACTIVAR CLIENTE DHCP EN INTERFAZ ===
R2(config)# interface GigabitEthernet0/1
R2(config-if)# no ip address
R2(config-if)# ip address 192.168.1.1 255.255.255.0
! Al asignar IP estática, se pierde el lease DHCP
💡 Dato de Examen: Método DHCP

En show ip interface brief, la columna Method indica cómo se obtuvo la IP:

  • DHCP → Obtenida dinámicamente vía DHCP
  • manual → Configurada estáticamente con ip address
  • NVRAM → Cargada desde startup-config
  • TFTP → Cargada desde servidor TFTP (configuración remota)
  • unset → Sin IP configurada

DHCP Relay — ip helper-address

Los mensajes DHCP Discover y Request se envían como broadcasts (255.255.255.255). Los routers NO reenvían broadcasts entre subredes por defecto. ip helper-address resuelve esto convirtiendo el broadcast en unicast dirigido al servidor DHCP.

Funcionamiento del DHCP Relay Agent

Configuración de ip helper-address

Reenvío de broadcasts DHCP hacia servidor remoto
! === ESCENARIO ===
! Servidor DHCP en VLAN 99 (192.168.99.5)
! Clientes en VLAN 10 (192.168.10.0/24)
! Router R1 hace inter-VLAN routing

! === CONFIGURAR SUBINTERFAZ VLAN 10 ===
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# encapsulation dot1Q 10
R1(config-subif)# ip address 192.168.10.1 255.255.255.0
R1(config-subif)# ip helper-address 192.168.99.5
! ⬆️ COMANDO CLAVE: Reenvía broadcasts UDP hacia 192.168.99.5
! Incluye DHCP, BOOTP, TFTP, DNS, NetBIOS, etc.
R1(config-subif)# no shutdown
R1(config-subif)# exit

! === CONFIGURAR SUBINTERFAZ VLAN 20 (OTRA VLAN CON CLIENTES) ===
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# encapsulation dot1Q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0
R1(config-subif)# ip helper-address 192.168.99.5
R1(config-subif)# no shutdown

! === CONFIGURAR INTERFAZ HACIA SERVIDOR (VLAN 99) ===
R1(config)# interface GigabitEthernet0/0.99
R1(config-subif)# encapsulation dot1Q 99
R1(config-subif)# ip address 192.168.99.1 255.255.255.0
R1(config-subif)# no shutdown

! === SERVIDOR DHCP (en 192.168.99.5 o router con pool) ===
! El servidor debe tener pools para 192.168.10.0 y 192.168.20.0

Puertos UDP reenviados por ip helper-address

Servicios que se reenvían automáticamente
! === PUERTOS UDP REENVIADOS POR DEFECTO ===
! Cuando configuras ip helper-address, el router reenvía:

! Puerto  Protocolo/Servicio
! ------  ------------------
!  37     Time
!  49     TACACS
!  53     DNS
!  67     DHCP/BOOTP (Server)
!  68     DHCP/BOOTP (Client)
!  69     TFTP
!  137    NetBIOS Name Service
!  138    NetBIOS Datagram Service
!  137    NetBIOS (otros)

! === CONTROLAR QUÉ PUERTOS SE REENVÍAN ===
! Si solo quieres reenviar DHCP (67 y 68), bloquea los demás:

R1(config)# no ip forward-protocol udp 37
R1(config)# no ip forward-protocol udp 49
R1(config)# no ip forward-protocol udp 53
R1(config)# no ip forward-protocol udp 69
R1(config)# no ip forward-protocol udp 137
R1(config)# no ip forward-protocol udp 138

! Ahora solo se reenviarán DHCP (67/68) hacia el helper-address
! Esto mejora la seguridad y reduce tráfico innecesario

! === AGREGAR PUERTOS PERSONALIZADOS ===
R1(config)# ip forward-protocol udp 3000
! Reenvía broadcasts UDP del puerto 3000 también

Múltiples helper-address ( redundancia )

Alta disponibilidad DHCP
! === DOS SERVIDORES DHCP PARA REDUNDANCIA ===
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# ip helper-address 192.168.99.5
R1(config-subif)# ip helper-address 192.168.99.6
! El router reenvía el broadcast a AMBOS servidores
! El cliente acepta la primera oferta que reciba
! ⚠️ Asegúrate de que ambos pools NO se solapen
⚠️ Error Crítico: Olvidar ip helper-address

En el examen CCNA, el escenario más común es: "Los clientes en VLAN 10 obtienen IP pero los de VLAN 20 no". La causa casi siempre es que se olvidó configurar ip helper-address en la subinterfaz/VLAN 20 del router, o el servidor DHCP no tiene pool para la red 192.168.20.0.

Regla mnemotécnica: Si el servidor DHCP NO está en la misma subred que los clientes, SIEMPRE se necesita ip helper-address en la interfaz gateway de los clientes.

Opciones Avanzadas del Servidor DHCP

Cisco IOS soporta múltiples opciones DHCP estándar (RFC 2132) y propietarias. Estas opciones permiten personalizar lo que el cliente recibe más allá de IP, máscara y gateway.

Opciones DHCP Comunes en Cisco IOS

Configuración de opciones personalizadas
! === OPCIÓN 6: SERVIDORES DNS ===
R1(dhcp-config)# dns-server 8.8.8.8 8.8.4.4 1.1.1.1
! Hasta 8 servidores DNS pueden especificarse

! === OPCIÓN 15: NOMBRE DE DOMINIO ===
R1(dhcp-config)# domain-name ciscoacademy.local

! === OPCIÓN 3: GATEWAY (DEFAULT-ROUTER) ===
R1(dhcp-config)# default-router 192.168.10.1 192.168.10.2
! Hasta 8 gateways (el primario y backups)

! === OPCIÓN 51: TIEMPO DE CONCESIÓN ===
R1(dhcp-config)# lease 3 12 30
! 3 días, 12 horas, 30 minutos
R1(dhcp-config)# lease infinite
! Nunca expira (NO recomendado en producción)

! === OPCIÓN 66/150: SERVIDOR TFTP (VoIP) ===
! Crítico para despliegue de teléfonos IP Cisco
R1(dhcp-config)# option 66 ascii http://192.168.99.10/phones/
R1(dhcp-config)# option 150 ip 192.168.99.10
! Opción 150: Lista de servidores TFTP para Cisco IP Phones
! Opción 66: URL del servidor de provisionamiento

! === OPCIÓN 43: VENDOR SPECIFIC (APs WiFi) ===
R1(dhcp-config)# option 43 hex f108c0a80a0a
! Usado para APs Cisco LWAP para encontrar WLC
! El valor hex codifica la IP del controlador

! === OPCIÓN 44: SERVIDORES WINS ===
R1(dhcp-config)# netbios-name-server 192.168.10.5
! Legacy, usado en redes Windows antiguas

! === OPCIÓN 46: TIPO DE NODO NETBIOS ===
R1(dhcp-config)# netbios-node-type h-node
! h-node = hybrid (WINS primero, broadcast después)

Opciones personalizadas con código numérico

option [código] [tipo] [valor]
! === SINTAXIS GENÉRICA PARA CUALQUIER OPCIÓN ===

! Opción 33: Static Routes (rutas estáticas por DHCP)

R1(dhcp-config)# option 42 ip 192.168.99.2
! Opción 42: NTP Servers (servidores de tiempo)

R1(dhcp-config)# option 28 ip 192.168.10.255

Configuración de clase y asignación por cliente

DHCP Class y Client-Identifier
! === CLASES DHCP (Asignación condicional) ===
! Permite asignar opciones diferentes según el cliente

R1(config)# ip dhcp class VOIP-PHONES
R1(dhcp-class)# relay agent information
R1(dhcp-class)# relay-information hex 0106000400010101
R1(dhcp-class)# exit

R1(config)# ip dhcp pool POOL-VOIP
R1(dhcp-config)# network 192.168.110.0 255.255.255.0
R1(dhcp-config)# class VOIP-PHONES
R1(dhcp-pool-class)# address range 192.168.110.10 192.168.110.100
R1(dhcp-pool-class)# exit

! === RESERVAR IP ESPECÍFICA POR MAC ===
! Cisco IOS NO soporta reservas estáticas directas en el pool
! como Windows Server, pero puedes usar exclusiones + IP estática
! o usar la opción client-identifier:

R1(config)# ip dhcp pool RESERVA-GERENTE
R1(dhcp-config)# host 192.168.10.50 255.255.255.0
R1(dhcp-config)# client-identifier 0100.1234.5678.ab
! Reserva 192.168.10.50 para MAC específica
R1(dhcp-config)# default-router 192.168.10.1
R1(dhcp-config)# exit

La verificación es esencial para confirmar que el servidor está operativo, que las IPs se asignan correctamente y que no hay conflictos en la red.

show ip dhcp pool
📋

show ip dhcp binding

Tabla de leases activos: IP asignada, MAC del cliente, lease expiration, tipo.

show ip dhcp binding
📈

Contadores de mensajes DHCP: discovers, offers, requests, acks, naks, declines.

show ip dhcp server statistics
🔍

Direcciones en conflicto detectadas por ping antes de ofrecerlas (si está habilitado).

show ip dhcp conflict

show dhcp lease

show ip dhcp conflict — Detección de Conflictos

IPs en conflicto detectadas
! === DETECCIÓN AUTOMÁTICA DE CONFLICTOS ===
! Por defecto, Cisco hace ping a una IP antes de ofrecerla
! Si hay respuesta, marca la IP como conflictiva

Troubleshooting DHCP — Guía de Diagnóstico

Los problemas de DHCP son frecuentes en redes conmutadas y enrutadas. Esta guía sistemática cubre los escenarios más comunes del examen CCNA y producción.

Problema 1: Cliente no obtiene IP (misma subred)

1
Verificar servicio DHCP activo
R1# show ip dhcp pool
Confirmar que el pool existe, tiene IPs disponibles y no está agotado.
2
Verificar exclusiones
R1# show run | section ip dhcp excluded-address
Asegurar que el gateway NO esté dentro del rango asignable.
3
Verificar interfaz del router
R1# show ip interface brief
La interfaz conectada a los clientes debe estar up/up y tener IP en la misma red del pool.
4
Verificar estadísticas
R1# show ip dhcp server statistics
¿Se reciben DHCPDISCOVER? Si no, el problema es físico/L2 o el cliente no envía solicitudes.

Problema 2: Cliente en VLAN remota no obtiene IP

Diagnóstico paso a paso
! === SÍNTOMA: PCs en VLAN 20 no obtienen IP ===
! Servidor DHCP en VLAN 10, clientes en VLAN 20

! Paso 1: Verificar ip helper-address en gateway VLAN 20
R1# show running-config interface GigabitEthernet0/0.20
! ¿Aparece "ip helper-address [ip-servidor]"?

! Paso 2: Si NO está, configurarlo:
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# ip helper-address 192.168.10.1
! IP del servidor DHCP (o del router con pool)

! Paso 3: Verificar que el servidor tiene pool para VLAN 20
R1# show ip dhcp pool
! Debe existir un pool con "network 192.168.20.0 255.255.255.0"

! Paso 4: Verificar conectividad entre VLANs
R1# ping 192.168.20.1 source 192.168.10.1
! El servidor debe poder alcanzar el gateway de VLAN 20

! Paso 5: Verificar trunks entre switches
SW1# show interfaces trunk
! ¿VLAN 20 está permitida en el trunk hacia el router?

Problema 3: IP asignada pero sin conectividad

Diagnóstico de parámetros incorrectos
! === SÍNTOMA: Cliente tiene IP pero no navega ===

! Paso 1: Verificar gateway en el cliente
PC> ipconfig
IP Address......................: 192.168.10.11
Subnet Mask.....................: 255.255.255.0
Default Gateway.................: 192.168.10.1

! Paso 2: Verificar que default-router en pool sea correcto
R1# show ip dhcp pool POOL-VLAN10
! ¿El default-router coincide con la IP de la interfaz del router?

! Paso 3: Verificar DNS
PC> ping 8.8.8.8
PC> ping google.com
! ¿Funciona por IP pero no por nombre? → Problema de DNS
R1# show run | section dns-server
! Verificar que dns-server esté configurado en el pool

! Paso 4: Verificar ACLs que bloqueen DHCP
R1# show ip interface GigabitEthernet0/0.10
! ¿Hay una ACL inbound que bloquee UDP 67/68?

Problema 4: Pool agotado (Agotamiento de IPs)

Cuando no hay más direcciones disponibles
! === DIAGNÓSTICO ===
R1# show ip dhcp pool
Pool POOL-VLAN10 :
 Utilization mark (high/low)    : 100 / 0
 Leased addresses               : 251
 Excluded addresses             : 3
 Total addresses                : 254
! 251 + 3 = 254 → POOL AGOTADO

! === SOLUCIONES ===
! 1. Reducir lease time para liberar IPs más rápido:
R1(dhcp-config)# lease 0 8 0
! 8 horas en lugar de 1 día

! 2. Ampliar la subred (cambiar máscara):
! De /24 (254 hosts) a /23 (510 hosts)
! ⚠️ Requiere cambiar máscara en TODOS los dispositivos

! 3. Crear segundo pool (si hay subredes disponibles):
! No aplicable si solo tienes una subred

! 4. Liberar leases manualmente (emergencia):
R1# clear ip dhcp binding 192.168.10.50
! Libera una IP específica (el cliente la perderá)
R1# clear ip dhcp binding *
! ⚠️ LIBERA TODAS LAS IPs — MUY PELIGROSO EN PRODUCCIÓN
💡 Checklist de Troubleshooting DHCP
  1. ¿El servicio DHCP está activo? (service dhcp)
  2. ¿El pool existe y tiene IPs disponibles? (show ip dhcp pool)
  3. ¿La interfaz del router está up/up? (show ip int brief)
  4. ¿El gateway está excluido del pool? (show run | section excluded)
  5. ¿Hay ip helper-address si el servidor está en otra subred?
  6. ¿El servidor tiene pool para esa subred remota?
  7. ¿Los trunks permiten la VLAN del cliente? (show int trunk)
  8. ¿Hay ACLs bloqueando UDP 67/68?
  9. ¿El cliente tiene IP duplicada? (show ip dhcp conflict)
  10. ¿El lease expiró y no hay IPs disponibles?

Seguridad en Implementaciones DHCP

DHCP es inherentemente inseguro si no se protege. Ataques como Rogue DHCP, DHCP Starvation y Man-in-the-Middle son comunes. Cisco proporciona mecanismos integrados para mitigar estos riesgos.

Ataques DHCP Comunes

⚠️ ATAQUES CONTRA DHCP
1 Rogue DHCP Server: Atacante conecta un servidor DHCP falso que asigna IPs maliciosas (gateway falso, DNS envenenado). Los clientes redirigen tráfico hacia el atacante.
2 DHCP Starvation: El atacante envía miles de DHCP Discover con MACs falsas (MAC flooding) agotando todas las IPs del pool. Denegación de servicio legítimo.
3 DHCP Spoofing / MITM: Combinación de Rogue DHCP + ARP spoofing. El atacante se convierte en gateway man-in-the-middle interceptando todo el tráfico.

DHCP Snooping en Switches Cisco

Protección contra Rogue DHCP
! === HABILITAR DHCP SNOOPING GLOBALMENTE ===
SW1(config)# ip dhcp snooping
! Activa la función en el switch

! === APLICAR A VLANs ESPECÍFICAS ===
SW1(config)# ip dhcp snooping vlan 10,20,30
! Solo monitorea DHCP en estas VLANs

! === CONFIGURAR PUERTO DE CONFIANZA (TRUSTED) ===
! Puerto hacia el servidor DHCP legítimo o router:
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# ip dhcp snooping trust
! Solo los puertos trusted pueden enviar DHCPOFFER y DHCPACK
! Puertos untrusted (default) solo pueden enviar DISCOVER/REQUEST

! === CONFIGURAR LÍMITE DE TASA (RATE LIMIT) ===
SW1(config)# interface range FastEthernet0/1 - 24
SW1(config-if-range)# ip dhcp snooping limit rate 10
! Máximo 10 paquetes DHCP por segundo por puerto
! Evita ataques de DHCP starvation

! === VERIFICAR DHCP SNOOPING ===
SW1# show ip dhcp snooping

IP Source Guard — Protección de Origen

Filtrado basado en bindings DHCP
! === HABILITAR IP SOURCE GUARD ===
! Requiere DHCP Snooping previamente configurado

SW1(config)# interface range FastEthernet0/1 - 24
SW1(config-if-range)# ip verify source port-security
! Filtra tráfico basado en IP+MAC aprendidas por DHCP
! Solo permite tráfico con la IP asignada por DHCP

! Alternativa solo por IP (menos estricta):
SW1(config-if-range)# ip verify source
! Verifica solo la dirección IP fuente

! === VERIFICAR IP SOURCE GUARD ===
SW1# show ip verify source

---------        -----------   -----------   ----------       -----------        ----

⚠️ Orden de Implementación de Seguridad L2/L3

Para una protección completa contra ataques DHCP y ARP, implementa en este orden:

  1. Port Security → Limita MACs por puerto
  2. DHCP Snooping → Valida mensajes DHCP y crea tabla de bindings
  3. Dynamic ARP Inspection → Valida ARP contra bindings de DHCP snooping
  4. IP Source Guard → Filtra tráfico IP basado en bindings

Sin DHCP Snooping, DAI e IP Source Guard no tienen tabla de referencia para validar.
