<div class="content-block">
  <h2 class="section-title">Ansible como herramienta de red team</h2>
  <p>La mayoría de los ingenieros asocian Ansible con <em>configuración</em>: desplegar VLANs, crear ACLs, modificar rutas. Pero su verdadero superpoder en operaciones de red es el <strong>troubleshooting y la auditoría</strong>. Un playbook de Ansible puede conectarse a 500 switches simultáneamente, verificar que todos cumplen una política de seguridad, detectar interfaces caídas, validar versiones de IOS y generar un reporte ejecutivo en minutos.</p>
  <p>Esto transforma el modelo de trabajo de <strong>reactivo</strong> (esperar que un usuario reporte un problema) a <strong>proactivo</strong> (detectar desviaciones antes de que causen incidentes). En el contexto del examen CCNA Domain 6.6, entender este caso de uso demuestra madurez en la gestión de infraestructura programable.</p>

  <div class="info-box tip">
    <div class="info-box-title">💡 Concepto clave del examen CCNA</div>
    <p>El dominio 6.6 evalúa el uso de herramientas de automatización para gestión de configuración <strong>y</strong> monitoreo. La pregunta puede ser: <em>"¿Qué módulo de Ansible permite verificar el estado de interfaces en múltiples switches simultáneamente?"</em> La respuesta es <strong>ios_facts</strong> (para datos estructurados) o <strong>ios_command</strong> (para salida raw), combinado con <strong>assert</strong> para validación.</p>
  </div>

  <h2 class="section-title">Casos de uso de troubleshooting con Ansible</h2>
  <div class="concept-grid">
    <div class="concept-card">
      <div class="concept-icon">🔍</div>
      <h3>Validación Post-Cambio</h3>
      <p>Después de un mantenimiento, verifica automáticamente que BGP neighbors estén up, que las VLANs existan y que los uplinks no hayan caído.</p>
    </div>

  <h2 class="section-title">Estrategia de agregación de datos</h2>
  <p>Un patrón clave para reportes es usar <code>set_fact</code> por host y luego agregar en el <code>post_tasks</code> con <code>delegate_to: localhost</code> y <code>run_once: yes</code>. Esto evita race conditions y genera un único archivo de salida:</p>

  <div class="info-box tip">
    <div class="info-box-title">💡 Patrón de diseño: Gather → Process → Aggregate → Render</div>
    <p><strong>1. Gather:</strong> Cada host recopila sus propios facts.<br>
    <strong>2. Process:</strong> Validaciones y transformaciones por host (set_fact).<br>
    <strong>3. Aggregate:</strong> Tarea única en localhost que lee hostvars de todos.<br>
    <strong>4. Render:</strong> Template Jinja2 itera sobre hostvars para generar el reporte final.</p>
  </div>

  <div class="table-wrap">
    <table class="ref-table">
      <thead>
        <tr><th>Módulo</th><th>Uso en Reportes</th><th>Ejemplo</th></tr>
      </thead>
      <tbody>
        <tr><td><code>template</code></td><td>Renderizar Jinja2 a cualquier formato</td><td>HTML, CSV, XML, LaTeX</td></tr>
        <tr><td><code>copy</code></td><td>Escribir contenido generado con filtros</td><td>JSON con to_nice_json</td></tr>
        <tr><td><code>lineinfile</code></td><td>Agregar líneas a un log centralizado</td><td>Append a auditoria.log</td></tr>
        <tr><td><code>mail</code></td><td>Enviar reporte por email</td><td>Adjuntar CSV al finalizar</td></tr>
        <tr><td><code>uri</code></td><td>POST a API externa</td><td>Enviar JSON a ServiceNow</td></tr>
        <tr><td><code>slack</code></td><td>Notificación a canal</td><td>Alertar si hay fallos</td></tr>
      </tbody>
    </table>

/* ── QUIZ ── */
function ansCheck(btn,isCorrect){
  const parent=btn.closest('.quiz-question');
  parent.querySelectorAll('.quiz-opt').forEach(b=>{
    b.disabled=true;
    b.style.cursor='default';
  });
  if(isCorrect){
    btn.classList.add('correct');
  }else{
    btn.classList.add('wrong');
    parent.querySelectorAll('.quiz-opt').forEach(b=>{
      if(b.onclick.toString().includes('true')) b.classList.add('correct');
    });
  }
  parent.querySelector('.quiz-explanation').style.display='block';
}
