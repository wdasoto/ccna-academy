ARP Address Resolution Protocol

El protocolo que traduce direcciones IP a direcciones MAC. Sin ARP, ningún paquete IP puede entregarse dentro de una red local. Comprender ARP a fondo es comprender la base de toda comunicación en redes Ethernet modernas.

📋 RFC 826 — 1982 🎯 CCNA 200-301 Exam Topic 1.3 ⏱ ~60 min lectura 🧪 Lab incluido 🔒 Seguridad DAI

¿Qué es ARP y por qué existe?

En toda comunicación de red existe un problema fundamental: las aplicaciones y los protocolos de capa 3 trabajan con direcciones IP, pero la red física Ethernet requiere direcciones MAC para entregar los frames de un dispositivo a otro. Nadie conoce de antemano la MAC del destinatario.

ARP (Address Resolution Protocol), definido en la RFC 826 (1982), resuelve exactamente este problema: dada una dirección IP conocida dentro del mismo segmento local, encontrar la dirección MAC que le corresponde. Sin ARP en operación, no existe comunicación de capa 2 dirigida.

💡 Analogía perfecta

Imagina que trabajas en una empresa y necesitas entregar un documento a “Laura Pérez” (la dirección IP). Para entregarlo físicamente debes saber en qué escritorio trabaja (la dirección MAC). Preguntas en voz alta a toda la oficina: “¿Alguien sabe dónde está Laura Pérez?” — eso es un ARP Request (broadcast). Laura responde: “¡Soy yo, escritorio B‑7!” — eso es el ARP Reply (unicast). Memorizas su ubicación y ya no necesitas preguntar de nuevo durante un tiempo.

Posición de ARP en el modelo OSI

ARP es un protocolo único porque opera en la frontera entre capa 2 y capa 3. Técnicamente:

  • Usa frames Ethernet de capa 2 para transportarse
  • Resuelve direcciones de capa 3 (IP)
  • Tiene su propio EtherType: 0x0806
  • No cruza routers — es estrictamente local al dominio de broadcast (VLAN)

Un mensaje ARP tiene exactamente 28 bytes de payload (para IPv4 sobre Ethernet). Cada campo tiene un propósito específico:

La Tabla ARP (ARP Cache)

Cada dispositivo mantiene una tabla ARP en memoria RAM que mapea IPs conocidas con sus MACs correspondientes. La tabla es dinámica y tiene un tiempo de vida (TTL) por cada entrada.

  • Entrada dinámica: aprendida automáticamente. TTL ≈ 240 s en IOS, 60 s en Windows, 20 min en macOS.
  • Entrada estática: configurada manualmente, nunca expira y es inmune a ARP Poisoning.
  • Entrada incompleta: Request enviado, Reply aún no recibido.
⚠️ Punto crítico CCNA

Cuando un router reenvía un paquete de un segmento a otro, reemplaza las MACs de origen y destino en cada salto. Las IPs de origen y destino permanecen iguales de extremo a extremo. En cada nuevo segmento se ejecuta un ARP para resolver la MAC del siguiente salto. Este concepto aparece frecuentemente en el examen.

ARP entre redes distintas — El rol del Default Gateway

Cuando el destino está en una red diferente, el host origen NO hace ARP por la IP destino. En cambio:

  1. Aplicar la máscara y determinar que el destino está fuera de la subred.
  2. Hacer ARP **por la IP del Default Gateway**.
  3. Envelopar el paquete IP con la MAC del router como destino.
  4. El router descapsula, busca la ruta y vuelve a hacer ARP en el siguiente salto.
  5. El proceso se repite hasta llegar al destino final.

El proceso ARP paso a paso

1
Consulta de la caché ARP local

Antes de generar tráfico, el dispositivo consulta su tabla ARP en RAM. Si ya existe una entrada válida, la MAC se usa inmediatamente y no se genera ningún mensaje ARP.

2
ARP Request — Broadcast de capa 2

Si no hay entrada en caché, el host construye un ARP Request y lo envía como broadcast:

  • MAC destino del frame: FF:FF:FF:FF:FF:FF
  • Target MAC en el payload: 00:00:00:00:00:00
  • Target IP = IP que se quiere resolver
  • Sender MAC/IP = datos del host que pregunta

Todos los hosts del segmento reciben este frame porque el switch lo inunda por todos los puertos del mismo dominio de broadcast (VLAN).

3
Procesamiento por cada host

Cada host que recibe el Request:

  • Aprendizaje pasivo: extrae Sender MAC/IP del Request y lo guarda en su caché.
  • Compara la Target IP con su propia IP.
  • Si no coincide descarta el Request.
  • Si coincide prepara y envía un ARP Reply.
4
ARP Reply — Unicast

El host dueño de la IP responde con un ARP Reply directamente al solicitante (unicast). Gracias al Sender MAC presente en el Request, el Reply sabe a quién enviarse.

5
Actualización de caché y envío del paquete pendiente

Al recibir el Reply, el solicitante:

  • Guarda la correspondencia IP → MAC en su caché (TTL configurable).
  • Encapsula el paquete IP pendiente usando la MAC recién aprendida como destino L2.
  • Transmite el frame inmediatamente.

Las comunicaciones posteriores a esa IP usarán la caché y no volverán a generar ARP mientras el TTL siga vigente.

Diagrama de flujo completo

💡 ¿Por qué el primer ping en Cisco IOS falla? (.!!!!)

En Cisco IOS, al ejecutar ping por primera vez el resultado frecuente es .!!!!. El primer punto (.) indica que el router tuvo que resolver la MAC vía ARP y el request tardó más que el timeout (2 s). Los siguientes pings (!) usan la caché ARP y son exitosos.

Impacto en rendimiento — ARP Storms

En redes grandes, el tráfico ARP puede convertirse en un “storm” (muchos hosts generan ARP Request al mismo tiempo). Consecuencias:

  • Los switches inundan todos los puertos con broadcasts.
  • Cada host procesa TODOS los ARP Requests recibidos.
  • CPU de switches y hosts se satura.
  • Latencia aumenta y paquetes de datos se pierden.

La solución típica es segmentar la red en VLANs, de forma que cada VLAN sea un dominio de broadcast independiente.

Tipos de ARP

📢

ARP Request

Solicitud estándar, broadcast FF:FF:FF:FF:FF:FF. “¿Quién tiene la IP X?”. Target MAC = ceros. Opcode = 1.

✅

ARP Reply

Respuesta unicast al solicitante. Opcode = 2. Contiene la MAC asociada a la IP solicitada.

📣

Gratuitous ARP

Sender IP = Target IP. Se usa para anunciar presencia, detectar IP duplicada y para HSRP/VRRP.

🔀

Proxy ARP

Un router responde ARP Requests en nombre de hosts en otras redes usando su propia MAC.

🔁

Reverse ARP (RARP)

Proceso inverso: dado una MAC, obtener la IP. Obsoleto, reemplazado por DHCP.

🔍

Inverse ARP (InARP)

Usado en Frame‑Relay y ATM. Dado un DLCI (identificador de circuito virtual) descubre la IP del extremo remoto.

Gratuitous ARP — Análisis en profundidad

🚀 Anuncio de IP al arrancar

Cuando un host configura su IP (manual o DHCP) envía un GARP para informar a todos los vecinos de su presencia. Los dispositivos que tenían una entrada anterior para esa IP la actualizan.

🔍 Detección de IP duplicada

Si otro host ya posee esa IP, responderá al GARP con un ARP Reply. El emisor detecta el conflicto y muestra error.

⚡ Alta disponibilidad – HSRP / VRRP / GLBP

Cuando el router standby asume el rol activo, envía inmediatamente un GARP con la IP virtual. Todos los hosts actualizan su caché ARP apuntando al nuevo router.

🔄 Actualización post‑migración

Tras cambiar la tarjeta de red (nueva MAC) pero manteniendo la misma IP, el GARP actualiza automáticamente todas las cachés vecinas.

Proxy ARP — Configuración y funcionamiento

Cisco IOS — Proxy ARP
! Verificar estado actual de Proxy ARP en una interfaz
Router# show ip interface GigabitEthernet0/0 | include proxy
  Proxy ARP is enabled

! Deshabilitar (recomendado en redes bien diseñadas)
Router(config)# interface GigabitEthernet0/0
Router(config-if)# no ip proxy-arp

! Habilitar (si se necesita compatibilidad legacy)
Router(config-if)# ip proxy-arp
⚠️ Seguridad con Proxy ARP

Un dispositivo con Proxy ARP activo puede responder a ARP Requests ajenos y redirigir tráfico. En redes con gateways bien configurados es innecesario; lo recomendable es no ip proxy-arp en todas las interfaces de producción.

ARP Spoofing / ARP Poisoning

ARP no tiene mecanismo de autenticación. Cualquier host puede enviar ARP Replies falsos y los receptores actualizarán su caché sin validar.

🔴 Ataque ARP Poisoning – Man in the Middle
1El atacante envía ARP Replies falsos a la víctima: "La IP del router 192.168.1.1 tiene MAC AA:BB:CC:ATACANTE"
2La víctima actualiza su caché ARP con la MAC falsa.
3El atacante también envenena la caché del router: "La víctima tiene MAC AA:BB:CC:ATACANTE"
4Todo el tráfico víctima↔router pasa ahora por el atacante → MITM completo.

Dynamic ARP Inspection (DAI)

DAI es la contramedida de Cisco contra ARP Spoofing. Valida cada mensaje ARP contra la tabla de DHCP Snooping. Si la combinación IP + MAC no está registrada, el frame se descarta.

Configuración DAI en Switch Cisco
! Paso 1 – habilitar DHCP Snooping (prerequisito)
SW(config)# ip dhcp snooping
SW(config)# ip dhcp snooping vlan 10

! Paso 2 – marcar puerto DHCP‑server como trusted
SW(config)# interface GigabitEthernet0/1
SW(config-if)# ip dhcp snooping trust
SW(config-if)# exit

! Paso 3 – habilitar DAI en la VLAN
SW(config)# ip arp inspection vlan 10

! Paso 4 – puertos de hosts: untrusted (por defecto)
! Puertos trunk/uplinks: trusted
SW(config)# interface GigabitEthernet0/24
SW(config-if)# ip arp inspection trust

! Paso 5 – hosts con IP estática (sin entrada DHCP‑Snooping)
SW(config)# arp access-list HOSTS-ESTATICOS
SW(config-arp-nacl)# permit ip host 192.168.10.5 mac host aabb.cc11.2233
SW(config-arp-nacl)# exit
SW(config)# ip arp inspection filter HOSTS-ESTATICOS vlan 10

! Paso 6 – verificación
SW# show ip arp inspection
SW# show ip arp inspection vlan 10
SW# show ip arp inspection statistics vlan 10

💡 Resumen de defensas ARP
  • DAI – protección automática en switches, la más eficaz.
  • ARP estático – para dispositivos críticos (gateway, servidores).
  • Port‑Security – limita MACs por puerto.
  • DHCP Snooping – base de DAI, también protege contra DHCP falsos.
  • VLANs – segmentan dominios de broadcast.
  • 802.1X – autenticación de puerto, impide dispositivos no autorizados.
  • Cifrado (TLS/HTTPS/SSH) – aunque el atacante intercepte, no podrá leer el tráfico.

Problemas comunes y sus soluciones

❌ Primer ping siempre falla en Cisco IOS (.!!!!)
El router no tenía la entrada ARP del destino; envió un ARP Request y esperó el Reply. El timeout del primer ping supera los 2 s de IOS.
✅ Normal. Si TODOS los pings fallan (.....) hay un problema real de conectividad.
❌ No hay conectividad aunque IPs y rutas sean correctas
Tabla ARP corrupta (entradas erróneas) – suele ocurrir tras ARP Poisoning o cambio de hardware.
✅ Ejecutar clear arp-cache y volver a comprobar con show arp.
❌ Comunicación intermitente (va y viene)
ARP Poisoning activo: la caché cambia constantemente por Replies falsos.
✅ Ver show arp repetidamente; si la MAC varía, habilitar DAI con DHCP Snooping.
❌ Host no puede comunicarse con el gateway tras cambio de NIC
Caché ARP contiene la MAC anterior del gateway.
✅ Limpiar la caché del host (arp -d * Windows / ip neigh flush all Linux) y, en el nuevo gateway, enviar un GARP.
❌ DAI descarta ARPs de un host con IP estática
DAI valida contra tabla DHCP Snooping; los hosts estáticos no aparecen allí.
✅ Crear una ARP ACL (arp access-list) y aplicarla con ip arp inspection filter.
❌ HSRP failover correcto pero el tráfico no conmuta al router activo
Los hosts siguen usando la MAC del router fallido; el GARP del nuevo router activo fue bloqueado por DAI (puerto no trusted).
✅ Marcar los puertos hacia los routers HSRP como ip arp inspection trust para que el GARP llegue.
