ARP Address Resolution Protocol

El protocolo que traduce direcciones IP a direcciones MAC. Sin ARP, ningún paquete IP puede entregarse dentro de una red local. Comprender ARP a fondo es comprender la base de toda comunicación en redes Ethernet modernas.

📋 RFC 826 — 1982 🎯 CCNA 200-301 Exam Topic 1.3 ⏱ ~60 min lectura 🧪 Lab incluido 🔒 Seguridad DAI

¿Qué es ARP y por qué existe?

En toda comunicación de red existe un problema fundamental: las aplicaciones y los protocolos de capa 3 trabajan con direcciones IP, pero la red física Ethernet requiere direcciones MAC para entregar los frames de un dispositivo a otro. Nadie conoce de antemano la MAC del destinatario.

ARP (Address Resolution Protocol), definido en la RFC 826 (1982), resuelve exactamente este problema: dada una dirección IP conocida dentro del mismo segmento local, encontrar la dirección MAC que le corresponde. Sin ARP en operación, no existe comunicación de capa 2 dirigida.

💡 Analogía perfecta

Imagina que trabajas en una empresa y necesitas entregar un documento a "Laura Pérez" (la dirección IP). Para entregarlo físicamente debes saber en qué escritorio trabaja (la dirección MAC). Preguntas en voz alta a toda la oficina: "¿Alguien sabe dónde está Laura Pérez?" — eso es un ARP Request (broadcast). Laura responde: "¡Soy yo, escritorio B-7!" — eso es el ARP Reply (unicast). Memorizas su ubicación y ya no necesitas preguntar de nuevo durante un tiempo.

Posición de ARP en el modelo OSI

ARP es un protocolo único porque opera en la frontera entre capa 2 y capa 3. Técnicamente:

  • Usa frames Ethernet de capa 2 para transportarse
  • Resuelve direcciones de capa 3 (IP)
  • Tiene su propio EtherType: 0x0806
  • NO es encapsulado dentro de IP — viaja directamente en el frame Ethernet
  • No cruza routers — es estrictamente local al dominio de broadcast (VLAN)

Un mensaje ARP tiene exactamente 28 bytes de payload (para IPv4 sobre Ethernet). Cada campo tiene un propósito específico:

La Tabla ARP (ARP Cache)

Cada dispositivo mantiene una tabla ARP en memoria RAM que mapea IPs conocidas con sus MACs correspondientes. Esta tabla es dinámica y tiene un tiempo de vida (TTL) por cada entrada.

  • Entrada dinámica: Aprendida automáticamente. TTL: 240 seg en Cisco IOS, 60 seg en Windows, 20 min en macOS.
  • Entrada estática: Configurada manualmente por el administrador. No expira. Inmune a ARP Poisoning.
  • Entrada incompleta: ARP Request enviado, Reply aún no recibido. Estado transitorio mientras se espera la respuesta.
⚠️ Punto crítico CCNA

Cuando un router reenvía un paquete de un segmento a otro, reemplaza las MACs de origen y destino en cada salto. Las IPs de origen y destino permanecen iguales de extremo a extremo. En cada nuevo segmento se ejecuta un ARP para resolver la MAC del siguiente salto. Este concepto es frecuente en el examen CCNA.

ARP entre redes distintas — El rol del Default Gateway

Cuando el destino está en una red diferente, el host origen NO hace ARP por la IP destino. En cambio:

  1. El host aplica su máscara de subred y determina que el destino está en una red diferente.
  2. No puede hacer ARP por la IP destino — hace ARP por la IP del Default Gateway.
  3. Envía el frame con la MAC del router como destino, pero con la IP final en el header IP.
  4. El router desencapsula el frame, consulta su tabla de routing y hace ARP en el segmento siguiente.
  5. Este proceso se repite en cada salto hasta llegar al destino final.

El proceso ARP paso a paso

1
Consulta de la caché ARP local

Antes de generar cualquier tráfico, el dispositivo consulta su tabla ARP en RAM. Si hay una entrada válida y no expirada para la IP destino, utiliza esa MAC de inmediato. No se genera ningún mensaje ARP. En redes activas, la mayoría de comunicaciones usan la caché directamente.

2
ARP Request — Broadcast de capa 2

Si no hay entrada en caché, el host construye un ARP Request y lo envía como broadcast de capa 2:

  • MAC destino del frame: FF:FF:FF:FF:FF:FF (broadcast)
  • Target MAC en el payload ARP: 00:00:00:00:00:00 (desconocida)
  • Target IP: la IP que queremos resolver
  • Sender MAC y Sender IP: los del host que pregunta

Todos los hosts del segmento reciben este frame porque es broadcast. Los switches lo inundan por todos los puertos del mismo dominio de broadcast (VLAN).

3
Procesamiento por cada host

Cada host que recibe el ARP Request hace lo siguiente:

  • Aprendizaje pasivo: extrae el Sender MAC e IP del Request y lo agrega a su propia caché ARP. Esto optimiza la comunicación futura sin necesidad de un nuevo ARP.
  • Comparación: coteja la Target IP con su propia dirección IP configurada.
  • Si NO coincide: descarta silenciosamente el request
  • Si SÍ coincide: prepara y envía un ARP Reply
4
ARP Reply — Unicast

El host dueño de la IP responde con un ARP Reply unicast directamente al solicitante:

  • MAC destino del frame: MAC del host que preguntó
  • Sender MAC: su propia MAC (la respuesta)
  • Opcode: 2 (Reply)

El Reply va directo al solicitante — no es broadcast. Ya se conoce la MAC destino gracias al Sender MAC del Request.

5
Actualización de caché y envío del paquete pendiente

Al recibir el ARP Reply, el solicitante:

  • Guarda la correspondencia IP→MAC en su caché con el TTL configurado (240 seg en IOS).
  • Encapsula el paquete IP que estaba pendiente usando la MAC recién aprendida como destino L2.
  • Transmite el frame inmediatamente.

Toda comunicación posterior hacia esa IP durante la vigencia del TTL usará la caché directamente — sin generar nuevo tráfico ARP.

Diagrama de flujo completo

💡 ¿Por qué el primer ping en Cisco IOS falla? (.!!!!)

En Cisco IOS, al ejecutar ping por primera vez el resultado frecuente es .!!!!. El primer punto (.) = timeout. El router necesitó resolver la MAC vía ARP antes de encapsular el primer ICMP. El proceso toma <2 segundos. Los siguientes pings (!) usan la caché ARP y son exitosos. Si todos los pings fallan (.....), ahí sí hay un problema real de conectividad.

Impacto en rendimiento — ARP Storms

En redes grandes con muchos hosts, el tráfico ARP puede volverse significativo. Un ARP Storm ocurre cuando muchos hosts generan ARP Requests simultáneamente (ej: después de un corte de luz cuando todos arrancan al mismo tiempo). Las consecuencias:

  • Los switches inundan todos los puertos con broadcasts
  • Cada host procesa TODOS los ARP Requests recibidos
  • La CPU de switches y hosts se satura
  • La latencia aumenta y los paquetes de datos se pierden

La solución es segmentar la red en VLANs — cada VLAN es un dominio de broadcast independiente, limitando el alcance del tráfico ARP.

Tipos de ARP

📢

ARP Request

Solicitud estándar. Enviado como broadcast (FF:FF:FF:FF:FF:FF). Pregunta: "¿Quién tiene la IP X? Dime tu MAC". Target MAC = ceros. Opcode = 1.

✅

ARP Reply

Respuesta unicast al solicitante. Opcode = 2. Contiene la MAC correspondiente a la IP preguntada. Solo el solicitante lo recibe — no inunda la red.

📣

Gratuitous ARP (GARP)

ARP donde Sender IP = Target IP. El host hace ARP de su propia IP. Usado para anuncio de presencia, detección de duplicados y alta disponibilidad (HSRP/VRRP).

🔀

Proxy ARP

Un router responde ARP Requests en nombre de hosts en otras redes, usando su propia MAC. Permite comunicación a hosts sin gateway bien configurado. Activo por defecto en IOS.

🔁

Reverse ARP (RARP)

El proceso inverso: dado una MAC, obtener la IP. Obsoleto, reemplazado por DHCP. Un host sin IP preguntaba al servidor RARP cuál IP le correspondía.

🔍

Inverse ARP (InARP)

Usado en redes Frame Relay y ATM. Dado un DLCI (identificador de circuito virtual), descubre la IP del extremo remoto. Diferente a ARP estándar.

Gratuitous ARP — Análisis en profundidad

El GARP es el tipo de ARP más importante para el examen CCNA. Sus casos de uso críticos en entornos de producción:

🚀 Anuncio de IP al arrancar

Cuando un host configura su IP (manual o por DHCP), envía un GARP para informar a todos los vecinos de su presencia. Los dispositivos que tenían una entrada anterior para esa IP en su caché la actualizan con la nueva MAC.

🔍 Detección de IP duplicada

Si otro host ya tiene esa IP, responderá al GARP con un ARP Reply. El host que envió el GARP detecta la colisión y genera un error. Windows muestra "Conflicto de dirección IP".

⚡ Alta Disponibilidad — HSRP / VRRP / GLBP

Cuando el router standby asume el rol activo en HSRP, VRRP o GLBP, envía inmediatamente un GARP usando la IP virtual del grupo. Todos los hosts del segmento actualizan su caché ARP apuntando al nuevo router activo, garantizando continuidad del servicio sin reconfiguración manual.

🔄 Actualización post-migración

Cuando se reemplaza una tarjeta de red (cambio de MAC) pero se mantiene la misma IP, el GARP actualiza todas las cachés vecinas automáticamente.

Proxy ARP — Configuración y funcionamiento

Con Proxy ARP habilitado, un router puede responder ARP Requests de hosts que no tienen default gateway configurado correctamente. El router actúa como "intermediario":

Cisco IOS — Proxy ARP
! Verificar estado actual de Proxy ARP en una interfaz
Router# show ip interface GigabitEthernet0/0 | include proxy
  Proxy ARP is enabled

! Deshabilitar Proxy ARP — recomendado en redes bien diseñadas
Router(config)# interface GigabitEthernet0/0
Router(config-if)# no ip proxy-arp

! Habilitar Proxy ARP (si se requiere compatibilidad legacy)
Router(config-if)# ip proxy-arp
⚠️ Seguridad con Proxy ARP

Un dispositivo mal configurado con Proxy ARP activo puede responder ARP Requests ajenos y redirigir tráfico inadvertidamente. En redes con gateways correctamente configurados, Proxy ARP es innecesario. Deshabilitar con no ip proxy-arp en todas las interfaces de producción es una práctica de hardening recomendada.

ARP Spoofing / ARP Poisoning

La vulnerabilidad fundamental de ARP es que no tiene ningún mecanismo de autenticación. Cualquier host puede enviar ARP Replies con información falsa y los receptores actualizarán su caché sin cuestionarlo. Esto es la base del ataque ARP Spoofing / ARP Cache Poisoning, que permite construir un ataque Man-in-the-Middle (MitM) completo en cualquier red LAN.

🔴 Ataque ARP Poisoning — Man in the Middle
1 El atacante envía ARP Replies falsos a la víctima: "La IP del router 192.168.1.1 tiene MAC AA:BB:CC:ATACANTE"
2 La víctima actualiza su caché ARP con la MAC falsa del atacante
3 El atacante también envenena la caché del router: "La víctima tiene MAC AA:BB:CC:ATACANTE"
4 Todo el tráfico víctima↔router ahora pasa por el atacante → Man in the Middle completo

Dynamic ARP Inspection (DAI)

DAI es la contramedida de Cisco contra ARP Spoofing. Opera en los switches y valida cada mensaje ARP contra la tabla de vinculaciones de DHCP Snooping. Si el par IP+MAC del ARP no coincide con la asignación legítima, el mensaje es descartado silenciosamente.

Configuración DAI en Switch Cisco
! Paso 1: Habilitar DHCP Snooping (requisito previo para DAI)
SW(config)# ip dhcp snooping
SW(config)# ip dhcp snooping vlan 10

! Paso 2: Configurar el puerto del DHCP server como trusted
SW(config)# interface GigabitEthernet0/1
SW(config-if)# ip dhcp snooping trust
SW(config-if)# exit

! Paso 3: Habilitar Dynamic ARP Inspection para la VLAN
SW(config)# ip arp inspection vlan 10

! PASO 4 — Puertos de hosts: untrusted por defecto (correcto)
! Puertos trunk / uplinks hacia otros switches: trusted
SW(config)# interface GigabitEthernet0/24
SW(config-if)# ip arp inspection trust

! PASO 5 — Para hosts con IP estática (sin entrada DHCP Snooping)
SW(config)# arp access-list HOSTS-ESTATICOS
SW(config-arp-nacl)# permit ip host 192.168.10.5 mac host aabb.cc11.2233
SW(config-arp-nacl)# exit
SW(config)# ip arp inspection filter HOSTS-ESTATICOS vlan 10

! Verificación de DAI
SW# show ip arp inspection
SW# show ip arp inspection vlan 10
SW# show ip arp inspection statistics vlan 10

ARP Estático — Defensa adicional

Las entradas ARP estáticas son inmunes a ARP Poisoning porque no pueden ser sobrescritas por ARP Replies recibidos. Se usan para dispositivos críticos como el default gateway.

! Agregar entrada ARP estática
Router(config)# arp 192.168.1.254 aabb.cc11.2233 arpa

! Ver tabla ARP — Age "-" = entrada estática (permanente)

Router(config)# no arp 192.168.1.254 aabb.cc11.2233 arpa
💡 Resumen de defensas ARP
  • DAI: Protección automática en switches — la más efectiva
  • ARP estático: Para dispositivos críticos como gateways
  • Port Security: Limita MACs por puerto, reduce superficie de ataque
  • DHCP Snooping: Base de DAI, también protege contra DHCP falsos
  • VLANs: Segmentan dominios de broadcast, limitan alcance del ataque
  • 802.1X: Autenticación de puerto, impide que dispositivos no autorizados se conecten.
  • Cifrado (TLS/HTTPS/SSH): aunque el atacante intercepte el tráfico, no puede descifrarlo.

Problemas comunes relacionados con ARP y sus soluciones

❌ Primer ping siempre falla en Cisco IOS (.!!!!)
Causa: El router no tenía la entrada ARP del destino. Debió enviar un ARP Request y esperar el Reply antes de encapsular el ICMP. El primer paquete supera el timeout de 2 segundos.
✅ Completamente normal en IOS. Si TODOS los pings fallan (.....) hay un problema real: verificar conectividad, tabla ARP y routing.
❌ No hay conectividad aunque IPs y rutas estén correctas
Causa probable: caché ARP corrupta con entradas incorrectas. Puede ocurrir tras ARP Poisoning, reemplazo de hardware o fallo de red.
✅ Ejecutar clear arp-cache en el router para forzar resolución nueva. Verificar con show arp que las MACs sean correctas.
❌ Comunicación intermitente — conectividad va y viene
Causa probable: ARP Poisoning activo. Las entradas ARP de la caché cambian frecuentemente por el atacante enviando Replies falsos repetidamente.
✅ Ejecutar show arp repetidamente y observar si las MACs cambian. Si hay variación, implementar DAI con DHCP Snooping para bloquear los Replies ilegítimos.
❌ Host no puede comunicarse con el gateway tras cambio de hardware
Causa: el host tiene cacheada la MAC antigua del gateway. La nueva NIC tiene otra MAC, la caché ARP todavía apunta a la anterior.
✅ En el host: limpiar caché ARP (arp -d * Windows / ip neigh flush all Linux). En el nuevo gateway: enviar un GARP para que todos los hosts actualicen su caché automáticamente.
❌ DAI descarta ARPs de un host con IP estática
Causa: DAI valida contra la tabla DHCP Snooping. Hosts con IP estática no tienen entrada en esa tabla (nunca usaron DHCP), por lo que sus ARPs son rechazados.
✅ Crear ARP ACL manual: arp access-list ESTATICOS → permit ip host [IP] mac host [MAC] → ip arp inspection filter ESTATICOS vlan [X].
❌ HSRP failover correcto pero el tráfico no conmuta hacia el nuevo router activo
Causa: los hosts mantienen en caché la MAC del router que falló. El nuevo router activo envió el GARP pero fue bloqueado por DAI (su puerto no está marcado como trusted).
✅ Verificar que los puertos hacia los routers HSRP estén marcados como ip arp inspection trust en el switch. El GARP del nuevo router activo debe poder llegar a los hosts.
