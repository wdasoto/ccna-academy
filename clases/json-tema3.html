JSON vs XML vs YAML

La guerra de los formatos de datos en networking. Aprende cuándo usa cada protocolo: RESTCONF (JSON/XML), NETCONF (XML), Ansible (YAML). Diferencias sintácticas clave y escenarios de uso para el examen CCNA.

⚖️ Comparativa 🎯 CCNA Domain 6.5-6.7 ⏱ ~55 min lectura 🌐 RESTCONF/NETCONF ⚙️ Ansible

Los Tres Pilares de la Automatización de Red

En el mundo del networking moderno, no existe un único formato de datos. Cada protocolo y herramienta eligió su lenguaje: RESTCONF habla JSON (y XML), NETCONF habla XML exclusivamente, y Ansible prefiere YAML por su legibilidad humana. Dominar las diferencias es clave para el examen CCNA y para la vida real.

{ }
JSON
RESTCONF APIs

Ligero, nativo de JavaScript, estructura clave-valor. Ideal para APIs HTTP modernas.

</>
XML
NETCONF / YANG

Verbos pero estricto. El estándar de NETCONF y configuraciones legacy. Schema-based.

⎽⎽
YAML
Ansible / Configs

Human-friendly. Indentación significativa. Playbooks de Ansible e inventarios.

💡 Clave para el examen CCNA

Si la pregunta menciona RESTCONF, piensa JSON. Si menciona NETCONF, piensa XML. Si menciona Ansible, piensa YAML. No hay excepciones en el examen: NETCONF siempre usa XML por especificación RFC 6241.

Mismo Dato, Tres Lenguajes

Compara cómo se representa la misma información (una interfaz de router) en los tres formatos. Observa la verbosidad y legibilidad.

📋 JSON
{
  "interface": {
    "name": "GigabitEthernet1",
    "ip": "192.168.1.1",
    "mask": "255.255.255.0",
    "enabled": true,
    "vlan": null
  }
}

<interface>
  <name>GigabitEthernet1</name>
  <ip>192.168.1.1</ip>
  <mask>255.255.255.0</mask>
  <enabled>true</enabled>
  <vlan>null</vlan>
</interface>
📄 YAML

  name: GigabitEthernet1
  ip: 192.168.1.1
  mask: 255.255.255.0
  enabled: true
  vlan: null

JSON — El Estándar REST

JSON es el formato por defecto de RESTCONF y la mayoría de APIs HTTP modernas. Su simplicidad lo hace ideal para JavaScript y Python.

✅ Ventajas JSON
  • Nativo en navegadores (JavaScript)
  • Más ligero que XML (menos bytes en red)
  • Tipos de datos nativos (strings, números, booleanos)
  • Parseo rápido en Python con json.loads()

Ejemplo RESTCONF Real (Cisco)

GET /restconf/data/ietf-interfaces:interfaces
{
  "ietf-interfaces:interfaces": {
    "interface": [
      {
        "name": "GigabitEthernet1",
        "description": "Uplink to Core",
        "type": "iana-if-type:ethernetCsmacd",
        "enabled": true,
        "ietf-ip:ipv4": {
          "address": [
            {
              "ip": "203.0.113.1",
              "netmask": "255.255.255.0"
            }
          ]
        }
      }
    ]
  }
}
⚡ JSON en el examen CCNA

Busca las llaves {} para identificar JSON. Recuerda: las claves van entre comillas dobles "clave", nunca simples. Los booleanos son true/false (minúsculas).

XML — El Lenguaje de NETCONF

XML es obligatorio para NETCONF (RFC 6241). Aunque verboso, proporciona validación estricta mediante esquemas (XSD) y es el formato nativo de YANG data models.

🌐 Características XML
  • Tags de apertura y cierre obligatorios <tag></tag>
  • Atributos en tags: <interface name="Gi0/0">
  • Namespaces para evitar colisiones: <nc:rpc>
  • Comentarios: <!-- esto es un comentario -->

Ejemplo NETCONF Real (RPC)

<rpc> message para obtener configuración
<rpc message-id="101" 
     xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get-config>
    <source>
      <running/>
    </source>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <interface>
          <name>GigabitEthernet1</name>
        </interface>
      </interfaces>
    </filter>
  </get-config>
</rpc>
🎯 Clave CCNA: NETCONF = XML

En el examen, NETCONF siempre usa XML. No existe NETCONF con JSON en la especificación base (aunque algunos vendors tienen extensiones). Si ves una pregunta sobre NETCONF, busca tags XML <>, no llaves JSON.

YAML — El Lenguaje de Ansible

YAML (YAML Ain't Markup Language) es el formato de los playbooks de Ansible. Es un superconjunto de JSON (todo JSON válido es YAML válido) pero mucho más legible para humanos.

📄 Reglas Críticas YAML
  • Indentación con espacios (no tabs). Generalmente 2 espacios.
  • Sin comillas obligatorias (aunque permitidas).
  • Listas con guión - y espacio.
  • Comentarios con #.
  • Booleanos: true/false o yes/no (cuidado: no se convierte en booleano, no string).

Ejemplo Ansible Playbook

Configurar VLANs en switches Cisco
# playbook-vlans.yml
---
- name: Configurar VLANs en switches
  hosts: switches
  gather_facts: no
  
  tasks:
    - name: Crear VLANs
      cisco.ios.ios_vlan:
        vlan_id: 10
        name: USUARIOS
        state: present
      register: resultado
      
    - name: Mostrar resultado
      ansible.builtin.debug:
        var: resultado

⚡ Trampa común en YAML

En YAML, no y yes se convierten en booleanos. Si necesitas el string "no", ponlo entre comillas: "no". Esto es un error clásico en contraseñas o nombres de interfaces.

Conversión Entre Formatos

En la práctica, convertirás entre estos formatos constantemente:

  • YAML → JSON: Ansible lee YAML pero APIs esperan JSON. Usa yaml.safe_load() → json.dumps()
  • XML → JSON: NETCONF devuelve XML pero tu app usa JSON. Librerías como xmltodict o xml.etree
  • JSON → YAML: Para hacer APIs legibles en documentación. yaml.dump()
Conversión YAML a JSON en Python
import yaml
import json

# Leer playbook de Ansible (YAML)
with open('playbook.yml') as f:
    playbook = yaml.safe_load(f)

# Convertir a JSON para enviar a API
json_payload = json.dumps(playbook, indent=2)
print(json_payload)

Parseo en Python de los Tres Formatos

JSON
import json

# Parsear
data = json.loads('{"ip": "1.1.1.1"}')

# Acceder
print(data["ip"])  # 1.1.1.1

# Serializar
json_str = json.dumps(data)
XML
import xml.etree.ElementTree as ET

# Parsear
root = ET.fromstring(xml_data)

# Acceder
ip = root.find("ip").text

# Crear
iface = ET.Element("interface")
ET.SubElement(iface, "name")
YAML
import yaml

# Parsear
with open('inv.yml') as f:
    data = yaml.safe_load(f)

# Acceder
print(data["routers"])

# Serializar
yaml_str = yaml.dump(data)
💡 Instalación de librerías

JSON viene en la librería estándar. XML también (xml.etree). Para YAML necesitas instalar: pip install PyYAML

Cheat Sheet: Identificación Rápida

  • Ve {} o [] → Es JSON
  • Ve <tag> → Es XML
  • Ve indentación con - lista → Es YAML
  • Ve xmlns= → Es XML (NETCONF)
  • Ve Content-Type: application/json → RESTCONF
