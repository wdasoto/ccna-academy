DNA CENTER — Intent-Based Networking API

El cerebro central de Cisco SD-Access. Aprende a consumir la Intent API para orquestar inventario, topología, assurance y políticas desde Python. Autenticación Token, Command Runner, Templates, Webhooks y Health Metrics. El estándar empresarial de automatización Cisco para el examen CCNA.

🧬 Intent API v1 🎯 CCNA Domain 6.5 ⏱ ~90 min lectura 💻 Always-On Sandbox 🐍 Python requests

¿Qué es Cisco DNA Center y por qué revoluciona la red?

Cisco DNA Center es el controlador centralizado de Cisco para Intent-Based Networking (IBN). En lugar de configurar dispositivos uno por uno vía CLI, defines qué quieres lograr (el "intent"): una política de acceso, una topología de fabric, un segmento de usuarios. DNA Center traduce esa intención en configuraciones concretas y las aplica automáticamente a través de la red.

Desde la perspectiva del examen CCNA Domain 6.5, DNA Center representa el modelo de controlador SDN aplicado a campus empresarial. Separa el plano de control del plano de datos, centraliza la inteligencia y expone todo su potencial a través de APIs REST que cualquier desarrollador puede consumir con Python y la librería requests.

💡 DNA Center vs. Controladores tradicionales
  • SD-WAN (vManage): Optimiza WAN y conectividad a la nube. APIs propias.
  • Wireless LAN Controllers: Gestionan APs y RF. Usan config-and-ejbca o REST.
  • DNA Center: Orquesta LAN, WLAN, políticas de acceso, assurance y provisioning. Intent API unificada en /dna/intent/api/v1/.
  • Para el examen CCNA: conoce que DNA Center centraliza gestión, políticas y monitoreo; expone REST APIs; y reduce el trabajo manual CLI.

Arquitectura de APIs en DNA Center

DNA Center expone tres familias de APIs. El examen CCNA se enfoca en la primera:

🎯

Intent API (REST)

La API principal. CRUD sobre inventario, topología, políticas, configuración y assurance. Usa JSON sobre HTTPS. Es la que consume el 90% de los scripts de automatización.

📡

Event API (Webhooks)

Notificaciones push asincrónicas. DNA Center envía un POST a tu endpoint cuando ocurre un evento (alarma, cambio de topología, rogue AP). Evita el polling constante.

🧪

Multi-Domain Integration

Integra DNA Center con SD-WAN (vManage), ISE, ServiceNow y Webex. Relevante para arquitecturas enterprise, fuera del alcance directo del CCNA.

🌐

Always-On Sandbox

⚡ Seguridad en producción vs. Sandbox

En el sandbox usamos verify=False para evitar errores de certificado SSL autofirmado. En producción, nunca desactives la verificación SSL. Debes usar certificados válidos o el certificado CA raíz de tu organización. Además, nunca hardcodees credenciales en scripts; usa variables de entorno o un vault de secretos.

Autenticación basada en Token (JWT)

DNA Center no usa autenticación Basic en cada petición. En su lugar, implementa un modelo de Token JWT: envías tus credenciales una sola vez al endpoint de autenticación, recibes un token temporal, y lo reutilizas en el header X-Auth-Token de todas las llamadas subsiguientes.

⏳

Duración del Token

En la mayoría de deployments, el token expira tras 1 hora (3600s) de inactividad. Si haces peticiones frecuentes, puedes reutilizarlo. Si expira, recibes HTTP 401 y debes re-autenticar.

🔄

Refresh implícito

La Intent API no expone un refresh_token separado. El "refresh" consiste simplemente en volver a llamar /auth/token con Basic Auth para obtener un token nuevo.

🛡️

Transporte seguro

Toda la comunicación viaja sobre HTTPS (443). El token es sensible; si se filtra, un atacante puede leer configuración o ejecutar comandos. Protégelo como una contraseña.

Obtención del Token con Python requests

auth_dna_center.py — Obtener y encapsular el token
import requests
import urllib3
from requests.auth import HTTPBasicAuth

# Desactivar warnings de SSL solo para sandbox/educación
urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

BASE_URL = "https://sandboxdnac.cisco.com"
USERNAME = "devnetuser"
PASSWORD = "Cisco123!"

def get_auth_token():
    """Autentica contra DNA Center y retorna el JWT token."""
    endpoint = "/dna/system/api/v1/auth/token"
    url = BASE_URL + endpoint

    response = requests.post(
        url,
        auth=HTTPBasicAuth(USERNAME, PASSWORD),
        verify=False   # Sandbox usa cert autofirmado
    )

    # DNA Center devuelve 200 con JSON {"Token": "eyJ0eXAi..."}
    response.raise_for_status()
    token = response.json()["Token"]
    print(f"✅ Token obtenido: {token[:20]}...")
    return token

# Uso
token = get_auth_token()
headers = {
    "X-Auth-Token": token,
    "Content-Type": "application/json",
    "Accept": "application/json"
}
💡 Patrón de producción: Clase wrapper

En lugar de pasar headers manualmente a cada función, crea una clase DNACenterAPI que gestione el token, su expiración y el re-intento automático ante un 401. Esto separa la lógica de transporte de la lógica de negocio y hace tu código testeable.

Manejo de expiración y errores 401

Re-autenticación automática ante token expirado
class DNACenterClient:
    def __init__(self, base_url, username, password):
        self.base_url = base_url
        self.auth = HTTPBasicAuth(username, password)
        self.headers = {}
        self._get_token()

    def _get_token(self):
        url = f"{self.base_url}/dna/system/api/v1/auth/token"
        r = requests.post(url, auth=self.auth, verify=False)
        r.raise_for_status()
        self.headers["X-Auth-Token"] = r.json()["Token"]

    def request(self, method, endpoint, **kwargs):
        url = f"{self.base_url}{endpoint}"
        kwargs.setdefault("headers", {}).update(self.headers)
        kwargs["verify"] = False

        resp = requests.request(method, url, **kwargs)
        if resp.status_code == 401:
            print("🔄 Token expirado. Re-autenticando...")
            self._get_token()
            kwargs["headers"].update(self.headers)
            resp = requests.request(method, url, **kwargs)
        resp.raise_for_status()
        return resp

# Uso limpio en todo el proyecto
api = DNACenterClient("https://sandboxdnac.cisco.com", "devnetuser", "Cisco123!")
devices = api.request("GET", "/dna/intent/api/v1/network-device").json()

Inventario de red y jerarquía de sites

El inventario es el pilar de cualquier operación en DNA Center. Antes de aplicar una configuración, debes saber qué dispositivos existen, dónde están (site, building, floor) y cómo están conectados (topología).

Endpoints esenciales de Inventario

GET/dna/intent/api/v1/network-device
Lista todos los dispositivos gestionados: routers, switches, APs, WLCs. Soporta filtros por family, type, series y managementIpAddress.
GET/dna/intent/api/v1/network-device/{deviceId}
Detalle completo de un dispositivo específico: interfaces, versión IOS, serial number, ubicación, health score y roles en fabric.
GET/dna/intent/api/v1/site
Jerarquía de sites definida en DNA Center: Global > Area > Building > Floor. Cada site tiene un siteId único.
GET/dna/intent/api/v1/topology/physical-topology
Topología física de la red: nodos (dispositivos) y enlaces (conexiones L1/L2). Útil para mapear visualmente la infraestructura.

def get_sites(headers):

def get_physical_topology(headers):
    url = "https://sandboxdnac.cisco.com/dna/intent/api/v1/topology/physical-topology"
    r = requests.get(url, headers=headers, verify=False)
    topo = r.json()["response"]

    nodes = topo["nodes"]
    links = topo["links"]

    print(f"🌐 Nodos descubiertos: {len(nodes)}")
    print(f"🔗 Enlaces detectados: {len(links)}")

    for node in nodes[:5]:
        print(f"  • {node['label']} ({node['deviceSeries']}) — {node['ip']}")

    for link in links[:3]:
        src = next(n["label"] for n in nodes if n["id"] == link["source"])
        dst = next(n["label"] for n in nodes if n["id"] == link["target"])
        print(f"  └─ {src} === {dst} [{link.get('linkStatus','up')}]")

El examen CCNA puede preguntar: "¿Qué controlador Cisco proporciona inventario centralizado, topología y health monitoring vía REST API?" La respuesta es Cisco DNA Center (no WLC ni vManage). También puede preguntar el formato de respuesta: JSON, nunca XML ni YAML.

Assurance: Health de red, cliente y sitio

Assurance es el módulo de DNA Center que transforma telemetría en inteligencia accionable. No solo te dice si un dispositivo está up/down; te da un health score (0-10) basado en KPIs: CPU, memoria, errores de interfaces, retardo, pérdida de paquetes, problemas de RF, etc.

🌐

Network Health

Agregado de salud de todos los dispositivos de red (routers, switches, APs). Score bajo indplica congestionamiento, errores de enlace o hardware degradado.

📱

Client Health

Experiencia de usuarios finales y dispositivos IoT. Mide asociación Wi-Fi, roaming, throughput y problemas de autenticación.

🏢

Site Health

Vista consolidada por ubicación física. Permite comparar el desempeño del edificio A vs. el edificio B, o identificar pisos problemáticos.

Endpoints de Assurance

GET/dna/intent/api/v1/network-health
Devuelve health score global, número de dispositivos por estado (healthy, fair, poor), y timestamp de la muestra.
GET/dna/intent/api/v1/client-health
Métricas de clientes: total de clientes conectados, distribución por banda (2.4GHz, 5GHz, 6GHz), y score de experiencia.
GET/dna/intent/api/v1/site-health
Health por site jerárquico. Incluye access health (usuarios), network health (infra) y application health (app-aware).

Command Runner — CLI a través de la API

No siempre existe un modelo YANG o un endpoint REST para la información que necesitas. Command Runner es el puente entre el mundo API y el mundo CLI: envías comandos show como si estuvieras en la consola, DNA Center los ejecuta en el dispositivo destino vía SSH/NETCONF interno, y te devuelve la salida como texto plano o estructurado.

Flujo de trabajo del Command Runner

1. ENVIAR
POST /cli/read-request
→
2. TASK ID
UUID de tarea
→
3. POLLING
GET /task/{id}
→
4. RESULTADO
fileId / progress

Ejecución de comandos show

command_runner.py — Patrón completo con polling
import time
import json

def run_show_command(headers, device_uuid, commands):
    """
    Ejecuta comandos 'show' en un dispositivo vía DNA Center.
    Retorna la salida cruda del CLI.
    """
    base = "https://sandboxdnac.cisco.com"

    # 1. Solicitar ejecución
    url = f"{base}/dna/intent/api/v1/network-device-poller/cli/read-request"
    payload = {
        "commands": commands,
        "deviceUuids": [device_uuid],
        "timeout": 30
    }
    r = requests.post(url, headers=headers, json=payload, verify=False)
    r.raise_for_status()
    task_id = r.json()["response"]["taskId"]
    print(f"🚀 Tarea creada: {task_id}")

    # 2. Polling del estado de la tarea
    task_url = f"{base}/dna/intent/api/v1/task/{task_id}"
    for _ in range(10):
        time.sleep(1)
        tr = requests.get(task_url, headers=headers, verify=False)
        task = tr.json()["response"]

        if task.get("isError"):
            raise Exception(f"Error en tarea: {task.get('progress')}")

        progress = task.get("progress", "")
        print(f"⏳ Progreso: {progress}")

        if task.get("endTime"):
            break

    # 3. Obtener resultado por fileId
    file_id = json.loads(progress).get("fileId")
    file_url = f"{base}/dna/intent/api/v1/file/{file_id}"
    fr = requests.get(file_url, headers=headers, verify=False)
    result = fr.json()

    for entry in result:
        print(f"\n🖥️  {entry['deviceUuid']}")
        print("─" * 50)
        print(entry.get("commandResponses", {}).get("SUCCESS", {}))
        return entry

# Uso práctico
device_id = "a3d1e2f8-1234-5678-9abc-def012345678"  # UUID real del inventario
run_show_command(headers, device_id, ["show version", "show ip interface brief"])
⚡ Command Runner ≠ Configuración masiva

Command Runner está diseñado para comandos show (lectura). Si intentas enviar config t, DNA Center lo bloqueará por seguridad. Para cambios de configuración masivos, usa Templates o la Intent API específica del servicio (ej. /network-device poller para CLI write no está soportado para cambios).

Obtener el UUID de un dispositivo

Los endpoints de Command Runner requieren deviceUuids (UUID), no IP ni hostname. Primero debes consultar el inventario:

Resolver hostname → UUID
def get_device_uuid_by_ip(headers, ip):
    url = "https://sandboxdnac.cisco.com/dna/intent/api/v1/network-device"
    r = requests.get(url, headers=headers, verify=False)
    for dev in r.json()["response"]:
        if dev.get("managementIpAddress") == ip:
            return dev["id"]   # Este es el UUID
    raise ValueError(f"Dispositivo con IP {ip} no encontrado")

Templates y Provisioning automatizado

Los Templates de DNA Center permiten definir configuraciones CLI como plantillas reutilizables usando el lenguaje Apache Velocity. Variables como ${VLAN_ID} o ${INTERFACE} se sustituyen en tiempo de despliegue. Esto elimina errores humanos de copy-paste y acelera el provisioning de Day 0 y Day N.

📋

Velocity Templates

Sintaxis ${variable}, condicionales #if, bucles #foreach. Soportado nativamente por DNA Center para generar CLI dinámico.

🚀

Day-0 Provisioning

Al encender un switch nuevo (PnP — Plug and Play), DNA Center le aplica automáticamente template + imagen + licencia sin intervención manual.

🔄

Day-N Changes

Ajustes posteriores: añadir VLANs, modificar ACLs, actualizar descripciones. Se versionan y pueden revertirse desde la interfaz.

Desplegar un template desde Python

deploy_template.py — Provisioning programático
def deploy_template(headers, template_id, device_uuids, variables):
    """Despliega un template Velocity a una lista de dispositivos."""
    url = "https://sandboxdnac.cisco.com/dna/intent/api/v1/template-programmer/template/deploy"

    payload = {
        "targetInfo": [
            {
                "id": dev_id,
                "type": "MANAGED_DEVICE_UUID",
                "params": variables
            }
            for dev_id in device_uuids
        ],
        "templateId": template_id
    }

    r = requests.post(url, headers=headers, json=payload, verify=False)
    deployment = r.json()["response"]
    print(f"📋 Deployment ID: {deployment['deploymentId']}")
    print(f"⏳ Estado: {deployment['status']}")
    return deployment

# Uso
template_vars = {"INTERFACE": "GigabitEthernet1/0/1", "VLAN_ID": "10", "DESCRIPTION": "USUARIOS_VENTAS"}
deploy_template(headers, "tpl-abc123", ["uuid-1", "uuid-2"], template_vars)
💡 Templates vs. NETCONF/RESTCONF

Para el examen CCNA es clave entender cuándo usar cada herramienta:

  • Templates (DNA Center): Útil para configuraciones masivas tipo CLI en dispositivos legacy o cuando no hay modelos YANG disponibles.
  • NETCONF/YANG: Transaccional, estructurado, ideal para cambios atómicos y rollback.
  • RESTCONF: Stateless, HTTP-based, perfecto para microservicios y dashboards.
  • DNA Center API: Orquestación de alto nivel, assurance, topología e inventario. No reemplaza NETCONF; lo complementa.

Event API: Webhooks y notificaciones push

El modelo de polling (preguntar cada X segundos si hay novedades) es ineficiente. DNA Center soporta Webhooks: tú registras una URL de tu aplicación, y DNA Center envía un POST automático cuando ocurre un evento relevante (alarma, rogue AP, cambio de topología, fallo de enlace).

🔔

Eventos de red

LINK_DOWN, INTERFACE_ERR_DISABLED, BGP_NEIGHBOR_DOWN. Notificación inmediata sin necesidad de SNMP traps complejos.

👤

Eventos de cliente

CLIENT_ASSOCIATION_FAILURE, ROUGE_AP_DETECTED. Útil para seguridad WLAN y troubleshooting proactivo.

⚙️

Eventos de configuración

CONFIGURATION_CHANGE, TEMPLATE_DEPLOYMENT_SUCCESS. Auditoría y compliance automatizada.

def create_webhook_subscription(headers, name, webhook_url):

        {
            "name": name,
            "description": "Suscripción a alarmas de red",
            "filter": {
                "eventIds": ["NETWORK-API-ERROR", "LINK-FAILURE"]
            },
            "subscriptionEndpoints": [
                {
                    "instanceId": "webhook-01",
                    "subscriptionDetails": {
                        "connectorType": "REST",
                        "name": "MiWebhook",
                        "description": "Recibe alarmas en mi app",
                        "url": webhook_url,
                        "method": "POST",
                        "trustCert": False,
                        "headers": [
                            {"name": "Content-Type", "value": "application/json"}
                        ]
                    }
                }
            ]
        }
    ]

    r = requests.post(url, headers=headers, json=payload, verify=False)
    print(f"🔔 Suscripción creada: {r.status_code}")
    return r.json()

# Tu endpoint receptor (ej. Flask/FastAPI) recibirá:

⚡ Seguridad en Webhooks

Tu endpoint expuesto en Internet debe validar la autenticidad del emisor. DNA Center permite incluir headers personalizados (ej. X-Webhook-Secret) que tu aplicación debe verificar. Nunca proceses un webhook sin validar firma o secret, o podrías recibir payloads falsificados.

dna_center_dashboard.py — Orquestador completo
#!/usr/bin/env python3
"""
dna_center_dashboard.py
CCNA Academy M8 T4 — Orquestador DNA Center Intent API
────────────────────────────────────────────────────────
- Autenticación con re-intento automático
- Inventario con exportación CSV
- Network & Client Health
- Command Runner para show version
- Logging profesional y manejo de errores
"""

import os
import csv
import sys
import json
import time
import logging
import requests
import urllib3
from datetime import datetime
from requests.auth import HTTPBasicAuth

urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

# ══════════════════════════════════════════════════════
# CONFIGURACIÓN
# ══════════════════════════════════════════════════════
BASE_URL      = os.getenv("DNAC_URL",      "https://sandboxdnac.cisco.com")
USERNAME      = os.getenv("DNAC_USER",     "devnetuser")
PASSWORD      = os.getenv("DNAC_PASS",     "Cisco123!")
COMMANDS      = ["show version"]
REPORT_DIR    = "./reports"

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%H:%M:%S"
)

# ══════════════════════════════════════════════════════
# CLIENTE DNA CENTER
# ══════════════════════════════════════════════════════
class DNACenterClient:
    def __init__(self):
        self.base_url = BASE_URL
        self.auth     = HTTPBasicAuth(USERNAME, PASSWORD)
        self.headers  = {"Content-Type": "application/json", "Accept": "application/json"}
        self._authenticate()

    def _authenticate(self):
        endpoint = "/dna/system/api/v1/auth/token"
        r = requests.post(f"{self.base_url}{endpoint}", auth=self.auth, verify=False)
        r.raise_for_status()
        self.headers["X-Auth-Token"] = r.json()["Token"]
        logging.info("🔐 Autenticado contra DNA Center")

    def request(self, method, endpoint, **kwargs):
        url = f"{self.base_url}{endpoint}"
        kwargs.setdefault("headers", {}).update(self.headers)
        kwargs["verify"] = False

        resp = requests.request(method, url, **kwargs)
        if resp.status_code == 401:
            logging.warning("🔄 Token expirado. Re-autenticando...")
            self._authenticate()
            kwargs["headers"].update(self.headers)
            resp = requests.request(method, url, **kwargs)
        resp.raise_for_status()
        return resp

# FUNCIONES DE NEGOCIO

def export_inventory_csv(api):
    logging.info("📦 Obteniendo inventario...")
    r = api.request("GET", "/dna/intent/api/v1/network-device")
    devices = r.json()["response"]

    os.makedirs(REPORT_DIR, exist_ok=True)
    fname = f"{REPORT_DIR}/inventory_{datetime.now():%Y%m%d_%H%M%S}.csv"
    with open(fname, "w", newline="") as f:
        writer = csv.writer(f)
        writer.writerow(["Hostname", "IP", "Platform", "SW Version", "Reachability"])
        for d in devices:
            writer.writerow([
                d.get("hostname"),
                d.get("managementIpAddress"),
                d.get("platformId"),
                d.get("softwareVersion"),
                d.get("reachabilityStatus")
            ])
    logging.info(f"✅ Inventario exportado: {fname}")
    return devices

def print_health_summary(api):
    logging.info("❤️  Obteniendo health metrics...")
    r = api.request("GET", "/dna/intent/api/v1/network-health")
    net = r.json()["response"][0]

    r2 = api.request("GET", "/dna/intent/api/v1/client-health")
    cli = r2.json()["response"]

    print("\n╔════════════════════════════════════════════╗")
    print("║     DNA CENTER DAILY HEALTH REPORT         ║")
    print("╠════════════════════════════════════════════╣")
    print(f"║ Network Score : {net.get('healthScore','N/A'):<6}/10               ║")
    print(f"║ Devices Total : {net.get('totalCount',0):<6}                 ║")
    print(f"║ Devices Poor  : {net.get('poorCount',0):<6}                 ║")
    total_clients = sum(c.get("clientCount",0) for c in cli)
    print(f"║ Total Clients : {total_clients:<6}                 ║")
    print("╚════════════════════════════════════════════╝\n")

def run_commands_on_cores(api, devices):
    cores = [d for d in devices if "CORE" in d.get("hostname", "").upper()]
    if not cores:
        logging.warning("No se encontraron dispositivos con 'CORE' en el hostname")
        return

    for dev in cores:
        uuid = dev["id"]
        ip   = dev.get("managementIpAddress")
        logging.info(f"⌨️  Command Runner en {dev['hostname']} ({ip})")

        endpoint = "/dna/intent/api/v1/network-device-poller/cli/read-request"
        payload  = {"commands": COMMANDS, "deviceUuids": [uuid], "timeout": 30}
        r = api.request("POST", endpoint, json=payload)
        task_id  = r.json()["response"]["taskId"]

        # Polling simple
        for _ in range(15):
            time.sleep(1)
            tr = api.request("GET", f"/dna/intent/api/v1/task/{task_id}")
            task = tr.json()["response"]
            if task.get("endTime"):
                progress = json.loads(task.get("progress", "{}"))
                file_id  = progress.get("fileId")
                if file_id:
                    fr = api.request("GET", f"/dna/intent/api/v1/file/{file_id}")
                    result = fr.json()[0]
                    version_output = result.get("commandResponses", {}).get("SUCCESS", {}).get("show version", "N/A")
                    logging.info(f"   📄 Versión: {version_output[:60]}...")
                break

# ══════════════════════════════════════════════════════

def main():

    api = DNACenterClient()

    devices = export_inventory_csv(api)
    print_health_summary(api)
    run_commands_on_cores(api, devices)

    logging.info("🏁 Proceso completado exitosamente.")

if __name__ == "__main__":
    try:
        main()
