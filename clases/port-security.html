Port Security

CAM Flooding · Modos de Violación · Sticky MAC · Err-Disabled · DHCP Snooping · DAI · Hardening

CCNA 200-301 Módulo 3 · Switching 🔒 Seguridad Capa 2

⚠️ El problema — Ataques a la Capa 2

La Capa 2 (Enlace de Datos) es la base de las redes LAN y, paradójicamente, la capa menos protegida en muchas organizaciones. Un atacante con acceso físico a un puerto del switch puede explotar vulnerabilidades que afectan a toda la red, independientemente de cuán bien esté protegida la Capa 3.

La razón: los switches confían ciegamente en las direcciones MAC — no verifican identidades, no requieren autenticación de Capa 2. Cualquier dispositivo conectado puede enviar frames con cualquier MAC origen.

💀 CAM Table Overflow

El atacante inunda el switch con miles de MACs falsas. La tabla CAM se llena. El switch empieza a inundar (flooding) todo el tráfico unicast a todos los puertos — se convierte en un hub. El atacante captura todas las conversaciones.

🔄 MAC Spoofing

El atacante clona la dirección MAC de un dispositivo legítimo. El switch actualiza su tabla CAM apuntando esa MAC al puerto del atacante. El tráfico destinado al dispositivo legítimo llega al atacante.

🎭 Rogue DHCP Server

El atacante conecta un servidor DHCP no autorizado. Los clientes reciben IPs con gateway falso (el propio atacante). Todo el tráfico pasa por el atacante — Man-in-the-Middle perfecto.

🔀 ARP Spoofing / Poisoning

El atacante envía respuestas ARP falsas asociando su MAC con la IP del gateway. Los hosts actualizan su caché ARP y envían todo el tráfico al atacante en lugar del router real.

💀 CAM Table Overflow Attack — Disección técnica

Este ataque es la razón principal de existencia de Port Security. Merece un análisis profundo.

¿Cómo funciona el ataque paso a paso?

📋 El mecanismo del ataque

  1. El atacante usa herramientas como macof (Linux) que genera y envía frames Ethernet con MACs origen completamente aleatorias y diferentes en cada frame.
  2. El switch recibe cada frame, lee la MAC origen y la agrega a la tabla CAM: [MAC_falsa_1 → Fa0/1], [MAC_falsa_2 → Fa0/1], [MAC_falsa_3 → Fa0/1]…
  3. La tabla CAM de un switch típico tiene capacidad para 4.000 a 128.000 entradas. macof puede generar ~155.000 entradas/minuto.
  4. La tabla se llena completamente en segundos. El switch ya no puede aprender nuevas MACs legítimas.
  5. Cuando llega un frame destinado a una MAC que ya no está en la tabla (porque fue desplazada), el switch realiza flooding a todos los puertos.
  6. El atacante, en ese mismo puerto, recibe copias de toda la comunicación unicast de la red — el switch actúa como un hub.

🔒 Port Security — Definición y mecanismo

Port Security es una función de Capa 2 de Cisco IOS que permite al administrador controlar qué dispositivos (MACs) pueden conectarse a cada puerto del switch, limitando el número de MACs permitidas y definiendo la acción a tomar cuando se detecta una violación.

🔑 Los tres elementos de Port Security

  1. MACs seguras: las direcciones MAC que tienen permitido usar el puerto. Pueden ser estáticas (configuradas manualmente), dinámicas (aprendidas automáticamente) o sticky (aprendidas y guardadas como estáticas).
  2. Máximo de MACs: el número máximo de MACs seguras que puede tener el puerto simultáneamente. Default = 1.
  3. Modo de violación: la acción que toma el switch cuando se detecta una MAC no autorizada. Opciones: Shutdown, Restrict, Protect.

✅ Requisito previo obligatorio — El error #1

⚠️ Port Security solo funciona en puertos de acceso o trunk ESTÁTICOS

No puede activarse en puertos con modo DTP dinámico (dynamic auto o dynamic desirable). Si intentas activar Port Security en un puerto dinámico, IOS devolverá este error:

! El puerto está en modo dynamic auto (default)
SW1(config)# interface FastEthernet0/1
SW1(config-if)# switchport port-security
Command rejected: FastEthernet0/1 is a dynamic port.
! ← Error: no se puede en puerto dinámico

! Solución: configurar el puerto como estático primero
SW1(config-if)# switchport mode access
SW1(config-if)# switchport port-security
! Ahora sí funciona ✅

🚨 Modos de violación — La decisión más crítica

Una violación ocurre cuando el switch detecta alguna de estas condiciones en un puerto con Port Security activo:

  • Una MAC origen que no está en la lista de MACs seguras intenta usar el puerto
  • Una MAC segura del puerto intenta usar un puerto diferente
  • Se supera el número máximo de MACs permitidas (maximum)

⚠️ Trampa del examen: Restrict vs Protect

Ambos modos descartan el tráfico violador y mantienen el puerto activo. La diferencia clave es:

  • Restrict: incrementa el contador + genera syslog/SNMP. El administrador sabe que hubo una violación.
  • Protect: solo descarta. Sin contador, sin syslog, sin SNMP. El administrador NO sabe que hubo una violación.

Protect es prácticamente inútil en producción porque no hay trazabilidad. El examen pregunta esta diferencia frecuentemente.

Port Security permite configurar un tiempo de envejecimiento para las MACs seguras. Esto es útil en entornos donde los dispositivos cambian frecuentemente (salas de reuniones, hotdesking).

📌 Sticky MAC — El aprendizaje inteligente

Sticky MAC es la característica más elegante de Port Security. Combina lo mejor de las MACs estáticas (persistencia) con la comodidad de las dinámicas (aprendizaje automático): el switch aprende las MACs de los dispositivos que se conectan y las guarda automáticamente como si fueran estáticas en la running-config.

🔑 El proceso Sticky MAC paso a paso

  1. El administrador habilita Port Security con mac-address sticky
  2. El primer dispositivo se conecta al puerto y envía un frame
  3. El switch aprende la MAC origen del frame
  4. En lugar de guardarla como MAC dinámica (volátil), la convierte en una entrada SecureSticky y la escribe en la running-config
  5. La entrada aparece en show running-config como switchport port-security mac-address sticky XXXX.XXXX.XXXX
  6. Si el administrador hace copy running-config startup-config, la MAC persiste al reiniciar
  7. Si conecta un segundo dispositivo diferente → violación según el modo configurado

🔴 Err-Disabled — El estado de cuarentena

Err-disabled es un estado especial de las interfaces Cisco que indica que el switch desactivó el puerto automáticamente como respuesta a un evento de error o seguridad. El puerto queda completamente inactivo hasta que el administrador lo recupere manualmente — o se configure la recuperación automática.

⚠️ Err-disabled es diferente a "shutdown"

  • Shutdown: el administrador apagó el puerto manualmente con shutdown. Se recupera con no shutdown.
  • Err-disabled: el switch apagó el puerto automáticamente por un error. no shutdown solo puede revertirlo si primero se hace shutdown. La causa debe investigarse y corregirse antes de recuperarlo.

🛡️ DHCP Snooping — Autenticación del servidor DHCP

DHCP Snooping es un mecanismo de seguridad que actúa como un filtro inteligente de mensajes DHCP. Clasifica cada puerto como trusted (confiable) o untrusted (no confiable) y decide qué mensajes DHCP puede enviar cada uno.

🔑 La lógica trusted/untrusted

  • Trusted: puerto hacia un servidor DHCP legítimo, router o switch de confianza. Puede enviar cualquier mensaje DHCP (Offer, ACK, NAK).
  • Untrusted: puerto hacia hosts clientes. Solo puede enviar mensajes DHCP iniciados por el cliente (Discover, Request, Release, Decline). Si envía un Offer o ACK → el switch lo descarta y puede poner el puerto en err-disabled.

⚠️ Ataque que DHCP Snooping previene: Rogue DHCP Server

Un atacante conecta un servidor DHCP no autorizado. Los clientes envían DHCP Discover (broadcast). El servidor legítimo Y el rogue server responden con DHCP Offer. El cliente acepta la oferta más rápida — que puede ser del atacante. El atacante asigna una IP con su propio gateway → Man-in-the-Middle: todo el tráfico pasa por el atacante.

Con DHCP Snooping: el Offer del rogue server llega por un puerto untrusted → el switch lo descarta → el cliente solo recibe la oferta del servidor legítimo (desde el puerto trusted).

📋 Binding Table — El registro forense de DHCP Snooping

DHCP Snooping mantiene una tabla de enlace (binding table) que registra cada asignación DHCP exitosa. Esta tabla es fundamental porque DAI la usa para validar las respuestas ARP.

📌 Contenido de la binding table

Para cada cliente DHCP: MAC Address | IP Address | Lease Time | Tipo | VLAN | Interface

Se actualiza en tiempo real con cada DHCP ACK. Cuando un lease expira o el cliente envía DHCP Release, la entrada se elimina.

⚙️ Configuración completa de DHCP Snooping

Configuración paso a paso

! Paso 1: Habilitar DHCP Snooping globalmente
SW1(config)# ip dhcp snooping
! ⚠️ Por sí solo no hace nada — necesitas especificar las VLANs

! Paso 2: Habilitar en las VLANs que tienen clientes DHCP
SW1(config)# ip dhcp snooping vlan 10,20,30
! Solo en estas VLANs se aplica DHCP Snooping

! Paso 3: Marcar el puerto hacia el servidor DHCP como trusted
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# ip dhcp snooping trust
! Este es el uplink hacia el servidor DHCP legítimo o el router
! TODOS los demás puertos quedan untrusted por defecto

! Paso 4 (opcional): Limitar la tasa de mensajes DHCP por puerto
SW1(config)# interface range FastEthernet0/1-20
SW1(config-if-range)# ip dhcp snooping limit rate 15
! Máximo 15 mensajes DHCP por segundo — previene DHCP Starvation
! Si se supera → err-disabled por dhcp-rate-limit

! Paso 5 (opcional): Deshabilitar la inserción de option 82
SW1(config)# no ip dhcp snooping information option
! Option 82 puede causar problemas con algunos servidores DHCP
! Deshabilitarlo si el servidor DHCP no lo soporta

! La binding table registra MAC → IP → Interface → VLAN

🔍 DAI — Dynamic ARP Inspection

DAI (Dynamic ARP Inspection) protege contra los ataques de ARP Spoofing/Poisoning — uno de los ataques Man-in-the-Middle más comunes en redes LAN. Valida cada respuesta ARP comparándola con la tabla de DHCP Snooping binding.

⚠️ ¿Cómo funciona el ataque ARP Spoofing?

  1. PC-A (192.168.1.10) quiere hablar con el Gateway (192.168.1.1). Envía ARP Request: "¿Quién tiene 192.168.1.1?"
  2. El Gateway responde: "Yo soy 192.168.1.1, mi MAC es AA:BB:CC:DD:EE:FF"
  3. El Atacante (192.168.1.99) también responde (gratuitous ARP): "Yo soy 192.168.1.1, mi MAC es 00:11:22:33:44:55"
  4. PC-A actualiza su tabla ARP con la MAC falsa del atacante para la IP del Gateway
  5. Todo el tráfico de PC-A hacia el Gateway va ahora al atacante → Man-in-the-Middle
  6. El atacante reenvía el tráfico al Gateway real → la comunicación parece normal

⚙️ Cómo DAI valida los mensajes ARP

DAI intercepta todas las respuestas ARP (ARP Reply y gratuitous ARP) en los puertos untrusted y las valida contra la tabla de DHCP Snooping binding:

Condición de la respuesta ARPResultado DAI
MAC en el ARP Reply coincide con la MAC en la binding table para esa IP✅ Permitido — el paquete se reenvía
MAC en el ARP Reply NO coincide con la binding table❌ Descartado — posible ARP Spoofing
La IP no está en la binding table (no obtuvo IP por DHCP)❌ Descartado (a menos que haya una entrada ARP estática)
El ARP llega por un puerto trusted✅ Siempre permitido sin validación

📌 DAI requiere DHCP Snooping — Son complementarios

DAI usa la binding table de DHCP Snooping como su base de datos de validación. Sin DHCP Snooping activo (o sin entradas en la binding table), DAI descartará los ARPs de todos los hosts aunque sean legítimos. Se deben configurar ambos juntos.

Excepción: se pueden agregar entradas ARP estáticas manualmente con arp access-list para hosts con IP estática que no aparecen en la binding table de DHCP.

Configuración DAI sobre DHCP Snooping

SW1(config-nacl)# exit
SW1(config)# ip arp inspection filter STATIC-HOSTS vlan 10

📋 Resumen Maestro — Port Security para el examen CCNA

✅ Los 28 puntos más críticos del examen

  1. CAM Flooding: la tabla CAM se llena con MACs falsas → switch actúa como hub → atacante captura tráfico.
  2. Port Security previene CAM Flooding y dispositivos no autorizados controlando qué MACs pueden usar el puerto.
  3. Port Security solo funciona en puertos access o trunk estáticos. No en dynamic auto/desirable.
  4. Los 3 tipos de MAC segura: Estática (manual), Dinámica (volátil), Sticky (aprendida y guardada).
  5. Valores por defecto: máximo=1 MAC, violación=Shutdown, tipo=dinámica.
  6. Shutdown: puerto → err-disabled. Contador sube. Syslog. Requiere recuperación.
  7. Restrict: descarta frame violador. Puerto activo. Contador sube. Syslog. ✅ Más útil en producción.
  8. Protect: descarta frame violador. Puerto activo. Contador NO sube. Sin Syslog. Silencioso.
  9. Restrict vs Protect: ambos descartan y mantienen el puerto activo. Solo Restrict genera log/contador.
  10. Sticky MAC: se guarda en running-config como mac-address sticky XXXX. Requiere copy run start para persistir.
  11. Si no haces copy run start, las sticky MACs se pierden al reiniciar el switch.
  12. Err-disabled: estado de cuarentena automático. Diferente a shutdown manual. Requiere ciclo shutdown/no shutdown para recuperar.
  13. errdisable recovery cause psecure-violation: recuperación automática configurable.
  14. errdisable recovery interval X: tiempo antes de la recuperación automática (default 300s).
  15. show port-security: resumen. show port-security interface X: detalle del puerto.
  16. show port-security address: todas las MACs seguras con su tipo (SecureSticky, SecureConfigured).
  17. DHCP Snooping: clasifica puertos trusted/untrusted. Previene rogue DHCP servers.
  18. Trusted = puerto hacia servidor DHCP legítimo. Untrusted (default) = hacia clientes.
  19. DHCP Offer/ACK desde puerto untrusted → descartado. Solo se permite desde trusted.
  20. DHCP Snooping binding table: MAC → IP → Interface → VLAN. La usa DAI para validación.
  21. ip dhcp snooping limit rate X: previene DHCP Starvation limitando mensajes/segundo.
  22. no ip dhcp snooping information option: desactivar option 82 si causa problemas.
  23. DAI: valida ARP Replies contra la DHCP Snooping binding table. Previene ARP Spoofing.
  24. DAI requiere DHCP Snooping activo (o entradas ARP estáticas) para funcionar correctamente.
  25. El puerto trusted de DAI debe coincidir con el de DHCP Snooping.
  26. ARP desde untrusted con MAC/IP que no coincide en binding table → descartado.
  27. Hosts con IP estática: necesitan arp access-list para que DAI no descarte sus ARPs.
  28. Orden de hardening: Port Security → DHCP Snooping → DAI. Son capas complementarias.

⚠️ Los 12 errores más comunes en el examen CCNA

  1. Intentar activar Port Security en un puerto dynamic auto → error "Command rejected"
  2. Creer que Protect genera syslog o incrementa el contador → NO, es completamente silencioso
  3. Creer que Restrict y Protect bloquean el puerto → NO, el puerto sigue activo
  4. No hacer copy run start después de que sticky aprende MACs → se pierden al reiniciar
  5. Confundir err-disabled con administratively down → son causas diferentes, recuperación diferente
  6. Olvidar que errdisable recovery con no shutdown sin shutdown previo no funciona
  7. Activar DHCP Snooping globalmente sin especificar las VLANs → no hace nada
  8. Marcar el puerto de usuario como trusted en DHCP Snooping → el punto del ataque
  9. Activar DAI sin tener DHCP Snooping binding table → descarta todos los ARPs legítimos
  10. No configurar arp access-list para hosts con IP estática cuando DAI está activo
  11. Usar el mismo puerto trusted para DAI y distinto para DHCP Snooping → inconsistencia
  12. Creer que Port Security reemplaza a DHCP Snooping o DAI → son complementarios
