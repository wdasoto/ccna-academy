📚 La Lógica Profunda de OSPF — Por qué funciona como funciona

OSPFv2 (Open Shortest Path First v2, RFC 2328) es un protocolo de routing de estado de enlace (link-state). La clave para comprender OSPF profundamente no es memorizar comandos, sino entender su filosofía: cada router construye un mapa completo de la red y calcula el camino óptimo de forma independiente. Esto lo hace fundamentalmente diferente de los protocolos distance-vector como RIP o EIGRP.

⭕ La Analogía del Mapa de la Ciudad

Imagina que eres un taxista. En un sistema distance-vector (RIP), solo conoces los tiempos de viaje que te dicen tus vecinos — no sabes los detalles del mapa. En OSPF, tienes el mapa completo de toda la ciudad. Conoces cada calle, cada semáforo (costo de enlace). Puedes calcular tú mismo la ruta más corta sin depender de lo que te digan los vecinos. Esto elimina los bucles de routing y converge mucho más rápido.

🧮 El Algoritmo SPF de Dijkstra — Cómo OSPF elige la mejor ruta

📌 El Proceso SPF Paso a Paso

  1. Punto de partida: el propio router es la raíz del árbol SPF. Costo hacia sí mismo = 0.
  2. Poblar el árbol: OSPF examina la LSDB y añade todos los vecinos directos al árbol con su costo de enlace correspondiente.
  3. Selección del nodo de menor costo: de todos los nodos pendientes, se selecciona el de menor costo acumulado desde la raíz.
  4. Expansión: se examinan los vecinos del nodo seleccionado. Si el costo acumulado hasta un vecino a través de este nodo es menor que el conocido actualmente, se actualiza.
  5. Repetición: se repite hasta que todos los nodos de la red estén en el árbol SPF.
  6. Resultado: el árbol SPF define la ruta de menor costo hacia cada red. Estas rutas se instalan en la tabla de routing como entradas tipo "O".

📌 Costo OSPF — La Métrica que Importa

El costo OSPF se calcula como: Costo = 100,000,000 / Bandwidth(bps)

El costo es acumulativo a lo largo del camino — se suman los costos de todos los enlaces hacia el destino.

⚠️ El Problema del Reference Bandwidth por Defecto

Con el reference bandwidth por defecto (100 Mbps), FastEthernet, GigabitEthernet y 10GE tienen todos costo 1. OSPF no puede distinguir entre ellos y puede elegir un camino subóptimo. Solución profesional: ajustar el reference bandwidth a 10 Gbps o superior en TODOS los routers del dominio OSPF.

📌 En OSPFv2 Single-Area — Solo LSAs Tipo 1 y 2

En una topología single-area (solo Área 0), solo existen LSAs Tipo 1 (Router LSA de cada router) y Tipo 2 (Network LSA del DR en cada segmento Ethernet). Todos los routers del área tienen exactamente la misma LSDB. Si dos routers tienen LSDBs diferentes, hay un problema grave de convergencia.

Antes de configurar OSPF, cada router debe tener una configuración base segura. En producción, un router sin hardening es una vulnerabilidad crítica. Estos pasos son la primera línea de defensa y una buena práctica que el CCNA evalúa.

⚙️ Configuración OSPFv2 — Paso a Paso con Justificación

1 R1 — Proceso OSPF Completo con Mejores Prácticas
! ══════════════════════════════════════════
! FASE OSPF — R1-CENTRAL
! ══════════════════════════════════════════

R1-CENTRAL(config)# router ospf 1
! Process ID = 1 (local al router — NO necesita coincidir con vecinos)
! Rango válido: 1-65535. Usar el mismo en todos para coherencia administrativa.

R1-CENTRAL(config-router)# router-id 1.1.1.1
! *** CRÍTICO: Configurar Router ID MANUALMENTE siempre ***
! Si no se configura: IOS usa la IP más alta de Loopback o interfaz activa
! Un cambio inesperado de RID reinicia TODAS las adyacencias OSPF
! Convención: RID = IP de Loopback = 1.1.1.1 para R1

R1-CENTRAL(config-router)# auto-cost reference-bandwidth 10000
! Ajustar el bandwidth de referencia a 10 Gbps
! *** OBLIGATORIO en todos los routers del área ***
! Con el default (100 Mbps): GigabitEthernet y FastEthernet tienen costo 1
! Con 10000 Mbps: GigabitEthernet=10, FastEthernet=100 (diferencia real)
! Si no se configura igual en todos, el SPF calcula rutas incorrectas

! ─── Anunciar redes en OSPFv2 ───────────────────────
R1-CENTRAL(config-router)# network 1.1.1.1 0.0.0.0 area 0
! Anunciar la Loopback0 en OSPF — RID y ruta de gestión
! Wildcard 0.0.0.0 = solo esa IP exacta

R1-CENTRAL(config-router)# network 10.10.1.0 0.0.0.127 area 0
! LAN HQ /25: wildcard = 255.255.255.255 - 255.255.255.128 = 0.0.0.127

R1-CENTRAL(config-router)# network 10.10.10.0 0.0.0.3 area 0
! WAN R1-R2 /30: wildcard = 0.0.0.3

R1-CENTRAL(config-router)# network 10.10.10.12 0.0.0.3 area 0
! WAN R1-R4 /30: inicia en .12

! ─── Passive-Interface — Mejores Prácticas ───────────
R1-CENTRAL(config-router)# passive-interface default
! Poner TODAS las interfaces en modo pasivo primero
! Passive-interface: la red se anuncia en OSPF pero NO se envían Hellos
! Esto previene que el router intente formar adyacencias en la LAN de usuarios

R1-CENTRAL(config-router)# no passive-interface GigabitEthernet0/1
! Activar Hellos SOLO en la interfaz WAN hacia R2
R1-CENTRAL(config-router)# no passive-interface GigabitEthernet0/2
! Activar Hellos en la interfaz WAN hacia R4
! Gi0/0 (LAN HQ) permanece pasiva — no hay routers OSPF en la LAN de usuarios

! ─── Ajuste de Timers para Convergencia Rápida ──────
R1-CENTRAL(config-router)# timers throttle spf 50 100 5000
! Ajuste agresivo del timer SPF para convergencia rápida:
! 50 ms  = delay inicial antes de ejecutar SPF
! 100 ms = delay mínimo entre ejecuciones SPF consecutivas
! 5000 ms = delay máximo (backoff exponencial)

! ─── Prioridad DR/BDR en Interfaces WAN ──────────────
! En interfaces P2P el DR/BDR no aplica, pero en Ethernet sí:
R1-CENTRAL(config-router)# end
R1-CENTRAL(config)# interface GigabitEthernet0/1
R1-CENTRAL(config-if)# ip ospf priority 255
! Máxima prioridad → R1 siempre será DR en este segmento
! Prioridad 255 = candidato preferido siempre
! Prioridad 0 = nunca participa en la elección DR/BDR

R1-CENTRAL(config-if)# ip ospf hello-interval 5
! Hello cada 5 segundos (default Ethernet = 10s)
! *** Ambos extremos del enlace deben tener el mismo timer ***
R1-CENTRAL(config-if)# ip ospf dead-interval 20
! Dead interval = 4x Hello (20s). Si no hay Hello en 20s → vecino caído
! *** Debe coincidir exactamente con el vecino ***

! ─── Autenticación MD5 en los enlaces OSPF ───────────
R1-CENTRAL(config-if)# ip ospf authentication message-digest
R1-CENTRAL(config-if)# ip ospf message-digest-key 1 md5 OspfKey@TC2024
! MD5 autentica los paquetes OSPF — previene routers no autorizados
! Key ID (1) y contraseña DEBEN coincidir en ambos extremos del enlace
! Sin autenticación: cualquier router OSPF puede inyectar rutas falsas

R1-CENTRAL(config)# interface GigabitEthernet0/2
R1-CENTRAL(config-if)# ip ospf priority 255
R1-CENTRAL(config-if)# ip ospf hello-interval 5
R1-CENTRAL(config-if)# ip ospf dead-interval 20
R1-CENTRAL(config-if)# ip ospf authentication message-digest
R1-CENTRAL(config-if)# ip ospf message-digest-key 1 md5 OspfKey@TC2024
2 R2, R3, R4 — Configuración OSPF (Resumen)
! ─── R2-NORTE ────────────────────────────────────────

🚨 Troubleshooting Avanzado — Escenarios Reales

El troubleshooting de OSPF requiere un método sistemático. No se puede resolver un problema de OSPF al azar — hay que seguir la jerarquía: conectividad física → adyacencia → LSDB → routing table → conectividad extremo a extremo.

⚠️ Escenario 1: La Adyacencia Nunca Pasa de INIT — ¿Por Qué?

Síntoma: show ip ospf neighbor muestra al vecino en estado INIT permanente.

Causa: Comunicación unidireccional — R1 ve los Hellos de R2 pero R2 no ve los de R1 (o viceversa).

! Paso 1: Verificar que los Hellos llegan
R1-CENTRAL# debug ip ospf hello
OSPF-1 HELLO Gi0/1: Rcv hello from 2.2.2.2 area 0 10.10.10.2
OSPF-1 HELLO Gi0/1: Rcv hello from 2.2.2.2 area 0 10.10.10.2
! Si se reciben Hellos pero el estado sigue en INIT:
! R1 NO aparece en la lista de vecinos del Hello de R2
! → R2 no está recibiendo los Hellos de R1

! Paso 2: Verificar que R1 SÍ envía Hellos
R1-CENTRAL# debug ip ospf packet
OSPF-1 PAK  Gi0/1: Send with youngest Key 1
! Hellos se envían → el problema está en que R2 los descarta

! Paso 3: Verificar posibles causas
R1-CENTRAL# show ip ospf interface GigabitEthernet0/1
! Buscar: Cryptographic authentication enabled? Key ID? Area?
! Si hay ACL bloqueando en la interfaz de R2 → los Hellos de R1 no llegan

! *** SIEMPRE apagar debug después ***
R1-CENTRAL# undebug all

Soluciones posibles: ACL bloqueando tráfico multicast 224.0.0.5 en un sentido, IPs en subredes diferentes, interfaz de R2 no configurada con OSPF.

⚠️ Escenario 2: Adyacencia en EXSTART/EXCHANGE — MTU Mismatch

Síntoma: La adyacencia oscila entre EXSTART y EXCHANGE, nunca llega a FULL.

! Síntoma en show ip ospf neighbor
R1-CENTRAL# show ip ospf neighbor
Neighbor ID  Pri  State      Dead Time  Address      Interface
2.2.2.2       1   EXSTART/-  00:00:35   10.10.10.2   Gi0/1
! Estado EXSTART = negociación Master/Slave y secuencias
! Si no avanza a EXCHANGE → MTU mismatch es la causa más común

! Verificar el MTU de las interfaces en ambos routers
R1-CENTRAL# show interfaces GigabitEthernet0/1 | include MTU
  MTU 1500 bytes
R2-NORTE# show interfaces GigabitEthernet0/1 | include MTU
  MTU 1480 bytes   ← MTU diferente → causa del EXSTART

! Solución 1: Igualar el MTU
R2-NORTE(config)# interface GigabitEthernet0/1
R2-NORTE(config-if)# mtu 1500

! Solución 2 (parche, no recomendado): ignorar el MTU en OSPF
R2-NORTE(config-if)# ip ospf mtu-ignore
! ip ospf mtu-ignore: OSPF ignora el MTU durante el intercambio DBD
! Solo usar si no puedes cambiar el MTU (GRE tunnels, por ejemplo)

⚠️ Escenario 3: Adyacencia FULL pero Faltan Rutas — Router ID Duplicado

Síntoma: Las adyacencias están en FULL pero la tabla de routing de algún router tiene rutas incorrectas o faltan destinos.

! Detectar Router ID duplicado
R1-CENTRAL# show ip ospf
 Routing Process "ospf 1" with ID 1.1.1.1
R3-SUR# show ip ospf
 Routing Process "ospf 1" with ID 1.1.1.1   ← DUPLICADO ← PROBLEMA

! En el log de R1 aparecerá:
%OSPF-4-DUP_RTRID: Detected router with duplicate router ID 1.1.1.1

! Solución: cambiar el Router ID y reiniciar el proceso
R3-SUR(config)# router ospf 1
R3-SUR(config-router)# router-id 3.3.3.3
R3-SUR(config-router)# end
R3-SUR# clear ip ospf process
Reset OSPF processes? [no]: yes
! ⚠️ clear ip ospf process reinicia TODAS las adyacencias de ese router
! En producción: coordinar con el equipo antes de ejecutar

⚠️ Escenario 4: Timers Hello/Dead No Coinciden

! show ip ospf interface revela los timers configurados
R1-CENTRAL# show ip ospf interface GigabitEthernet0/1
  Timer intervals configured, Hello 5, Dead 20
R2-NORTE# show ip ospf interface GigabitEthernet0/1
  Timer intervals configured, Hello 10, Dead 40   ← Diferente
! OSPF no forma adyacencia si Hello o Dead timers difieren
! En el debug: "Dead timer mismatch" en los logs

! Solución: igualar timers en ambos extremos
R2-NORTE(config)# interface GigabitEthernet0/1
R2-NORTE(config-if)# ip ospf hello-interval 5
R2-NORTE(config-if)# ip ospf dead-interval 20

⚠️ Escenario 5: Autenticación Falla — Adyacencia No Se Establece

! El debug muestra rechazos de autenticación
R1-CENTRAL# debug ip ospf adj
OSPF-1 ADJ Gi0/1: Rcv DBD from 2.2.2.2 with invalid auth
OSPF-1 ADJ Gi0/1: Drop due to auth error

! Verificar la configuración de autenticación
R1-CENTRAL# show ip ospf interface GigabitEthernet0/1 | include auth
  Message digest authentication enabled
    Youngest key id is 1
R2-NORTE# show ip ospf interface GigabitEthernet0/1 | include auth
  Simple password authentication enabled   ← Tipo de auth diferente
! R1 usa MD5, R2 usa simple password → incompatibles

! Solución: igualar tipo y contraseña
R2-NORTE(config)# interface GigabitEthernet0/1
R2-NORTE(config-if)# no ip ospf authentication
R2-NORTE(config-if)# ip ospf authentication message-digest
R2-NORTE(config-if)# ip ospf message-digest-key 1 md5 OspfKey@TC2024
R2-NORTE(config-if)# end
R1-CENTRAL# undebug all

🏆 Desafío de Consolidación — Extiende la Red TecnoCorp

🎯 Misión: Integrar R5 como Router de Internet con Default Route OSPF

TecnoCorp ha contratado un ISP. Debes integrar un nuevo router R5 conectado a R1, configurarlo como ASBR (AS Boundary Router), inyectar la default route hacia OSPF y verificar que todos los routers internos puedan llegar a "Internet" (simulado por Loopback8.8.8.8 en R5).

📋 Escenario del Desafío

📡 R5 — Router ISP/ASBR

Cisco 2911
Gi0/0 → Enlace R1-R5 WAN
Lo0 = 5.5.5.5/32 (RID)
Lo8 = 8.8.8.8/32 (simula Internet)

🔗 Nuevo Enlace WAN

R1 Gi0/3 ↔ R5 Gi0/0
Red: 10.10.10.16/30
R1 = .17 (ASBR interno)
R5 = .18 (Router ISP)

🎯 Objetivo de Verificación

ping 8.8.8.8 desde PC-HQ ✅
ping 8.8.8.8 desde PC-NORTE ✅
ping 8.8.8.8 desde PC-DC ✅
Ruta O*E2 en todos ✅

⚙️ Solución del Desafío — Paso a Paso

1 Configurar R5 como Router ISP
! R5 NO participará en OSPF — es el "ISP externo"
R5-ISP(config)# hostname R5-ISP
R5-ISP(config)# interface Loopback0
R5-ISP(config-if)# ip address 5.5.5.5 255.255.255.255
R5-ISP(config)# interface Loopback8
R5-ISP(config-if)# ip address 8.8.8.8 255.255.255.255
! Simula el servidor DNS de Google — objetivo del desafío
R5-ISP(config)# interface GigabitEthernet0/0
R5-ISP(config-if)# description ENLACE-HACIA-R1-TECNOCORP
R5-ISP(config-if)# ip address 10.10.10.18 255.255.255.252
R5-ISP(config-if)# no shutdown

! R5 necesita ruta de retorno hacia la red interna de TecnoCorp
! (o bien una ruta resumen, o una default route dirigida a R1)
R5-ISP(config)# ip route 10.10.0.0 255.255.0.0 10.10.10.17
! Ruta resumida hacia toda la red 10.10.0.0/16 de TecnoCorp vía R1
2 Configurar R1 como ASBR — Inyectar Default Route en OSPF
! Nueva interfaz hacia R5 (ISP)
R1-CENTRAL(config)# interface GigabitEthernet0/3
R1-CENTRAL(config-if)# description ENLACE-ISP-R5
R1-CENTRAL(config-if)# ip address 10.10.10.17 255.255.255.252
R1-CENTRAL(config-if)# no shutdown

! Configurar la default route hacia el ISP
R1-CENTRAL(config)# ip route 0.0.0.0 0.0.0.0 10.10.10.18
! 10.10.10.18 = IP de R5 (ISP gateway)

! *** La clave del desafío: propagar la default route en OSPF ***
R1-CENTRAL(config)# router ospf 1
R1-CENTRAL(config-router)# default-information originate
! Este comando convierte a R1 en ASBR (AS Boundary Router)
! R1 genera un LSA Tipo 5 con la ruta 0.0.0.0/0
! Todos los routers del área aprenden la default route como O*E2
!
! Variante con "always":
R1-CENTRAL(config-router)# default-information originate always
! "always" = propagar aunque R1 no tenga la default route en su tabla
! Sin "always": si cae el enlace con ISP, R1 deja de propagar → convergencia
! Con "always": R1 siempre propaga (puede enviar tráfico a un enlace caído)
! En este desafío: usar sin "always" para comportamiento correcto
3 Verificación del Desafío
! En R2 (y todos los demás routers) verificar la default route OSPF
R2-NORTE# show ip route | include 0.0.0.0

! O*E2 = OSPF External Type 2 (candidata a default route)
! Gateway of last resort = 10.10.10.1 (R1) ✅
! Métrica E2 = siempre es la del ASBR (no acumulativa)

! Verificar que R1 tiene el LSA Tipo 5

! Advertised by 1.1.1.1 = R1 es el ASBR que la origina

! Ping de verificación extremo a extremo

! Ruta: R3→R2→R1→R5→8.8.8.8 (o R3→R4→R1→R5→8.8.8.8 según SPF)

  2  10.10.10.6  (R3)  35ms
  3  10.10.3.1   (R3 LAN)  42ms
! Ruta actual: R1→R2→R3 (pasando por R2)

! 2. Simular fallo del enlace R1-R2
R1-CENTRAL(config)# interface GigabitEthernet0/1
R1-CENTRAL(config-if)# shutdown
%OSPF-5-ADJCHG: Process 1, Nbr 2.2.2.2 on GigabitEthernet0/1 from FULL to DOWN
! El mensaje de log confirma la caída de la adyacencia

! 3. Esperar la reconvergencia (segundos con timers agresivos)
! 4. Verificar la nueva ruta — OSPF recalculó el SPF
R1-CENTRAL# traceroute 10.10.3.1
  1  10.10.10.14 (R4)  18ms
  2  10.10.10.10 (R3)  25ms
  3  10.10.3.1   (R3 LAN)  32ms
! Nueva ruta: R1→R4→R3 — OSPF encontró el camino alternativo ✅
! La redundancia del enlace R1-R4 funcionó perfectamente

! 5. Restaurar el enlace
R1-CENTRAL(config)# interface GigabitEthernet0/1
R1-CENTRAL(config-if)# no shutdown
%OSPF-5-ADJCHG: Process 1, Nbr 2.2.2.2 on GigabitEthernet0/1 from LOADING to FULL
! La adyacencia se restableció — la ruta original puede volver
! Depende del costo: si ambas tienen el mismo costo → ECMP (load balancing)

📋 Resumen Masterclass — OSPFv2 Single-Area Expert

✅ Los 30 Puntos Maestros — Nivel Expert

  1. OSPFv2 es un protocolo link-state (RFC 2328). Cada router tiene el mapa completo de la topología (LSDB). No hay bucles de routing posibles.
  2. El algoritmo SPF de Dijkstra calcula el árbol de caminos de menor costo desde el router local hacia todos los destinos.
  3. Costo OSPF = 100,000,000 / BW(bps). Costo es acumulativo. Siempre ajustar auto-cost reference-bandwidth 10000 en todos los routers.
  4. Los 7 estados de adyacencia: DOWN → ATTEMPT → INIT → 2-WAY → EXSTART → EXCHANGE → LOADING → FULL.
  5. Estado INIT = comunicación unidireccional. Estado 2-WAY = bidireccional. Estado FULL = LSDBs sincronizadas ✅.
  6. En EXSTART: el router con mayor RID se convierte en Master y establece el número de secuencia de los DBDs.
  7. MTU mismatch causa que la adyacencia se quede en EXSTART/EXCHANGE. Solución: ip ospf mtu-ignore o igualar MTU.
  8. El Router ID siempre debe configurarse manualmente con router-id X.X.X.X. Usar la IP de Loopback.
  9. Un cambio de Router ID requiere clear ip ospf process para aplicarse (reinicia todas las adyacencias).
  10. El Process ID de OSPF es local al router — no necesita coincidir entre vecinos.
  11. Los Hello/Dead timers deben coincidir exactamente entre vecinos. Hello default Ethernet = 10s, Dead = 40s.
  12. LSA Tipo 1 (Router LSA): generado por todos los routers. Tipo 2 (Network LSA): generado por el DR. Tipo 5: externas del ASBR.
  13. En single-area solo existen LSAs Tipo 1 y 2. La LSDB debe ser idéntica en todos los routers del área.
  14. El DR/BDR se elige en redes broadcast. Elección: mayor prioridad → mayor Router ID. Prioridad 0 = nunca DR.
  15. La elección DR/BDR es NO apropiativa: un nuevo router con mayor prioridad no desplaza al DR actual.
  16. passive-interface default + no passive-interface IFACE: la mejor práctica para controlar dónde se forman adyacencias.
  17. La autenticación MD5 en OSPF previene routers maliciosos de inyectar rutas falsas. Configurar en cada interfaz OSPF.
  18. default-information originate: convierte al router en ASBR y propaga la default route como LSA Tipo 5 (O*E2).
  19. La variante default-information originate always propaga la default aunque no esté en la tabla local.
  20. Hardening OSPF: autenticación MD5, passive-interface en LAN, deshabilitar HTTP, SSH v2, contraseñas SHA256.
  21. show ip ospf neighbor: verificación principal. show ip ospf database: ver la LSDB.
  22. show ip ospf interface brief: ver costo, estado DR/BDR y vecinos por interfaz.
  23. debug ip ospf hello: verificar recepción/envío de Hellos. debug ip ospf adj: ver el proceso de adyacencia.
  24. Siempre ejecutar undebug all después de cualquier debug en producción.
  25. Router ID duplicado genera %OSPF-4-DUP_RTRID en el log. Solución: cambiar RID + clear ospf process.
  26. El SPF timer controla la rapidez de convergencia: timers throttle spf 50 100 5000 para convergencia agresiva.
  27. OSPF multicast 224.0.0.5 = AllSPFRouters. 224.0.0.6 = AllDRouters (solo DR y BDR).
  28. IP número de protocolo OSPF = 89 (directo sobre IP, no TCP ni UDP).
  29. VLSM con /30 para enlaces WAN, /25-/28 para LANs según tamaño, /32 para Loopbacks de Router ID.
  30. En Packet Tracer: simular fallo con shutdown → observar log de adyacencia caída → verificar nueva ruta SPF → no shutdown → verificar reconvergencia.

⚠️ Los 10 Errores Críticos que Arruinan el Examen

  1. No configurar auto-cost reference-bandwidth igual en todos los routers — costos inconsistentes
  2. Confundir Process ID con AS Number — el Process ID es local, no necesita coincidir
  3. Olvidar passive-interface en interfaces LAN — intenta formar adyacencias con los PCs
  4. No usar router-id manual — un cambio inesperado de IP reinicia todas las adyacencias
  5. Creer que la elección DR/BDR es apropiativa — el DR no cambia cuando llega uno con mayor prioridad
  6. Olvidar que clear ip ospf process es necesario después de cambiar el Router ID
  7. Usar default-information originate always cuando el ISP link puede caer — sigue enviando tráfico al vacío
  8. No igualar Hello/Dead timers en ambos extremos del enlace — la adyacencia no se forma
  9. Diferente key ID en la autenticación MD5 — OSPF descarta los paquetes sin mostrar error obvio
  10. No verificar el MTU cuando la adyacencia se queda en EXSTART — causa muy frecuente y difícil de diagnosticar
