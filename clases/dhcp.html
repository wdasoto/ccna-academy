DHCP

Protocolo · Proceso DORA · Configuración IOS · Relay Agent · Snooping · Troubleshooting

📡 ¿Qué es DHCP?

DHCP (Dynamic Host Configuration Protocol) es un protocolo de capa de aplicación que asigna automáticamente parámetros de red a los hosts: dirección IP, máscara, gateway y DNS. Opera en el puerto UDP 67 (servidor) y UDP 68 (cliente).

💡 Sin DHCP — cada dispositivo necesitaría configuración manual: IP, máscara, gateway, DNS. En redes de 500 dispositivos, esto es inmanejable. DHCP automatiza todo el proceso.

Parámetros que entrega DHCP

Tipos de asignación DHCP

🔄 Dinámica
El servidor asigna una IP del pool por tiempo limitado (lease). Al vencer, el cliente puede renovar o se libera para otro host. Más común.
📌 Estática / Reserva
El servidor siempre asigna la misma IP a un host basándose en su dirección MAC. Ideal para servidores e impresoras.
✋ Manual
El administrador configura la IP manualmente en el dispositivo. Sin intervención del servidor DHCP. Usada en routers, switches y servidores críticos.
🎯 CCNA Examen: DHCP usa UDP — puerto 67 (servidor) y puerto 68 (cliente). Los mensajes DHCP son broadcasts en la LAN. El lease time por defecto en IOS Cisco es 1 día (86400 segundos).

🔄 El Proceso DORA

DHCP funciona mediante un intercambio de 4 mensajes conocidos como DORA: Discover → Offer → Request → Acknowledgement. Todo ocurre en broadcast inicialmente ya que el cliente no tiene IP.

Renovación del lease (Renew)

Al llegar al 50% del lease time, el cliente envía un DHCP Request en unicast al servidor para renovar. Si no responde, al llegar al 87.5% lo intenta en broadcast. Si aún no hay respuesta, al expirar el 100% el cliente debe reiniciar el proceso DORA completo.

DHCP NACK y DECLINE

⚠️ DHCP NACK
El servidor rechaza la solicitud del cliente. Ocurre cuando la IP solicitada ya no está disponible o el cliente se movió a otra red.
⛔ DHCP DECLINE
El cliente rechaza la IP ofrecida porque detectó (mediante ARP) que ya está en uso en la red por otro dispositivo.
🎯 CCNA Examen: El REQUEST es broadcast — no unicast — porque necesita informar a todos los servidores DHCP en el segmento que aceptó una oferta específica y que rechaza las demás. Recuerda: D-O-R-A.

⌨️ Configuración DHCP en IOS Cisco

Un router Cisco puede actuar como servidor DHCP directamente en IOS. Se configura un pool con los parámetros de la red y opcionalmente se excluyen IPs para uso estático.

Configuración completa — Servidor DHCP

Router — Servidor DHCP básico
Router(config)#
! Paso 1: Excluir IPs estáticas (routers, switches, servidores)
Router(config)# ip dhcp excluded-address 192.168.1.1 192.168.1.20
! Paso 2: Crear el pool DHCP
Router(config)# ip dhcp pool LAN_POOL
Router(dhcp-config)# network 192.168.1.0 255.255.255.0
Router(dhcp-config)# default-router 192.168.1.1
Router(dhcp-config)# dns-server 8.8.8.8 8.8.4.4
Router(dhcp-config)# domain-name empresa.local
Router(dhcp-config)# lease 1 0 0
! lease = días horas minutos (1 día)
Router(dhcp-config)# exit
💡 Orden importante: siempre configura ip dhcp excluded-address ANTES de crear el pool. Las IPs excluidas no serán asignadas a clientes, reservándolas para dispositivos con IP estática.

Múltiples pools — Varias subredes

Router — DHCP para múltiples VLANs
Router(config)# ip dhcp excluded-address 10.10.10.1 10.10.10.10
Router(config)# ip dhcp excluded-address 10.10.20.1 10.10.20.10
! Pool para VLAN 10 — Ventas
Router(config)# ip dhcp pool VLAN10_VENTAS
Router(dhcp-config)# network 10.10.10.0 255.255.255.0
Router(dhcp-config)# default-router 10.10.10.1
Router(dhcp-config)# dns-server 10.10.10.5
Router(dhcp-config)# exit
! Pool para VLAN 20 — Ingeniería
Router(config)# ip dhcp pool VLAN20_INGENIERIA
Router(dhcp-config)# network 10.10.20.0 255.255.255.0
Router(dhcp-config)# default-router 10.10.20.1
Router(dhcp-config)# dns-server 10.10.10.5
Router(dhcp-config)# exit

Reservas estáticas por MAC (Binding manual)

Router — IP fija para un host por su MAC
Router(config)# ip dhcp pool SERVIDOR_WEB
Router(dhcp-config)# host 192.168.1.100 255.255.255.0
Router(dhcp-config)# client-identifier 01aa.bbcc.dd11.22
! O usar hardware-address para MAC directa:
Router(dhcp-config)# hardware-address aabb.ccdd.1122
Router(dhcp-config)# default-router 192.168.1.1
Router(dhcp-config)# dns-server 8.8.8.8

Verificación del servidor DHCP

Comandos show — Verificar DHCP
! Ver bindings activos (IPs asignadas)
Router# show ip dhcp binding
! Ver estadísticas del servidor
Router# show ip dhcp statistics
! Ver pools configurados
Router# show ip dhcp pool
! Ver IPs excluidas
Router# show running-config | section dhcp

Deshabilitar DHCP en el router

Deshabilitar servicio DHCP
Router(config)# no service dhcp
! Para re-habilitar:
Router(config)# service dhcp
🎯 CCNA Examen: El comando ip dhcp excluded-address acepta rango (inicio fin) o IP única. Si omites el rango, excluye solo esa IP. El pool necesita mínimo network y default-router para funcionar. Sin dns-server, los clientes no resolverán nombres.

🔀 DHCP Relay Agent (ip helper-address)

Los mensajes DHCP son broadcasts — no atraviesan routers por defecto. El DHCP Relay Agent (comando ip helper-address) permite que un router reenvíe el broadcast del cliente al servidor DHCP centralizado como unicast.

Configuración del Relay Agent en el router

Router — Habilitar ip helper-address
! Configurar en la interfaz HACIA los clientes (LAN)
Router(config)# interface GigabitEthernet0/0
! Esta interfaz conecta con los clientes que necesitan DHCP
Router(config-if)# ip address 192.168.1.1 255.255.255.0
Router(config-if)# ip helper-address 10.0.0.50
! 10.0.0.50 = IP del servidor DHCP centralizado
Router(config-if)# no shutdown
! Múltiples helper-address (múltiples servidores):
Router(config-if)# ip helper-address 10.0.0.51

Relay en topología Router-on-a-Stick (VLANs)

Router — helper-address en subinterfaces VLAN
! VLAN 10
Router(config)# interface GigabitEthernet0/0.10
Router(config-subif)# encapsulation dot1Q 10
Router(config-subif)# ip address 10.10.10.1 255.255.255.0
Router(config-subif)# ip helper-address 10.0.0.50
! VLAN 20
Router(config)# interface GigabitEthernet0/0.20
Router(config-subif)# encapsulation dot1Q 20
Router(config-subif)# ip address 10.10.20.1 255.255.255.0
Router(config-subif)# ip helper-address 10.0.0.50
🎯 CCNA Examen: ip helper-address se configura en la interfaz del router que mira hacia los clientes, NO hacia el servidor. Es el error más común en el examen. Recuerda: el relay convierte el broadcast del cliente en unicast hacia el servidor.

🛡️ DHCP Snooping

DHCP Snooping es una función de seguridad en switches Cisco que filtra mensajes DHCP no autorizados y previene ataques como el DHCP Spoofing. Clasifica los puertos del switch en Trusted (confiables) y Untrusted (no confiables).

El problema: DHCP Spoofing

Un atacante conecta un servidor DHCP falso (rogue) en la red. Los clientes pueden recibir IPs con un gateway falso (el atacante), permitiendo un ataque Man-in-the-Middle (MitM). DHCP Snooping previene esto.

Puertos Trusted vs Untrusted

TipoDescripciónMensajes permitidos
Trusted (Confiable) Puertos hacia servidores DHCP legítimos, otros switches o routers Todos los mensajes DHCP (Offer, ACK, NAK)
Untrusted (No confiable) Puertos hacia clientes, hosts finales (por defecto todos) Solo DHCP Discover y Request. Bloquea Offer y ACK

Configuración DHCP Snooping en switch

Switch — Habilitar DHCP Snooping
! Paso 1: Habilitar DHCP Snooping globalmente
Switch(config)# ip dhcp snooping
! Paso 2: Habilitar en VLANs específicas
Switch(config)# ip dhcp snooping vlan 10,20,30
! Paso 3: Deshabilitar inserción de option 82 (recomendado en acceso)
Switch(config)# no ip dhcp snooping information option
! Paso 4: Marcar puertos hacia servidor/uplink como trusted
Switch(config)# interface GigabitEthernet0/1
! Este puerto va hacia el router/servidor DHCP
Switch(config-if)# ip dhcp snooping trust
Switch(config-if)# exit
! Los puertos de acceso (clientes) quedan untrusted por defecto
! Opcional: limitar rate de mensajes DHCP por puerto
Switch(config)# interface range FastEthernet0/1-24
Switch(config-if-range)# ip dhcp snooping limit rate 15
! Máximo 15 paquetes DHCP por segundo por puerto

DHCP Option 82 (Relay Information)

La Option 82 agrega información del switch (puerto y VLAN) al mensaje DHCP, permitiendo al servidor asignar IPs según la ubicación física del cliente. Se usa en redes empresariales grandes. En topologías simples con router IOS como relay, se recomienda deshabilitar con no ip dhcp snooping information option.

🎯 CCNA Examen: DHCP Snooping debe habilitarse globalmente Y por VLAN — ambos comandos son necesarios. Los puertos uplink (hacia routers/servidores) deben marcarse como trusted o los clientes no recibirán IPs.

🔧 Troubleshooting DHCP

Los problemas de DHCP son frecuentes en el examen CCNA y en producción. La metodología es: verificar conectividad → revisar configuración del pool → verificar relay → revisar snooping.

Problemas comunes y soluciones

SíntomaCausa probableSolución
Cliente obtiene 169.254.x.x (APIPA) No alcanza el servidor DHCP Verificar ip helper-address, conectividad, pool activo
Pool agotado — no asigna más IPs Todas las IPs del pool están en uso clear ip dhcp binding * o ampliar el pool
Servidor DHCP no responde Servicio deshabilitado service dhcp en configuración global
Conflicto de IPs en la red IP ya en uso no excluida del pool Agregar ip dhcp excluded-address
Clientes en otra VLAN no reciben IP Falta ip helper-address en subinterfaz Configurar helper-address en cada SVI/subinterfaz
DHCP Snooping bloquea IPs legítimas Puerto uplink marcado como untrusted ip dhcp snooping trust en interfaz uplink

Comandos de troubleshooting esenciales

Router/Switch — Diagnóstico DHCP
! Ver IPs asignadas actualmente
Router# show ip dhcp binding
! Ver estadísticas (mensajes enviados/recibidos)
Router# show ip dhcp statistics
! Ver configuración del pool
Router# show ip dhcp pool
! Verificar conflictos detectados
Router# show ip dhcp conflict
! Debug en tiempo real (¡usar con cuidado en producción!)
Router# debug ip dhcp server events
Router# debug ip dhcp server packet
! Desactivar debug
Router# undebug all
! Limpiar bindings (forzar renovación de todos los clientes)
Router# clear ip dhcp binding *
! Limpiar conflictos registrados
Router# clear ip dhcp conflict *

Verificar ip helper-address

Verificar configuración de relay
Router# show running-config | section interface
! Verificar que la interfaz correcta tiene ip helper-address
Router# show ip interface GigabitEthernet0/0
! Buscar en la salida: "Helper address is X.X.X.X"

Flujo de diagnóstico — DHCP no funciona

1
¿El servidor DHCP está activo?
show ip dhcp pool — debe mostrar el pool con IPs disponibles.
2
¿Hay IPs disponibles en el pool?
show ip dhcp binding — si todas están asignadas, limpiar o ampliar el pool.
3
¿El cliente está en otra subred?
Verificar ip helper-address en la interfaz del router hacia la LAN del cliente.
4
¿DHCP Snooping bloqueando?
show ip dhcp snooping statistics — revisar drops. Verificar puertos trusted.
5
Debug en tiempo real
debug ip dhcp server events — ver exactamente qué mensajes procesa el servidor.
🎯 CCNA Examen: Si un cliente muestra IP 169.254.x.x (APIPA), significa que no recibió respuesta DHCP. Causas más frecuentes en el examen: falta de ip helper-address, pool agotado, o DHCP Snooping sin trusted port. Siempre verifica con show ip dhcp binding.
