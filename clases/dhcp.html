📖 DHCP — Asignación automática de configuración de red

DHCP (Dynamic Host Configuration Protocol, RFC 2131) es el protocolo que permite a los dispositivos obtener automáticamente su configuración de red al conectarse: dirección IP, máscara de subred, gateway predeterminado, servidor DNS y otros parámetros. Sin DHCP, cada dispositivo requeriría configuración manual — inviable en redes con cientos o miles de dispositivos.

🎯 Asignación dinámica

El servidor DHCP asigna IPs de un pool disponible. Cuando el lease expira o el cliente se desconecta, la IP regresa al pool y puede asignarse a otro dispositivo. Ideal para entornos con muchos dispositivos temporales.

📌 Asignación estática (reserva)

Se asocia permanentemente una IP específica a la MAC address de un dispositivo. El dispositivo siempre recibe la misma IP via DHCP. Útil para impresoras, servidores, cámaras IP que necesitan IP fija.

⏱️ Lease time

Cada asignación DHCP tiene una duración (lease). El cliente debe renovar antes de que expire. Si no renueva, la IP regresa al pool. El lease time es configurable según el entorno.

🌐 DHCP Options

Además de la IP, DHCP puede entregar docenas de parámetros adicionales: DNS, gateway, NTP, servidor WINS, nombre de dominio, TFTP server para provisioning de teléfonos IP, etc.

⚠️ APIPA — Automatic Private IP Addressing

Si un cliente no recibe respuesta DHCP, Windows/macOS asignan automáticamente una IP del rango 169.254.0.0/16. Un dispositivo con IP APIPA solo puede comunicarse con otros en el mismo segmento con APIPA — no tiene gateway y no puede acceder a Internet ni a otras subredes.

Ver IP 169.254.x.x en un dispositivo = el servidor DHCP no respondió → problema de conectividad DHCP.

🔄 El proceso DORA — Cómo obtiene IP un cliente

DORA es el acrónimo de los 4 mensajes que intercambian el cliente y el servidor DHCP para asignar una dirección IP. Es el proceso fundamental que debes conocer perfectamente para el CCNA.

📌 Los 4 mensajes DORA

  1. D — Discover: el cliente no tiene IP, broadcast a 255.255.255.255 buscando servidores DHCP
  2. O — Offer: el servidor responde ofreciendo una IP disponible del pool
  3. R — Request: el cliente solicita formalmente la IP ofrecida (broadcast — informa a otros servidores)
  4. A — Acknowledge: el servidor confirma la asignación con todos los parámetros

📨 Análisis técnico de cada mensaje DORA

1️⃣ DHCP Discover — El cliente busca un servidor

CampoValorPor qué

2️⃣ DHCP Offer — El servidor propone una IP

CampoValorPor qué

3️⃣ DHCP Request — El cliente acepta la oferta

📌 ¿Por qué el Request es también un broadcast?

Si hay múltiples servidores DHCP en la red, todos hacen un Offer. El cliente elige uno (normalmente el primero) y envía el Request como broadcast. Esto sirve para dos cosas:

  • Confirmar al servidor elegido que acepta su oferta
  • Notificar a los demás servidores que rechaza sus ofertas (para que liberen las IPs reservadas)

El campo "Server Identifier" en el Request indica qué servidor fue elegido.

4️⃣ DHCP Acknowledge — El servidor confirma

📌 El ACK contiene toda la configuración final

El servidor registra el lease en su base de datos y envía el ACK con:

  • IP asignada y máscara
  • Default gateway (opción 3)
  • Servidores DNS (opción 6)
  • Lease time total (opción 51)
  • Nombre de dominio (opción 15)
  • Cualquier otra opción configurada

Si el servidor no puede cumplir la Request → envía DHCP NAK (Negative Acknowledge) → el cliente reinicia el proceso DORA.

🔁 Renovación del lease — T1 y T2

📌 Dos momentos críticos de renovación

Cisco IOS incluye un servidor DHCP completo. Es suficiente para redes medianas y eliminra la necesidad de un servidor dedicado para DHCP.

Configuración básica — Pool DHCP completo

! ─── PASO 1: Excluir IPs que NO se asignarán dinámicamente ───
! Siempre excluir ANTES de crear el pool
R1(config)# ip dhcp excluded-address 192.168.1.1 192.168.1.20
! Excluye de .1 hasta .20 → reservadas para routers, switches, servidores
R1(config)# ip dhcp excluded-address 192.168.1.254
! Excluye una IP individual (la dirección de broadcast - 1)

! ─── PASO 2: Crear el pool DHCP ──────────────────────
R1(config)# ip dhcp pool LAN-VENTAS
! Nombre descriptivo del pool
R1(dhcp-config)# network 192.168.1.0 255.255.255.0
! Red para la que aplica este pool
R1(dhcp-config)# default-router 192.168.1.1
! Gateway predeterminado para los clientes (opción DHCP 3)
R1(dhcp-config)# dns-server 8.8.8.8 8.8.4.4
! Servidores DNS primario y secundario (opción DHCP 6)
R1(dhcp-config)# domain-name empresa.com
! Nombre de dominio DNS (opción DHCP 15)
R1(dhcp-config)# lease 7 0 0
! Lease time: 7 días, 0 horas, 0 minutos (default: 1 día)
! Formato: lease DIAS HORAS MINUTOS | infinite
R1(dhcp-config)# netbios-name-server 192.168.1.5
! Servidor WINS para redes Windows (opción DHCP 44)

Asignación estática (reserva por MAC address)

! Reserva: asociar IP fija a una MAC específica

🔁 DHCP Relay Agent — Cruzar subredes con ip helper-address

El proceso DHCP usa broadcasts (255.255.255.255). Los routers, por defecto, no reenvían broadcasts — los descartan al cruzar a otra subred. Esto significa que si el servidor DHCP está en una subred diferente a los clientes, los Discover nunca llegarán al servidor.

La solución es el DHCP Relay Agent, configurado con ip helper-address en la interfaz del router más cercana a los clientes. El relay convierte el broadcast en unicast hacia el servidor DHCP.

📌 Cómo funciona ip helper-address

  1. El cliente envía un DHCP Discover (broadcast 255.255.255.255)
  2. El router recibe el broadcast en la interfaz con ip helper-address
  3. El router convierte el broadcast en unicast hacia la IP del servidor DHCP (la que especificaste en helper-address)
  4. El router agrega la opción DHCP 82 (Relay Agent Information) con la IP de la interfaz de entrada
  5. El servidor DHCP recibe la solicitud unicast. Ve la opción 82 y sabe qué pool usar para esa subred
  6. El servidor responde en unicast al relay agent
  7. El relay reenvía la respuesta al cliente

⚙️ Configuración ip helper-address

Escenario: Cliente en 192.168.1.0/24, servidor DHCP en 10.0.0.10

! El router R1 conecta la LAN de clientes y tiene ruta al servidor DHCP
! Interfaz Gi0/0 → LAN clientes (192.168.1.0/24)
! El servidor DHCP está en 10.0.0.10 (otra subred)

R1(config)# interface GigabitEthernet0/0
! La interfaz HACIA los clientes DHCP
R1(config-if)# ip helper-address 10.0.0.10
! IP del servidor DHCP — el relay enviará los Discovers aquí

! ¿Múltiples servidores DHCP? Configurar múltiples helper-address
R1(config-if)# ip helper-address 10.0.0.10
R1(config-if)# ip helper-address 10.0.0.11
! El relay reenvía el Discover a AMBOS servidores
! El cliente aceptará el Offer más rápido

Inter-VLAN con helper-address en subinterfaces

! Topología: Router-on-a-Stick con 3 VLANs, servidor DHCP en 10.0.0.10
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# encapsulation dot1q 10
R1(config-subif)# ip address 192.168.10.1 255.255.255.0
R1(config-subif)# ip helper-address 10.0.0.10
! Relay para VLAN 10 hacia el servidor central

R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# encapsulation dot1q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0
R1(config-subif)# ip helper-address 10.0.0.10
! Relay para VLAN 20 — el servidor debe tener pool para 192.168.20.0/24

R1(config)# interface GigabitEthernet0/0.30
R1(config-subif)# encapsulation dot1q 30
R1(config-subif)# ip address 192.168.30.1 255.255.255.0
R1(config-subif)# ip helper-address 10.0.0.10
! El servidor DHCP usa la IP origen del relay (giaddr) para seleccionar el pool
! giaddr=192.168.10.1 → usa pool 192.168.10.0/24
! giaddr=192.168.20.1 → usa pool 192.168.20.0/24

💻 Router Cisco como cliente DHCP

Un router Cisco también puede actuar como cliente DHCP — obtener su IP automáticamente de un servidor upstream. Esto es común en interfaces WAN hacia el ISP, o cuando el router es el dispositivo del cliente (CPE) en una red con servidor DHCP central.

Configurar interfaz como cliente DHCP

! Configurar la interfaz para obtener IP via DHCP
R1(config)# interface GigabitEthernet0/1
R1(config-if)# ip address dhcp
! La interfaz solicita IP al servidor DHCP upstream
R1(config-if)# no shutdown

! Opcionalmente especificar el hostname del cliente para el servidor
R1(config-if)# ip address dhcp client-id GigabitEthernet0/1 hostname R1-CPE

6️⃣ Configuración automática IPv6 — 3 métodos

En IPv6 existen tres mecanismos para que los hosts obtengan su configuración de red automáticamente. A diferencia de IPv4 (donde DHCP es la única opción práctica), IPv6 añade SLAAC — un método completamente autónomo que no requiere servidor.

MétodoIPGatewayDNSServidor requerido
SLAAC✅ Auto (EUI-64)✅ Via RA❌ No (sin DNS)❌ No — solo router con RA
SLAAC + DHCPv6 Stateless✅ Auto (EUI-64)✅ Via RA✅ Via DHCPv6✅ Servidor DHCPv6 (solo DNS)
DHCPv6 Stateful✅ Del pool DHCPv6✅ Via RA✅ Via DHCPv6✅ Servidor DHCPv6 completo

🌐 SLAAC — Stateless Address Autoconfiguration (RFC 4862)

📌 Cómo el host genera su propia IPv6 con SLAAC

  1. El host genera una dirección link-local (FE80::/10) usando su MAC (EUI-64).
  2. El host envía un Router Solicitation (RS) a FF02::2 (all routers) para descubrir routers.
  3. El router responde con un Router Advertisement (RA) anunciando el prefijo de la red (/64) y los flags.
  4. El host combina el prefijo /64 del RA con su identificador EUI-64 para construir la dirección global.
  5. Antes de usar la IP, ejecuta DAD (Duplicate Address Detection) para verificar que no hay duplicados.

📌 Flags del Router Advertisement — M y O bits

FlagNombreValorSignificado
M bitManaged0 → SLAAC0 = usa SLAAC para la IP; 1 = usa DHCPv6 stateful para la IP
O bitOther0 → solo SLAAC0 = sin otro servidor; 1 = usa DHCPv6 stateless para otros parámetros (DNS)

M=0, O=0 → SLAAC puro | M=0, O=1 → SLAAC + DHCPv6 stateless | M=1, O=1 → DHCPv6 stateful

⚙️ Configuración DHCPv6 en Cisco IOS

DHCPv6 Stateless — IP por SLAAC, DNS por DHCPv6

! ─── Servidor DHCPv6 Stateless ───────────────────────
! El cliente usa SLAAC para su IP pero obtiene DNS de DHCPv6
R1(config)# ipv6 dhcp pool STATELESS-POOL
R1(config-dhcpv6)# dns-server 2001:DB8::53
R1(config-dhcpv6)# domain-name empresa.com
! Solo DNS y dominio — sin asignación de IPs (stateless)

! Configurar la interfaz del router hacia los clientes
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ipv6 address 2001:DB8:1::1/64
R1(config-if)# ipv6 dhcp server STATELESS-POOL
! Asociar el pool DHCPv6 a la interfaz
R1(config-if)# ipv6 nd other-config-flag
! Activa el O bit en el RA → clientes buscan DNS en DHCPv6
! M bit = 0 (default) → IP por SLAAC

DHCPv6 Stateful — IP y DNS por DHCPv6

! ─── Servidor DHCPv6 Stateful ────────────────────────
! El servidor asigna IPs del pool (como DHCP IPv4)
R1(config)# ipv6 dhcp pool STATEFUL-POOL

🔧 Troubleshooting DHCP — Diagnóstico sistemático

Los problemas DHCP son muy comunes en el CCNA y en producción. El diagnóstico debe seguir un proceso lógico: verificar el servidor → verificar la red → verificar el relay → verificar el cliente.

⚠️ Los problemas DHCP más frecuentes

SíntomaCausa probableVerificación
Cliente obtiene 169.254.x.x (APIPA)No llega respuesta DHCP al cliente¿Está el servicio DHCP activo? ¿Hay pool configurado? ¿ip helper-address correcto?
Pool agotado — no hay IPs disponiblesMuchos clientes o lease muy largoshow ip dhcp pool — ver "Leased addresses"
Conflicto de IPIP asignada ya estaba en usoshow ip dhcp conflict — excluir la IP conflictiva
Clientes en VLAN diferente no obtienen IPFalta ip helper-address en esa subinterfazshow ip interface Gi0/X.Y → buscar "Helper address"
Servidor DHCP no respondeservice dhcp desactivado o pool mal configuradoshow ip dhcp server statistics
Gateway incorrectodefault-router mal configurado en el poolshow ip dhcp pool NOMBRE

🧪 Laboratorio Magistral — DHCP Multipool con Relay Agent

📋 Topología: Router-on-a-Stick con servidor DHCP centralizado

R1 — Router central (servidor DHCP + relay para subinterfaces):
Gi0/0.10 → VLAN 10 VENTAS 192.168.10.0/24 (gateway .1)
Gi0/0.20 → VLAN 20 IT 192.168.20.0/24 (gateway .1)
Gi0/0.99 → VLAN 99 GESTION 192.168.99.0/24 (gateway .1)
Lo0 → 1.1.1.1/32 (gestión del router)
SW1 — Switch acceso con VLANs 10, 20, 99 y trunk hacia R1

Fase 1 — Configuración base R1

R1(config)# hostname R1
R1(config)# interface Loopback0
R1(config-if)# ip address 1.1.1.1 255.255.255.255

! Subinterfaz VLAN 10 VENTAS
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# encapsulation dot1q 10
R1(config-subif)# ip address 192.168.10.1 255.255.255.0

! Subinterfaz VLAN 20 IT
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# encapsulation dot1q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0

! Subinterfaz VLAN 99 GESTION (native)
R1(config)# interface GigabitEthernet0/0.99
R1(config-subif)# encapsulation dot1q 99 native
R1(config-subif)# ip address 192.168.99.1 255.255.255.0

R1(config)# interface GigabitEthernet0/0
R1(config-if)# no shutdown

Fase 2 — Pools DHCP para cada VLAN

! ─── Exclusiones (SIEMPRE antes de los pools) ─────────
R1(config)# ip dhcp excluded-address 192.168.10.1 192.168.10.10
R1(config)# ip dhcp excluded-address 192.168.20.1 192.168.20.20
R1(config)# ip dhcp excluded-address 192.168.99.1 192.168.99.10

! ─── Pool VLAN 10 VENTAS ─────────────────────────────
R1(config)# ip dhcp pool VLAN10-VENTAS
R1(dhcp-config)# network 192.168.10.0 255.255.255.0
R1(dhcp-config)# default-router 192.168.10.1
R1(dhcp-config)# dns-server 8.8.8.8 8.8.4.4
R1(dhcp-config)# domain-name empresa.com
R1(dhcp-config)# lease 1

! ─── Pool VLAN 20 IT ─────────────────────────────────
R1(config)# ip dhcp pool VLAN20-IT
R1(dhcp-config)# network 192.168.20.0 255.255.255.0
R1(dhcp-config)# default-router 192.168.20.1
R1(dhcp-config)# dns-server 192.168.20.5
! IT usa DNS interno
R1(dhcp-config)# domain-name empresa.com
R1(dhcp-config)# lease 3

! ─── Reserva estática para servidor de IT ─────────────
R1(config)# ip dhcp pool SERVER-IT
R1(dhcp-config)# host 192.168.20.5 255.255.255.0
R1(dhcp-config)# hardware-address 00aa.bbcc.ddee
R1(dhcp-config)# default-router 192.168.20.1
R1(dhcp-config)# client-name DNS-SERVER-IT

Fase 3 — Switch SW1

SW1(config)# vlan 10
SW1(config-vlan)# name VENTAS
SW1(config)# vlan 20
SW1(config-vlan)# name IT
SW1(config)# vlan 99
SW1(config-vlan)# name GESTION

! Puerto trunk hacia R1
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk native vlan 99
SW1(config-if)# switchport trunk allowed vlan 10,20,99

! Puertos de acceso para clientes
SW1(config)# interface range FastEthernet0/1-10
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 10
SW1(config)# interface range FastEthernet0/11-20
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 20

Verificación final completa

R1# show ip dhcp pool
Pool VLAN10-VENTAS:
  Total addresses: 244, Leased addresses: 8
Pool VLAN20-IT:
  Total addresses: 234, Leased addresses: 5

📋 Resumen Maestro — DHCP CCNA 200-301

✅ Los 28 puntos más críticos del examen

  1. DHCP (RFC 2131) asigna automáticamente: IP, máscara, gateway, DNS y más. Opera en UDP puertos 67 (servidor) y 68 (cliente).
  2. Proceso DORA: Discover (broadcast) → Offer → Request (broadcast) → Acknowledge.
  3. IP origen del Discover = 0.0.0.0 (cliente sin IP). Destino = 255.255.255.255 (broadcast).
  4. El Request es broadcast para notificar a TODOS los servidores (el cliente puede haber recibido múltiples Offers).
  5. DHCP NAK: el servidor rechaza la Request → el cliente reinicia DORA.
  6. T1 = 50% del lease time → renovación unicast. T2 = 87.5% → renovación broadcast.
  7. APIPA: si no hay servidor DHCP → cliente asigna 169.254.x.x (autoconfiguración). Sin gateway — no enrutable.
  8. Configurar exclusiones SIEMPRE antes de crear el pool: ip dhcp excluded-address.
  9. Comandos del pool: network, default-router, dns-server, lease, domain-name.
  10. Reserva estática: pool con host IP MASK + hardware-address MAC → lease "Manual/Infinite".
  11. Option 150 = servidor TFTP para IP Phones Cisco (provisioning de configuración).
  12. Los routers descartan broadcasts por defecto → DHCP no cruza subredes sin Relay Agent.
  13. ip helper-address: en la interfaz del router HACIA los clientes. Convierte broadcast DHCP en unicast hacia el servidor.
  14. ip helper-address usa la IP de la interfaz (giaddr) para que el servidor elija el pool correcto.
  15. ip helper-address reenvía por defecto 8 tipos de tráfico UDP (no solo DHCP): 67, 68, 53, 69, 37, 137, 138, 49.
  16. Router como cliente DHCP: ip address dhcp en la interfaz.
  17. no service dhcp: deshabilita completamente el servidor DHCP y el relay agent.
  18. IPv6: tres métodos de autoconfiguración: SLAAC, SLAAC+DHCPv6 stateless, DHCPv6 stateful.
  19. SLAAC: IP = prefijo del RA + EUI-64 de la MAC. No requiere servidor. No proporciona DNS.
  20. Flags RA: M=0 → SLAAC; M=1 → DHCPv6 stateful. O=1 → usar DHCPv6 para parámetros adicionales (DNS).
  21. EIGRPv6 stateless: solo DNS de DHCPv6. EIGRPv6 stateful: IP + DNS de DHCPv6.
  22. DHCPv6: puertos UDP 546 (cliente) y 547 (servidor).
  23. ipv6 nd managed-config-flag → M bit=1. ipv6 nd other-config-flag → O bit=1.
  24. EIGRPv6 stateful: ipv6 dhcp pool + address prefix. EIGRPv6 stateless: solo DNS, sin address prefix.
  25. show ip dhcp binding: leases activos. show ip dhcp pool: utilización. show ip dhcp conflict: conflictos.
  26. clear ip dhcp binding *: libera todos los leases. clear ip dhcp conflict *: limpia conflictos.
  27. debug ip dhcp server events: ver el proceso DORA en tiempo real.
  28. El servidor DHCP detecta conflictos usando ping antes de ofrecer una IP.

⚠️ Los 10 errores más comunes en el examen

  1. Configurar el pool antes de las exclusiones — las IPs excluidas aún podrían asignarse si el pool se creó antes
  2. Poner ip helper-address en la interfaz HACIA el servidor en lugar de la interfaz HACIA los clientes
  3. Creer que el DHCP Discover usa la IP del cliente como origen — usa 0.0.0.0
  4. Creer que el DHCP Request es unicast — es broadcast para notificar a todos los servidores
  5. Confundir APIPA (169.254.x.x) con una IP válida — no tiene gateway, no es enrutable
  6. "no service dhcp" también deshabilita el relay agent — no solo el servidor
  7. Olvidar que ip helper-address reenvía más servicios además de DHCP (DNS, TFTP, etc.)
  8. En SLAAC, creer que el cliente obtiene DNS — SLAAC puro no proporciona DNS (necesita DHCPv6 stateless)
  9. Confundir puertos IPv4 DHCP (67/68) con IPv6 DHCPv6 (547/546)
  10. Creer que la opción 150 es para servidores DNS — es para servidor TFTP de IP Phones Cisco
