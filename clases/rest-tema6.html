APIs CON PYTHON — Librería requests

El estándar de facto para consumir APIs REST desde Python. Desde el primer GET hasta autenticación compleja, manejo de JSON, errores HTTP, SSL, sesiones persistentes y paginación. La herramienta indispensable del ingeniero de redes moderno para automatizar contra DNA Center, Meraki, SD-WAN controllers y cualquier servicio web.

🐍 Python 3.8+ 🎯 CCNA Domain 6.5 ⏱ ~90 min lectura 📦 pip install requests 🌐 HTTP/1.1 + JSON

¿Por qué requests y no urllib?

Python incluye en su librería estándar urllib para realizar peticiones HTTP. Sin embargo, urllib es verboso, poco intuitivo y requiere mucho código boilerplate para tareas simples. requests, creada por Kenneth Reitz, se ha convertido en el estándar de facto porque sigue la filosofía "HTTP for Humans".

❌ urllib (stdlib)

  • Verbos: necesitas urllib.request.urlopen
  • JSON manual: json.loads(response.read())
  • Manejo de errores complejo con URLError
  • Headers como diccionarios planos sin helpers
  • Encoding y decodificación manual del body

✅ requests (3rd party)

  • API limpia: requests.get(url)
  • JSON integrado: response.json()
  • Excepciones claras: ConnectionError, Timeout
  • Headers, auth, cookies como parámetros nativos
  • Keep-alive y pooling de conexiones automático

💡 Dato clave del examen

En el CCNA Domain 6.5 se evalúa que entiendas el intercambio request-response. requests abstrae todo el stack TCP/IP y HTTP, pero debes saber que debajo viaja: IP → TCP → TLS (si es HTTPS) → HTTP → payload JSON.

Anatomía de un objeto Response

Cada llamada a requests.get/post/put/delete retorna un objeto Response que contiene toda la información del servidor:

  • status_code: Entero HTTP (200, 201, 404, 500...).
  • headers: Diccionario con los headers de respuesta (Content-Type, Server, X-Rate-Limit...).
  • text: Body crudo como string Unicode.
  • content: Body en bytes (útil para imágenes o binarios).
  • json(): Método que parsea el body como JSON y retorna dict/list.
  • ok: Boolean, True si status_code < 400.
  • encoding: Charset detectado (utf-8 por defecto).
response_anatomy.py — Inspeccionando la respuesta
import requests

r = requests.get("https://httpbin.org/get")

print(f"¿Éxito? {r.ok}")           # True si 2xx
print(f"Código: {r.status_code}")   # 200
print(f"Encoding: {r.encoding}")  # utf-8
print(f"Servidor: {r.headers.get('Server')}")
print(f"Tipo: {r.headers.get('content-type')}")

# Si la respuesta es JSON, .json() la convierte a Python nativo
data = r.json()
print(data["origin"])  # Tu IP pública

Nunca hagas verify=False en producción (lo veremos en detalle más adelante). En labs locales con certificados autofirmados (DNA Center sandbox, controllers de prueba) está justificado, pero siempre documenta el riesgo de Man-in-the-Middle.

Métodos HTTP y su semántica

REST no es un protocolo, es un estilo arquitectónico que usa HTTP como transporte. Cada método tiene un significado preciso que requests expone como funciones de primer nivel:

MétodorequestsUso en redes/APIsIdempotente
GETrequests.get()Obtener datos: inventario, clientes, configuración.✅ Sí
POSTrequests.post()Crear recurso: nueva VLAN, nuevo usuario, webhook.❌ No
PUTrequests.put()Reemplazar recurso completo: actualizar política.✅ Sí
PATCHrequests.patch()Actualización parcial: cambiar solo el nombre.❌ No*
DELETErequests.delete()Eliminar: borrar ACL, remover dispositivo.✅ Sí
HEADrequests.head()Verificar si recurso existe sin descargar body.✅ Sí
OPTIONSrequests.options()Consultar métodos permitidos por el servidor (CORS).✅ Sí

* PATCH puede ser idempotente dependiendo de la implementación, pero no está garantizado por la especificación.

Status Codes: qué dice el servidor

El código de estado es el primer indicador de salud de tu petición. Un ingeniero de redes debe reconocerlos al instante:

2️⃣

2xx — Éxito

200 OK: GET exitoso.
201 Created: POST exitoso, recurso creado.
204 No Content: DELETE exitoso, sin body.

3️⃣

3xx — Redirección

301 Moved: URL cambió permanentemente.
302 Found: Redirección temporal.
Requests sigue redirecciones automáticamente por defecto.

4️⃣

4xx — Error cliente

400 Bad Request: JSON malformado.
401 Unauthorized: Falta token/API key.
403 Forbidden: Sin permisos.
404 Not Found: Recurso inexistente.

5️⃣

5xx — Error servidor

500 Internal Error: Bug del servidor.
502 Bad Gateway: Proxy/intermediario caído.
503 Unavailable: Servidor sobrecargado.
504 Gateway Timeout: upstream no responde.

Headers HTTP: el metadata invisible

Los headers son pares clave-valor que transportan metadatos sobre la petición o la respuesta. En APIs REST modernas controlan todo: autenticación, formato, caché, rate limiting.

headers_fundamentals.py — Enviar y recibir headers
import requests

# Headers de petición (Request Headers)
headers = {
    "Accept": "application/json",
    "User-Agent": "CCNA-Academy-NetworkBot/1.0",
    "X-Custom-Header": "valor-personalizado"
}

r = requests.get("https://httpbin.org/headers", headers=headers)

# Headers de respuesta (Response Headers)
print(f"Date: {r.headers.get('Date')}")
print(f"Content-Type: {r.headers.get('Content-Type')}")
print(f"X-RateLimit-Remaining: {r.headers.get('X-RateLimit-Remaining', 'N/A')}")

# httpbin devuelve los headers que recibió
print(r.json()["headers"])
💡 Headers imprescindibles en redes/APIs
  • Accept: application/json: Indica que esperas JSON, no XML ni HTML.
  • Content-Type: application/json: Obligatorio en POST/PUT cuando envías body JSON.
  • Authorization: Bearer <token>: Estándar OAuth 2.0 y API Keys.
  • X-Auth-Token: Usado por DNA Center (ver Tema 4).
  • X-Cisco-Meraki-API-Key: Header propietario de Meraki (ver Tema 5).

GET con Query Parameters

Los parámetros de consulta (query string) se añaden a la URL después del ?. Se usan para filtrar, paginar, buscar o limitar resultados. requests permite pasarlos como diccionario vía el parámetro params, evitando errores manuales de URL-encoding.

get_params.py — Parámetros de consulta
import requests

# Ejemplo: obtener solo los primeros 5 posts de userId=1
url = "https://jsonplaceholder.typicode.com/posts"

payload = {
    "userId": 1,
    "_limit": 5
}

r = requests.get(url, params=payload)

print(f"URL final: {r.url}")
# → https://jsonplaceholder.typicode.com/posts?userId=1&_limit=5

posts = r.json()
for post in posts:
    print(f"  {post['id']}: {post['title'][:40]}...")

Timeouts: nunca esperes infinito

En redes, los paquetes pueden perderse y las conexiones quedarse colgadas. El parámetro timeout en requests puede ser un número (segundos totales) o una tupla (connect_timeout, read_timeout).

timeout_config.py — Control de tiempos de espera
import requests
from requests.exceptions import Timeout, ConnectionError

try:
    # 3s para conectar, 27s para leer la respuesta
    r = requests.get(
        "https://httpbin.org/delay/2",
        timeout=(3, 27)
    )
    print(r.json())

except Timeout:
    print("⏱️ El servidor tardó demasiado en responder.")

except ConnectionError:
    print("🔌 No se pudo establecer conexión TCP. ¿Firewall? ¿URL incorrecta?")
❌ Error común crítico

Si no especificas timeout, requests esperará indefinidamente (o hasta que el SO corte el socket). En scripts de monitoreo o automatización, esto bloquea tu ejecución. Siempre define timeout.

Paginación: cuando los datos no caben en una página

Las APIs de redes (DNA Center, Meraki, etc.) no devuelven 10,000 dispositivos en una sola respuesta. Usan paginación. Los tres patrones más comunes:

  • Offset/Limit: ?offset=0&limit=100, luego ?offset=100&limit=100.
  • Page/PerPage: ?page=1&per_page=100.
  • Cursor/Link: El header Link contiene la URL de la siguiente página (usado por Meraki, GitHub).
pagination_loop.py — Consumir todas las páginas
import requests

def get_all_items(base_url, per_page=100):
    """Patrón offset/limit para APIs tipo DNA Center."""
    items = []
    offset = 0

    while True:
        r = requests.get(
            base_url,
            params={"offset": offset, "limit": per_page},
            timeout=30
        )
        r.raise_for_status()
        page = r.json()

        if not page:
            break

        items.extend(page)
        offset += per_page

        # Si la última página trae menos de per_page, terminamos
        if len(page) < per_page:
            break

    return items

# Uso
todos = get_all_items("https://api.ejemplo.com/devices")
print(f"Total de dispositivos: {len(todos)}")

Descarga de archivos: configuraciones, backups, imágenes

En automatización de redes, a menudo necesitas descargar firmware, backups de configuración o logs. Usa response.content (bytes) y escribe en modo binario.

download_file.py — Guardar respuesta binaria
import requests

url = "https://httpbin.org/bytes/1024"

r = requests.get(url, timeout=30, stream=True)
r.raise_for_status()

with open("backup.bin", "wb") as f:
    for chunk in r.iter_content(chunk_size=8192):
        if chunk:
            f.write(chunk)

print("✅ Descarga completada con streaming.")

POST: Crear recursos con JSON

Para enviar datos al servidor (crear una VLAN, un usuario, un webhook), usamos requests.post(). El parámetro json= automáticamente: (1) serializa el dict a string JSON, (2) añade el header Content-Type: application/json, y (3) codifica el body.

post_json.py — Crear recurso vía JSON
import requests

url = "https://jsonplaceholder.typicode.com/posts"

nuevo_post = {
    "title": "Configuración VLAN 10",
    "body": "Añadida VLAN 10 a switch core-dist-01",
    "userId": 1
}

r = requests.post(url, json=nuevo_post, timeout=10)

print(f"Status: {r.status_code}")  # 201 Created
print(f"ID asignado: {r.json()['id']}")  # 101
💡 json= vs data= vs files=
  • json=payload: Envía JSON. Automáticamente serializa y pone Content-Type: application/json.
  • data=payload: Envía form-encoded (como un formulario HTML). Content-Type: application/x-www-form-urlencoded.
  • files={"archivo": open(...)}: Multipart/form-data para subir imágenes, firmware, etc.

PUT: Reemplazo completo

PUT envía la representación completa del recurso. Si omites un campo, el servidor podría borrarlo (depende de la implementación). Es idempotente: hacer PUT dos veces con los mismos datos deja el servidor en el mismo estado.

put_replace.py — Actualización total
import requests

url = "https://jsonplaceholder.typicode.com/posts/1"

update_total = {
    "id": 1,
    "title": "Nueva política QoS",
    "body": "Reemplazo completo del recurso",
    "userId": 1
}

r = requests.put(url, json=update_total, timeout=10)
print(f"PUT status: {r.status_code}")  # 200

PATCH: Actualización parcial

Cuando solo quieres cambiar un campo sin tocar el resto del recurso. Ideal para cambios rápidos: renombrar un dispositivo, cambiar una IP, activar/desactivar una política.

patch_partial.py — Actualización selectiva
import requests

url = "https://jsonplaceholder.typicode.com/posts/1"

r = requests.patch(
    url,
    json={"title": "Título actualizado sin tocar el body"},
    timeout=10
)
print(r.json())

Autenticación en APIs REST

Las APIs de redes Cisco (DNA Center, Meraki, SD-WAN) requieren autenticación. requests soporta los tres mecanismos principales: Basic Auth, Bearer Token (API Key / OAuth) y Custom Headers.

🔑

Basic Auth

Usuario:contraseña codificados en Base64 en el header Authorization. Usado por switches/routers para APIs legacy y algunos controladores on-prem.

🎫

Bearer Token

Token de sesión o JWT enviado en Authorization: Bearer <token>. Usado por DNA Center y la mayoría de APIs modernas.

🛡️

Custom Header

API Keys propietarias como X-Cisco-Meraki-API-Key o X-Auth-Token. No usan el header Authorization estándar.

Basic Auth

basic_auth.py — Usuario y contraseña
import requests
from requests.auth import HTTPBasicAuth

url = "https://api.example.com/networks"

# Opción A: tupla (usuario, password)
r = requests.get(url, auth=("admin", "C1sco123!"), timeout=10)

# Opción B: objeto HTTPBasicAuth (explícito)
r = requests.get(
    url,
    auth=HTTPBasicAuth("admin", "C1sco123!"),
    timeout=10
)

# requests codifica automáticamente en Base64:
# Authorization: Basic YWRtaW46QzFzY28xMjMh

Bearer Token / API Key

El patrón más común en APIs cloud-modernas. Obtienes un token (por login, API key o OAuth) y lo envías en cada petición.

bearer_token.py — Patrón estándar
import requests
import os

# Nunca hardcodees tokens. Usa variables de entorno.
TOKEN = os.getenv("API_TOKEN", "tu-token-aqui")
BASE_URL = "https://api.example.com/v1"

headers = {
    "Authorization": f"Bearer {TOKEN}",
    "Content-Type": "application/json",
    "Accept": "application/json"
}

def get_devices():
    r = requests.get(
        f"{BASE_URL}/devices",
        headers=headers,
        timeout=30
    )
    r.raise_for_status()
    return r.json()

devices = get_devices()
print(f"📡 Dispositivos descubiertos: {len(devices)}")

DNA Center: obtención de token + uso

DNA Center requiere un paso previo: hacer POST a /dna/system/api/v1/auth/token con Basic Auth para recibir un X-Auth-Token. Luego usas ese token en todas las llamadas subsiguientes.

dnac_auth_pattern.py — Flujo completo
import requests

DNAC = "https://sandboxdnac.cisco.com"
USER = "devnetuser"
PASS = "Cisco123!"

# ── PASO 1: Obtener token ──
auth_resp = requests.post(
    f"{DNAC}/dna/system/api/v1/auth/token",
    auth=(USER, PASS),
    timeout=10
)
auth_resp.raise_for_status()
token = auth_resp.json()["Token"]

# ── PASO 2: Usar token en endpoints ──
headers = {
    "X-Auth-Token": token,
    "Content-Type": "application/json"
}

r = requests.get(
    f"{DNAC}/dna/intent/api/v1/network-device",
    headers=headers,
    timeout=30
)

for dev in r.json()["response"]:
    print(f"{dev['hostname']} — {dev['managementIpAddress']}")

Manejo profesional de errores HTTP

Un script de red que se detiene con un traceback feo no es automatización, es un desastre. requests lanza excepciones específicas que debes capturar para reintentar, loggear o alertar.

ExcepciónCuándo ocurreAcción recomendada
ConnectionErrorNo se puede resolver DNS o establecer TCP.Verificar URL, firewall, interfaz de red.
TimeoutEl servidor no responde en el tiempo límite.Aumentar timeout o verificar carga del servidor.
TooManyRedirectsBucle de redirecciones HTTP.Verificar URL final, desactivar redirects si es necesario.
HTTPErrorStatus 4xx o 5xx tras raise_for_status().Loggear status, leer mensaje de error, reintentar si es 5xx.
JSONDecodeErrorLa respuesta no es JSON válido.Verificar Content-Type, leer response.text crudo.
SSLErrorCertificado TLS inválido o no confiable.En labs: verify=False. En prod: usar certificado válido.

raise_for_status(): la línea que nunca debe faltar

Después de cada petición, r.raise_for_status() verifica si r.ok es False. Si el status es 4xx o 5xx, lanza HTTPError con el código y la URL. Es la forma más limpia de fallar rápido (fail fast).

error_handling.py — Patrón robusto de producción
import requests
from requests.exceptions import (
    ConnectionError, Timeout, HTTPError,
    RequestException, JSONDecodeError
)

def safe_api_call(method, url, **kwargs):
    """Ejecuta una llamada REST con manejo completo de errores."""
    try:
        r = requests.request(method, url, timeout=30, **kwargs)
        r.raise_for_status()

        # Solo intenta parsear JSON si el content-type lo indica
        if "application/json" in r.headers.get("Content-Type", ""):
            return r.json()
        return r.text

    except ConnectionError:
        print(f"🔌 [CONN] No se pudo conectar a {url}")
        return None

    except Timeout:
        print(f"⏱️ [TIME] Timeout en {url}")
        return None

    except HTTPError as e:
        print(f"❌ [HTTP {r.status_code}] {e}")
        # Aquí puedes agregar lógica específica por código
        if r.status_code == 429:
            print("   → Rate limit excedido. Reintentar con backoff.")
        elif r.status_code == 401:
            print("   → Token expirado o inválido. Refrescar credenciales.")
        return None

    except JSONDecodeError:
        print(f"📄 [JSON] Respuesta no es JSON válido: {r.text[:200]}")
        return None

    except RequestException as e:
        print(f"⚠️  [REQ] Error inesperado: {e}")
        return None

# Uso
resultado = safe_api_call("GET", "https://httpbin.org/status/404")

Reintentos inteligentes con backoff exponencial

Los errores 5xx y los timeouts transitorios son comunes en controladores sobrecargados. Un retry con backoff exponencial (esperar 1s, 2s, 4s, 8s...) reduce la carga y aumenta la resiliencia.

retry_backoff.py — Resiliencia en automatización
import time
import requests
from requests.exceptions import RequestException

def request_with_retry(method, url, max_retries=3, **kwargs):
    """Reintenta con backoff exponencial ante errores 5xx o excepciones."""
    for intento in range(max_retries + 1):
        try:
            r = requests.request(method, url, timeout=30, **kwargs)

            # Si es 5xx, reintentar
            if 500 <= r.status_code < 600 and intento < max_retries:
                wait = 2 ** intento
                print(f"⚠️  {r.status_code} — Reintento {intento+1}/{max_retries} en {wait}s...")
                time.sleep(wait)
                continue

            r.raise_for_status()
            return r

        except RequestException as e:
            if intento == max_retries:
                raise
            wait = 2 ** intento
            print(f"⚠️  {e} — Reintento {intento+1}/{max_retries} en {wait}s...")
            time.sleep(wait)

    return None

# Uso
r = request_with_retry("GET", "https://httpbin.org/status/500")
if r:
    print("✅ Éxito tras reintentos")
💡 Tip: la librería urllib3.retry

Para casos complejos, requests usa urllib3 debajo. Puedes configurar un HTTPAdapter con Retry(total=5, backoff_factor=1, status_forcelist=[500, 502, 503, 504]) para retries automáticos a nivel de sesión sin escribir loops manuales.

SSL/TLS y el parámetro verify

Por defecto, requests verifica el certificado TLS del servidor (verify=True). Esto protege contra ataques Man-in-the-Middle. Sin embargo, en laboratorios, sandboxes y controladores internos con certificados autofirmados, la verificación fallará con SSLError.

verify=False: uso en laboratorios

Deshabilitar la verificación SSL es útil en entornos controlados, pero genera una advertencia de seguridad. Suprímela solo si entiendes el riesgo.

ssl_lab.py — Desactivar verificación en sandbox
import requests
from urllib3.exceptions import InsecureRequestWarning

# Suprimir advertencia en labs (NO en producción)
requests.packages.urllib3.disable_warnings(InsecureRequestWarning)

# DNA Center sandbox usa cert autofirmado
r = requests.get(
    "https://sandboxdnac.cisco.com/dna/system/api/v1/auth/token",
    auth=("devnetuser", "Cisco123!"),
    verify=False,
    timeout=10
)

print(f"Status: {r.status_code}")
🛑 NUNCA en producción

verify=False expone tu script a ataques Man-in-the-Middle. Un atacante en la misma red (ARP spoofing) podría interceptar credenciales y tokens. En producción, usa siempre verify=True (por defecto) y, si es necesario, apunta verify al archivo CABundle de tu empresa:

requests.get(url, verify="/path/to/enterprise-ca-bundle.crt")

Certificados de cliente (mTLS)

Algunas APIs de red requieren autenticación mutua TLS: el cliente debe presentar un certificado X.509 válido. requests lo soporta vía el parámetro cert.

mtls_auth.py — Autenticación con certificado cliente
import requests

# cert=(certificado_cliente, clave_privada)
r = requests.get(
    "https://api.segura.empresa.com/networks",
    cert=("client.crt", "client.key"),
    verify="ca-empresa.crt",
    timeout=30
)

print(r.status_code)

Inspección del certificado del servidor

Para troubleshooting de conectividad HTTPS, a veces necesitas verificar qué certificado presenta el servidor, su CN, SAN y fecha de expiración.

inspect_cert.py — Ver certificado sin requests
import ssl
import socket
import datetime

hostname = "api.meraki.com"
context = ssl.create_default_context()

with socket.create_connection((hostname, 443), timeout=5) as sock:
    with context.wrap_socket(sock, server_hostname=hostname) as ssock:
        cert = ssock.getpeercert()
        expire = cert["notAfter"]
        print(f"Subject: {cert['subject']}")
        print(f"Expira: {expire}")
        print(f"Cipher: {ssock.cipher()[0]}")

Sessions: persistencia y eficiencia

Una Session reutiliza la conexión TCP subyacente (keep-alive) entre peticiones. Esto reduce la latencia del handshake TLS y permite compartir cookies, headers y autenticación entre llamadas. Es obligatorio en scripts que hacen 50+ peticiones.

session_persistence.py — Reutilizar conexiones
import requests

# Sin session: cada requests.get() abre y cierra TCP+TLS
# Con session: keep-alive automático

session = requests.Session()
session.headers.update({
    "Accept": "application/json",
    "User-Agent": "CCNA-NetworkBot/1.0"
})
session.auth = ("admin", "C1sco123!")

urls = [
    "https://httpbin.org/get",
    "https://httpbin.org/ip",
    "https://httpbin.org/user-agent"
]

for url in urls:
    r = session.get(url, timeout=10)
    print(f"{url}: {r.status_code}")

session.close()

Cookies y estado de sesión

Las cookies son headers Set-Cookie que el servidor envía y el cliente debe devolver en peticiones subsiguientes. Session las maneja automáticamente.

cookies_session.py — Persistencia de estado
import requests

s = requests.Session()

# Login que devuelve cookie de sesión
login = s.post(
    "https://httpbin.org/cookies/set/sessionid/abc123",
    timeout=10
)

# La siguiente petición envía automáticamente la cookie
r = s.get("https://httpbin.org/cookies", timeout=10)
print(r.json())  # {"cookies": {"sessionid": "abc123"}}

Rate Limiting manual + throttling

Además de manejar 429, es mejor prevenir el rate limit controlando la frecuencia de envío desde el cliente.

rate_limit_client.py — Throttling proactivo
import time
import requests

class ThrottledClient:
    def __init__(self, requests_per_second=5):
        self.min_interval = 1.0 / requests_per_second
        self.last_request = 0
        self.session = requests.Session()

    def request(self, method, url, **kwargs):
        elapsed = time.time() - self.last_request
        if elapsed < self.min_interval:
            time.sleep(self.min_interval - elapsed)

        r = self.session.request(method, url, timeout=30, **kwargs)
        self.last_request = time.time()
        return r

# Uso: máximo 2 req/s para no saturar al controller
client = ThrottledClient(requests_per_second=2)
for i in range(10):
    r = client.request("GET", "https://httpbin.org/get")
    print(f"{i}: {r.status_code}")

Hooks: interceptar request/response

Los hooks permiten ejecutar código automáticamente en cada respuesta. Útil para logging centralizado, métricas o validaciones globales.

response_hooks.py — Logging automático
import requests

def log_response(response, *args, **kwargs):
    print(f"[HOOK] {response.request.method} {response.url} → {response.status_code}")

session = requests.Session()
session.hooks["response"].append(log_response)

r = session.get("https://httpbin.org/get", timeout=10)
# Imprime automáticamente antes de retornar

🚀 Proyecto Final — Cliente REST Genérico para Redes

Este proyecto consolida todo: sesiones persistentes, autenticación configurable (Basic, Bearer, API Key), manejo robusto de errores, reintentos automáticos, logging, paginación y exportación a JSON/CSV. Es un cliente reutilizable que puedes apuntar a DNA Center, Meraki, o cualquier API REST de infraestructura.

💡 Escenario de producción

Eres el líder de automatización de red. Tu equipo usa 3 plataformas distintas: DNA Center (Bearer/X-Auth), Meraki (API Key header) y un IPAM interno (Basic Auth). Necesitas un cliente unificado que cualquier compañero pueda usar sin entender los detalles de cada API. Esta clase RestClient es la solución.

rest_client.py — Cliente universal de red
#!/usr/bin/env python3
"""
rest_client.py
CCNA Academy M8 T6 — Cliente REST Genérico para Automatización de Redes
────────────────────────────────────────────────────────────────────────
Soporta: Basic Auth, Bearer Token, API Key custom, reintentos, paginación,
logging estructurado y exportación CSV/JSON.
"""

import csv
import json
import logging
import time
from typing import Optional, Dict, Any
import requests
from requests.auth import HTTPBasicAuth
from requests.adapters import HTTPAdapter
from requests.packages.urllib3.util.retry import Retry

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s"
)


class RestClient:
    """Cliente REST genérico para APIs de infraestructura de red."""

    def __init__(
        self,
        base_url: str,
        auth_type: str = "none",
        credentials: Optional[Dict[str, str]] = None,
        verify_ssl: bool = True,
        timeout: int = 30,
        max_retries: int = 3
    ):
        self.base_url = base_url.rstrip("/")
        self.timeout = timeout
        self.verify_ssl = verify_ssl
        self.session = requests.Session()

        # ── Configurar retries a nivel de conexión ──
        retry_strategy = Retry(
            total=max_retries,
            backoff_factor=1,
            status_forcelist=[500, 502, 503, 504],
            allowed_methods=["GET", "POST", "PUT", "DELETE", "PATCH"]
        )
        adapter = HTTPAdapter(max_retries=retry_strategy)
        self.session.mount("https://", adapter)
        self.session.mount("http://", adapter)

        # ── Configurar autenticación ──
        if auth_type == "basic":
            self.session.auth = HTTPBasicAuth(
                credentials["username"],
                credentials["password"]
            )

        elif auth_type == "bearer":
            self.session.headers.update({
                "Authorization": f"Bearer {credentials['token']}"
            })

        elif auth_type == "api_key":
            header_name = credentials.get("header", "X-API-Key")
            self.session.headers.update({
                header_name: credentials["key"]
            })

        # Headers globales
        self.session.headers.update({
            "Accept": "application/json",
            "Content-Type": "application/json",
            "User-Agent": "CCNA-RestClient/1.0"
        })

        if not verify_ssl:
            requests.packages.urllib3.disable_warnings()
            self.session.verify = False

    def request(
        self,
        method: str,
        endpoint: str,
        params: Optional[Dict] = None,
        json_data: Optional[Dict] = None,
        extra_headers: Optional[Dict] = None
    ) -> Optional[Dict[str, Any]]:
        """Ejecuta una petición REST con manejo completo de errores."""
        url = f"{self.base_url}/{endpoint.lstrip('/')}"
        headers = extra_headers or {}

        try:
            logging.info(f"{method} {url}")
            r = self.session.request(
                method, url,
                params=params,
                json=json_data,
                headers=headers,
                timeout=self.timeout
            )
            r.raise_for_status()

            if r.status_code == 204:
                return {"success": True, "status": 204}

            return r.json()

        except requests.exceptions.HTTPError as e:
            logging.error(f"HTTP {r.status_code}: {e}")
            return {"error": f"HTTP {r.status_code}", "detail": str(e)}

        except requests.exceptions.ConnectionError:
            logging.error(f"ConnectionError: {url}")
            return {"error": "Connection failed"}

        except requests.exceptions.Timeout:
            logging.error(f"Timeout: {url}")
            return {"error": "Timeout"}

        except Exception as e:
            logging.error(f"Unexpected: {e}")
            return {"error": "Unexpected exception"}

    def get_all_pages(
        self,
        endpoint: str,
        params: Optional[Dict] = None,
        limit_key: str = "limit",
        offset_key: str = "offset",
        page_size: int = 100
    ) -> list:
        """Paginación offset/limit genérica."""
        all_items = []
        params = dict(params or {})
        params[limit_key] = page_size
        params[offset_key] = 0

        while True:
            data = self.request("GET", endpoint, params=params)
            if not data or "error" in data:
                break

            # Soporte para respuestas envueltas (DNA Center style)
            items = data if isinstance(data, list) else data.get("response", [])
            if not items:
                break

            all_items.extend(items)
            params[offset_key] += page_size

            if len(items) < page_size:
                break

        return all_items

    def export_json(self, data, filename):
        with open(filename, "w", encoding="utf-8") as f:
            json.dump(data, f, indent=2, ensure_ascii=False)
        logging.info(f"💾 JSON exportado: {filename}")

    def export_csv(self, data: list, filename: str):
        if not data:
            return
        with open(filename, "w", newline="", encoding="utf-8") as f:
            writer = csv.DictWriter(f, fieldnames=data[0].keys())
            writer.writeheader()
            writer.writerows(data)
        logging.info(f"📊 CSV exportado: {filename}")


# ══════════════════════════════════════════════════════
# DEMOSTRACIÓN DE USO
# ══════════════════════════════════════════════════════
if __name__ == "__main__":
    # Ejemplo contra JSONPlaceholder (API pública de prueba)
    client = RestClient(
        base_url="https://jsonplaceholder.typicode.com",
        auth_type="none",
        verify_ssl=True
    )

    # GET simple
    post = client.request("GET", "/posts/1")
    print(f"📄 Post: {post['title']}")

    # POST con JSON
    nuevo = client.request(
        "POST", "/posts",
        json_data={"title": "Nueva VLAN", "body": "VLAN 100 creada", "userId": 1}
    )
    print(f"✅ Creado con ID: {nuevo.get('id')}")

    # Exportar
    client.export_json(post, "post_1.json")
▶️ Cómo adaptar a tu entorno
  1. DNA Center: auth_type="bearer" pero primero obtén el token vía POST /auth/token y luego pásalo en credentials.
  2. Meraki: auth_type="api_key", credentials={"header": "X-Cisco-Meraki-API-Key", "key": "tu-key"}.
  3. Controladores legacy: auth_type="basic" con username/password.
  4. Para paginación por cursor (Meraki, GitHub), sobrescribe get_all_pages() para leer el header Link en lugar de offset/limit.
