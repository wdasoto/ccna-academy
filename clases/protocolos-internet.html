M1 · FUNDAMENTOS · SUITE TCP/IP COMPLETA

PROTOCOLOS DE INTERNET

La suite completa de protocolos que hace funcionar Internet. Desde IP y TCP/UDP hasta HTTP, DNS, SSH, SNMP y NTP. Cada byte que viaja por la red obedece estas reglas.

📋 RFC 791 · 793 · 768 · 792 🎯 CCNA 200-301 Topics 1.1 · 1.3 · 4.6 ⏱ ~80 min lectura 🌐 12 protocolos cubiertos

La Suite de Protocolos TCP/IP

Internet funciona gracias a una suite de protocolos complementarios, cada uno responsable de una función específica. Esta suite se organiza en capas donde cada capa usa los servicios de la inferior y provee servicios a la superior.

Modelo TCP/IP — 4 capas y sus protocolos
4 · APLICACIÓN
HTTP/S · FTP · SSH · Telnet · DNS · SMTP · POP3 · IMAP · SNMP · NTP · TFTP · DHCP
3 · TRANSPORTE
TCP (confiable, orientado a conexión) · UDP (rápido, sin conexión)
2 · INTERNET
IP (IPv4/IPv6) · ICMP · ARP · OSPF · EIGRP · BGP
1 · ACCESO RED
Ethernet · Wi-Fi (802.11) · PPP · HDLC · Frame Relay

💡 Regla de oro — ¿Qué cambia en cada salto?

Cuando un router reenvía un paquete: las IPs de origen y destino NO cambian (permanecen las de los extremos finales). Las MACs SÍ cambian en cada salto (se actualizan con las del segmento local). El TTL se decrementa en 1.

IP — Internet Protocol (RFC 791)

IP es el protocolo de capa 3 que provee entrega de paquetes de extremo a extremo. Características esenciales:

🔀

Sin Conexión

Cada paquete se enruta independientemente. No hay circuito previo. Pueden llegar desordenados.

⚡

Best-Effort

IP no garantiza entrega, orden ni tiempo. La fiabilidad es responsabilidad de TCP (capa 4).

🌐

Direccionamiento Lógico

Direcciones de 32 bits (IPv4) identifican origen y destino de forma única y jerárquica.

✂️

Fragmentación

Si el paquete supera el MTU (1500B en Ethernet), IP lo fragmenta. Se reensambla en el destino.

💡 TTL en troubleshooting

El TTL es clave para troubleshooting. traceroute envía paquetes con TTL=1,2,3... — cada router responde con ICMP "Time Exceeded", revelando la ruta completa. El TTL del reply revela cuántos hops hay en el retorno. Cisco IOS usa TTL=255 por defecto.

TCP — Transmission Control Protocol (RFC 793)

TCP es el protocolo de transporte orientado a conexión que garantiza entrega confiable, ordenada y sin errores.

🤝

Orientado a conexión

Establece sesión con 3-Way Handshake antes de enviar datos. Ambos extremos sincronizan números de secuencia.

✅

Entrega confiable

Usa números de secuencia y ACKs. Retransmite segmentos no confirmados automáticamente.

📋

Entrega ordenada

Reordena segmentos que llegan fuera de orden usando números de secuencia.

🌊

Control de flujo

Window Size — el receptor anuncia cuántos bytes puede aceptar. El emisor no excede ese límite.

El 3-Way Handshake

1. SYN
Cliente → Servidor
SYN=1, Seq=100 · "Quiero conectarme. Mi secuencia inicial es 100"
2. SYN-ACK
Servidor → Cliente
SYN=1, ACK=1, Seq=300, Ack=101 · "Acepto. Mi seq es 300. Espero tu byte 101"
3. ACK
Cliente → Servidor
ACK=1, Seq=101, Ack=301 · "Confirmado. ✅ Conexión establecida — datos pueden fluir"

Flags TCP — Los 6 bits de control

SYNSynchronize. Inicia conexión. Solo en primeros mensajes del handshake.
ACKAcknowledge. Confirma recepción. Presente en casi todos los segmentos.
FINFinish. Cierre ordenado. El extremo no enviará más datos.
RSTReset. Cierre abrupto. Indica error o conexión inválida.
PSHPush. Entrega inmediata a la aplicación sin esperar llenar buffer.
URGUrgent. Datos prioritarios apuntados por Urgent Pointer.

UDP — User Datagram Protocol (RFC 768)

UDP es el protocolo de transporte sin conexión. Sin handshake, sin ACKs, sin orden garantizado. Solo 8 bytes de header. Ideal donde la velocidad importa más que la perfección.

💡 ¿Por qué VoIP usa UDP?

En voz IP, una retransmisión TCP llegaría demasiado tarde para ser audible. Es mejor perder un paquete (pequeño glitch de 20ms) que retransmitir y causar retraso. UDP permite latencia constante y baja, que es exactamente lo que requiere la voz humana para ser inteligible.

Cabecera UDP — Solo 8 bytes

CampoBitsDescripción
Source Port16Puerto origen (opcional — puede ser 0)
Destination Port16Puerto destino del servicio
Length16Longitud total datagrama (header + datos). Mínimo: 8
Checksum16Integridad opcional en IPv4, obligatorio en IPv6

ICMP — Internet Control Message Protocol (RFC 792)

ICMP transporta mensajes de control y error sobre el funcionamiento de IP. Viaja encapsulado en IP (Protocol=1) pero conceptualmente es capa 3. No transporta datos de usuario.

ping y traceroute — Herramientas ICMP

Cómo funciona ping internamente
ping 8.8.8.8 desde 192.168.1.10:

→ Envía: IP[src=192.168.1.10, dst=8.8.8.8, proto=1, TTL=128]
         ICMP[Type=8 Echo Request, ID=1234, Seq=1]

← Recibe: IP[src=8.8.8.8, dst=192.168.1.10]
          ICMP[Type=0 Echo Reply, ID=1234, Seq=1]

Resultado: Reply from 8.8.8.8: bytes=32 time=12ms TTL=118
TTL=118 → 128(inicial) - 10(hops retorno) = 118 → hay ~10 routers de retorno
Cómo traceroute usa el TTL
traceroute 8.8.8.8 — envía paquetes con TTL incremental:

TTL=1: Router1 decrementa a 0 → ICMP Type=11 (Time Exceeded) → IP de R1
TTL=2: R1 reenvía, R2 decrementa a 0 → ICMP Time Exceeded → IP de R2
TTL=N: Paquete llega a 8.8.8.8 → ICMP Type=0 (Echo Reply) → FIN

Output:
  1  192.168.1.1    1ms    ← Gateway local
  2  10.100.0.1     5ms    ← ISP router
  3  * * *                 ← Filtra ICMP TTL Exceeded (normal)
  4  8.8.8.8        12ms   ← Destino alcanzado
⚠️ Seguridad ICMP — No bloquear todo

Bloquear todo ICMP rompe funcionalidad: sin Type 3 Code 4 falla Path MTU Discovery y las conexiones TCP se cuelgan. Lo correcto es filtrar selectivamente: bloquear Echo Request desde Internet pero permitir TTL Exceeded, Unreachable y Fragmentation Needed.

Protocolos de Capa de Aplicación

🌐
HTTP / HTTPS
TCP 80 / TCP 443

HyperText Transfer Protocol — el protocolo de la Web. Petición/respuesta.

  • HTTP: Puerto 80, texto plano — inseguro para datos sensibles
  • HTTPS: Puerto 443, HTTP sobre TLS/SSL — todo cifrado
  • Métodos: GET · POST · PUT · DELETE · HEAD · OPTIONS
  • Códigos: 200 OK · 301/302 Redirect · 404 Not Found · 500 Server Error
  • HTTP/2: Multiplexación múltiples peticiones en una sola conexión TCP
🌐
DNS
UDP 53 / TCP 53

Domain Name System — traduce nombres a IPs. Indispensable para Internet.

  • Resolución: Cache local → Servidor DNS local → Root servers → TLD → Authoritative DNS
  • Registros: A (IPv4) · AAAA (IPv6) · MX (mail) · CNAME (alias) · PTR (reverse) · NS · SOA
  • UDP para consultas normales, TCP para transferencias de zona (AXFR)
  • Cisco: ip name-server 8.8.8.8 · ip domain-name empresa.com
  • Sin DNS, deben usarse IPs — por eso DNS es crítico en SSH (necesita domain-name para RSA)
📁
FTP / SFTP / TFTP
TCP 20/21 · TCP 22 · UDP 69

Protocolos de transferencia de archivos — cada uno con diferente propósito.

  • FTP: Puerto 21 (control) + 20 (datos en modo activo). Sin cifrado. Obsoleto en producción.
  • FTP Pasivo: Cliente inicia ambas conexiones. Mejor con NAT/firewalls.
  • SFTP: FTP sobre SSH. Puerto 22. Cifrado completo. Estándar moderno.
  • TFTP: UDP 69. Sin autenticación, sin listado. Usado para boot de dispositivos, imágenes IOS Cisco, configs de red.
🔐
SSH / Telnet
TCP 22 / TCP 23

Acceso remoto CLI a dispositivos de red.

  • Telnet: TCP 23. Todo en texto plano — contraseñas visibles. SOLO para labs aislados.
  • SSH v2: TCP 22. Cifrado asimétrico para intercambio de clave + simétrico para datos.
  • Cisco: transport input ssh prohíbe Telnet. ip ssh version 2 fuerza v2.
  • Requiere: hostname, ip domain-name, crypto key generate rsa, login local
⚙️
SNMP
UDP 161 (agente) · 162 (trap)

Simple Network Management Protocol — gestión centralizada de dispositivos.

  • Componentes: Manager (NMS) · Agent (dispositivo) · MIB (base de objetos)
  • Operaciones: GET (leer) · SET (modificar) · TRAP (alerta proactiva sin solicitud)
  • v1/v2c: Community string en texto plano — inseguro
  • v3: Autenticación MD5/SHA + cifrado AES — estándar recomendado
⏰
NTP
UDP 123

Network Time Protocol — sincronización de relojes precisa en la red.

  • Stratum: Nivel de precisión. Stratum 0 = reloj atómico. Stratum 1 = conectado a stratum 0. Stratum 15 = máximo útil.
  • Precisión: ±1-50ms en LAN, ±50-100ms en Internet
  • Crítico para: logs forenses, certificados TLS, Kerberos, HSRP
  • Cisco: ntp server 216.239.35.0 · show ntp status · show ntp associations
📋
DHCP
UDP 67 (servidor) · 68 (cliente)

Dynamic Host Configuration Protocol — asignación automática de parámetros IP.

  • DORA: Discover (broadcast) → Offer → Request → Acknowledge
  • Asigna: IP · Máscara · Gateway · DNS · Lease time
  • ip helper-address: Reenvía DHCP Discover hacia servidores en otras redes
  • DHCP Snooping: protege contra servidores DHCP falsos (base de DAI)
📝
Syslog
UDP 514

Protocolo de logs — centraliza mensajes de eventos de dispositivos de red.

  • Niveles de severidad (0-7): 0=Emergency · 1=Alert · 2=Critical · 3=Error · 4=Warning · 5=Notice · 6=Informational · 7=Debug
  • Cisco: logging host 192.168.1.200 · logging trap informational
  • service timestamps log datetime msec localtime: timestamps en los logs
  • Por defecto, Cisco envía logs a la consola (nivel 7) y al buffer interno

Puertos TCP/UDP — Tabla completa CCNA

Los puertos son identificadores de 16 bits (0–65535) que permiten multiplexar múltiples servicios sobre una misma IP. Rangos:

🏛️

Well-Known (0–1023)

Reservados para servicios estándar. Requieren privilegios de administrador.

📋

Registered (1024–49151)

Registrados por aplicaciones específicas con IANA. RDP=3389, MySQL=3306.

🔀

Ephemeral (49152–65535)

Dinámicos. Los clientes los usan aleatoriamente como puerto de origen.

Puertos que DEBES memorizar para CCNA

El concepto de Socket

Un socket = IP + Puerto + Protocolo. Una conexión TCP se identifica por la 4-tupla única: (IP origen, Puerto origen, IP destino, Puerto destino).

Ejemplo de socket TCP
Conexión HTTPS cliente→servidor:
Cliente: 192.168.1.10:54321 (TCP)
Servidor: 93.184.216.34:443 (TCP)

4-tupla única: (192.168.1.10, 54321, 93.184.216.34, 443, TCP)
Permite identificar esta sesión específica entre millones simultáneas

Configuración de protocolos en Cisco IOS

DNS en Cisco IOS
! Configurar servidores DNS
Router(config)# ip name-server 8.8.8.8 8.8.4.4
Router(config)# ip domain-name empresa.com
! Deshabilitar DNS lookup (evitar esperas en typos de comandos)
Router(config)# no ip domain-lookup
! Verificar
Router# show hosts
NTP en Cisco IOS
! Sincronizar con servidor NTP externo
Router(config)# ntp server 216.239.35.0 prefer
Router(config)# ntp server 216.239.35.4
Router(config)# clock timezone COT -5
! Este router como servidor NTP para otros dispositivos
Router(config)# ntp master 3
! Verificar
Router# show ntp status
Router# show ntp associations
Router# show clock
SSH — Configuración segura completa
! Prerrequisitos obligatorios
R1(config)# hostname R1
R1(config)# ip domain-name empresa.com
! Generar par de claves RSA (mínimo 1024, recomendado 2048)
R1(config)# crypto key generate rsa modulus 2048
! Forzar SSHv2 únicamente
R1(config)# ip ssh version 2
R1(config)# ip ssh time-out 60
R1(config)# ip ssh authentication-retries 3
! Crear usuario local
R1(config)# username admin privilege 15 secret Cisco@2024
! Configurar VTY para SSH solamente
R1(config)# line vty 0 4
R1(config-line)# login local
R1(config-line)# transport input ssh
R1(config-line)# exec-timeout 10 0
! Verificar
R1# show ip ssh
R1# show ssh
SNMP y Syslog en Cisco IOS
! SNMP v2c (básico)
Router(config)# snmp-server community PUBLIC ro
Router(config)# snmp-server host 192.168.1.100 version 2c PUBLIC
Router(config)# snmp-server enable traps
Router(config)# snmp-server location "Datacenter Piso 3"

! SNMP v3 (recomendado — autenticado y cifrado)
Router(config)# snmp-server group ADMINGROUP v3 priv
Router(config)# snmp-server user ADMINUSER ADMINGROUP v3 auth sha AuthPass priv aes 128 PrivPass

! Syslog
Router(config)# logging host 192.168.1.200
Router(config)# logging trap informational
Router(config)# service timestamps log datetime msec localtime
Router(config)# logging buffered 16384 informational
Router# show logging
