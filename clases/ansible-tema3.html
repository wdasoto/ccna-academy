PLAYBOOKS YAML — Escribir Automatizaciones

La unidad central de Ansible. Aprende a escribir plays y tasks en YAML para configurar cientos de dispositivos Cisco simultáneamente con el mismo esfuerzo que uno. Domina la idempotencia, los handlers, los condicionales y los bucles para crear automatizaciones de red robustas, legibles y mantenibles.

📋 YAML 🎯 CCNA Domain 6.6 ⏱ ~90 min lectura ⚡ Plays / Tasks / Handlers 🔁 When / Loop

¿Qué es un Playbook en Ansible?

Un playbook es un archivo YAML que contiene una serie de "plays" ordenados. Cada play mapea un grupo de hosts a un conjunto de tareas (tasks) que se ejecutan secuencialmente. Si el inventario es el mapa de tu infraestructura, el playbook es el plan de batalla: define qué hacer, en qué orden, y con qué herramientas.

A diferencia de los comandos ad-hoc (de un solo uso en línea de comandos), los playbooks son repetibles, versionables y auditables. Puedes guardarlos en Git, revisarlos en pull requests y ejecutarlos en pipelines CI/CD. Un comando ad-hoc sirve para una prueba rápida; un playbook es la automatización de producción.

En el contexto de redes Cisco, un playbook puede configurar simultáneamente VLANs en 50 switches, aplicar ACLs en 20 routers, verificar el estado de interfaces y guardar la configuración —todo en una sola ejecución idempotente.

💡 Concepto clave del examen CCNA

El dominio 6.6 evalúa si entiendes la diferencia entre comandos ad-hoc y playbooks. La pregunta clásica es: "¿Qué archivo YAML declarativo de Ansible contiene plays, tasks y handlers para gestionar configuración?" La respuesta es el playbook.

Anatomía mínima de un Playbook

Todo playbook comienza con tres guiones (---) y contiene una lista de plays. Cada play es un diccionario YAML con al menos dos claves: hosts (dónde ejecutar) y tasks (qué hacer).

playbook_minimo.yml — Hola mundo de red
# --- Inicio del documento YAML
---
- name: "Verificar conectividad con switches Cisco"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Ejecutar show version"
      cisco.ios.ios_command:
        commands:
          - "show version"

Los cuatro pilares de un playbook de red

🎭

Plays

Mapean grupos de hosts a tareas. Un playbook puede tener múltiples plays: primero routers, luego switches, luego firewalls.

⚡

Tasks

Llamadas a módulos que realizan trabajo. Cada task usa un módulo (ios_config, ios_command) y reporta ok/changed/failed.

🔔

Handlers

Tareas especiales que solo se ejecutan cuando son notificadas por un cambio. Ideales para write memory tras modificar config.

🔁

When / Loop

Condicionales (when) y bucles (loop) que adaptan la ejecución al contexto. Aplicar VLANs solo en switches core, por ejemplo.

🎯

Idempotencia

Ejecuta el playbook una o mil veces: el estado final es idéntico. Ansible solo cambia lo que difiere del estado deseado.

🏷️

Tags

Etiqueta tareas para ejecutar subsets. ansible-playbook site.yml --tags "vlan" solo configura VLANs, saltando el resto.

Anatomía de un Play

Un play es un mapeo entre un conjunto de hosts y las tareas que se les aplicarán. Es la unidad de ejecución de nivel superior dentro de un playbook. Un playbook puede contener múltiples plays, ejecutándose secuencialmente de arriba hacia abajo.

Cada play es un diccionario YAML con directivas que controlan el comportamiento global de esa fase de la automatización.

play_completo.yml — Directivas de un play
---
- name: "Configurar routers Cisco WAN"
  hosts: cisco_routers
  gather_facts: no
  connection: network_cli

  vars:
    ntp_server_primario: "10.0.0.254"
    syslog_server: "10.0.0.253"

  vars_files:
    - "secrets/vault.yml"

  become: yes
  become_method: enable

  strategy: linear
  serial: 3

  tasks:
    - name: "Configurar NTP"
      cisco.ios.ios_config:
        lines:
          - "ntp server {{ ntp_server_primario }}"

  handlers:
    - name: "Guardar configuración"
      cisco.ios.ios_command:
        commands: "write memory"

Múltiples plays en un solo playbook

La verdadera potencia de Ansible aparece cuando orquestas diferentes tipos de dispositivos en secuencia. Primero configuras routers, luego switches, luego verificas conectividad end-to-end:

multiples_plays.yml — Orquestación completa
---
- name: "PASO 1: Configurar routers WAN"
  hosts: cisco_routers
  gather_facts: no
  tasks:
    - name: "Aplicar configuración BGP"
      cisco.ios.ios_config:
        src: "templates/bgp_config.j2"

- name: "PASO 2: Configurar switches core"
  hosts: cisco_switches
  gather_facts: no
  tasks:
    - name: "Configurar VLANs de producción"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 10
            name: "VENTAS"

- name: "PASO 3: Verificar conectividad"
  hosts: localhost
  gather_facts: no
  tasks:
    - name: "Ping desde servidor de monitoreo"
      ansible.builtin.ping:
        data: "alive"
ℹ️ Estrategia de ejecución: linear vs. free

Por defecto, Ansible usa strategy: linear: cada task debe completarse en TODOS los hosts antes de pasar a la siguiente task. Con strategy: free, cada host avanza independientemente. Para redes, linear es más seguro porque te permite detectar fallos temprano antes de propagar cambios.

gather_facts en entornos de red

En servidores Linux, gather_facts: yes recopila información del sistema (CPU, memoria, OS). En dispositivos de red Cisco, el módulo equivalente es ios_facts o nxos_facts. Por defecto, desactiva gather_facts en plays de red para ahorrar tiempo, y usa los módulos específicos de facts cuando los necesites.

facts_red.yml — Recopilación selectiva en Cisco
- name: "Obtener información de hardware"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Recopilar facts del router"
      cisco.ios.ios_facts:
        gather_subset:
          - "hardware"
          - "interfaces"
      register: "router_facts"

    - name: "Mostrar versión de IOS"
      ansible.builtin.debug:
        msg: "El router {{ inventory_hostname }} ejecuta {{ router_facts.ansible_facts.ansible_net_version }}"

Tasks: la unidad atómica de trabajo

Una task es una llamada a un módulo de Ansible con argumentos específicos. Es la acción más pequeña que puedes ejecutar. Las tasks se procesan secuencialmente dentro de un play, y Ansible espera a que cada task termine en todos los hosts antes de pasar a la siguiente.

Cada task debe tener un name descriptivo. Aunque es opcional, es obligatorio para mantenibilidad: el output de Ansible mostrará este nombre mientras ejecuta, facilitando la depuración.

tasks_ejemplo.yml — Estructura de tasks
---
- name: "Configurar switches de acceso"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Asegurar que la VLAN 10 existe"
      cisco.ios.ios_config:
        lines:
          - "vlan 10"
          - " name VENTAS"

    - name: "Configurar descripción de interfaz"
      cisco.ios.ios_config:
        lines:
          - "interface GigabitEthernet1/0/1"
          - " description Enlace a Router WAN"
        parents: "interface GigabitEthernet1/0/1"

    - name: "Verificar estado de interfaces"
      cisco.ios.ios_command:
        commands:
          - "show ip interface brief"
      register: "output_interfaces"

Estados de retorno de una task

Ansible reporta el resultado de cada task con un color y estado específico. Entender estos estados es crucial para interpretar la salida de un playbook:

🟢

ok

La tarea se ejecutó pero no realizó cambios (idempotencia). El estado deseado ya existía.

🟡

changed

La tarea realizó un cambio en el dispositivo. Se notifica a los handlers si hay notify.

🔴

failed

La tarea falló (comando inválido, credenciales incorrectas, dispositivo inalcanzable). Detiene el play por defecto.

🔵

skipped

La tarea fue omitida por un condicional when que evaluó falso.

⚫

unreachable

Ansible no pudo conectarse al host (SSH caído, IP inalcanzable). Se reintenta según retries.

🟣

rescued

La tarea falló pero fue capturada por un bloque rescue (manejo de errores).

Idempotencia real en redes Cisco

La idempotencia significa que puedes ejecutar el mismo playbook múltiples veces sin efectos secundarios. Los módulos ios_config y nxos_config logran esto comparando la configuración deseada con el running-config actual antes de enviar comandos.

idempotencia_real.yml — Solo cambia si es necesario
---
- name: "Configuración idempotente de SNMP"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Asegurar comunidad SNMP readonly"
      cisco.ios.ios_config:
        lines:
          - "snmp-server community {{ snmp_community }} RO"
        parents: "snmp-server community {{ snmp_community }} RO"
        before: "no snmp-server community {{ snmp_community }}"
        match: "exact"

  vars:
    snmp_community: "NetManage2024"
💡 Tip de idempotencia en ios_config

Usa las directivas parents, before, after y match para controlar exactamente cómo Ansible compara la configuración. match: exact asegura que la sección coincida palabra por palabra; match: line es más permisivo y solo verifica que cada línea exista.

Tags: ejecutar solo lo que necesitas

Los tags permiten etiquetar tareas y ejecutar solo un subconjunto del playbook. Esto es vital cuando tienes un playbook monolítico y solo quieres aplicar una parte (ej. solo NTP, solo VLANs):

tags_ejemplo.yml — Ejecución selectiva
---
- name: "Configuración completa de switches"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Configurar VLANs"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 10
            name: "VENTAS"
      tags: ["vlan", "produccion"]

    - name: "Configurar NTP"
      cisco.ios.ios_ntp:
        config:
          - server: "10.0.0.254"
            version: 4
      tags: ["ntp", "produccion"]

    - name: "Configurar Syslog"
      cisco.ios.ios_logging:
        config:
          - dest: "host"
            name: "10.0.0.253"
      tags: ["syslog"]
Ejecución con tags
# Ejecutar solo tareas etiquetadas como "vlan"
ansible-playbook site.yml --tags "vlan"

# Ejecutar todo EXCEPTO syslog
ansible-playbook site.yml --skip-tags "syslog"

# Listar todas las tags disponibles en un playbook
ansible-playbook site.yml --list-tags

Handlers: acciones reactivas ante cambios

Un handler es una task especial que solo se ejecuta cuando es notificada por otra task que reportó un estado changed. Los handlers se acumulan durante todo el play y se disparan una única vez al final, después de que todas las tasks normales hayan terminado.

En redes Cisco, el caso de uso más común es: "Si cambié la configuración de cualquier interfaz, VLAN o ruta, guarda la configuración con write memory al final". Sin handlers, tendrías que incluir write memory en cada task, lo cual es ineficiente y peligroso.

TASK 1
Configura VLAN
changed → notify
→
TASK 2
Configura NTP
ok → no notify
→
TASK 3
Configura SNMP
changed → notify
→
FIN DEL PLAY
Handlers ejecutados
write memory (1 sola vez)

Sintaxis de notify y handlers

handlers_basico.yml — Guardar config solo si hubo cambios
---
- name: "Configurar switches con persistencia"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Asegurar VLAN 20"
      cisco.ios.ios_config:
        lines:
          - "vlan 20"
          - " name IT"
      notify: "Guardar configuración"

    - name: "Configurar descripción de uplink"
      cisco.ios.ios_config:
        lines:
          - "description Uplink a Core"
        parents: "interface GigabitEthernet1/0/24"
      notify: "Guardar configuración"

  handlers:
    - name: "Guardar configuración"
      cisco.ios.ios_command:
        commands: "write memory"
      listen: "Guardar configuración"

listen: notificación por nombre genérico

Además de notify: "Nombre exacto del handler", puedes usar listen en el handler para crear canales de notificación. Esto permite que múltiples handlers respondan a un mismo evento, o que notifiques con nombres genéricos como "config changed".

handlers_listen.yml — Múltiples acciones por cambio
---
- name: "Configuración con múltiples reacciones"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Actualizar ACL de entrada"
      cisco.ios.ios_config:
        lines:
          - "ip access-list extended BLOCK_BAD"
          - " deny ip 10.0.99.0 0.0.0.255 any"
      notify: "config_changed"

  handlers:
    - name: "Persistir configuración"
      cisco.ios.ios_command:
        commands: "write memory"
      listen: "config_changed"

    - name: "Enviar log a syslog"
      cisco.ios.ios_command:
        commands: "send log Cambio aplicado por Ansible"
      listen: "config_changed"
⚡ Comportamiento crítico de los handlers

Los handlers se ejecutan una sola vez por play, incluso si son notificados por múltiples tasks. Si un handler falla, Ansible lo marca como fallido y no lo reintenta para otros hosts a menos que uses force_handlers: true en el play. Además, si un play posterior en el mismo playbook notifica un handler del play anterior, ese handler no se ejecutará: los handlers son locales a su play.

force_handlers: ejecutar aunque falle una task

Por defecto, si una task falla en un host, Ansible aborta el play para ese host y no ejecuta los handlers acumulados. Esto puede dejar el dispositivo con cambios en running-config pero sin guardar. Usa force_handlers: true para garantizar la persistencia:

force_handlers.yml — Persistencia garantizada
---
- name: "Configuración crítica con persistencia forzada"
  hosts: cisco_switches
  gather_facts: no
  force_handlers: true

  tasks:
    - name: "Configurar VLANs"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 99
            name: "NATIVA"
      notify: "Guardar config"

    - name: "Esta task puede fallar"
      cisco.ios.ios_command:
        commands: "show inventario"

  handlers:
    - name: "Guardar config"
      cisco.ios.ios_command:
        commands: "write memory"

When: condicionales que adaptan la ejecución

La directiva when permite ejecutar una task solo si se cumple una condición. En redes Cisco, esto es fundamental para aplicar configuraciones diferentes según el modelo del switch, la ubicación, el rol del dispositivo o el resultado de una task anterior.

Ansible evalúa la expresión de when para cada host individualmente. Si la condición es falsa, la task se marca como skipped para ese host y continúa con la siguiente.

when_basico.yml — Condicionales por grupo y facts
---
- name: "Configuración condicional por sitio"
  hosts: cisco_devices
  gather_facts: no

  tasks:
    - name: "Configurar NTP para Madrid"
      cisco.ios.ios_config:
        lines: "ntp server 10.1.0.254"
      when: "site == 'madrid'"

    - name: "Configurar NTP para Barcelona"
      cisco.ios.ios_config:
        lines: "ntp server 10.2.0.254"
      when: "site == 'barcelona'"

    - name: "Aplicar solo en switches core"
      cisco.ios.ios_config:
        lines: "ip routing"
      when:
        - "'cisco_switches' in group_names"
        - "device_role == 'core'"

Loop: iterar sobre listas y diccionarios

Los loops permiten ejecutar la misma task múltiples veces con diferentes valores. En redes Cisco, esto es esencial para configurar VLANs, interfaces, rutas estáticas o servidores NTP/Syslog sin repetir código.

loop_lista.yml — Crear múltiples VLANs
---
- name: "Configurar VLANs masivamente"
  hosts: cisco_switches
  gather_facts: no

  vars:
    vlans_red:
      - { id: 10, nombre: "VENTAS" }
      - { id: 20, nombre: "IT" }
      - { id: 30, nombre: "RRHH" }
      - { id: 40, nombre: "GUEST" }

  tasks:
    - name: "Asegurar VLANs existen"
      cisco.ios.ios_config:
        lines:
          - "vlan {{ item.id }}"
          - " name {{ item.nombre }}"
      loop: "{{ vlans_red }}"
      notify: "Guardar config"

  handlers:
    - name: "Guardar config"
      cisco.ios.ios_command:
        commands: "write memory"

Loop con diccionarios: with_dict

Cuando tus datos están estructurados como diccionarios (clave-valor), puedes iterar sobre ellos usando loop con la filtro dict2items o el lookup with_dict:

loop_diccionario.yml — Configurar interfaces desde dict
---
- name: "Configurar interfaces desde diccionario"
  hosts: cisco_routers
  gather_facts: no

  vars:
    interfaces_wan:
      GigabitEthernet0/0/0:
        desc: "Enlace a ISP Primario"
        ip: "203.0.113.1 255.255.255.252"
      GigabitEthernet0/0/1:
        desc: "Enlace a ISP Backup"
        ip: "198.51.100.1 255.255.255.252"

  tasks:
    - name: "Configurar interfaz WAN"
      cisco.ios.ios_config:
        lines:
          - "description {{ item.value.desc }}"
          - "ip address {{ item.value.ip }}"
        parents: "interface {{ item.key }}"
      loop: "{{ interfaces_wan | dict2items }}"

loop_control: personalizar la iteración

La directiva loop_control permite modificar el comportamiento del loop: cambiar el nombre de la variable item, mostrar labels personalizadas, o pausar entre iteraciones:

loop_control.yml — Control avanzado de bucles
---
- name: "Actualización controlada de switches"
  hosts: cisco_switches
  gather_facts: no
  serial: 1

  vars:
    switches_por_actualizar:
      - "SW-Acc-01"
      - "SW-Acc-02"
      - "SW-Acc-03"

  tasks:
    - name: "Aplicar cambio en switch"
      cisco.ios.ios_config:
        lines: "spanning-tree portfast default"
      loop: "{{ switches_por_actualizar }}"
      loop_control:
        loop_var: "switch_item"
        label: "Procesando {{ switch_item }}"
        pause: 5
      when: "inventory_hostname == switch_item"
💡 Combinación when + loop

Puedes combinar when y loop en la misma task. Ansible evalúa el when para CADA iteración del loop. Esto permite filtrar elementos de la lista dinámicamente. Ejemplo: loop: "{{ all_vlans }}" when: "item.site == site" solo procesa las VLANs del sitio actual.

YAML avanzado para playbooks profesionales

Dominar YAML más allá de la sintaxis básica te permite escribir playbooks más compactos, reutilizables y libres de errores. Estas técnicas son especialmente útiles cuando trabajas con configuraciones de red extensas.

Anchors (&) y Aliases (*): reutilizar bloques

Los anchors permiten marcar una sección de YAML para reutilizarla posteriormente con un alias. Esto evita repetir configuraciones comunes (como credenciales o parámetros de conexión) en múltiples plays.

yaml_anchors.yml — Reutilización de bloques
---
# Definimos un anchor con credenciales comunes
conexion_cisco: &cisco_defaults
  ansible_connection: network_cli
  ansible_network_os: ios
  ansible_user: admin
  ansible_become: yes
  ansible_become_method: enable

conexion_nxos: &nxos_defaults
  ansible_connection: network_cli
  ansible_network_os: nxos
  ansible_user: admin

---
- name: "Configurar routers IOS"
  hosts: cisco_routers
  vars:
    # Reutilizamos el anchor con el alias *
    <<<: *cisco_defaults
  tasks:
    - name: "Configurar hostname"
      cisco.ios.ios_config:
        lines: "hostname {{ inventory_hostname }}"

- name: "Configurar switches Nexus"
  hosts: nexus_switches
  vars:
    <<<: *nxos_defaults
  tasks:
    - name: "Crear VLAN"
      cisco.nxos.nxos_vlans:
        config:
          - vlan_id: 10

Multilinea: literal (|) y folded (>)

En redes Cisco, a menudo necesitas pasar bloques de configuración multilinea. YAML ofrece dos operadores:

  • | (pipe): Preserva saltos de línea y espacios. Cada línea del bloque se convierte en un elemento de lista o un string con newlines.
  • > (folded): Reemplaza saltos de línea por espacios. Útil para descripciones largas en una sola línea lógica.
yaml_multilinea.yml — Bloques de configuración
---
- name: "Aplicar ACL multilinea"
  hosts: cisco_routers
  gather_facts: no

  vars:
    # Literal: preserva saltos de línea para ios_config
    acl_entrada: |
      ip access-list extended ENTRADA_INTERNET
       permit tcp any any eq 80
       permit tcp any any eq 443
       deny ip 10.0.0.0 0.255.255.255 any log
       permit ip any any

    # Folded: descripción larga en una línea
    banner_motd: >
      Acceso autorizado unicamente.
      Todo intento de conexion no autorizada sera registrado
      y reportado a las autoridades competentes.

  tasks:
    - name: "Aplicar ACL"
      cisco.ios.ios_config:
        lines: "{{ acl_entrada | split('\n') }}"

Cuidado con tipos de datos implícitos

YAML 1.1 (usado por Ansible) interpreta ciertos strings como booleanos. Esto es una fuente clásica de bugs en configuraciones de red:

Valor escritoYAML lo interpreta comoSolución
yes, noBooleano true/false"yes", "no" (entre comillas)
on, offBooleano true/false"on", "off"
1.10Float 1.1"1.10" (preservar el cero final)
10.08Error (octal inválido)"10.08"
192.168.1.010Octal (pierde el 10)"192.168.1.010"
⚡ Trampa clásica en versiones IOS

Si defines version: 1.10 en YAML para una variable de IOS, Ansible lo envía como 1.1 (float). Si la versión exacta es crítica (ej. para validación de compatibilidad), siempre usa comillas: version: "1.10".

Playbooks prácticos para redes Cisco

Esta sección contiene tres playbooks completos, documentados línea por línea, que puedes usar directamente en producción o adaptar a tu entorno. Representan los casos de uso más comunes: backup masivo, configuración inicial estandarizada y verificación post-cambio.

Playbook 1: Backup masivo de configuraciones

Este playbook se conecta a todos los dispositivos Cisco, extrae el running-config y lo guarda localmente con timestamp. Es el primer playbook que debe implementar cualquier equipo de red antes de tocar configuraciones.

backup_configuraciones.yml — Backup con timestamp
---
- name: "BACKUP: Guardar configuraciones de todos los dispositivos"
  hosts: cisco_routers:cisco_switches
  gather_facts: no
  connection: network_cli

  vars:
    backup_dir: "./backups/{{ ansible_date_time.date }}"

  tasks:
    - name: "Asegurar que el directorio de backup existe"
      ansible.builtin.file:
        path: "{{ backup_dir }}"
        state: directory
      delegate_to: localhost
      run_once: yes

    - name: "Obtener running-config del dispositivo"
      cisco.ios.ios_command:
        commands:
          - "show running-config"
      register: "running_config"

    - name: "Guardar configuración en archivo local"
      ansible.builtin.copy:
        content: "{{ running_config.stdout[0] }}"
        dest: "{{ backup_dir }}/{{ inventory_hostname }}.cfg"
      delegate_to: localhost

    - name: "Confirmar backup exitoso"
      ansible.builtin.debug:
        msg: "Backup guardado: {{ backup_dir }}/{{ inventory_hostname }}.cfg"

Playbook 2: Configuración inicial estandarizada

Este playbook aplica una configuración base (baseline) a switches de acceso nuevos: VLANs, NTP, SNMP, SSH hardening y banner. Usa variables por grupo y handlers para persistir.

baseline_switches.yml — Configuración base completa
---
- name: "BASELINE: Configurar switches de acceso nuevos"
  hosts: cisco_switches
  gather_facts: no
  connection: network_cli

  vars:
    vlans:
      - { id: 10, name: "VENTAS" }
      - { id: 20, name: "IT" }
      - { id: 99, name: "NATIVA" }
    ntp_servers:
      - "10.0.0.254"
      - "10.0.0.253"
    snmp_community: "NetManage2024"
    syslog_server: "10.0.0.100"

  tasks:
    - name: "1. Configurar hostname"
      cisco.ios.ios_config:
        lines: "hostname {{ inventory_hostname }}"
      notify: "Guardar configuracion"

    - name: "2. Crear VLANs"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: "{{ item.id }}"
            name: "{{ item.name }}"
      loop: "{{ vlans }}"
      notify: "Guardar configuracion"

    - name: "3. Configurar NTP"
      cisco.ios.ios_config:
        lines: "ntp server {{ item }}"
      loop: "{{ ntp_servers }}"
      notify: "Guardar configuracion"

    - name: "4. Configurar SNMP readonly"
      cisco.ios.ios_config:
        lines: "snmp-server community {{ snmp_community }} RO"
      notify: "Guardar configuracion"

    - name: "5. Configurar Syslog"
      cisco.ios.ios_config:
        lines:
          - "logging {{ syslog_server }}"
          - "logging trap informational"
      notify: "Guardar configuracion"

    - name: "6. Hardening SSH v2"
      cisco.ios.ios_config:
        lines:
          - "ip ssh version 2"
          - "ip ssh time-out 60"
          - "ip ssh authentication-retries 3"
      notify: "Guardar configuracion"

    - name: "7. Configurar banner MOTD"
      cisco.ios.ios_banner:
        banner: motd
        text: |
          Acceso autorizado unicamente.
          Desconexion inmediata si no tiene permiso.
        state: present
      notify: "Guardar configuracion"

  handlers:
    - name: "Guardar configuracion"
      cisco.ios.ios_command:
        commands: "write memory"

Playbook 3: Verificación post-cambio con assert

Este playbook ejecuta comandos show después de un cambio y valida que el resultado contenga los valores esperados. Si algo falla, el playbook reporta error inmediatamente:

verificacion_post_cambio.yml — Validación automática
---
- name: "VERIFICACION: Validar cambios aplicados"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Verificar que VLAN 10 existe"
      cisco.ios.ios_command:
        commands: "show vlan brief"
      register: "vlan_output"

    - name: "Validar que VLAN 10 esta presente"
      ansible.builtin.assert:
        that:
          - "'10  VENTAS' in vlan_output.stdout[0]"
        fail_msg: "ERROR: La VLAN 10 no se encuentra configurada"
        success_msg: "VLAN 10 verificada correctamente"

    - name: "Verificar estado de interfaz uplink"
      cisco.ios.ios_command:
        commands: "show ip interface brief GigabitEthernet1/0/24"
      register: "int_status"

    - name: "Validar que uplink esta UP/UP"
      ansible.builtin.assert:
        that:
          - "'up' in int_status.stdout[0]"
          - "'up' in int_status.stdout[0].split()[-1]"
        fail_msg: "CRITICO: El uplink no esta levantado"
💡 Patrón de orquestación recomendado

En producción, ejecuta siempre en este orden: 1) Backup (playbook 1), 2) Cambio (playbook 2 con --check --diff primero), 3) Verificación (playbook 3). Solo si la verificación pasa, considera el cambio exitoso. Este patrón elimina el 90% de los rollbacks manuales.
