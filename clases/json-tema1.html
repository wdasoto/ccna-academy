Fundamentos de JSON

Domina la sintaxis JSON desde cero — el lenguaje de datos universal de las APIs modernas. Objetos, arrays, tipos de datos, anidamiento y validación RFC 7159, con ejemplos reales de configuraciones de red Cisco y respuestas de APIs.

📋 RFC 7159 🎯 CCNA Domain 6.7 ⏱ ~60 min lectura 🔤 Sintaxis estricta 🧪 Validador incluido

¿Qué es JSON y por qué en Redes?

JSON (JavaScript Object Notation) es el formato de intercambio de datos dominante en la industria. Inventado por Douglas Crockford en 2001, se estandarizó como RFC 7159 en 2014. En networking, es el lenguaje que hablan Cisco DNA Center, RESTCONF, Meraki Dashboard y prácticamente cualquier API REST moderna.

La clave de JSON es su simplicidad brutal: seis tipos de construcciones gramaticales que representan cualquier estructura de datos. Es "auto-documentado" — un humano puede leerlo sin esquema previo — y "auto-validable" — una máquina puede verificar su sintaxis sin ambigüedades.

💡 JSON en el examen CCNA (Domain 6.7)

El examen CCNA 200-301 evalúa tu capacidad para identificar componentes JSON: distinguir objetos ({}) de arrays ([]), reconocer tipos de datos válidos (string, number, boolean, null, object, array), y detectar errores de sintaxis como comillas simples o comas finales. No escribirás JSON complejo, pero debes leerlo con fluidez.

Sintaxis General — Las 6 Estructuras

🗂️

Objeto {}

Colección desordenada de pares clave: valor. Equivalente a diccionarios en Python o hash maps.

📚

Array []

Lista ordenada de valores. Equivalente a listas en Python. Indexada numéricamente desde 0.

" "

String

Texto Unicode entre comillas dobles obligatorias. Comillas simples (' ') son inválidas en JSON.

123

Number

Números enteros o decimales. No hay distinción entre int y float. Notación científica permitida.

✅

Boolean

Valores lógicos: true o false. En minúsculas obligatoriamente (diferente de Python).

∅

Null

Valor vacío explícito. Representa ausencia de dato. En minúscula: null.

Primer ejemplo: Router en JSON

router.json — Estructura básica de dispositivo
{
  "hostname": "R1-CORE",
  "ip_management": "192.168.1.1",
  "modelo": "Cisco ISR 4321",
  "version_ios": "16.9.4",
  "interfaces_up": 4,
  "ssh_habilitado": true,
  "ultimo_backup": null
}
⚡ Diferencias críticas con Python
  • JSON usa comillas dobles (") obligatoriamente. Python permite simples (').
  • JSON no permite comentarios. Python sí (# o """).
  • JSON no permite coma final en el último elemento. Python lo tolera.
  • JSON booleanos son true/false (minúscula). Python usa True/False (mayúscula inicial).
  • JSON null es null. Python usa None.

Objetos JSON — Pares Clave-Valor

Un objeto JSON es una colección desordenada de pares clave: valor encerrados entre llaves {}. Las claves deben ser strings (entre comillas dobles). Los valores pueden ser cualquier tipo JSON válido. Los pares se separan con comas.

Sintaxis de objeto — Reglas estrictas
{
  "clave": "valor",
  "otra_clave": 123,
  "clave_final": true
}

Configuración de Interfaz como Objeto

interfaz.json — Objeto de configuración Cisco
{
  "nombre": "GigabitEthernet0/0",
  "ip_address": "192.168.1.1",
  "subnet_mask": "255.255.255.0",
  "estado": "up",
  "protocolo": "up",
  "mtu": 1500,
  "duplex": "full",
  "speed": 1000,
  "descripcion": "Enlace principal a Internet",
  "shutdown": false
}
💡 Claves JSON — Buenas prácticas
  • Sin espacios: usa snake_case o camelCase. Ej: ip_address o ipAddress.
  • Únicas dentro del mismo objeto: no puedes repetir una clave.
  • Case-sensitive: HostName y hostname son claves diferentes.
  • Evita caracteres especiales excepto guiones bajos (_) en APIs empresariales.

Acceso en Python (adelanto Tema 2)

Acceso a datos del objeto
import json

# Parsear JSON a diccionario Python
with open('interfaz.json') as f:
    interfaz = json.load(f)

# Acceder por clave (como diccionario Python)
print(interfaz["ip_address"])     # "192.168.1.1"
print(interfaz["speed"])            # 1000
print(interfaz.get("vlan", "N/A")) # "N/A" (clave no existe, default)

Arrays JSON — Listas Ordenadas

Un array es una colección ordenada de valores encerrados entre corchetes []. Los valores pueden ser de cualquier tipo (incluso mezclados, aunque no es recomendado) y se separan con comas. El orden se preserva siempre.

Sintaxis de array
[
  "valor1",
  "valor2",
  123,
  true,
  null
]

💡 Indexación de Arrays

Los arrays JSON son accedidos por índice numérico (base 0). En Python, data["routes"][0] accede al primer elemento del array. Los arrays JSON son equivalentes a listas de Python.

Números — Precisión y Límites

Números válidos en JSON
{
  "vlan_id": 4094,
  "temperatura": -5.5,
  "latencia_ms": 1.23,
  "ancho_banda": 1.25e9,
  "contador": 0
}
⚡ Números en JSON vs Python

JSON no distingue entre int y float (todo es number). Python convertirá 42 en int y 42.0 en float. JSON no admite NaN, Infinity o -Infinity (aunque algunos parsers los toleran, violan RFC 7159).

Anidamiento Profundo — Estructuras Complejas

La verdadera potencia de JSON radica en combinar objetos y arrays arbitrariamente. Un objeto puede contener arrays, que contienen objetos, que contienen más arrays... Esto representa cualquier topología de red jerárquica.

topologia_completa.json — Red enterprise anidada
{
  "datacenter": {
    "nombre": "DC-Principal",
    "routers_core": [
      {
        "hostname": "R1-CORE",
        "interfaces": [
          {
            "nombre": "Gi0/0",
            "ip": "10.0.0.1/30",
            "vecinos_ospf": [
              {"router_id": "2.2.2.2", "estado": "FULL"},
              {"router_id": "3.3.3.3", "estado": "FULL"}
            ]
          }
        ],
        "bgp_peers": [
          {"as": 64512, "ip": "198.51.100.1", "estado": "Established"}
        ]
      }
    ]
  }
}
💡 Navegación de estructuras anidadas

Para acceder al estado del primer vecino OSPF de R1-CORE en Python:

data["datacenter"]["routers_core"][0]["interfaces"][0]["vecinos_ospf"][0]["estado"]

Resultado: "FULL"

Validación RFC 7159 — Errores Comunes

JSON es estricto. Un solo error de sintaxis hace que todo el documento sea inválido. Aquí los errores más comunes que debes evitar en el examen CCNA y en producción.

Error Ejemplo inválido ❌ Corrección válida ✅ Explicación
Comillas simples {'key': 'value'} {"key": "value"} JSON solo permite comillas dobles
Coma final {"a": 1,} {"a": 1} No puede haber coma después del último elemento
Comentarios {// comentario
"a": 1}		 {"a": 1} JSON no admite comentarios
Números con ceros iniciales {"vlan": 0010} {"vlan": 10} Los números no pueden empezar con 0 (excepto 0 mismo)
Booleanos mayúsculas {"activo": True} {"activo": true} Debe ser true/false minúsculas
Null mayúscula {"valor": NULL} {"valor": null} Debe ser null minúscula
Sin comillas en clave {clave: "valor"} {"clave": "valor"} Las claves deben ser strings entre comillas
Trailing comma en array [1, 2, 3,] [1, 2, 3] Sin coma después del último elemento

Validación con Python (try/except)
import json

def validar_json(cadena):
    try:
        json.loads(cadena)
        return "✅ JSON válido"
    except json.JSONDecodeError as e:
        return f"❌ Error: {e}"

# Tests
print(validar_json('{"nombre": "R1", "up": true}'))        # ✅ Válido
print(validar_json("{'nombre': 'R1'}"))                     # ❌ Comillas simples
print(validar_json('{"nombre": "R1",}'))                    # ❌ Coma final

YANG + JSON = RESTCONF

RESTCONF usa JSON como codificación por defecto para transportar datos modelados por YANG. Las claves siguen el modelo YANG (namespaces como ietf-interfaces:).

💡 Clave para el examen CCNA

Si ves una pregunta con {"clave": "valor"} es JSON. Si ves <clave>valor</clave> es XML (NETCONF). RESTCONF usa JSON; NETCONF usa XML.
