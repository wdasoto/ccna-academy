MÓDULOS cisco.ios — Automatización IOS

La colección oficial de Ansible para Cisco IOS. Domina ios_command, ios_config, ios_facts, ios_interfaces e ios_vlans para gestionar routers y switches Cisco con idempotencia real. Desde comandos show hasta configuración declarativa completa de VLANs e interfaces.

🧩 cisco.ios 🎯 CCNA Domain 6.6 ⏱ ~90 min lectura 📡 ios_command / ios_config 🔍 ios_facts / ios_vlans

La colección cisco.ios: automatización nativa

La colección cisco.ios es el conjunto oficial de módulos de Ansible mantenido por Cisco y Red Hat. Proporciona una interfaz declarativa y idempotente para gestionar dispositivos que ejecutan Cisco IOS, IOS-XE e IOS-XR. Desde routers de borde hasta switches Catalyst, estos módulos traducen tu intención (YAML) en comandos CLI ejecutados vía SSH.

A diferencia de enviar comandos raw por SSH (como harías con un script Python básico), los módulos cisco.ios entienden el contexto de configuración de Cisco. Saben que interface GigabitEthernet1/0/1 requiere entrar en modo de configuración de interfaz, que las VLANs deben verificarse contra show vlan brief, y que un cambio en una ACL puede requerir recrearla completamente.

Esta colección es la que más preguntas genera en el examen CCNA dentro del Domain 6.6, porque demuestra la diferencia entre automatización declarativa (dices QUÉ quieres) versus imperativa (dices CÓMO hacerlo paso a paso).

💡 Concepto clave del examen CCNA

El dominio 6.6 evalúa si reconoces las herramientas de gestión de configuración. La pregunta frecuente es: "¿Qué módulo de Ansible se usa para ejecutar comandos show en un router Cisco?" La respuesta es ios_command. Para enviar configuración declarativa, la respuesta es ios_config.

Instalación y verificación de la colección

Ansible separa el motor core de las colecciones específicas de plataformas. Para usar cualquier módulo cisco.ios, primero debes instalar la colección desde Ansible Galaxy:

Instalación y verificación
# Instalar la colección oficial (última versión estable)
ansible-galaxy collection install cisco.ios

# Instalar versión específica (recomendado en producción)
ansible-galaxy collection install cisco.ios:==9.0.0

# Ver colecciones instaladas y sus rutas
ansible-galaxy collection list | grep cisco.ios

# Documentación de un módulo específico
ansible-doc cisco.ios.ios_config
ansible-doc cisco.ios.ios_command

Módulos principales de la colección

📡

ios_command

Ejecuta comandos show y exec en modo privilegiado. Devuelve la salida en stdout para análisis o validación. No modifica configuración.

⚙️

ios_config

Envía configuración al dispositivo de forma idempotente. Soporta líneas sueltas, parents, before/after y templates Jinja2.

🔍

ios_facts

Recopila información del hardware, software, interfaces y configuración. Equivalente a ejecutar múltiples show commands y parsear la salida.

🔌

ios_interfaces

Gestión declarativa de interfaces: habilitar/deshabilitar, descripciones, velocidad, dúplex, encapsulación. Idempotente por diseño.

🏷️

ios_vlans

Crea, modifica y elimina VLANs en switches Catalyst. Valida contra el VLAN database y reporta cambios solo cuando es necesario.

🛡️

ios_acls / ios_banner

Módulos especializados para listas de acceso, banners MOTD/login, logging, NTP, usuarios y más. Cada uno gestiona un subsistema específico de IOS.

Requisitos previos en el dispositivo Cisco

Para que los módulos funcionen, el dispositivo Cisco debe cumplir:

  • SSH habilitado: ip ssh version 2, crypto key generate rsa
  • Usuario con privilegios: Nivel 15 o enable password configurado para escalado
  • Terminal length 0: Algunos módulos requieren que el prompt no pagine (Ansible lo configura automáticamente en la mayoría de casos)
  • Almacenamiento suficiente: Para módulos que copian archivos (config backups, IOS upgrades)
⚡ Diferencia clave: ios_command vs ios_config

ios_command ejecuta comandos en modo EXEC (show, ping, traceroute). ios_config entra en modo de configuración global o de interfaz. Nunca uses ios_command para enviar configure terminal — usa ios_config que maneja los modos de configuración de forma segura y verifica errores de sintaxis.

ios_command: ejecutar show y validar estado

El módulo ios_command es el más simple y el que más se usa para operaciones de lectura. Ejecuta comandos en modo privilegiado (enable) y devuelve la salida como texto plano en la variable stdout. Es ideal para:

  • Recolección de información (inventario, versiones, contadores)
  • Validación post-cambio (verificar que una interfaz está up, que una ruta existe)
  • troubleshooting automatizado (ping extendido, traceroute)
ios_command_basico.yml — Comandos show simples
---
  - name: "Obtener información de routers"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Ejecutar show version"
      cisco.ios.ios_command:
        commands:
          - "show version"
      register: "version_output"

    - name: "Mostrar versión de IOS"
      ansible.builtin.debug:
        msg: "{{ version_output.stdout[0] | regex_search('Version ([^,]+)') }}"

wait_for: validar condiciones antes de continuar

El parámetro wait_for permite especificar condiciones que deben cumplirse en la salida del comando antes de que Ansible considere la task exitosa. Es fundamental para:

  • Esperar a que una interfaz esté up/up después de un cambio
  • Verificar que un vecino BGP aparezca en show ip bgp summary
  • Confirmar que una VLAN existe antes de continuar
ios_command_wait_for.yml — Validación de estado
---
- name: "Validar que interfaz WAN está levantada"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Verificar estado de GigabitEthernet0/0/0"
      cisco.ios.ios_command:
        commands:
          - "show ip interface brief GigabitEthernet0/0/0"
        wait_for:
          - "result[0] contains 'up'"
          - "result[0] contains 'up'"
        retries: 5
        interval: 10
      register: "wan_status"

    - name: "Fallar si la interfaz no está operativa"
      ansible.builtin.fail:
        msg: "La interfaz WAN no está levantada"
      when: "'down' in wan_status.stdout[0]"

Comandos múltiples y parseo con regex

Puedes pasar una lista de comandos y luego analizar la salida usando filtros Jinja2 como regex_search, regex_findall o split:

ios_command_multiple.yml — Parseo avanzado
---
- name: "Auditoría de interfaces"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Obtener estado de todas las interfaces"
      cisco.ios.ios_command:
        commands:
          - "show ip interface brief"
          - "show interfaces description"
      register: "interfaces_data"

    - name: "Contar interfaces en estado up"
      ansible.builtin.debug:
        msg: "Interfaces activas: {{ interfaces_data.stdout[0] | regex_findall(' up  +up ') | length }}"
💡 Tip de examen CCNA

El módulo ios_command es de solo lectura para comandos EXEC. Si el examen pregunta cómo verificar el estado de una interfaz o recolectar información de un router Cisco sin modificar configuración, la respuesta es siempre ios_command.

ios_config: configuración declarativa e idempotente

El módulo ios_config es el más poderoso de la colección. Permite enviar configuración a dispositivos Cisco de forma que Ansible verifica primero si el cambio es necesario. Si la configuración deseada ya existe, reporta ok sin enviar comandos. Si no existe o difiere, aplica los cambios y reporta changed.

Este módulo soporta múltiples estrategias de inserción: líneas sueltas (lines), configuración por secciones (parents), reemplazo completo de secciones (replace: config), y templates Jinja2 (src).

ios_config_lines.yml — Configuración simple con lines
---
- name: "Configurar hostname y banner"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Aplicar hostname"
      cisco.ios.ios_config:
        lines:
          - "hostname {{ inventory_hostname }}"

    - name: "Configurar banner MOTD"
      cisco.ios.ios_config:
        lines:
          - "banner motd #Acceso autorizado unicamente#"

Parents: configurar submodos de IOS

Cuando necesitas configurar dentro de un submodo (interfaz, línea, router o vlan), usa parents para indicar el contexto. Ansible entra en ese modo, aplica las líneas y sale:

ios_config_parents.yml — Configuración de interfaz
---
- name: "Configurar interfaces de acceso"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Configurar puerto de acceso VLAN 10"
      cisco.ios.ios_config:
        lines:
          - "switchport mode access"
          - "switchport access vlan 10"
          - "spanning-tree portfast"
          - "no shutdown"
        parents: "interface GigabitEthernet1/0/5"

    - name: "Configurar interfaz WAN"
      cisco.ios.ios_config:
        lines:
          - "description Enlace a ISP"
          - "ip address 203.0.113.1 255.255.255.252"
          - "no shutdown"
        parents: "interface GigabitEthernet0/0/0"

Idempotencia con match y before/after

Para garantizar idempotencia en secciones complejas, ios_config ofrece:

  • match: line — Verifica que cada línea exista (default, más permisivo)
  • match: strict — Las líneas deben estar en el orden exacto
  • match: exact — La sección debe coincidir palabra por palabra, eliminando líneas extra
  • before — Comandos a ejecutar ANTES de aplicar los cambios
  • after — Comandos a ejecutar DESPUÉS
ios_config_idempotencia.yml — Control preciso
---
- name: "Configurar ACL de forma idempotente"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Asegurar ACL ENTRADA_INTERNET"
      cisco.ios.ios_config:
        lines:
          - "permit tcp any any eq 80"
          - "permit tcp any any eq 443"
          - "deny ip 10.0.0.0 0.255.255.255 any log"
          - "permit ip any any"
        parents: "ip access-list extended ENTRADA_INTERNET"
        before:
          - "no ip access-list extended ENTRADA_INTERNET"
        match: "exact"
        replace: "block"
⚡ Peligro de match: exact

match: exact eliminará cualquier línea en la sección que no esté en tu lista lines. Si la ACL ya tenía entradas adicionales manuales, serán borradas. Úsalo solo cuando quieras un estado 100% definido por el playbook.

Templates Jinja2: configuración masiva

Para configuraciones extensas o repetitivas, usa el parámetro src para cargar un template Jinja2. Esto separa la lógica (playbook) de los datos (variables) y la presentación (template):

ios_config_template.yml — Uso de plantillas
---
- name: "Aplicar configuración desde template"
  hosts: cisco_routers
  gather_facts: no

  vars:
    interfaces_wan:
      - { name: "GigabitEthernet0/0/0", ip: "203.0.113.1 255.255.255.252", desc: "ISP Primario" }
      - { name: "GigabitEthernet0/0/1", ip: "198.51.100.1 255.255.255.252", desc: "ISP Backup" }

  tasks:
    - name: "Aplicar template de interfaces"
      cisco.ios.ios_config:
        src: "templates/interfaces_wan.j2"
      notify: "Guardar config"

  handlers:
    - name: "Guardar config"
      cisco.ios.ios_command:
        commands: "write memory"
templates/interfaces_wan.j2 — Plantilla Jinja2
{% for intf in interfaces_wan %}
interface {{ intf.name }}
 description {{ intf.desc }}
 ip address {{ intf.ip }}
 no shutdown
{% endfor %}

ios_facts: inventario inteligente del dispositivo

El módulo ios_facts recopila información del dispositivo Cisco y la expone como variables de Ansible (facts). Estos facts incluyen datos de hardware, software, interfaces, configuración y más. Son fundamentales para tomar decisiones condicionales basadas en el estado real del equipo.

A diferencia de ios_command donde tú parseas manualmente la salida con regex, ios_facts devuelve datos estructurados listos para usar: diccionarios de interfaces, listas de neighbors, versión de IOS, modelo de hardware, etc.

ios_facts_basico.yml — Recopilación completa
---
- name: "Recopilar facts de routers Cisco"
  hosts: cisco_routers
  gather_facts: no

  tasks:
    - name: "Obtener todos los facts"
      cisco.ios.ios_facts:
        gather_subset:
          - "all"
      register: "facts_result"

    - name: "Mostrar versión de IOS"
      ansible.builtin.debug:
        msg: "Versión: {{ ansible_net_version }}"

    - name: "Mostrar modelo de hardware"
      ansible.builtin.debug:
        msg: "Modelo: {{ ansible_net_model }}"

    - name: "Mostrar hostname"
      ansible.builtin.debug:
        msg: "Hostname: {{ ansible_net_hostname }}"

Subconjuntos de facts (gather_subset)

Puedes limitar la recolección para acelerar la ejecución:

SubsetInformación recopilada
hardwareModelo, memoria, flash, slots, serial number
interfacesLista de interfaces, IPs, estados, contadores, MACs
configRunning-config completo (texto plano)
minHostname, versión, imagen (mínimo indispensable)
allTodo lo anterior (default)

Facts clave disponibles

Facts más utilizados en automatización

ansible_net_version         # Versión de IOS/IOS-XE
ansible_net_image           # Ruta de la imagen boot

ℹ️ Uso condicional basado en facts

Puedes usar facts en condicionales when para adaptar el playbook al hardware. Ejemplo: aplicar una configuración solo en modelos C9300: when: "'C9300' in ansible_net_model". O verificar la versión de IOS antes de aplicar una feature: when: "ansible_net_version is version('16.9', '>=')".

ios_interfaces: gestión declarativa de interfaces

El módulo ios_interfaces proporciona una interfaz completamente declarativa para gestionar interfaces físicas y lógicas en dispositivos Cisco IOS. En lugar de enviar comandos CLI imperativos, defines el estado deseado de cada propiedad y Ansible se encarga de alcanzarlo.

Este módulo es idempotente por diseño: si la interfaz ya tiene la configuración deseada, no se envían comandos. Si falta algo, aplica solo los cambios necesarios.

ios_interfaces_basico.yml — Configuración declarativa
---
- name: "Configurar interfaces de switches"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Asegurar estado de interfaces de acceso"
      cisco.ios.ios_interfaces:
        config:
          - name: "GigabitEthernet1/0/1"
            description: "Puerto usuario VLAN 10"
            enabled: true
            mode: "layer2"
          - name: "GigabitEthernet1/0/2"
            description: "Puerto usuario VLAN 20"
            enabled: true
            mode: "layer2"
          - name: "GigabitEthernet1/0/24"
            description: "Uplink a Core"
            enabled: true
            mode: "layer2"
            duplex: "full"
            speed: 1000
        state: "replaced"

Parámetros clave de ios_interfaces

ParámetroValoresDescripción
nameStringNombre exacto de la interfaz (ej. GigabitEthernet1/0/1)
descriptionStringDescripción de la interfaz
enabledtrue / falseAdministratively up o shutdown
speed10, 100, 1000, autoVelocidad en Mbps
duplexfull, half, autoModo dúplex
mtuIntegerMTU de la interfaz
modelayer2, layer3Modo de capa (switchport vs routed)

Estados de operación: merged, replaced, overridden, deleted

El parámetro state define cómo Ansible aplica la configuración:

➕

merged (default)

Añade o modifica solo los parámetros especificados, preservando el resto de la configuración existente de la interfaz.

🔄

replaced

Reemplaza la configuración de las interfaces listadas, eliminando parámetros no mencionados en esas interfaces específicas.

⚠️

overridden

Reemplaza TODAS las interfaces del dispositivo. Las interfaces no listadas vuelven a su estado default. ¡Peligroso!

🗑️

deleted

Elimina la configuración de las interfaces especificadas, devolviéndolas a valores de fábrica.

⚡ Precaución con state: overridden

state: overridden afecta a TODAS las interfaces del switch, no solo las que listas. Si omites la interfaz de management o el uplink, podrías dejar el dispositivo inaccesible. Úsalo solo en playbooks de provisión inicial (ZTP).

ios_vlans: gestión declarativa del VLAN database

El módulo ios_vlans permite crear, modificar y eliminar VLANs en switches Cisco Catalyst de forma completamente declarativa. En lugar de enviar vlan 10 y name VENTAS como comandos sueltos, defines una lista de VLANs con sus propiedades y Ansible garantiza que el switch termine con exactamente esa configuración.

Este módulo es especialmente valioso porque maneja automáticamente el modo VTP: si el switch está en modo server o transparent, crea las VLANs localmente. Si está en modo client, falla con un mensaje claro.

ios_vlans_basico.yml — VLANs declarativo
---
- name: "Configurar VLANs en switches core"
  hosts: cisco_switches
  gather_facts: no

  tasks:
    - name: "Asegurar VLANs de producción"
      cisco.ios.ios_vlans:
        config:
          - vlan_id: 10
            name: "VENTAS"
            state: "active"
          - vlan_id: 20
            name: "IT"
            state: "active"
          - vlan_id: 30
            name: "RRHH"
            state: "active"
          - vlan_id: 99
            name: "NATIVA"
            state: "active"
        state: "merged"
      notify: "Guardar configuración"

  handlers:
    - name: "Guardar configuración"
      cisco.ios.ios_command:
        commands: "write memory"

Parámetros de ios_vlans

ParámetroValoresDescripción
vlan_id1-4094Número de VLAN
nameString (32 chars)Nombre descriptivo de la VLAN
stateactive, suspendEstado operativo de la VLAN
shutdownenabled, disabledApagar la VLAN localmente
mtuIntegerMTU específico de la VLAN
remote_spantrue, falseVLAN para Remote SPAN

Comparativa: ios_config vs ios_vlans

¿Cuándo usar cada módulo para VLANs?

Característicaios_config (lines)ios_vlans (resource)
EnfoqueImperativo (comandos)Declarativo (estado deseado)
IdempotenciaRequiere match/regexNativa y automática
VelocidadRápido (pocos comandos)Más lento (parsea VLAN database)
ValidaciónManual (asserts)Automática (diff integrado)
RecomendadoCambios puntualesGestión completa del VLAN database
💡 Tip de producción

Para operaciones masivas de VLANs (crear 50 VLANs en 20 switches), ios_vlans es más seguro porque valida automáticamente que no haya duplicados, que los IDs estén en rango, y que los nombres no excedan 32 caracteres. Con ios_config, un error de tipeo en un comando podría dejar el VLAN database inconsistente.
