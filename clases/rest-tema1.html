HTTP y REST — Fundamentos

Domina el protocolo que mueve la Internet moderna y las APIs de red. Desde los verbos HTTP y códigos de estado hasta los principios REST, el modelo cliente-servidor y su aplicación directa en Cisco IOS-XE RESTCONF, DNA Center y Meraki.

🌐 HTTP/1.1 & HTTP/2 🎯 CCNA Domain 6.5 ⏱ ~90 min lectura 💻 Labs con Python requests 🧪 Proyecto CRUD real

¿Qué es una API y por qué REST?

Una API (Application Programming Interface) es un conjunto de reglas que permite que dos aplicaciones se comuniquen entre sí. En redes, las APIs son el puente entre tu código Python y los dispositivos Cisco: routers, switches, controladores SDN y plataformas cloud como Meraki.

Antes de las APIs, todo se hacía vía CLI (SSH/Telnet): conectarte a cada dispositivo, ejecutar comandos, parsear texto con expresiones regulares. Era frágil, lento y propenso a errores. Las APIs cambiaron las reglas del juego: ahora los dispositivos exponen sus datos y configuraciones en formatos estructurados (JSON, XML) que tu código puede consumir y modificar de forma predecible.

💡 APIs en el ecosistema Cisco (Domain 6.3–6.5)
  • RESTCONF (RFC 8040): API REST nativa en IOS-XE 16.6+. Usa HTTP para manipular modelos YANG. Es el futuro de la gestión programática de routers y switches.
  • NETCONF (RFC 6241): API RPC sobre SSH usando XML. Más antigua y robusta, pero no usa HTTP/REST.
  • DNA Center API: REST API del controlador SD-Access. Inventario, topología, políticas, provisioning.
  • Meraki Dashboard API: REST pura para gestión cloud de APs, switches y firewalls.

No todas las APIs son REST. Entender las diferencias es clave para el examen CCNA y para la vida real:

La analogía del restaurante

Imagina una API REST como un restaurante:

  • El menú (API Documentation / OpenAPI): enumera todos los recursos disponibles y qué operaciones puedes hacer con ellos.
  • El camarero (HTTP): lleva tu pedido a la cocina y trae la respuesta. Habla un protocolo estándar (GET, POST, etc.).
  • La cocina (Servidor): procesa tu solicitud, accede a la base de datos (running-config, interfaces, etc.) y prepara la respuesta.
  • El cliente (Tú / Script Python): no necesitas saber cómo funciona la cocina internamente. Solo necesitas saber leer el menú y hacer pedidos correctos.
⚡ Lo que evalúa el CCNA 200-301 (Domain 6.5)

El examen no te pide programar una API desde cero, pero sí exige que sepas: (1) qué verbo HTTP usar para cada operación CRUD, (2) interpretar códigos de estado 2xx/4xx/5xx, (3) reconocer que REST es stateless, (4) diferenciar RESTCONF de NETCONF, y (5) entender el rol del controlador SDN. Todo eso lo cubrimos aquí.

El Protocolo HTTP en profundidad

HTTP (HyperText Transfer Protocol) es el lenguaje que hablan los navegadores, las apps y ahora los routers Cisco. Es un protocolo de la capa de aplicación (OSI L7) que funciona sobre TCP (puerto 80) o TLS/SSL (puerto 443 como HTTPS). HTTP es stateless: cada mensaje es independiente; el servidor no recuerda nada de conversaciones anteriores a menos que se lo digas explícitamente (via headers o cookies).

Anatomía de una URL

Toda petición REST apunta a una URL (Uniform Resource Locator) que identifica un recurso. En redes Cisco, las URLs de RESTCONF siguen estructuras YANG muy específicas:

URL Structure
https://router.local:443/restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0
│      │              │  │         │                                      │
│      │              │  │         └── root resource (datastores)         └── instancia
│      │              │  └────────── API root (RFC 8040)
│      │              └───────────── Puerto HTTPS (443 por defecto)
│      └──────────────────────────── Hostname o IP del dispositivo
└─────────────────────────────────── Esquema: https (obligatorio para RESTCONF)
🔒

HTTPS

RESTCONF exige TLS. El puerto por defecto es 443. El certificado puede ser autofirmado (usar verify=False en desarrollo).

🌳

Resource Path

La ruta identifica el recurso YANG. /restconf/data/... accede al datastore running. Todo es jerárquico.

❓

Query String

Parámetros opcionales: ?depth=1, ?fields=interfaces, ?content=config. Filtran o limitan la respuesta.

#️⃣

Fragmento (#)

No se envía al servidor. Se usa en el cliente para navegación local. Irrelevante en APIs de red.

Versiones: HTTP/1.1 vs HTTP/2

  • HTTP/1.1: la versión dominante durante décadas. Texto plano, persistent connections (keep-alive), pipelining limitado. Es lo que usa la mayoría de APIs de red Cisco hoy.
  • HTTP/2: binario, multiplexación de streams sobre una sola conexión TCP, compresión de headers (HPACK), server push. Usado por plataformas cloud modernas. RESTCONF en IOS-XE soporta HTTP/1.1 principalmente.
  • HTTP/3: sobre QUIC (UDP). Aún poco común en infraestructura de red tradicional.

Los Verbos HTTP — El alfabeto de REST

Los verbos (métodos) HTTP definen la acción que quieres realizar sobre un recurso. En REST, cada verbo tiene una semántica precisa que el servidor debe respetar. Para el CCNA y para trabajar con RESTCONF, debes saberlos al dedillo:

GET
READ

Obtiene un recurso o una colección. No debe tener efectos secundarios en el servidor. Es safe y idempotente.

✅ Idempotente · ✅ Safe
POST
CREATE

Crea un nuevo recurso subordinado. El servidor asigna la URI. No es idempotente: ejecutarlo 2 veces crea 2 recursos.

❌ No idempotente · ❌ No safe
PUT
UPDATE (total)

Reemplaza por completo un recurso existente o lo crea si no existe (cuando conoces la URI). Es idempotente.

✅ Idempotente · ❌ No safe
PATCH
UPDATE (parcial)

Modifica parcialmente un recurso. Solo envías los campos que cambian. Puede no ser idempotente según la implementación.

⚠️ Generalmente idempotente · ❌ No safe
DELETE
DELETE

Elimina un recurso. Después de DELETE, un GET al mismo recurso debería retornar 404. Es idempotente.

✅ Idempotente · ❌ No safe

Idempotencia: el concepto clave

Una operación es idempotente si producirá el mismo resultado si la ejecutas 1 vez o 100 veces. Esto es crítico en redes porque los scripts automatizados reintentan peticiones ante fallos de conectividad.

  • GET: pedir la tabla de routing 1 o 100 veces no cambia el router. ✅
  • PUT: configurar la IP de una interfaz 1 o 100 veces deja la misma IP. ✅
  • POST: crear una interfaz Loopback 100 veces crea 100 loopbacks. ❌

Ejemplo práctico — Verbos contra un router Cisco (RESTCONF)
# GET — Obtener configuración de una interfaz
GET /restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0
→ 200 OK + JSON con IP, mask, description, shutdown...

# POST — Crear una nueva Loopback (el servidor asigna el nombre si es colección)
POST /restconf/data/Cisco-IOS-XE-native:native/interface
Body: {"Loopback": [{"name": "100", "ip": {"address": {"primary": {"address": "1.1.1.1", "mask": "255.255.255.255"}}}}]}
→ 201 Created + Location: /interface/Loopback=100

# PUT — Reemplazar TODA la configuración de esa interfaz
PUT /restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0
Body: {JSON completo con TODOS los campos}
→ 204 No Content (éxito sin datos de retorno)

# PATCH — Cambiar solo la descripción
PATCH /restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0
Body: {"description": "ENLACE-PRINCIPAL-AL-CORE"}
→ 204 No Content

# DELETE — Eliminar la Loopback 100
DELETE /restconf/data/Cisco-IOS-XE-native:native/interface/Loopback=100
→ 204 No Content
💡 Trampa común del examen CCNA

El examen pregunta: "¿Qué verbo HTTP se usa para modificar parcialmente un recurso?" La respuesta es PATCH, no PUT. PUT reemplaza el recurso completo. Si envías un PUT con solo 2 campos, el servidor puede borrar todos los demás campos que no incluiste.

Códigos de Estado HTTP

El servidor siempre responde con un código de estado numérico de 3 dígitos que indica el resultado de la petición. En automatización de red, tu script debe reaccionar diferente ante un 200 (éxito) que ante un 401 (credenciales malas) o 500 (el router falló internamente).

2xx — ÉXITO
  • 200 OK: petición exitosa. Viene datos en el body.
  • 201 Created: recurso creado exitosamente. Header Location con la URI.
  • 204 No Content: éxito pero sin body. Común en PUT/PATCH/DELETE.
3xx — REDIRECCIÓN
  • 301 Moved Permanently: recurso cambió de URL permanentemente.
  • 302 Found: redirección temporal (uso histórico).
  • 307 Temporary Redirect: redirección temporal; conservar método.
  • 308 Permanent Redirect: redirección permanente; conservar método.
4xx — ERROR DEL CLIENTE
  • 400 Bad Request: JSON malformado o datos inválidos.
  • 401 Unauthorized: falta autenticación o credenciales incorrectas.
  • 403 Forbidden: autenticado, pero sin permisos para el recurso.
  • 404 Not Found: el recurso solicitado no existe.
  • 406 Not Acceptable: servidor no puede generar el formato pedido en Accept.
  • 409 Conflict: conflicto de estado (ej. intentar crear recurso que ya existe con POST).
  • 415 Unsupported Media Type: Content-Type no soportado.
5xx — ERROR DEL SERVIDOR
  • 500 Internal Server Error: error genérico del servidor (bug o condición inesperada).
  • 501 Not Implemented: servidor no soporta la funcionalidad requerida.
  • 502 Bad Gateway: gateway/proxy recibió respuesta inválida del upstream.
  • 503 Service Unavailable: servidor sobrecargado o en mantenimiento.

⚡ Diferencia crítica: 401 vs 403

401 Unauthorized: "¿Quién eres?" — No te autenticaste o lo hiciste mal. 403 Forbidden: "Sé quién eres, pero no puedes entrar aquí" — Estás autenticado pero sin privilegios suficientes. En IOS-XE, un usuario con privilegio 1 puede obtener configuración (GET 200) pero no modificarla (PUT 403).

Headers específicos de RESTCONF (RFC 8040)

Cuando trabajas con routers Cisco vía RESTCONF, estos headers son obligatorios o altamente recomendados:

  • Accept: application/yang-data+json — pides datos en formato JSON según modelo YANG.
  • Content-Type: application/yang-data+json — informas que envías JSON YANG.
  • Authorization: Basic ... — RESTCONF usa siempre HTTPS + Basic Auth (o certificados cliente en setups avanzados).
💡 Errores comunes de headers
  • Olvidar Content-Type al hacer POST/PUT: el servidor devuelve 415 Unsupported Media Type.
  • Enviar Accept: application/xml a un servidor solo-JSON: devuelve 406 Not Acceptable.
  • No incluir Authorization: 401 Unauthorized inmediato.
  • En RESTCONF, usar application/json genérico en lugar de application/yang-data+json puede funcionar en algunas versiones, pero es mejor ser explícito según RFC 8040.

Principios REST de Fielding

REST no es un protocolo ni una tecnología: es un estilo arquitectónico definido por Roy Fielding en su tesis doctoral (2000). Una API que dice ser "RESTful" debe cumplir ciertos constraints. Para el examen CCNA y para diseñar integraciones limpias, estos son los 6 constraints:

🖥️

1. Cliente-Servidor

Separa las responsabilidades: el cliente maneja la UI/lógica de presentación; el servidor maneja datos y estado. Permiten evolucionar independientemente.

Stateless

2. Stateless

Cada petición del cliente al servidor debe contener toda la información necesaria para ser entendida. El servidor no guarda contexto de sesión entre requests.

⚡

3. Cacheable

Las respuestas deben etiquetarse explícitamente como cacheables o no. Un cliente puede reutilizar respuestas previas para mejorar rendimiento.

🏗️

4. Interfaz Uniforme

Identificación de recursos por URI, manipulación de recursos mediante representaciones, mensajes autodescriptivos y HATEOAS (hipermedia).

🥞

5. Sistema por Capas

El cliente no puede saber si está conectado directamente al servidor final o a un intermediario (load balancer, proxy, cache). Las capas son transparentes.

📜

6. Código Bajo Demanda (opcional)

Los servidores pueden extender funcionalidad del cliente enviando código ejecutable (ej. JavaScript). Raro en APIs de red; más común en web.

Recursos y Representaciones

En REST, todo es un recurso. Una interfaz GigabitEthernet0/0 es un recurso. Una ruta OSPF es un recurso. Una VLAN es un recurso. Cada recurso tiene una URI única:

  • /restconf/data/interface/GigabitEthernet=0%2F0 → recurso "interfaz gi0/0"
  • /restconf/data/router/ospf/process=1 → recurso "proceso OSPF 1"
  • /api/v1/network-device → colección "todos los dispositivos" (DNA Center)

Un recurso puede tener múltiples representaciones: JSON, XML, YAML. El cliente elige la representación via header Accept.

Stateless en el contexto de redes

Stateless significa que el router (servidor) no recuerda que tu script acaba de hacer un GET antes de tu POST. Cada request es una isla. Esto tiene implicaciones prácticas:

  • Ventaja: puedes distribuir requests entre múltiples routers sin preocuparte por "sesiones".
  • Desventaja: si necesitas transaccionalidad (ej. "solo configura si la versión IOS es X"), debes enviar toda esa lógica en el propio request o manejarla en el cliente.
  • NETCONF vs REST: NETCONF sí mantiene estado a través de la sesión SSH (locks, candidate datastore). RESTCONF es stateless sobre HTTP.
Ejemplo de stateless vs stateful
# RESTCONF (Stateless) — Cada request es independiente
Request 1: GET /restconf/data/interface/GigabitEthernet=0%2F0
  → El router no "recuerda" que preguntaste.

Request 2: PATCH /restconf/data/interface/GigabitEthernet=0%2F0
  → Debes enviar auth de nuevo. El router valida desde cero.

# NETCONF (Stateful) — Sesión persistente SSH
<rpc message-id="101"><get-config>...</get-config></rpc>
  → Sesión SSH activa. Puedes hacer lock, edit, commit en secuencia.
  → El servidor mantiene estado de la sesión.

Modelo Cliente-Servidor en APIs de Red

El modelo cliente-servidor es la base de toda comunicación REST. En infraestructura de red, los roles son claros:

💻

CLIENTE

Quien pide. Scripts Python, Postman, Ansible, Terraform, navegador, aplicación NMS. Construye requests HTTP y parsea responses.

🌐

SERVIDOR

Quien responde. Proceso RESTCONF en IOS-XE, API Gateway de DNA Center, Meraki Dashboard Cloud. Recibe requests, accede al running-config, devuelve JSON.

El Cliente: herramientas y librerías

  • Python + requests: el estándar de facto para automatización. Sencillo, potente, legible.
  • Postman / Insomnia: GUI para probar APIs manualmente. Ideal para explorar endpoints antes de automatizar.
  • cURL: herramienta de línea de comandos. Útil para debugging rápido desde un servidor Linux.
  • Ansible (uri module): automatización declarativa. El módulo uri hace peticiones HTTP desde playbooks.

💡 Arquitecturas SDN y el rol del controlador

En SD-Access y SD-WAN, el modelo cliente-servidor evoluciona: ya no hablas directamente con 500 routers. Hablas con UN controlador (DNA Center o vManage) que actúa como servidor/proxy. El controlador es el servidor REST; los edge devices son sus "agentes" internos. Tu script hace un GET al controlador y este consulta a todos los dispositivos por ti.

Seguridad en el canal

RESTCONF exige TLS (HTTPS). Nunca configures RESTCONF sobre HTTP plano en producción. Además:

  • Certificados autofirmados: en laboratorio, el router genera su propio cert. Python rechazará la conexión por defecto. Usa verify=False solo en lab.
  • Certificados CA: en producción, usa una PKI interna para firmar certificados de los routers.
  • RBAC: IOS-XE permite definir qué usuarios REST pueden leer (GET) y cuáles pueden escribir (PUT/POST/DELETE).

"""rest_crud_lab.py
Laboratorio HTTP/REST — CCNA Academy M8 T1
Demuestra: GET, POST, PUT, PATCH, DELETE, status codes, headers, JSON.
"""

import requests
import json

BASE_URL = "https://httpbin.org"

# ═══════════════════════════════════════════════════
# 1. GET — Obtener datos (equivalente a leer config)
# ═══════════════════════════════════════════════════
print("
=== GET /get ===")
resp = requests.get(f"{BASE_URL}/get", timeout=10)
print(f"Status Code: {resp.status_code}")        # 200
print(f"Content-Type: {resp.headers.get('Content-Type')}")
data = resp.json()                                   # Parsear JSON automáticamente
print(f"URL origen: {data['url']}")

# ═══════════════════════════════════════════════════
# 2. POST — Crear recurso (equivalente a crear VLAN)
# ═══════════════════════════════════════════════════
print("
=== POST /post ===")
payload = {
    "hostname": "R1-CORE",
    "ip_mgmt": "192.168.1.1",
    "vlans": [10, 20, 30]
}
headers = {"Content-Type": "application/json"}
resp = requests.post(
    f"{BASE_URL}/post",
    headers=headers,
    data=json.dumps(payload),
    timeout=10
)
print(f"Status Code: {resp.status_code}")        # 200 (httpbin devuelve 200 en lugar de 201)
print(f"JSON enviado: {resp.json()['data']}")

# ═══════════════════════════════════════════════════
# 3. PUT — Reemplazar recurso completo
# ═══════════════════════════════════════════════════
print("
=== PUT /put ===")
update_total = {
    "interface": "GigabitEthernet0/0",
    "ip": "10.0.0.1",
    "mask": "255.255.255.0",
    "description": "WAN-PRIMARY"
}
resp = requests.put(
    f"{BASE_URL}/put",
    headers=headers,
    data=json.dumps(update_total),
    timeout=10
)
print(f"Status: {resp.status_code} | Idempotente: Sí")

# ═══════════════════════════════════════════════════
# 4. PATCH — Modificación parcial
# ═══════════════════════════════════════════════════
print("
=== PATCH /patch ===")
parche = {"description": "WAN-PRIMARY-BACKUP"}
resp = requests.patch(
    f"{BASE_URL}/patch",
    headers=headers,
    data=json.dumps(parche),
    timeout=10
)
print(f"Status: {resp.status_code} | Cambio parcial aplicado")

# ═══════════════════════════════════════════════════
# 5. DELETE — Eliminar recurso
# ═══════════════════════════════════════════════════
print("
=== DELETE /delete ===")
resp = requests.delete(f"{BASE_URL}/delete", timeout=10)
print(f"Status: {resp.status_code} | Recurso eliminado (simulado)")

# ═══════════════════════════════════════════════════
# 6. Manejo de errores y status codes
# ═══════════════════════════════════════════════════
print("
=== Manejo de errores ===")
for code in [404, 500, 401]:
    resp = requests.get(f"{BASE_URL}/status/{code}", timeout=10)
    if resp.status_code == 404:
        print(f"[{code}] Recurso no encontrado — revisar URL")
    elif resp.status_code == 500:
        print(f"[{code}] Error interno del servidor — reintentar más tarde")
    elif resp.status_code == 401:
        print(f"[{code}] No autorizado — revisar credenciales")

# ═══════════════════════════════════════════════════
# 7. Ejemplo real: conexión a RESTCONF (estructura)
# ═══════════════════════════════════════════════════
print("
=== Plantilla RESTCONF Cisco (estructura real) ===")

def restconf_get_interface(ip_router, usuario, password, interfaz):
    """Obtiene configuración de interfaz vía RESTCONF RFC 8040"""
    url = (
        f"https://{ip_router}/restconf/data/"
        f"Cisco-IOS-XE-native:native/interface/"
        f"GigabitEthernet={interfaz}"
    )
    headers = {
        "Accept": "application/yang-data+json",
        "Content-Type": "application/yang-data+json",
    }
    try:
        resp = requests.get(
            url,
            headers=headers,
            auth=(usuario, password),
            verify=False,      # ⚠️ Solo en laboratorio (cert autofirmado)
            timeout=10
        )
        if resp.status_code == 200:
            return resp.json()
        elif resp.status_code == 404:
            return {"error": "Interfaz no existe"}
        else:
            return {"error": f"HTTP {resp.status_code}"}
    except requests.exceptions.RequestException as e:
        return {"error": str(e)}
