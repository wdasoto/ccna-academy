Comandos para Solución de Problemas

Domina el diagnóstico sistemático de fallas en redes Cisco. Desde la capa física hasta la capa de aplicación, pasando por switching, routing, VLANs, NAT, ACLs y servicios IP. Metodología Cisco, comandos esenciales y escenarios reales para el examen CCNA 200-301.

📋 IOS 15.x / IOS XE 🎯 CCNA 200-301 Exam Topic 1.6, 2.5, 3.7, 4.4, 5.2, 6.3 ⏱ ~100 min lectura 🧪 Lab: 2 Switches + 2 Routers + PCs

Fundamentos del Troubleshooting en Redes Cisco

El troubleshooting es una habilidad crítica para cualquier ingeniero de redes. No se trata solo de conocer comandos, sino de aplicar una metodología sistemática para identificar, aislar y resolver problemas de manera eficiente.

Modelo OSI y Troubleshooting por Capas

Metodologías de Troubleshooting

MétodoDescripciónCuándo Usar
Bottom-UpDesde capa 1 (física) hacia arriba hasta capa 7Problemas nuevos, sin información previa. Más exhaustivo.
Top-DownDesde capa 7 (aplicación) hacia abajo hasta capa 1Problemas de aplicación específica (HTTP, DNS, etc.).
Divide & ConquerVerificar en capa media (3 o 4) y dividir el problemaRedes grandes, problema desconocido. Más rápido.
Follow the PathSeguir la ruta del paquete hop-by-hopProblemas de routing, latencia, pérdida de paquetes.
Compare ConfigurationsComparar configuración funcional vs no funcionalCuando hay un "antes y después" de un cambio.
Swap ComponentsReemplazar cables, puertos, módulos sospechososFallos de hardware, capa física.

Comandos Universales de Diagnóstico

Comandos básicos aplicables a cualquier problema
! === COMANDOS DE VERIFICACIÓN RÁPIDA ===

! 1. Estado de interfaces (Capa 1 y 2)
SW1# show ip interface brief
! Status: up/up = OK, down/down = capa 1, up/down = capa 2

! 2. Conectividad básica (Capa 3)
SW1# ping 192.168.1.1
SW1# ping 192.168.1.1 source vlan 10
! Extended ping permite especificar origen, tamaño, repeticiones

! 3. Seguimiento de ruta (Capa 3)
SW1# traceroute 8.8.8.8
SW1# traceroute 8.8.8.8 source 192.168.1.10
! Identifica en qué salto se detiene el tráfico

! 4. Tabla ARP (Resolución IP → MAC)
SW1# show ip arp
! Si falta una MAC, el problema puede ser capa 2

! 5. Verificación de tiempo y logs
SW1# show clock
SW1# show logging
! Los logs muestran eventos recientes: errores de interfaz, STP, etc.
⚠️ Regla de Oro del Troubleshooting

Nunca asumas, siempre verifica. Un problema reportado como "no hay Internet" puede ser:

  • Cable desconectado (Capa 1)
  • Puerto en err-disabled por Port Security (Capa 2)
  • VLAN incorrecta en el puerto (Capa 2)
  • Gateway no configurado en el PC (Capa 3)
  • Ruta estática faltante en el router (Capa 3)
  • ACL bloqueando tráfico DNS (Capa 4)
  • Servidor DNS caído (Capa 7)

Documenta cada paso y cambio. Si no sabes qué cambió, show archive config differences o revisa logs de syslog.

Troubleshooting Capa 1 — Física

La capa física es el fundamento de toda comunicación de red. Problemas aquí afectan todo lo demás. Los comandos de esta sección verifican cables, velocidad, dúplex, LEDs y estado de interfaces.

Diagnóstico rápido de capa física

Velocidad y Dúplex (Speed/Duplex)

Configuración y verificación de autonegociación
! === CONFIGURAR VELOCIDAD Y DÚPLEX ===
SW1(config)# interface GigabitEthernet0/1
SW1(config-if)# speed 100
SW1(config-if)# duplex full
! Fuerza 100 Mbps full-duplex (NO autonegociar)

SW1(config-if)# speed auto
SW1(config-if)# duplex auto
! Autonegociación (recomendado en general)

! === VERIFICAR NEGOCIACIÓN ===
SW1# show interfaces GigabitEthernet0/1 | include duplex
  Full-duplex, 100Mb/s, media type is RJ45

SW1# show interfaces status
Port      Name               Status       Vlan       Duplex  Speed Type
Gi0/1     PC-VENTAS-01       connected    10         a-full  a-100 10/100/1000BaseTX
Gi0/2     PC-IT-01           notconnect   20           auto   auto 10/100/1000BaseTX
! a-full = autonegociado full-duplex, a-100 = autonegociado 100Mbps
! notconnect = cable desconectado o problema físico

! === PROBLEMA CLÁSICO: DÚPLEX MISMATCH ===
! Un lado en auto, otro en full → resultado: half-duplex
! Síntomas: Lentitud extrema, colisiones, errores CRC

SW1# show interfaces GigabitEthernet0/1 | include duplex
  Half-duplex, 100Mb/s, media type is RJ45
! Si el otro lado está en full, esto es un MISMATCH

! SOLUCIÓN: Configurar ambos lados igual (auto o fijo)

Errores de Interfaz y Contadores

Interpretación de errores físicos
! === CONTADORES DE ERROR ===
SW1# show interfaces GigabitEthernet0/1 | include error
     1524 packets input, 0 no buffer
     Received 45 broadcasts
     0 runts, 0 giants, 0 throttles
     234 input errors, 45 CRC, 12 frame, 0 overrun, 0 ignored
     0 watchdog, 0 multicast, 0 pause input
     0 output errors, 0 collisions, 0 interface resets

! TIPOS DE ERROR:
! CRC       = Error de verificación de redundancia cíclica (cable dañado, interferencia)
! Frame     = Frame mal formado (dúplex mismatch, problema de clocking)
! Runts     = Frame menor a 64 bytes (colisión, problema físico)
! Giants    = Frame mayor a 1518 bytes (MTU mismatch, jumbo mal configurado)
! Overrun   = Buffer de entrada lleno (sobrecarga de tráfico)
! Ignored   = Frame descartado por falta de buffer
! Collisions= Colisiones en half-duplex (normal), en full-duplex (problema)

! === LIMPIAR CONTADORES ===
SW1# clear counters GigabitEthernet0/1
! Útil para verificar si los errores siguen incrementándose

💡 Checklist Capa 1
  1. ¿El cable está conectado firmemente? (ambos extremos)
  2. ¿Los LEDs de enlace están encendidos (verde/naranja)?
  3. ¿La interfaz muestra up/up en show ip int brief?
  4. ¿La velocidad y dúplex coinciden en ambos extremos?
  5. ¿Hay errores CRC o frame incrementándose?
  6. ¿El puerto está en shutdown administrativo?
  7. ¿CDP/LLDP muestra el vecino esperado?

Troubleshooting Capa 2 — Enlace de Datos

La capa de enlace de datos maneja direccionamiento MAC, VLANs, trunking, STP y EtherChannel. Los problemas aquí son comunes y afectan la comunicación dentro del mismo segmento de red.

SW1# show mac address-table vlan 10

! === FILTRAR POR PUERTO ===
SW1# show mac address-table interface FastEthernet0/1

SW1# clear mac address-table dynamic

SW1(config-if-range)# spanning-tree bpduguard enable

⚠️ Problema Clásico Capa 2: VLAN Mismatch

Síntoma: Dos PCs en la "misma red" no se comunican.

Causa: Un puerto está en VLAN 10, el otro en VLAN 20.

Verificación: show vlan brief y show interfaces switchport.

Solución: Asegurar que ambos puertos access estén en la misma VLAN.

Troubleshooting Capa 3 — Red

La capa de red maneja direccionamiento IP, routing, ARP y protocolos ICMP. Los problemas aquí afectan la comunicación entre diferentes redes (subredes).

Tabla ARP y Resolución de Direcciones

Diagnosticar problemas de mapeo IP-MAC
! === TABLA ARP ===
R1# show ip arp
Protocol  Address          Age (min)  Hardware Addr   Type   Interface
Internet  192.168.1.1             -   0000.0c12.3456  ARPA   GigabitEthernet0/0
Internet  192.168.1.10            5   0000.0c78.9abc  ARPA   GigabitEthernet0/0
Internet  192.168.1.254           0   Incomplete      ARPA
! "Incomplete" = ARP request enviado, no hubo respuesta
! Causa: Host apagado, IP incorrecta, firewall bloqueando ARP

! === LIMPIAR ENTRADA ARP ESPECÍFICA ===
R1# clear ip arp 192.168.1.10

! === LIMPIAR TABLA ARP COMPLETA ===
R1# clear arp-cache

! === ARP ESTÁTICO ===
R1(config)# arp 192.168.1.100 0000.0c12.3456 arpa
! Útil para hosts que no responden ARP (firewall) o seguridad

! === PROXY ARP ===
R1(config)# interface GigabitEthernet0/0
R1(config-if)# ip proxy-arp
! Responde ARP en nombre de otros hosts (útil en migraciones)

MTU y Fragmentación IP

Diagnosticar problemas de tamaño de paquete
! === VERIFICAR MTU DE INTERFAZ ===
R1# show interfaces GigabitEthernet0/0 | include MTU
  MTU 1500 bytes, BW 1000000 Kbit/sec, DLY 10 usec,

! === PROBAR MTU CON PING ===
R1# ping 192.168.2.1 size 1472 df-bit
! 1472 payload + 8 ICMP + 20 IP = 1500 bytes (sin fragmentar)
! Si falla: MTU menor en algún enlace intermedio

! === PATH MTU DISCOVERY ===
R1# traceroute 192.168.2.1
! Usar con DF bit para encontrar el MTU mínimo de la ruta

! === TUNNEL MTU (GRE, IPsec) ===
! GRE añade 24 bytes, IPsec añade 56+ bytes
! MTU efectivo = 1500 - overhead = ~1400 bytes
! Síntoma: TCP funciona para HTTP pero no para transferencias grandes
💡 Checklist Capa 3
  1. ¿La IP y máscara son correctas? (show ip interface brief)
  2. ¿El gateway por defecto está configurado en el host?
  3. ¿Existe ruta hacia la red destino? (show ip route)
  4. ¿El next-hop es alcanzable? (ping next-hop)
  5. ¿La tabla ARP tiene la MAC del next-hop? (show ip arp)
  6. ¿Hay ACLs bloqueando ICMP o tráfico IP?
  7. ¿El MTU es suficiente en toda la ruta?
  8. ¿NAT está traduciendo correctamente?

Troubleshooting de Switching Avanzado

Los switches son el corazón de la red local. Problemas como port security, err-disabled, VTP, y spanning tree pueden causar interrupciones masivas si no se diagnostican rápidamente.

Port Security y Err-Disabled

Diagnosticar y recuperar puertos bloqueados
! === VERIFICAR PUERTOS ERR-DISABLED ===
SW1# show interfaces status err-disabled
Port      Name               Status       Reason
Fa0/1     PC-VENTAS-01       err-disabled psecure-violation
Fa0/5     PRINTER-HR         err-disabled dhcp-rate-limit
Fa0/12    AP-FLOOR2          err-disabled loopback

! CAUSAS COMUNES DE ERR-DISABLED:
! - psecure-violation: MAC no autorizada conectada
! - loopback: Loop detectado (loopguard, bpdu-guard)
! - dhcp-rate-limit: DHCP snooping detectó flood de DHCP
! - pagp-flap: EtherChannel inestable
! - dtp-flap: Negociación DTP errática

! === RECUPERACIÓN MANUAL ===
SW1(config)# interface FastEthernet0/1
SW1(config-if)# shutdown
SW1(config-if)# no shutdown

! === RECUPERACIÓN AUTOMÁTICA ===
SW1(config)# errdisable recovery cause psecure-violation
SW1(config)# errdisable recovery cause loopback
SW1(config)# errdisable recovery interval 300
! Recupera automáticamente después de 300 segundos (5 min)

! === VERIFICAR CONFIGURACIÓN DE PORT SECURITY ===

⚠️ Loop de Capa 2: Síntomas y Detección

Síntomas de un loop de switching:

  • Todos los LEDs de todos los switches parpadean al unísono (broadcast storm)
  • CPU del switch al 100%
  • Tráfico extremadamente lento o inexistente
  • Múltiples MACs en múltiples puertos (flapping)

Acción inmediata: Desconectar cables redundantes hasta que STP estabilice. Verificar show spanning-tree y habilitar spanning-tree portfast bpduguard en puertos de acceso.

Las ACLs (Access Control Lists) son la causa número uno de problemas de conectividad "misteriosos". Un deny implícito o una máscara wildcard incorrecta pueden bloquear tráfico legítimo sin advertencias.

Verificación de ACLs

R1# show access-lists

    10 permit 192.168.1.0, wildcard bits 0.0.0.255 (45 matches)
    20 deny   any (12 matches)

Extended IP access list 100
    10 permit tcp 192.168.1.0 0.0.0.255 any eq www (234 matches)
    20 permit udp 192.168.1.0 0.0.0.255 any eq dns (89 matches)
    30 deny ip any any log (5 matches)
! "(matches)" indica cuántos paquetes coincidieron con esa línea
! Si una línea tiene 0 matches, nunca se usó (posible error de lógica)

! === VER ACLs APLICADAS EN INTERFACES ===

  Inbound  access list is 100
  Outbound access list is not set

R1(config)# access-list 100 permit icmp 192.168.1.0 0.0.0.255 any

ACL Estándar vs Extendida

Diferencias y troubleshooting
! === ACL ESTÁNDAR (1-99, 1300-1999) ===
! Filtra SOLO por IP origen
! Debe aplicarse lo más CERCA del destino posible

R1(config)# access-list 10 permit 192.168.1.0 0.0.0.255
R1(config)# access-list 10 deny any

! === ACL EXTENDIDA (100-199, 2000-2699) ===
! Filtra por IP origen, IP destino, protocolo, puerto
! Debe aplicarse lo más CERCA del origen posible

R1(config)# access-list 100 permit tcp 192.168.1.0 0.0.0.255 any eq 80
R1(config)# access-list 100 permit tcp 192.168.1.0 0.0.0.255 any eq 443
R1(config)# access-list 100 permit udp 192.168.1.0 0.0.0.255 any eq 53
R1(config)# access-list 100 deny ip any any log

! === ACL CON NOMBRE (NAMED ACL) ===
R1(config)# ip access-list extended BLOCK-TELNET
R1(config-ext-nacl)# permit tcp 192.168.1.0 0.0.0.255 any eq 22
R1(config-ext-nacl)# deny tcp any any eq 23
R1(config-ext-nacl)# permit ip any any

! Ventaja: Se pueden editar líneas individuales (no en numbered ACL)

Wildcards y Máscaras Invertidas

Errores comunes en máscaras ACL
! === WILDCARD MASK ===
! 0 = bit debe coincidir exactamente
! 1 = bit puede ser cualquier valor (ignorado)

! Ejemplo: Permitir 192.168.1.0/24
! Máscara de subred:  255.255.255.0
! Wildcard mask:      0.0.0.255

R1(config)# access-list 1 permit 192.168.1.0 0.0.0.255

! === ERROR COMÚN: USAR MÁSCARA DE SUBRED EN LUGAR DE WILDCARD ===
! INCORRECTO (pero IOS lo acepta en algunas versiones):
R1(config)# access-list 1 permit 192.168.1.0 255.255.255.0
! Esto significa: "coincide si los bits NO coinciden" (lógica invertida)
! Resultado: Comportamiento inesperado o bloqueo total

! === WILDCARD PARA HOST ESPECÍFICO ===
R1(config)# access-list 1 permit 192.168.1.10 0.0.0.0
! O abreviado:
R1(config)# access-list 1 permit host 192.168.1.10

! === WILDCARD PARA CUALQUIER IP ===
R1(config)# access-list 1 permit any
! Equivalente a 0.0.0.0 255.255.255.255

💡 Regla del "Implicit Deny"

Toda ACL termina con un deny any (estándar) o deny ip any any (extendida) implícito e invisible. Si tu tráfico no coincide con ninguna línea permit, será bloqueado sin logs (a menos que añadas log al final).

Best practice: Siempre termina ACLs extendidas con permit ip any any log o deny ip any any log para poder ver qué tráfico está siendo afectado.

Metodología Cisco de Troubleshooting

Una metodología estructurada reduce el tiempo de resolución y evita cambios innecesarios que puedan empeorar el problema. Cisco recomienda un enfoque de 6 pasos.

Los 6 Pasos del Troubleshooting

1
Identificar el problema

Recopilar síntomas: ¿Qué no funciona? ¿Para quién? ¿Desde cuándo? ¿Qué cambió?

show logging | include changed
show archive config differences
2
Establecer una teoría de la causa probable

Aplicar Bottom-Up, Top-Down o Divide & Conquer. No asumir, verificar cada capa.

show ip interface brief
show vlan brief
show ip route
3
Probar la teoría

Usar comandos de verificación para confirmar o descartar la hipótesis.

ping, traceroute, show, debug
4
Establecer un plan de acción

Si la teoría se confirma, planificar la solución. Si no, volver al paso 2.

Copiar config antes de cambiar: copy running-config backup-config
5
Implementar la solución

Aplicar cambios de forma controlada. Un cambio a la vez.

configure terminal → cambio → verificación inmediata
6
Verificar la funcionalidad completa

Confirmar que el problema se resolvió y no se generaron nuevos problemas.

ping end-to-end, traceroute, monitoreo de usuarios

⚠️ Precaución con Debug en Producción

Los comandos debug consumen mucha CPU y pueden colapsar un router o switch en producción. Reglas:

  • Siempre usar service timestamps debug datetime msec para timestamps.
  • Usar debug condicional: debug ip packet [ACL] para filtrar.
  • Programar reload in 15 antes de debug para auto-recuperación.
  • Deshabilitar inmediatamente con undebug all o no debug all.
  • Preferir show commands sobre debug cuando sea posible.

Parte 1
Diagnóstico Capa 1 y 2
! === EN SW2 ===
SW2# show interfaces status
! Observar: Fa0/1 a-full, 100 (manual) vs PC3 a-half, 100 (auto)
! SOLUCIÓN:
SW2(config)# interface Fa0/1
SW2(config-if)# speed auto
SW2(config-if)# duplex auto

! === EN SW2: VTP ===
SW2# show vtp status
! Domain Name: CCNA-LAB-BAD (incorrecto, debe ser CCNA-LAB)
! SOLUCIÓN:
SW2(config)# vtp domain CCNA-LAB

! === EN SW1: PORT SECURITY ===
SW1# show interfaces status err-disabled
! Fa0/2 err-disabled psecure-violation
! SOLUCIÓN:
SW1(config)# interface Fa0/2
SW1(config-if)# shutdown
SW1(config-if)# no shutdown
Parte 2
Diagnóstico Capa 3 y Servicios
! === EN R1: RUTA ESTÁTICA ===
R1# show ip route 192.168.20.0
! Muestra next-hop incorrecto o inexistente
! SOLUCIÓN:
R1(config)# no ip route 192.168.20.0 255.255.255.0 10.1.1.5
R1(config)# ip route 192.168.20.0 255.255.255.0 192.168.1.2

! === EN R1: ACL ===
R1# show access-lists 100
! Falta línea para permitir UDP 53
! SOLUCIÓN:
R1(config)# access-list 100 permit udp any any eq 53

! === EN R1: NAT ===
R1# show ip nat statistics
! No muestra translations
R1# show running-config | include ip nat
! ip nat inside source list 1 interface GigabitEthernet0/1
! FALTA "overload"
! SOLUCIÓN:
R1(config)# no ip nat inside source list 1 interface GigabitEthernet0/1
R1(config)# ip nat inside source list 1 interface GigabitEthernet0/1 overload
Parte 3
Verificación End-to-End
! === DESDE PC1 ===
PC> ping 192.168.20.10      ! Inter-VLAN routing
PC> ping 8.8.8.8            ! Internet via NAT
PC> nslookup google.com     ! DNS resolution

! === DESDE PC2 ===
PC> ping 192.168.10.10      ! Inter-VLAN
PC> ping 192.168.10.20      ! Misma VLAN, diferente switch

! === DESDE PC3 ===
PC> ping 192.168.10.10      ! Misma VLAN
PC> ping 192.168.20.10      ! Diferente VLAN

! === COMANDOS FINALES ===
SW1# show vlan brief
SW1# show mac address-table
SW1# show interfaces trunk
SW2# show vtp status
R1# show ip route
R1# show ip nat translations
R1# show access-lists
✅ Checklist de Éxito del Laboratorio
  • ✓ Todas las interfaces muestran a-full / a-100 o a-1000
  • ✓ SW2 aprendió VLANs 10 y 20 por VTP
  • ✓ Puerto Fa0/2 de SW1 está up/up (no err-disabled)
  • ✓ R1 tiene ruta correcta hacia 192.168.20.0/24
  • ✓ ACL 100 permite DNS (UDP 53)
  • ✓ NAT overload muestra translations activas
  • ✓ Todos los PCs pueden hacer ping entre sí y a Internet
