INVENTARIOS y Variables

El mapa de tu infraestructura. Domina el corazón de Ansible: inventarios estáticos en INI y YAML, jerarquías de grupos, variables por host y por grupo, y la magia de los inventarios dinámicos con AWS, Azure y NetBox. Sin un inventario sólido, no hay automatización posible.

📄 INI / YAML 🎯 CCNA Domain 6.6 ⏱ ~90 min lectura 🔐 host_vars / group_vars ☁️ Inventarios Dinámicos

¿Qué es el Inventario en Ansible?

El inventario es el archivo más importante de cualquier proyecto Ansible. Es el mapa que le dice a Ansible qué dispositivos existen en tu red, dónde se encuentran y cómo conectarse a ellos. Sin inventario, Ansible es un piloto sin navegador: sabe qué hacer pero no a dónde ir.

En el contexto de redes Cisco, el inventario no solo lista IPs. Define grupos lógicos como [cisco_routers], [cisco_switches] o [firewalls_asa], asigna credenciales específicas por ubicación, establece el método de conexión (SSH, NETCONF, HTTPAPI) y almacena variables personalizadas como la versión de IOS deseada, servidores NTP o comunidades SNMP.

Ansible soporta dos fuentes de inventario: estáticos (archivos INI o YAML que tú mantienes manualmente) y dinámicos (plugins que consultan APIs de cloud, CMDBs o orquestadores como AWS, Azure, NetBox o VMware vCenter para construir el inventario en tiempo real).

💡 Concepto clave del examen CCNA

El dominio 6.6 evalúa si entiendes que Ansible necesita un inventario para operar. La pregunta clásica es: "¿Qué archivo define los hosts gestionados por Ansible?" La respuesta es el inventory file, ya sea en formato INI o YAML.

Tipos de Inventarios: Estático vs. Dinámico

📝

Estático INI

Formato clásico tipo .ini. Rápido de escribir, ideal para redes pequeñas y medianas. Usa corchetes para grupos y clave=valor para variables.

📋

Estático YAML

Estructura jerárquica anidada. Más legible para inventarios complejos. Permite listas y diccionarios nativos. Recomendado para equipos grandes.

☁️

Dinámico Cloud

Consulta APIs de AWS EC2, Azure RM o Google Cloud. Los hosts aparecen y desaparecen automáticamente. Esencial para infraestructura cloud híbrida.

🔍

Dinámico CMDB

Integración con NetBox, ServiceNow o CMDBs internos. La fuente de verdad es tu base de datos de activos, no un archivo plano.

🗂️

Directorios

Ansible puede usar un directorio como inventario, combinando múltiples archivos INI/YAML y scripts dinámicos en una sola fuente.

⚡

En Línea de Comandos

Puedes pasar hosts directamente con ansible -i "host1,host2," sin archivo. Útil para pruebas de emergencia (no recomendado en producción).

¿Por qué el inventario decide el éxito de tu automatización?

Un inventario mal diseñado genera:

  • Fallos de conexión: IPs incorrectas, credenciales mal agrupadas o métodos de conexión incompatibles con el modelo de switch.
  • Configuraciones inconsistentes: Aplicar una plantilla de router a un switch porque ambos están en el grupo genérico [cisco] sin subgrupos.
  • Problemas de seguridad: Contraseñas en texto plano dentro del inventario sin usar Vault o variables separadas.
  • Dificultad de escala: Un archivo INI de 500 líneas sin grupos ni variables estructuradas es imposible de mantener.

En este tema aprenderás a construir inventarios que escalan desde 5 dispositivos hasta 5,000, manteniendo la claridad y la seguridad.

Formato INI: el clásico de Ansible

El formato INI es el más antiguo y el que más rápido se escribe. Se basa en secciones encerradas entre corchetes. Cada línea debajo de una sección define un host. Las variables se asignan con el operador = (a diferencia de YAML que usa :).

Es ideal cuando comienzas con Ansible o cuando tu infraestructura es relativamente estable. Sin embargo, para estructuras muy anidadas o listas complejas, YAML ofrece mejor legibilidad.

inventory.ini — Red Cisco básica
[cisco_routers]
R1 ansible_host=192.168.1.1
R2 ansible_host=192.168.1.2
R3 ansible_host=192.168.1.3

[cisco_switches]
SW-Core-01 ansible_host=192.168.1.10
SW-Acc-01  ansible_host=192.168.1.11
SW-Acc-02  ansible_host=192.168.1.12

[firewalls]
FW-ASA-01 ansible_host=192.168.1.254

# Grupo padre que contiene a otros grupos
[cisco:children]
cisco_routers
cisco_switches

[all:vars]
ansible_python_interpreter="/usr/bin/python3"
ansible_connection=network_cli

[cisco:vars]
ansible_network_os=ios
ansible_user=admin
ansible_become=yes
ansible_become_method=enable

[firewalls:vars]
ansible_network_os=asa
ansible_user=fwadmin

Rangos numéricos y patrones de hosts

Ansible permite definir múltiples hosts similares usando rangos numéricos y alfabéticos, ahorrando líneas y reduciendo errores de tipeo:

inventory_ranges.ini — Rangos y patrones
[switches_acceso]
# SW-Acc-01 hasta SW-Acc-48
SW-Acc-[01:48] ansible_host=10.0.1.[1:48]

[routers_wan]
# R1, R2, R3 ... R12
R[1:12] ansible_host=10.0.0.[1:12]

[ap_wifi]
# AP-Floor-A, AP-Floor-B, AP-Floor-C
AP-Floor-[A:C] ansible_host=10.0.2.[1:3]
⚡ Trampa frecuente en el examen

Los rangos en INI usan dos puntos (:), no guiones. [01:48] es válido, pero [01-48] fallará. Además, los rangos IP como 10.0.1.[1:48] solo funcionan si los octetos finales son numéricos consecutivos.

Variables inline vs. Variables de grupo

Puedes asignar variables directamente en la línea del host (inline) o en secciones [grupo:vars]. Las inline tienen mayor precedencia y sobrescriben las de grupo.

inventory_precedence.ini — Precedencia de variables
[cisco_switches]
# Variable inline: este switch usa usuario 'backup' en lugar del grupo
SW-Backup ansible_host=192.168.1.99 ansible_user=backup

# Variable inline: conexión diferente para este modelo antiguo
SW-Legacy ansible_host=192.168.1.100 ansible_connection=local

[cisco_switches:vars]
# Variables de grupo: aplican a TODOS los switches menos los que tengan inline
ansible_user=admin
ansible_network_os=ios
ansible_become=yes

Hosts múltiples y alias

El nombre a la izquierda es el alias de Ansible (cómo tú lo llamas en los playbooks). La IP real va en ansible_host. Si omites ansible_host, Ansible intentará resolver el alias vía DNS.

inventory_alias.ini — Alias y DNS
[routers]
# Alias 'Router-Madrid' apunta a IP fija
Router-Madrid ansible_host=10.1.0.1

# Alias 'Router-Barcelona' confía en DNS (ansible_host omitido)
Router-Barcelona

# Múltiples aliases apuntando a la misma IP (útil para labs)
R1-TEST ansible_host=192.168.56.101
R1-PROD ansible_host=10.0.0.1

Formato YAML: estructura jerárquica y legible

El formato YAML es la evolución natural del inventario INI. Aunque requiere más indentación, permite estructuras anidadas complejas, listas nativas, diccionarios y comentarios más descriptivos. Es el formato recomendado por Red Hat para inventarios de más de 50 hosts o con múltiples niveles de grupos.

En YAML, todo el inventario es un diccionario con las claves all, children y hosts. Las variables se anidan bajo vars en lugar de usar la sintaxis [grupo:vars].

inventory.yml — Equivalente YAML al INI anterior
all:
  children:
    cisco_routers:
      hosts:
        R1:
          ansible_host: 192.168.1.1
        R2:
          ansible_host: 192.168.1.2
        R3:
          ansible_host: 192.168.1.3

    cisco_switches:
      hosts:
        SW-Core-01:
          ansible_host: 192.168.1.10
        SW-Acc-01:
          ansible_host: 192.168.1.11
        SW-Acc-02:
          ansible_host: 192.168.1.12

    firewalls:
      hosts:
        FW-ASA-01:
          ansible_host: 192.168.1.254

    cisco:
      children:
        cisco_routers:
        cisco_switches:

  vars:
    ansible_python_interpreter: "/usr/bin/python3"
    ansible_connection: network_cli

    cisco:
      vars:
        ansible_network_os: ios
        ansible_user: admin
        ansible_become: yes
        ansible_become_method: enable

    firewalls:
      vars:
        ansible_network_os: asa
        ansible_user: fwadmin

Ventajas de YAML sobre INI

🎯

Listas Nativas

Puedes definir listas de DNS, NTP o syslog servers directamente como arrays YAML sin necesidad de strings separadas por comas.

🧩

Diccionarios Anidados

Variables complejas como interfaces, VLANs o ACLs se representan como diccionarios YAML, imposibles de expresar limpiamente en INI.

📐

Consistencia Visual

La indentación obligatoria fuerza un formato uniforme. Los editores modernos (VS Code, Vim) validan la sintaxis YAML en tiempo real.

🔀

Fusión de Inventarios

Ansible puede combinar múltiples archivos YAML en un directorio de inventario, fusionando automáticamente grupos y variables.

YAML avanzado: variables complejas por host

En YAML puedes definir estructuras de datos ricas directamente en el inventario. Esto es especialmente útil para configurar interfaces, rutas estáticas o VLANs específicas por dispositivo:

inventory_advanced.yml — Estructuras anidadas
all:
  children:
    core_switches:
      hosts:
        SW-Core-01:
          ansible_host: 10.1.0.1
          vlans:
            - { id: 10, name: "VENTAS", subnet: "192.168.10.0/24" }
            - { id: 20, name: "IT", subnet: "192.168.20.0/24" }
            - { id: 30, name: "GUEST", subnet: "192.168.30.0/24" }
          uplink_interfaces:
            - "GigabitEthernet1/0/1"
            - "GigabitEthernet1/0/2"
          ntp_servers:
            - "10.1.0.254"
            - "10.1.0.253"

        SW-Core-02:
          ansible_host: 10.1.0.2
          vlans:
            - { id: 10, name: "VENTAS" }
            - { id: 20, name: "IT" }
          uplink_interfaces:
            - "GigabitEthernet1/0/1"
💡 Tip de productividad

Usa la herramienta de línea de comandos ansible-inventory -i inventory.yml --graph para visualizar la jerarquía de grupos y hosts antes de ejecutar playbooks. Detecta errores de indentación o grupos huérfanos al instante.

Jerarquías de grupos: organización lógica

Los grupos son la clave para escalar Ansible. No gestionas 500 switches individualmente; gestionas 5 grupos lógicos. Un grupo puede contener hosts o, mediante children, otros grupos. Esto crea una jerarquía que hereda variables de arriba hacia abajo.

La estrategia de agrupamiento más efectiva para redes Cisco combina dos dimensiones: función del dispositivo (routers, switches, firewalls) y ubicación física o lógica (Madrid, Barcelona, Datacenter, WAN).

Patrones de selección de hosts

Al ejecutar playbooks o comandos ad-hoc, puedes apuntar a hosts usando patrones que combinan grupos, exclusiones y wildcards:

PatrónSignificadoEjemplo de uso
allTodos los hosts del inventarioansible all -m ping
cisco_switchesTodos los miembros del grupoansible cisco_switches -m ios_command
R1-MAD:R2-BCNUnión de dos hosts (o grupos)Apunta a ambos routers simultáneamente
cisco:&ubicacion_madridIntersección: hosts en ambos gruposSolo dispositivos Cisco en Madrid
cisco:!firewallsExclusión: Cisco excepto firewallsTodo Cisco menos ASA/FTD
SW*Wildcard: hosts que empiecen con SWTodos los switches por nombre
~REGEXExpresión regular~(R1|R2).*MAD para R1-MAD y R2-MAD
ℹ️ Concepto de intersección y exclusión

Los operadores & (intersección) y ! (exclusión) son extremadamente poderosos. Por ejemplo, para aplicar un parcheo solo a switches de acceso en producción (no en laboratorio): sw_acceso:&produccion:!laboratorio. Esto selecciona hosts que estén en ambos grupos (acceso Y producción) pero no en laboratorio.

Variables: el alma del inventario

Las variables en Ansible permiten personalizar el comportamiento por host, por grupo o globalmente. En redes Cisco, las variables no solo definen credenciales: almacenan subnets, VLANs, interfaces, políticas QoS, comunidades SNMP y cualquier dato que diferencie a un dispositivo de otro.

Ansible aplica una precedencia estricta (variable precedence). Una variable definida en host_vars siempre ganará sobre la misma variable definida en group_vars. Entender esta jerarquía es fundamental para evitar sobrescrituras accidentales.

host_vars y group_vars: archivos separados

En lugar de saturar el inventario con variables inline, la mejor práctica es crear dos directorios paralelos al inventario:

  • host_vars/ — Contiene un archivo YAML por cada host (nombre del archivo = alias del host).
  • group_vars/ — Contiene un archivo YAML por cada grupo (nombre del archivo = nombre del grupo).
proyecto_red/ ├── inventory.yml ├── host_vars/ │ ├── R1-MAD.yml # Variables exclusivas de R1-MAD │ ├── SW-Core-01.yml # Variables exclusivas del core switch │ └── FW-ASA-01.yml # Variables del firewall ├── group_vars/ │ ├── all.yml # Variables globales (todas las conexiones) │ ├── cisco_routers.yml # Solo routers Cisco │ ├── cisco_switches.yml # Solo switches Cisco │ └── firewalls.yml # Variables específicas de ASA/FTD └── playbooks/ └── configurar_ntp.yml
group_vars/all.yml — Variables globales de conexión
# Aplican a TODOS los hosts del inventario
ansible_connection: network_cli
ansible_python_interpreter: "/usr/bin/python3"
ansible_user: admin
ansible_ssh_pass: "{{ vault_cisco_pass }}"
ansible_become: yes
ansible_become_method: enable
ansible_network_os: ios
group_vars/cisco_switches.yml — Variables por función
# Solo aplican a switches Cisco
vlans_default:
  - { id: 10, name: "NATIVA" }
  - { id: 999, name: "BLACKHOLE" }

spanning_tree_mode: rapid-pvst
portfast_default: true

syslog_servers:
  - "10.0.0.100"
  - "10.0.0.101"
host_vars/SW-Core-01.yml — Variables exclusivas de un host
# Este switch es el VTP Server y tiene VLANs adicionales
vtp_mode: server
vtp_domain: "EMPRESA_CORP"

vlans_extra:
  - { id: 100, name: "FINANZAS" }
  - { id: 200, name: "RRHH" }

uplink_trunk_ports:
  - "GigabitEthernet1/0/24"
  - "GigabitEthernet1/0/25"

# Sobrescribo la variable global solo para este host
ansible_user: "core_admin"

Jerarquía de precedencia de variables (simplificada)

Ansible evalúa más de 20 niveles de precedencia. Para el examen CCNA y la práctica diaria, estos son los que debes memorizar (de menor a mayor prioridad):

  1. all (variables más genéricas)
  2. group_vars/all
  3. group_vars/<grupo> (grupos más amplios primero)
  4. group_vars/<subgrupo>
  5. host_vars/<host> (variables más específicas)
  6. Variables inline en el inventario (mayor prioridad)
  7. Extra vars con -e en línea de comandos (ganan a todo)
⚡ Trampa de precedencia

Un error común es definir ansible_user: admin en group_vars/all.yml y luego sorprenderse porque un switch específico usa ansible_user: backup definido en host_vars/SW1.yml. Recuerda: más específico siempre gana sobre más genérico.

Inventarios Dinámicos: la nube y el CMDB como fuente de verdad

Los inventarios dinámicos no son archivos estáticos. Son plugins o scripts que consultan una fuente externa (API de cloud, base de datos, orquestador) y generan el inventario en tiempo real durante la ejecución del playbook. Esto es esencial en entornos cloud donde las instancias EC2 o VMs de Azure aparecen y desaparecen constantemente.

Ansible soporta decenas de plugins oficiales. Para redes empresariales, los más relevantes son AWS EC2, Azure Resource Manager, VMware y NetBox (el CMDB open-source preferido por ingenieros de red).

☁️

AWS EC2

Filtra instancias por tags, regiones o VPCs. Ideal para routers virtuales (CSR 1000v) o firewalls en cloud. Requiere credenciales AWS configuradas.

Ⓜ️

Azure RM

Consulta Resource Groups, VMs y scale sets. Soporta autenticación por Service Principal o Managed Identity. Perfecto para Azure Network Virtual Appliances.

🔍

NetBox

El CMDB open-source diseñado específicamente para infraestructura de red. Almacena racks, dispositivos, interfaces, IPs, VLANs y cables. Integración nativa con Ansible vía plugin.

🖥️

VMware vCenter

Descubre VMs, templates y clusters. Útil para entornos híbridos donde routers virtuales (CSR, Nexus 9000v) conviven con servidores en vSphere.

📋

ServiceNow

Integración con CMDBs enterprise. Consulta la tabla cmdb_ci_netgear para obtener routers, switches y firewalls registrados en tu ITSM.

⚡

Custom Scripts

Puedes escribir tu propio script en Python que devuelva JSON con el formato requerido por Ansible. Útil para bases de datos internas o APIs propietarias.

Plugin AWS EC2: filtrado por tags

El plugin amazon.aws.aws_ec2 es el más utilizado para infraestructura cloud. En lugar de listar IPs manualmente, defines filtros que seleccionan instancias basadas en tags de AWS:

aws_ec2.yml — Inventario dinámico AWS
plugin: amazon.aws.aws_ec2
regions:
  - eu-west-1
  - us-east-1

# Filtros: solo instancias con tag Environment=Production
filters:
  tag:Environment: Production
  instance-state-name: running

# Agrupamiento por tags
keyed_groups:
  - key: tags.Role
    prefix: aws
    separator: "_"
  - key: tags.Location
    prefix: site

# Variables compuestas
compose:
  ansible_host: public_ip_address | default(private_ip_address)
  ansible_network_os: "ios"
Comandos de verificación
# Listar hosts descubiertos por el plugin
ansible-inventory -i aws_ec2.yml --graph

# Ver variables asignadas a un host específico
ansible-inventory -i aws_ec2.yml --host i-0a1b2c3d4e5f6789a

# Ejecutar playbook contra grupos generados dinámicamente
ansible-playbook -i aws_ec2.yml backup_routers.yml --limit aws_csr1000v

Plugin NetBox: la fuente de verdad para redes

NetBox es un CMDB (Configuration Management Database) open-source desarrollado específicamente para ingenieros de red. Almacena racks, dispositivos, interfaces, IPs, prefixos, VLANs, cables y circuitos. Su integración con Ansible es impecable mediante el plugin netbox.netbox.nb_inventory.

netbox_inventory.yml — Inventario dinámico NetBox
plugin: netbox.netbox.nb_inventory
api_endpoint: "https://netbox.tuempresa.com"
token: "{{ lookup('env', 'NETBOX_TOKEN') }}"
validate_certs: true

# Qué dispositivos incluir (filtrado por plataforma)
filters:
  platform: "cisco-ios"

# Agrupar por sitio y rol
group_by:
  - site.slug
  - device_role.slug
  - platform.slug

# Variables compuestas desde NetBox
compose:
  ansible_host: "primary_ip.address | ansible.utils.ipaddr('address')"
  ansible_network_os: "platform.slug"
  site_code: "site.slug"

# Incluir custom fields como variables
query_filters:
  - has_primary_ip: true
💡 Ventajas de NetBox + Ansible
  • Fuente única de verdad: Si un técnico cambia la IP de un switch en NetBox, Ansible lo descubre automáticamente sin modificar archivos.
  • Documentación viva: El inventario siempre refleja la realidad física: racks, conexiones, interfaces.
  • Filtrado inteligente: Puedes ejecutar playbooks solo en "todos los routers Cisco del sitio Madrid que tengan primary_ip".
  • Custom fields: Extiende NetBox con campos personalizados (número de ticket, fecha de mantenimiento) y úsalos como variables en Ansible.

Azure Resource Manager: redes híbridas

Para entornos Microsoft Azure, el plugin azure.azcollection.azure_rm descubre VMs, scale sets y network interfaces. Es ideal cuando tienes Network Virtual Appliances (NVAs) como Cisco CSR 1000v o FortiGate en Azure:

azure_rm.yml — Inventario dinámico Azure
plugin: azure.azcollection.azure_rm
subscription_id: "12345678-1234-1234-1234-123456789abc"
resource_groups:
  - "RG-NETWORK-PROD"
  - "RG-NETWORK-DR"

# Filtrar por tags
include_vm_resource_groups:
  - "RG-NETWORK*"

keyed_groups:
  - key: tags.environment
    prefix: env
  - key: tags.role
    prefix: role

conditional_groups:
  cisco_routers: "tags.role == 'csr1000v'"
  firewalls: "tags.role == 'nva-firewall'"

Combinando estáticos y dinámicos

Puedes usar un directorio como inventario, colocando dentro archivos estáticos y dinámicos. Ansible los fusiona en un único inventario unificado:

inventario_produccion/ ├── inventory.yml # Estático: routers físicos on-premise ├── aws_ec2.yml # Dinámico: CSR 1000v en AWS ├── azure_rm.yml # Dinámico: NVAs en Azure ├── netbox_inventory.yml # Dinámico: todo desde CMDB ├── group_vars/ │ ├── all.yml │ ├── cisco_routers.yml │ └── firewalls.yml └── host_vars/ └── R1-MAD.yml
Ejecución con directorio de inventario
# Apuntar a un directorio completo como inventario
ansible-playbook -i inventario_produccion/ site.yml

# Ver la fusión completa de todos los orígenes
ansible-inventory -i inventario_produccion/ --graph --vars
⚡ Precaución con inventarios dinámicos

Los inventarios dinámicos dependen de conectividad a APIs externas. Si NetBox o AWS están caídos, tu playbook fallará antes de comenzar. Para operaciones críticas, genera una caché estática periódicamente con ansible-inventory -i dinamico.yml --yaml --output estatico_cache.yml.

Mejores prácticas para inventarios de red

Un inventario bien diseñado es la diferencia entre una automatización que escala elegantemente y un desastre de mantenimiento. Estas prácticas han sido validadas en producción en redes de miles de dispositivos:

1. Nunca almacenes contraseñas en texto plano

Usa Ansible Vault para cifrar variables sensibles. Vault usa AES-256 y se integra transparentemente con los playbooks:

Cifrado con Ansible Vault
# Crear archivo cifrado para contraseñas
ansible-vault create group_vars/all/vault.yml

# Editar archivo cifrado existente
ansible-vault edit group_vars/all/vault.yml

# Contenido de vault.yml (cifrado en disco)
vault_cisco_pass: "SuperSecreta123!"
vault_snmp_community: "C0mmun1ty_Str0ng"

# Referencia desde variables no cifradas
# group_vars/all.yml:
ansible_ssh_pass: "{{ vault_cisco_pass }}"

# Ejecutar playbook con vault
ansible-playbook site.yml --ask-vault-pass
# O con archivo de contraseña (automatización CI/CD)
ansible-playbook site.yml --vault-password-file ~/.vault_pass.txt

2. Usa nombres descriptivos, no IPs solas

Un inventario con 192.168.1.1 como alias es inmantenible. Usa convenciones de nomenclatura:

  • R1-MAD-EDGE — Router 1, Madrid, borde WAN
  • SW-CORE-BCN-01 — Switch core, Barcelona, número 01
  • FW-ASA-DR — Firewall ASA, sitio Disaster Recovery

3. Separa entornos con directorios

Nunca mezcles producción y laboratorio en el mismo archivo. Usa estructura de directorios:

inventories/ ├── production/ │ ├── inventory.yml │ ├── group_vars/ │ └── host_vars/ ├── staging/ │ ├── inventory.yml │ ├── group_vars/ │ └── host_vars/ └── laboratory/ ├── inventory.yml ├── group_vars/ └── host_vars/

4. Documenta con comentarios y metadata

Ansible ignora los comentarios, pero los ingenieros no. Documenta el propósito de cada grupo:

inventory_documentado.yml
# ============================================================
# GRUPO: cisco_wan_routers
# PROPOSITO: Routers de borde WAN con BGP a ISP
# RESPONSABLE: Equipo Core Network
# ULTIMO CAMBIO: 2024-03-15 — Añadido R3-MAD
# ============================================================
cisco_wan_routers:
  hosts:
    R1-MAD-EDGE:
      ansible_host: 10.1.0.1
      bgp_asn: 64512
      isp_peer: "Telefonica"

5. Valida antes de ejecutar

Ansible proporciona herramientas para verificar tu inventario antes de tocar dispositivos de producción:

Comandos de validación esenciales
# Ver estructura de grupos y hosts
ansible-inventory -i inventory.yml --graph

# Ver todas las variables de un host específico
ansible-inventory -i inventory.yml --host R1-MAD --yaml

# Verificar conectividad sin ejecutar cambios
ansible all -i inventory.yml -m ping --check

# Simulación completa de playbook (dry run)
ansible-playbook -i inventory.yml site.yml --check --diff

6. Variables de conexión: el checklist obligatorio

Antes de declarar un inventario "listo", verifica estos puntos para cada grupo de dispositivos:

✅

IP Reachable

Verifica conectividad ICMP y SSH desde el Control Node. Usa ping y ssh -v para diagnosticar.

✅

Credenciales válidas

Confirma que el usuario tiene privilegios suficientes (level 15 en Cisco) y que la contraseña no ha expirado.

✅

Protocolo correcto

IOS antiguo → network_cli. IOS-XE moderno → httpapi o netconf. Verifica con show ip ssh.

✅

Ansible_network_os

Debe coincidir exactamente: ios, nxos, iosxr, asa. Un typo aquí causa fallos crípticos.

ℹ️ Convención de nomenclatura recomendada

Establece un estándar de nombres para aliases: <TIPO>-<FUNCIÓN>-<SITE>-<NÚMERO>. Ejemplos: R-WAN-MAD-01, SW-CORE-BCN-01, FW-ASA-DR-01. Esto permite usar patrones como SW-*-MAD-* para seleccionar todos los switches de Madrid.
