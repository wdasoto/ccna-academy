Routers Cisco

Arquitectura · Boot Sequence · IOS CLI · Interfaces · SSH v2 · NTP · Syslog · Recuperación de Contraseña

CCNA 200-301 Módulo 2 · Dispositivos Cisco 📡 Capa 3 — Routing

📡 ¿Qué hace un router y por qué existe?

Un router es el dispositivo que interconecta redes distintas. Mientras un switch opera en Capa 2 usando MACs para reenviar frames dentro de la misma LAN, el router opera en Capa 3 usando direcciones IP para tomar decisiones de reenvío entre redes diferentes. Sin un router, dos subredes distintas (192.168.1.0/24 y 10.0.0.0/8) no pueden comunicarse jamás.

🗺️ Routing

Consulta la tabla de routing (RIB) para cada paquete entrante. Elige el mejor camino aplicando el algoritmo de longest-prefix match y reenvía el paquete por la interfaz correspondiente.

🔀 Separación de broadcasts

Cada interfaz del router es un dominio de broadcast separado. Los frames broadcast Ethernet nunca cruzan un router — quedan confinados a su red local.

🔐 Seguridad de perímetro

Aplica ACLs, NAT/PAT, firewall stateful e IPsec VPN. Es el guardián entre la red interna y el mundo exterior.

⚡ Servicios de red

DHCP server/relay, DNS proxy, NTP, Syslog, SNMP, QoS, NetFlow. Un router Cisco es una plataforma de servicios completa, no solo un dispositivo de reenvío.

🔑 Diferencia crítica: running-config vs startup-config

  • running-config: vive en RAM. Es la configuración activa ahora mismo. Cualquier comando que ejecutes modifica el running-config inmediatamente.
  • startup-config: vive en NVRAM. Es la configuración que se carga al arrancar. Si apagas el router sin guardar, los cambios del running-config se pierden.
  • copy running-config startup-config (o wr) = guardar los cambios permanentemente.
  • copy startup-config running-config = restaurar la última configuración guardada (no reinicia).

⚡ CEF — Cisco Express Forwarding

CEF es el mecanismo de reenvío de alta velocidad de Cisco. En lugar de procesar cada paquete en la CPU (process switching), CEF precalcula toda la información necesaria en hardware.

MecanismoCómo funcionaVelocidadEstado
Process SwitchingCada paquete interrumpe la CPU. Consulta la tabla de routing completa.Muy lentoLegado
Fast SwitchingCache del primer paquete de cada flujo. Los siguientes usan el cache.MedioObsoleto
CEF (default)FIB (Forwarding Information Base) + Adjacency Table precalculadas. Reenvío en hardware sin intervenir la CPU.Muy rápido — línea✅ Default en IOS moderno

📌 CEF en la práctica

La FIB es una copia optimizada de la tabla de routing para reenvío rápido. La Adjacency Table contiene las cabeceras de Capa 2 (MAC destino + MAC origen) precalculadas para cada next-hop. CEF combina ambas y reenvía el paquete sin tocar la CPU.

⚠️ Diferencia crítica router vs switch — interfaces

Las interfaces de un router están administratively DOWN por defecto. Requieren no shutdown explícito para activarse. Las interfaces de un switch están up por defecto al conectar un cable. Esta diferencia aparece frecuentemente en el examen CCNA.

⚡ Secuencia de Arranque del Router Cisco

Cada vez que enciendes un router Cisco, sigue un proceso de arranque determinístico en 4 fases. Entender este proceso es fundamental para el CCNA, especialmente para la recuperación de contraseñas.

Fase 1 — POST (Power-On Self Test)

Lo primero que hace el router al encender. El firmware en ROM verifica que el hardware funciona correctamente: CPU, RAM, interfaces. Si detecta un fallo crítico, el arranque se detiene y muestra un error.

System Bootstrap, Version 15.1(4)M4, RELEASE SOFTWARE
Copyright (c) 1994-2012 by Cisco Systems, Inc.
Cisco IOS Software, C1900 Software

Running default software
PC = 0x7fef4050, Vector = 0x500, SP = 0x680127b0
C1900 platform with 512 Mbytes of main memory
Main memory is configured to 64 bit mode with ECC disabled

Readonly ROMMON initialized

Fase 2 — Bootstrap: leer el Configuration Register

El bootstrap lee el configuration register (16 bits en NVRAM). Este valor determina dónde buscar el IOS y qué configuración cargar. El valor por defecto es 0x2102.

Config RegisterComportamientoUso
0x2102Carga IOS desde Flash + startup-config desde NVRAM✅ Operación normal (default)
0x2142Carga IOS desde Flash pero ignora startup-config🔑 Recuperación de contraseña
0x2100Arranca directamente en ROM Monitor (ROMmon)Diagnóstico / recuperación IOS
0x2101Carga el mini-IOS de ROM (RxBoot)Router sin IOS en Flash
 
! Ver el configuration register actual
R1# show version
...
Configuration register is 0x2102

! Cambiar el configuration register
R1(config)# config-register 0x2142
! El cambio tiene efecto en el PRÓXIMO reinicio

R1# show version | include register
Configuration register is 0x2102 (will be 0x2142 at next reload)

Fase 3 — Localizar y cargar el IOS

El bootstrap busca el IOS en este orden (determinado por los bits del config register):

  1. Flash — busca el primer archivo .bin disponible (o el especificado en boot system)
  2. TFTP server — si está configurado boot system tftp
  3. ROM — mini-IOS de emergencia (funcionalidad muy limitada)

Fase 4 — Cargar la startup-config desde NVRAM

Con el IOS cargado y corriendo, el router busca la startup-config en NVRAM. Si existe, la copia a RAM como running-config y el router queda operativo. Si no existe (o si config-register = 0x2142), lanza el Setup Dialog.

🖥️ ROM Monitor (ROMmon) — La sala de emergencias

ROMmon es el entorno de bajo nivel que se activa cuando el router no puede cargar el IOS normalmente. Es la última línea de defensa para recuperar un router.

⚠️ ¿Cuándo aparece ROMmon?

  • Config-register = 0x2100
  • Flash corrupta o vacía (sin IOS)
  • Se presiona Ctrl+Break durante los primeros 60 segundos del boot (método de recuperación de contraseña)
! El prompt cambia a rommon 1 >
rommon 1 >

! Ver variables de entorno de boot
rommon 1 > set
PS1=rommon ! >
?=0
IP_ADDRESS=
IP_SUBNET_MASK=
DEFAULT_GATEWAY=
TFTP_SERVER=
TFTP_FILE=
BOOT=flash:c1900-universalk9-mz.SPA.151-4.M4.bin

! Arrancar con el IOS en Flash
rommon 2 > boot flash:c1900-universalk9-mz.SPA.151-4.M4.bin

! Arrancar desde TFTP (recuperación)
rommon 3 > IP_ADDRESS=192.168.1.1
rommon 4 > IP_SUBNET_MASK=255.255.255.0
rommon 5 > DEFAULT_GATEWAY=192.168.1.254
rommon 6 > TFTP_SERVER=192.168.1.100
rommon 7 > TFTP_FILE=c1900-universalk9-mz.SPA.151-4.M4.bin
rommon 8 > tftpdnld

⚠️ Trampa del examen: up/down en Serial

En interfaces Serial, el estado up/down frecuentemente indica que falta el clock rate en el lado DCE, o que la encapsulación (HDLC/PPP) no coincide en ambos extremos del enlace.

📡 Interfaces Serial — WAN

Configuración de interfaz Serial con HDLC y PPP

! ── LADO DCE (pone el clock) ────────────────────────

R1(config-if)# description WAN-link-a-ISP --- R2-Se0/0/0
R1(config-if)# ip address 10.0.0.1 255.255.255.252

🔄 Interfaces Loopback — Siempre activas

Las interfaces Loopback son virtuales — no tienen existencia física. Su ventaja principal: siempre están en estado up/up mientras el router esté encendido. No dependen de un cable ni de un vecino.

🆔 Router ID

OSPF y EIGRP usan la IP más alta de una Loopback como Router ID. Se recomienda configurar una Loopback específica para el Router ID.

🔧 Gestión out-of-band

Asignar una IP de gestión en Loopback garantiza que el router siempre sea alcanzable por SSH/Telnet, independientemente de qué interfaces físicas estén up o down.

🧪 Pruebas y simulación

En labs, las Loopbacks simulan redes remotas para probar routing sin necesitar dispositivos adicionales.

! Crear y configurar una interfaz Loopback
R1(config)# interface Loopback0
R1(config-if)# description Router-ID-y-Gestion
R1(config-if)# ip address 1.1.1.1 255.255.255.255
! /32 para Loopback de gestión — no crea una subred, solo el host
! No necesita "no shutdown" — las Loopbacks están up por defecto

R1# show interface Loopback0

  Hardware is Loopback

🏷️ Subinterfaces — Router-on-a-Stick

! La interfaz física: activa, sin IP
R1(config)# interface GigabitEthernet0/0
R1(config-if)# no ip address
R1(config-if)# no shutdown

! Subinterfaz para VLAN 10
R1(config)# interface GigabitEthernet0/0.10
R1(config-subif)# description GATEWAY-VLAN10-VENTAS
R1(config-subif)# encapsulation dot1q 10
! encapsulation dot1q X es OBLIGATORIO — sin esto no hay VLAN ID
R1(config-subif)# ip address 192.168.10.1 255.255.255.0

! Subinterfaz para VLAN 20
R1(config)# interface GigabitEthernet0/0.20
R1(config-subif)# encapsulation dot1q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0

! VLAN nativa — usar "native" keyword
R1(config)# interface GigabitEthernet0/0.99
R1(config-subif)# encapsulation dot1q 99 native
R1(config-subif)# ip address 192.168.99.1 255.255.255.0

⚠️ Error frecuente: subinterfaz sin encapsulation dot1q

Si olvidas el comando encapsulation dot1q X en la subinterfaz, el router no sabe a qué VLAN corresponde y los paquetes no llegan correctamente. Es el error más común al configurar RoaS.

🔐 Seguridad del dispositivo — Capas de protección

La seguridad de un router Cisco tiene múltiples capas. Una mala configuración de seguridad puede exponer toda la red. El examen CCNA verifica específicamente el uso correcto de enable secret, SSH v2 y la protección de las líneas VTY.

⚠️ enable password vs enable secret — La diferencia que importa

ComandoAlmacenamientoAlgoritmoSeguridad
enable password PASSTexto plano en config (o tipo 7 con service password-encryption)Rot/Vigenère débil❌ Nunca usar — fácilmente reversible
enable secret PASSHash MD5 en configMD5⚠️ Aceptable — standard histórico
enable algorithm-type sha256 secret PASSHash SHA-256SHA-256✅ Recomendado en IOS 15.3+

Si existen ambos (enable password y enable secret), enable secret siempre tiene prioridad y enable password es ignorado.

🔒 SSH v2 — Acceso remoto seguro completo

Telnet envía todo en texto plano incluyendo contraseñas. SSH v2 cifra la sesión completamente. En producción, Telnet debe estar deshabilitado y solo SSH debe estar activo.

Configuración SSH v2 paso a paso

🛠️ NTP — Sincronización de tiempo

NTP (Network Time Protocol) sincroniza los relojes de todos los dispositivos de la red. Un reloj correcto es esencial para que los logs de Syslog, los certificados SSL, los timestamps de debugging y la correlación de eventos funcionen correctamente.

📌 Estratos NTP (Stratum)

NTP usa una jerarquía de estratos. Stratum 0 = fuente de referencia (reloj atómico, GPS). Stratum 1 = servidor conectado directamente al stratum 0. Stratum 2 = sincronizado con stratum 1, etc. Un router Cisco configura como NTP server tiene stratum = (stratum de su fuente) + 1.

Configurar NTP client y server

! ── R1 COMO CLIENTE NTP ─────────────────────────────
R1(config)# ntp server 216.239.35.0 prefer
! "prefer": usar este servidor como primera opción
R1(config)# ntp server 216.239.35.4
! Servidor secundario (redundancia)
R1(config)# ntp update-calendar
! Actualiza el reloj hardware del router

! ── R1 COMO SERVIDOR NTP PARA LA LAN ────────────────
R1(config)# ntp master 3
! Configura R1 como NTP master con stratum 3
! Los demás routers de la red pueden sincronizarse con R1

! ── ZONA HORARIA ────────────────────────────────────
R1(config)# clock timezone COT -5
! Colombia Time: UTC-5
R1(config)# clock summer-time CDT recurring
! Horario de verano (si aplica en tu país)

🔑 Recuperación de contraseña perdida — Procedimiento completo

Si un administrador olvida la contraseña enable secret de un router Cisco, el procedimiento de recuperación implica interrumpir el boot normal para que el router ignore la startup-config (que contiene la contraseña) y arranque sin ella. Luego se modifica la contraseña y se restaura la configuración.

⚠️ Requiere acceso físico a la consola

Este procedimiento requiere conectarse al puerto de consola físico del router. No puede realizarse por SSH/Telnet (precisamente porque no tienes la contraseña). Esto garantiza que solo alguien con acceso físico puede recuperar el dispositivo.

Paso 1 — Interrumpir el boot con Ctrl+Break

Conectado a la consola, reinicia el router y presiona Ctrl+Break durante los primeros 60 segundos del boot. El router entra en ROMmon.

! El router reinicia. Durante el boot:
System Bootstrap, Version 15.1(4)M4...
Loading "flash:c1900-universalk9-mz.SPA.151-4.M4.bin"

! Presionar Ctrl+Break AQUÍ (primeros 60 segundos)

Readonly ROMMON initialized

rommon 1 >
! Estamos en ROMmon — el IOS NO se ha cargado todavía

Paso 2 — Cambiar el configuration register a 0x2142

El valor 0x2142 hace que el router ignore la startup-config al arrancar. Esto significa que no cargará la contraseña (ni ninguna otra configuración).

rommon 1 > confreg 0x2142
You must reset or power cycle for new config to take effect

rommon 2 > reset
! El router se reinicia. Esta vez arranca SIN cargar startup-config.

Paso 3 — Arranque sin configuración y copiar startup → running

--- System Configuration Dialog ---
Would you like to enter the initial configuration dialog? [yes/no]: no

Router>
Router> enable
! No pide contraseña — la startup-config fue ignorada
Router#

! IMPORTANTE: copiar startup-config a running-config
! Esto restaura TODA la configuración (VLANs, rutas, interfaces...)
! EXCEPTO la contraseña — la startup-config está en NVRAM sin modificar
Router# copy startup-config running-config
Destination filename [running-config]? (Enter)
1234 bytes copied in 0.456 secs

R1#
! El hostname aparece ahora — la config fue cargada, excepto la contraseña

Paso 4 — Cambiar la contraseña

R1# configure terminal
R1(config)# enable secret NuevaContraseña@2024!
! También cambiar contraseñas de usuarios si es necesario
R1(config)# username admin algorithm-type sha256 secret NuevoAdmin@2024!

Paso 5 — Restaurar config-register a 0x2102 y guardar

! CRÍTICO: volver el config-register al valor normal
! Si olvidas esto, en el próximo reinicio volverá a ignorar la startup-config
R1(config)# config-register 0x2102
R1(config)# end

! Guardar todo (nueva contraseña + config-register restaurado)
R1# copy running-config startup-config
Building configuration...
[OK]

! Verificar que el config-register quedó bien
R1# show version | include Configuration register
Configuration register is 0x2102

! Reiniciar para confirmar que todo funciona
R1# reload
! El router arranca normalmente con la nueva contraseña

📊 Resumen visual del proceso

PasoAcciónPropósitoConfig-Register
1Ctrl+Break en bootEntrar a ROMmonAún 0x2102
2confreg 0x2142 + resetReiniciar ignorando startup-configCambia a 0x2142
3copy startup-config running-configCargar la config existente sin la contraseña0x2142 (activo)
4enable secret NUEVAEstablecer nueva contraseña0x2142 (activo)
5config-register 0x2102 + copy run startRestaurar arranque normal y guardarVuelve a 0x2102

⚠️ Los 3 errores más comunes en la recuperación

  1. Olvidar hacer copy startup-config running-config en el paso 3 — el router arranca sin ninguna configuración y pierdes VLANs, rutas, etc.
  2. Olvidar restaurar config-register 0x2102 en el paso 5 — en el próximo reinicio vuelve a ignorar la startup-config.
  3. Olvidar copy running-config startup-config al final — los cambios se pierden al recargar.

📋 Resumen Maestro — Routers Cisco para el examen CCNA

✅ Los 25 puntos más importantes del examen

  1. Router = Capa 3. Interconecta redes distintas usando IPs. Switch = Capa 2, misma red con MACs.
  2. RAM: running-config, tabla routing, ARP, CEF. Volátil — se pierde al apagar.
  3. NVRAM: startup-config + config-register. No volátil.
  4. Flash: IOS (.bin). No volátil. Como un disco SSD.
  5. ROM: POST + Bootstrap + ROMmon. Solo lectura. Firmware fijo.
  6. Interfaces del router: administratively down por defecto. Requieren no shutdown.
  7. Interfaces del switch: activas por defecto al conectar cable.
  8. Config-register 0x2102 = operación normal. 0x2142 = ignora startup-config.
  9. Boot sequence: POST → Bootstrap → IOS desde Flash → startup-config desde NVRAM.
  10. ROMmon: entorno de recuperación. Se accede con Ctrl+Break durante los primeros 60s del boot.
  11. enable secret (MD5/SHA) siempre tiene prioridad sobre enable password (texto plano).
  12. service password-encryption aplica cifrado tipo 7 (débil). No protege enable secret.
  13. SSH v2 requiere: hostname ≠ Router, ip domain-name, crypto key generate rsa modulus 2048, ip ssh version 2.
  14. VTY con transport input ssh: deshabilita Telnet completamente.
  15. login local: autentica con base de datos local. login: autentica con contraseña de línea.
  16. Loopback: siempre up/up. Usada para Router ID OSPF/EIGRP y gestión estable.
  17. Serial DCE: necesita clock rate. DTE: NO poner clock rate.
  18. show controllers Serial X: indica si es DCE o DTE y el clock rate.
  19. Subinterfaz RoaS: interfaz física sin IP + no shutdown. Subinterfaces con encapsulation dot1q X + IP.
  20. CEF (default): FIB + Adjacency Table → reenvío en hardware, sin interrumpir CPU.
  21. NTP: ntp server IP (cliente). ntp master X (servidor con stratum X).
  22. Syslog niveles: 0-Emergency hasta 7-Debug. logging trap informational = envía niveles 0-6.
  23. service timestamps log datetime msec: agrega timestamp a todos los logs.
  24. Recuperación contraseña: Ctrl+Break → confreg 0x2142 → reset → copy start run → nueva contraseña → config-reg 0x2102 → copy run start.
  25. copy run start = wr = copy running-config startup-config: guardar configuración.

⚠️ Los 10 errores más comunes en el examen CCNA

  1. Poner clock rate en el lado DTE (solo DCE lo necesita)
  2. Olvidar no shutdown en la interfaz física del router (están down por defecto)
  3. Olvidar ip domain-name antes de generar claves RSA para SSH
  4. Olvidar encapsulation dot1q X en la subinterfaz de RoaS
  5. Usar enable password en lugar de enable secret
  6. Olvidar restaurar config-register 0x2102 tras recuperar contraseña
  7. Olvidar hacer copy startup-config running-config en el paso 3 de recuperación
  8. Confundir login (contraseña de línea) con login local (base de datos local)
  9. No poner no ip domain-lookup — el router intenta resolver typos como DNS y cuelga la CLI
  10. Modificar running-config y no guardar con copy run start — se pierde al recargar
