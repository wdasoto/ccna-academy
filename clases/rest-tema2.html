RESTCONF — REST para IOS-XE

El protocolo estándar RFC 8040 que lleva REST directamente a los routers y switches Cisco. Aprende a manipular configuración YANG, datastores y modelos de datos nativos vía HTTPS para automatizar infraestructura real sin SSH ni parseo de CLI.

🔧 RFC 8040 🎯 CCNA Domain 6.5 ⏱ ~100 min lectura 💻 Labs con IOS-XE 🧪 Proyecto CRUD real

¿Qué es RESTCONF y por qué cambia todo?

RESTCONF es el protocolo definido en la RFC 8040 que permite gestionar dispositivos de red usando los principios REST (HTTP + verbs) sobre los modelos de datos YANG. Es la evolución natural de NETCONF: mientras NETCONF usa sesiones SSH persistentes y XML-RPC, RESTCONF abre la puerta a las herramientas modernas de desarrollo.

Con RESTCONF, configurar un router Cisco deja de ser un ejercicio de expresiones regulares sobre texto plano. Se convierte en una conversación estructurada: tu script Python envía JSON semánticamente validado contra un modelo YANG, y el router responde con códigos HTTP estándar. Si algo falla, obtienes un 400 Bad Request con detalles, no un % Ambiguous command en la CLI.

💡 ¿Por qué RESTCONF y no solo CLI/SSH?
  • Tipado fuerte: YANG valida tipos de datos (IPv4, uint16, boolean) antes de aplicar la configuración.
  • Idempotencia nativa: PUT y DELETE son idempotentes; reintentar es seguro.
  • Datos estructurados: JSON/XML con schema conocido, no texto libre que hay que parsear.
  • Datastores separados: Puedes editar candidate y hacer commit, o editar running directamente.
  • Estándar abierto: No es propietario de Cisco; es un RFC de la IETF soportado por múltiples vendors.

RESTCONF está disponible en IOS-XE 16.6+ (Denali y posterior) y en la mayoría de plataformas modernas: ISR 4000, Catalyst 8000, CSR 1000V, C9200/C9300 con capa de routing. Para habilitarlo:

RFC 8040 — La especificación formal

La RFC 8040: "RESTCONF Protocol" fue publicada en enero de 2017 por la IETF. Define cómo mapear operaciones sobre modelos YANG a los verbos HTTP estándar. No inventa un protocolo nuevo: usa HTTP/1.1 o HTTP/2 como transporte, JSON o XML como serialización, y YANG como lenguaje de modelado de datos.

Datastores: running, startup, candidate

RESTCONF hereda de NETCONF el concepto de datastores. Un datastore es una instancia de la base de datos de configuración del dispositivo. No todos los dispositivos soportan todos los datastores:

⚡

running

La configuración activa del dispositivo. Siempre presente. Operaciones GET/PUT/PATCH/DELETE apuntan aquí por defecto. Equivalente a show running-config.

💾

startup

Configuración que se carga al bootear. Presente en la mayoría de plataformas. Para persistir cambios en running, necesitas copy running-config startup-config (o su equivalente YANG).

📝

candidate

Copia editable sin impacto inmediato. Permite hacer múltiples cambios y luego un commit atómico. No todos los dispositivos Cisco lo soportan vía RESTCONF (más común en NETCONF puro).

Raíces de la API RESTCONF

La RFC 8040 define dos puntos de entrada (roots) obligatorios:

  • {+restconf}/data — Acceso a datastores y recursos de configuración/estado. Es donde vives el 99% del tiempo.
  • {+restconf}/operations — Invocación de RPCs ( Remote Procedure Calls ) definidos en YANG. Equivalente a <rpc> en NETCONF.
https://router.local:443 / restconf / data / Cisco-IOS-XE-native:native / interface / GigabitEthernet=0%2F0

Content-Type según RFC 8040

RESTCONF define sus propios tipos MIME, no usa genéricos:

Tipo MIMEUsoContexto

application/yang-data+xmlDatos YANG en XMLAlternativa para plataformas XML-centric
application/yang-patch+jsonPATCH conforme a YANG PatchCuando usas PATCH con formato YANG-Patch (no solo campos sueltos)
⚡ Trampa del examen CCNA

El examen pregunta cuál es el Content-Type correcto para RESTCONF. La respuesta no es application/json genérico. Es application/yang-data+json (o application/yang-data+xml). Este detalle distingue a un candidato que memorizó de uno que entiende RFC 8040.

YANG — El lenguaje de modelado de datos

YANG (Yet Another Next Generation, RFC 6020/7950) es un lenguaje de modelado de datos usado para definir la estructura, tipos y restricciones de la configuración y el estado operacional de dispositivos de red. Piensa en YANG como el schema de una base de datos: define qué tablas existen, qué columnas tienen y qué tipos de datos aceptan.

Cuando haces un GET a RESTCONF, el JSON que recibes no es arbitrario: está rigidamente definido por un modelo YANG. Cuando haces un PUT, el router valida tu JSON contra ese schema antes de aplicar cualquier cambio.

Anatomía de un modelo YANG (simplificado)

No necesitas escribir YANG, pero sí entenderlo para interpretar JSON y construir payloads:

Cisco-IOS-XE-native:native/interface (extracto conceptual)
module Cisco-IOS-XE-native {
  namespace "http://cisco.com/ns/yang/Cisco-IOS-XE-native";
  prefix native;

  container native {
    container interface {
      list GigabitEthernet {
        key "name";
        leaf name { type string; }
        leaf description { type string; }
        leaf shutdown {
          type empty;
          presence "Interface is shut down";
        }
        container ip {
          container address {
            container primary {
              leaf address { type inet:ipv4-address; }
              leaf mask { type inet:ipv4-address; }
            }
          }
        }
      }
    }
  }
}

Este modelo se traduce en el JSON que verás en RESTCONF:

JSON resultante para GigabitEthernet0/0
{
  "Cisco-IOS-XE-native:interface": {
    "GigabitEthernet": [
      {
        "name": "0/0",
        "description": "WAN-PRIMARY",
        "ip": {
          "address": {
            "primary": {
              "address": "192.168.1.1",
              "mask": "255.255.255.0"
            }
          }
        }
      }
    ]
  }
}

Tipos de nodos YANG

  • container: Agrupa nodos relacionados. No tiene valor propio. Ej: ip, address.
  • list: Secuencia ordenada de entradas con una key. Ej: GigabitEthernet (key = name).
  • leaf: Nodo con un solo valor de tipo simple. Ej: description (string), shutdown (empty).
  • leaf-list: Lista de valores simples sin keys. Ej: lista de DNS servers.
  • rpc: Operación invocable vía /operations. Ej: clear-arp.

Construcción de Paths RESTCONF

En RESTCONF, la URI identifica de forma única cada recurso YANG. La construcción de paths sigue reglas estrictas derivadas del modelo YANG. Un error en la URI produce un 404 Not Found o un 400 Bad Request inmediato.

Reglas de codificación

  • Module prefix: Se usa modulename: antes del nombre del nodo. Ej: Cisco-IOS-XE-native:native.
  • List keys: Se pasan como nombrerecurso=valor. Si el valor contiene / (como en interfaces), debe codificarse como %2F.
  • Namespace URI encoding: Los dos puntos del módulo van tal cual; los slashes internos de valores van URL-encoded.

Ejemplos de paths por recurso

RecursoPath RESTCONFNotas
Configuración completa /restconf/data/Cisco-IOS-XE-native:native GET retorna todo el running-config como JSON YANG masivo.
Interfaz específica /restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0 %2F es el slash. El key es el nombre 0/0.
Todas las interfaces /restconf/data/Cisco-IOS-XE-native:native/interface GET retorna array de todas las interfaces configuradas.
IP de interfaz .../interface/GigabitEthernet=0%2F0/ip/address/primary Navegación profunda por contenedores YANG.
Hostname /restconf/data/Cisco-IOS-XE-native:native/hostname Leaf simple; GET retorna string directamente.
OSPF proceso 1 /restconf/data/Cisco-IOS-XE-native:native/router/ospf/ospf=1 List key = process ID. En IOS-XE OSPF se modela como list.
VLAN 10 /restconf/data/Cisco-IOS-XE-native:native/vlan/vlan-list=10 En switches Layer 3 IOS-XE.
Usuario admin /restconf/data/Cisco-IOS-XE-native:native/username=admin Key del list username es el nombre.

Query parameters útiles

Filtros y modificadores de URI
# depth — controla qué tan profundo recorrer la jerarquía
GET .../interface/GigabitEthernet=0%2F0?depth=1
  → Solo el primer nivel de hijos (nombre, descripción, IP primaria...)

GET .../interface/GigabitEthernet=0%2F0?depth=unbounded
  → Todo el árbol completo (por defecto)

# fields — filtra qué nodos retornar
GET .../native?fields=hostname;interface
  → Solo hostname e interfaces, ignora routing, usuarios, etc.

# content — filtra tipo de datos
GET .../native?content=config     → solo configuración
GET .../native?content=nonconfig  → solo estado operacional (stats)
GET .../native?content=all      → ambos (por defecto)
⚡ Error común: olvidar codificar el slash

Si escribes GigabitEthernet=0/0 en la URL sin codificar, el servidor interpreta 0 como key y 0 como un path adicional, resultando en 404 Not Found. Siempre codifica: GigabitEthernet=0%2F0. En Python usa urllib.parse.quote('0/0', safe='').

CRUD en IOS-XE vía RESTCONF

Aquí aplicamos los verbos HTTP estudiados en el Tema 1 al contexto real de un router Cisco. Cada operación incluye la URI exacta, los headers obligatorios y el payload JSON esperado por YANG.

CREATE — POST vs PUT

En RESTCONF, POST se usa principalmente para crear recursos dentro de una colección (list) donde el servidor asigna la key. Sin embargo, muchos recursos de red tienen keys predecibles (nombre de interfaz, ID de proceso OSPF). Para esos, PUT es más común porque es idempotente.

PUT — Crear/Reemplazar Loopback 100
PUT /restconf/data/Cisco-IOS-XE-native:native/interface/Loopback=100
Content-Type: application/yang-data+json
Accept: application/yang-data+json

{
  "Cisco-IOS-XE-native:Loopback": {
    "name": "100",
    "description": "MANAGEMENT-LOOPBACK",
    "ip": {
      "address": {
        "primary": {
          "address": "1.1.1.1",
          "mask": "255.255.255.255"
        }
      }
    }
  }
}
POST — Crear nueva VLAN (cuando el server asigna o es colección)
POST /restconf/data/Cisco-IOS-XE-native:native/vlan
Content-Type: application/yang-data+json

{
  "Cisco-IOS-XE-native:vlan-list": {

UPDATE — PATCH parcial

PATCH es el verbo más útil para cambios rápidos: solo envías lo que quieres modificar. En IOS-XE, PATCH sobre un leaf o container reemplaza solo ese subárbol.

PATCH — Cambiar solo descripción y activar interfaz
PATCH /restconf/data/Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0
Content-Type: application/yang-data+json

{
  "Cisco-IOS-XE-native:GigabitEthernet": {
    "description": "ENLACE-AL-CORE-AKT1",
    "shutdown": [null]
  }
}

# Nota: "shutdown": [null] en JSON es el truco para eliminar un leaf de tipo empty.
# Equivale a "no shutdown" en CLI. El valor [null] representa delete en PATCH/PUT.

DELETE — Eliminar recursos

DELETE — Remover Loopback 100
DELETE /restconf/data/Cisco-IOS-XE-native:native/interface/Loopback=100

# Respuesta esperada:
HTTP/1.1 204 No Content

# Si intentas borrar algo que no existe:
HTTP/1.1 404 Not Found

Operaciones (RPC) vía /operations

Algunas acciones no encajan en CRUD: reiniciar un proceso, limpiar tabla ARP, aplicar copy run start. Estas se exponen como RPCs en /restconf/operations:

POST — Guardar configuración (copy run start vía RPC)
POST /restconf/operations/cisco-ia:save-config
Content-Type: application/yang-data+json

{ "cisco-ia:input": {} }

# Equivalente a "write memory" o "copy running-config startup-config".
# Retorna 204 No Content si tiene éxito.
💡 Idempotencia en la práctica con IOS-XE

Si tu script automatizado pierde conexión después de enviar un PUT, ¿lo reintentas? Con RESTCONF sí: PUT es idempotente. Enviar la misma configuración 10 veces no daña nada. Con CLI/SSH vía Netmiko, un reintento podría duplicar comandos o fallar en contextos cambiantes. Esta es la razón principal por la que RESTCONF es superior para automatización confiable.

Python + requests contra RESTCONF

La librería requests es todo lo que necesitas para hablar RESTCONF desde Python. No requieres librerías específicas de Cisco (aunque existen wrappers). Comprender HTTP a bajo nivel te da control total y te prepara para cualquier API REST de red.

restconf_client.py — Cliente base reutilizable
import requests
import json
import urllib3
urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

class RestconfClient:
    """Cliente RESTCONF RFC 8040 para IOS-XE"""

    def __init__(self, host, username, password, port=443):
        self.base_url = f"https://{host}:{port}/restconf/data"
        self.headers = {
            "Content-Type": "application/yang-data+json",
            "Accept": "application/yang-data+json",
        }
        self.auth = (username, password)
        self.verify = False   # ⚠️ Solo laboratorio

    def get(self, path):
        url = f"{self.base_url}/{path}"
        resp = requests.get(url, headers=self.headers, auth=self.auth,
                            verify=self.verify, timeout=10)
        resp.raise_for_status()
        return resp.json()

    def put(self, path, payload):
        url = f"{self.base_url}/{path}"
        resp = requests.put(url, headers=self.headers, auth=self.auth,
                            verify=self.verify, timeout=10,
                            data=json.dumps(payload))
        resp.raise_for_status()
        return resp.status_code

    def patch(self, path, payload):
        url = f"{self.base_url}/{path}"
        resp = requests.patch(url, headers=self.headers, auth=self.auth,
                              verify=self.verify, timeout=10,
                              data=json.dumps(payload))
        resp.raise_for_status()
        return resp.status_code

    def delete(self, path):
        url = f"{self.base_url}/{path}"
        resp = requests.delete(url, headers=self.headers, auth=self.auth,
                               verify=self.verify, timeout=10)
        resp.raise_for_status()
        return resp.status_code

# ── USO ─────────────────────────────────────────────
router = RestconfClient("192.168.1.1", "admin", "Cisco123")

# Leer hostname
data = router.get("Cisco-IOS-XE-native:native/hostname")
print(f"Hostname: {data['Cisco-IOS-XE-native:hostname']}")

# Cambiar descripción de interfaz
router.patch(
    "Cisco-IOS-XE-native:native/interface/GigabitEthernet=0%2F0",
    {"Cisco-IOS-XE-native:GigabitEthernet": {"description": "WAN-VIA-RESTCONF"}}
)

Manejo de errores específicos de IOS-XE

Los errores de RESTCONF en Cisco incluyen a veces un body JSON con detalles del modelo YANG que falló:

Manejo robusto de errores
from requests.exceptions import HTTPError, ConnectionError, Timeout

def safe_restconf_call(method_func, *args, **kwargs):
    """Wrapper que traduce errores HTTP a mensajes útiles"""
    try:
        return method_func(*args, **kwargs)
    except HTTPError as e:
        status = e.response.status_code
        if status == 400:
            detail = e.response.json().get("errors", {}).get("error", [{}])[0].get("error-message", "JSON malformado")
            print(f"[400] YANG validation failed: {detail}")
        elif status == 401:
            print("[401] Credenciales inválidas o usuario sin privilegio 15")
        elif status == 404:
            print("[404] Recurso no encontrado — revisa el path YANG")
        elif status == 409:
            print("[409] Conflicto — el recurso ya existe (usar PUT) o dependencia faltante")
        elif status == 500:
            print("[500] Error interno del daemon RESTCONF en el router")
        else:
            print(f"[{status}] Error HTTP: {e}")
    except ConnectionError:
        print("[NET] No se pudo conectar — verificar IP, HTTPS habilitado y reachability")
    except Timeout:
        print("[NET] Timeout — el router no respondió a tiempo")

Seguridad en RESTCONF

RESTCONF no negocia: HTTPS es obligatorio. Nunca envíes credenciales ni configuración por HTTP plano. La RFC 8040 asume TLS como transporte seguro y recomienda autenticación vía certificados de cliente o HTTP Basic Auth sobre TLS.

Autenticación soportada

🔑

HTTP Basic Auth

Usuario:contraseña codificados en Base64 dentro del header Authorization. Requiere HTTPS obligatoriamente. Soportado por todos los dispositivos IOS-XE.

🛡️

Certificados de Cliente

mTLS: el cliente presenta un certificado X.509 firmado por la CA del router. Sin contraseñas en el script. Máxima seguridad, configuración más compleja.

🎫

Tokens (OAuth/JWT)

No nativo en IOS-XE RESTCONF básico. Usado por DNA Center y controladores SDN donde RESTCONF actúa como backend interno.

Roles y privilegios en IOS-XE

RESTCONF respeta el nivel de privilegio del usuario:

  • Privilege 15: Acceso total. GET, PUT, PATCH, DELETE sobre cualquier modelo YANG.
  • Privilege < 15: Acceso limitado. Puede hacer GET sobre estado operacional pero recibe 403 Forbidden al intentar escritura.
  • RBAC con AAA/TACACS+: Se pueden definir comandos permitidos/denegados por usuario, y estos se mapean a operaciones RESTCONF.

Certificados autofirmados vs CA

Por defecto, IOS-XE genera un certificado autofirmado al habilitar ip http secure-server. Esto produce un warning en Python:

Opciones para manejar SSL en desarrollo
# Opción A: Deshabilitar verificación (SOLO LABORATORIO)
requests.get(url, verify=False)

# Opción B: Exportar certificado del router e importarlo
# En el router:
crypto pki export TP-self-signed-XXXX certificate pem terminal
# Copiar el PEM a un archivo local, por ejemplo router.pem

# En Python:
requests.get(url, verify="router.pem")

# Opción C: Usar PKI interna (Producción)
# Inscribir routers con SCEP/EST contra CA corporativa.
# Python usa el bundle de CA del sistema (verify=True por defecto).
⚡ Nunca en producción

Nunca uses verify=False en producción. Expones tu script a ataques MITM donde un atacante puede suplantar el router, capturar credenciales Basic Auth (aunque sean Base64, se decodifican trivialmente) y modificar tu infraestructura. En producción: mTLS o al menos verificación de certificado contra una CA interna.

🚀 Proyecto Final — Configurador IOS-XE vía RESTCONF

Este proyecto integra todo lo aprendido en una aplicación Python completa que se conecta a un router Cisco IOS-XE vía RESTCONF, obtiene su estado actual, aplica configuraciones y genera un reporte de auditoría.

💡 Escenario

Eres el ingeniero de automatización de una empresa con sucursales. Debes desplegar 50 routers ISR 4331 con configuración base idéntica: hostname, interfaz WAN con IP/DG, OSPF proceso 1, área 0, y una Loopback de management. En lugar de conectarte vía SSH a cada uno, usas RESTCONF.

proyecto_restconf_iosxe.py — Despliegue masivo
#!/usr/bin/env python3
"""proyecto_restconf_iosxe.py
CCNA Academy M8 T2 — Configuración masiva vía RESTCONF RFC 8040
Despliega configuración base en múltiples routers ISR.
"""

import requests
import json
import sys

urllib3.disable_warnings(urllib3.exceptions.InsecureRequestWarning)

# ══════════════════════════════════════════════════════
# CONFIGURACIÓN GLOBAL
# ══════════════════════════════════════════════════════
USER = "admin"
PASS = "Cisco123"
BASE_MODULE = "Cisco-IOS-XE-native:native"
HEADERS = {
    "Content-Type": "application/yang-data+json",
    "Accept": "application/yang-data+json",
}

ROUTERS = [
    {"id": 1, "ip": "10.0.0.1", "hostname": "R1-SUC-BOG", "wan_ip": "192.168.100.1", "mask": "255.255.255.252"},
    {"id": 2, "ip": "10.0.0.2", "hostname": "R2-SUC-MED", "wan_ip": "192.168.100.5", "mask": "255.255.255.252"},
    {"id": 3, "ip": "10.0.0.3", "hostname": "R3-SUC-CAL", "wan_ip": "192.168.100.9", "mask": "255.255.255.252"},
]

# ══════════════════════════════════════════════════════
# FUNCIONES CRUD REUTILIZABLES
# ══════════════════════════════════════════════════════
def restconf_url(host, path):
    return f"https://{host}/restconf/data/{path}"

def get_config(router, path):
    """Obtener configuración vía GET"""
    url = restconf_url(router["ip"], path)
    try:
        r = requests.get(url, headers=HEADERS, auth=HTTPBasicAuth(USER, PASS),
                         verify=False, timeout=10)
        r.raise_for_status()
        return r.json()
    except requests.exceptions.RequestException as e:
        print(f"❌ [{router['hostname']}] GET error: {e}")
        return None

def put_config(router, path, payload):
    """Aplicar configuración vía PUT (idempotente)"""
    url = restconf_url(router["ip"], path)
    try:
        r = requests.put(url, headers=HEADERS, auth=HTTPBasicAuth(USER, PASS),
                         verify=False, timeout=10, data=json.dumps(payload))
        r.raise_for_status()
        print(f"✅ [{router['hostname']}] PUT {path} → {r.status_code}")
        return True
    except requests.exceptions.HTTPError as e:
        print(f"⚠️  [{router['hostname']}] PUT falló: {e}")
        return False

# ══════════════════════════════════════════════════════
# TAREAS DE CONFIGURACIÓN
# ══════════════════════════════════════════════════════
def task_hostname(router):
    payload = {"Cisco-IOS-XE-native:hostname": router["hostname"]}
    return put_config(router, f"{BASE_MODULE}/hostname", payload)

def task_interface_wan(router):
    path = f"{BASE_MODULE}/interface/GigabitEthernet=0%2F0"
    payload = {
        "Cisco-IOS-XE-native:GigabitEthernet": {
            "name": "0/0",
            "description": "WAN-PRIMARY-RESTCONF",
            "ip": {
                "address": {
                    "primary": {
                        "address": router["wan_ip"],
                        "mask": router["mask"]
                    }
                }
            }
        }
    }
    return put_config(router, path, payload)

def task_loopback_mgmt(router):
    path = f"{BASE_MODULE}/interface/Loopback=100"
    payload = {
        "Cisco-IOS-XE-native:Loopback": {
            "name": "100",
            "description": "MGMT-LOOPBACK",
            "ip": {
                "address": {
                    "primary": {
                        "address": f"1.1.1.{router['id']}",
                        "mask": "255.255.255.255"
                    }
                }
            }
        }
    }
    return put_config(router, path, payload)

def task_ospf(router):
    path = f"{BASE_MODULE}/router/ospf/ospf=1"
    payload = {
        "Cisco-IOS-XE-native:ospf": {
            "id": 1,
            "router-id": f"1.1.1.{router['id']}",
            "network": [
                {
                    "ip": router["wan_ip"],
                    "mask": "0.0.0.3",
                    "area": "0"
                },
                {
                    "ip": f"1.1.1.{router['id']}",
                    "mask": "0.0.0.0",
                    "area": "0"
                }
            ]
        }
    }
    return put_config(router, path, payload)

# ══════════════════════════════════════════════════════
# ORQUESTADOR PRINCIPAL
# ══════════════════════════════════════════════════════
def desplegar_router(router):
    print(f"
{'='*60}")
    print(f"🚀 Desplegando {router['hostname']} ({router['ip']})")
    print(f"{'='*60}")

    # Pre-check: ¿responde RESTCONF?
    health = get_config(router, "Cisco-IOS-XE-native:native/hostname")
    if not health:
        print(f"⛔ {router['hostname']} no responde. Saltando.")
        return False

    resultados = []
    resultados.append(("Hostname", task_hostname(router)))
    resultados.append(("WAN Gi0/0", task_interface_wan(router)))
    resultados.append(("Loopback100", task_loopback_mgmt(router)))
    resultados.append(("OSPFv2", task_ospf(router)))

    ok = sum(1 for _, r in resultados if r)
    total = len(resultados)
    print(f"
📊 Resumen: {ok}/{total} tareas exitosas")
    return ok == total

def main():
    print("CCNA Academy — Proyecto RESTCONF Despliegue Masivo
")
    exitosos = 0
    for r in ROUTERS:
        if desplegar_router(r):
            exitosos += 1

    print(f"
{'='*60}")
    print(f"🎯 DESPLIEGUE FINALIZADO: {exitosos}/{len(ROUTERS)} routers configurados")
    print(f"{'='*60}")

if __name__ == "__main__":
    main()

  2. Actualiza la lista ROUTERS con las IPs reales de tu laboratorio.
  3. Ejecuta: python proyecto_restconf_iosxe.py
  4. Observa cómo cada tarea PUT retorna 201 (creado) o 204 (sin contenido) y el resumen final.

Este script es 100% idempotente: puedes ejecutarlo 10 veces y el resultado final será el mismo. Si un router ya tenía la IP configurada, PUT la sobrescribe con el mismo valor. No hay duplicados.
