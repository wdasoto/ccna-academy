YANG Data Models

El lenguaje de modelado que gobierna la configuración de red moderna. Aprende cómo YANG (RFC 7950) define la estructura, tipos y constraints de los datos que NETCONF y RESTCONF transportan. Módulos, containers, lists, leafs y la jerarquía de modelos Cisco.

🌿 RFC 7950 🔗 NETCONF/RESTCONF ⏱ ~80 min lectura 🏗️ Modelado de datos 🎯 CCNA Domain 6.5

¿Qué es YANG y por qué cambia todo?

YANG (Yet Another Next Generation) es un lenguaje de modelado de datos estructurados, estandarizado en RFC 7950 (2016). A diferencia de JSON, XML o YAML —que son formatos de serialización— YANG describe qué datos existen, cómo se relacionan y qué reglas deben cumplir, antes de que se transporten.

🌿 Analogía: El Plan de Arquitectura

Si NETCONF/RESTCONF son los camiones que transportan la mercancía, YANG es el plano arquitectónico de la bodega: define dónde va cada caja (container), qué contienen (leafs), y las reglas de validación (constraints) antes de que el camión llegue.

Jerarquía Conceptual

📋
MODULE
Namespace único. Ej: ietf-interfaces
🗂️
CONTAINER
Agrupación lógica. No tiene valor propio.
📚
LIST
Array de nodos. Requiere key única.
🍃
LEAF
Valor simple único. Tipo escalar.
🌿
LEAF-LIST
Array de valores simples.
🔗
AUGMENT
Extiende modelos existentes.

YANG vs Schema (XML/JSON)

YANG es agnóstico al formato de codificación. El mismo modelo YANG puede generar XML (para NETCONF) o JSON (para RESTCONF). El modelo define la semántica; la codificación define la sintaxis de transporte.

Visión dual: Un modelo, dos transportes
# YANG define el modelo conceptual:
container interfaces {
  list interface {
    key "name";
    leaf name { type string; }
    leaf ip { type inet:ipv4-address; }
  }
}

# Se codifica como XML (NETCONF):
<interfaces>
  <interface>
    <name>Gi0/0</name>
    <ip>192.168.1.1</ip>
  </interface>
</interfaces>

# O como JSON (RESTCONF):
{
  "ietf-interfaces:interfaces": {
    "interface": [{"name": "Gi0/0", "ip": "192.168.1.1"}]
  }
}

Anatomía de un Módulo YANG

Todo modelo YANG empieza con un module que define el namespace y la versión. Es similar a un "import" de Python pero para la definición de datos de red.

Estructura base de un módulo YANG (simplificado)
module cisco-router-config {
  yang-version 1.1;              // RFC 7950
  namespace "http://cisco.com/ns/yang/router";
  prefix router;
  
  import ietf-inet-types {     // Tipos estándar de IPs
    prefix inet;
  }
  
  organization "Cisco Systems";
  description "Modelo de configuración de routers";
  
  revision "2024-01-15" {
    description "Versión inicial";
  }
  
  // Aquí van los containers, lists y leafs...
}

Namespaces y Prefixes

En YANG, el namespace (URI) identifica únicamente al módulo, evitando colisiones entre "interface" de Cisco vs "interface" de IETF. El prefix es el alias corto usado en las instancias.

Uso de prefijos en instancias XML
<!-- Ejemplo real de NETCONF con namespaces -->
<rpc message-id="101"
    xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <edit-config>
    <target><candidate/></target>
    <config>
      <!-- Aquí usamos el prefix definido en YANG -->
      <if:interfaces 
         xmlns:if="urn:ietf:params:xml:ns:yang:ietf-interfaces">
        <if:interface>
          <if:name>eth0</if:name>
        </if:interface>
      </if:interfaces>
    </config>
  </edit-config>
</rpc>
💡 Tip de Examen

Cuando veas xmlns:if="..." en XML de NETCONF, está usando el namespace definido en el módulo YANG. El prefijo if: indica que esos datos obedecen al modelo ietf-interfaces.

Los Cuatro Pilares de YANG

1. Container — El Organizador

Agrupa nodos relacionados lógicamente. No tiene valor propio, solo existe para contener otros nodos. Equivalente a una carpeta.

Definición de Container
container system {
  description "Información del sistema";
  
  leaf hostname {
    type string;
  }
  leaf domain {
    type string;
  }
}

2. List — La Base de Datos

Una lista es una secuencia de entradas del mismo tipo, cada una identificada por una key única. Es el equivalente a una tabla SQL o un array de objetos con ID.

List con clave compuesta
list interface {
  key "name";                    // Identificador único
  unique "ip-address";          // Constraint adicional
  
  leaf name {
    type string;
  }
  leaf ip-address {
    type inet:ipv4-address;   // Tipo especializado
  }
  leaf mtu {
    type uint16;
    default 1500;               // Valor por defecto
  }
}

3. Leaf — El Valor Escalar

Representa un valor simple (string, número, booleano, enumeración). No puede contener otros nodos. Es la "hoja" del árbol.

Tipos de Leaf avanzados
leaf admin-status {
  type enumeration {           // Enum tipo C
    enum up { value 1; }
    enum down { value 2; }
    enum testing { value 3; }
  }
  default up;
  config true;                 // Es configurable (vs state)
}

leaf prefix-length {
  type uint8 {
    range "0..32";           // Constraint de rango
  }
}

4. Leaf-List — El Array de Scalares

Un array ordenado de valores simples, todos del mismo tipo. Ejemplo: lista de DNS servers o NTP servers.

Leaf-list para servidores
leaf-list name-server {
  type inet:ip-address;
  ordered-by user;           // Orden significativo
  max-elements 3;            // Máximo 3 servidores
}

Visualización del Árbol YANG

module: cisco-router
+-- container system
+-- leaf hostname (string) +-- leaf domain (string)
+-- list interface [name]
+-- leaf name (string) +-- leaf ip (inet:ipv4) +-- leaf status (enum) +-- container statistics
+-- leaf in-octets (uint64) +-- leaf out-octets (uint64)

NETCONF: YANG codificado en XML

NETCONF (RFC 6241) es el protocolo de gestión de red que usa YANG como modelo conceptual pero transporta los datos en XML. Cada elemento XML corresponde a un nodo YANG.

Operación NETCONF <get> sobre modelo YANG
<!-- RPC de solicitud -->
<rpc message-id="101" 
     xmlns="urn:ietf:params:xml:ns:netconf:base:1.0">
  <get>
    <filter type="subtree">
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
  </get>
</rpc>

<!-- Respuesta (instancia de datos YANG) -->
<rpc-reply message-id="101">
  <data>
    <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces">
      <interface>
        <name>GigabitEthernet1</name>
        <type>iana-if-type:ethernetCsmacd</type>
        <enabled>true</enabled>
        <ipv4 xmlns="urn:ietf:params:xml:ns:yang:ietf-ip">
          <address>
            <ip>192.168.1.1</ip>
            <netmask>255.255.255.0</netmask>
          </address>
        </ipv4>
      </interface>
    </interfaces>
  </data>
</rpc-reply>
🔗 Mapping YANG → XML
  • container → Elemento XML anidado (<interfaces>)
  • list → Múltiples elementos del mismo nombre (<interface>...</interface>)
  • leaf → Elemento XML con valor de texto (<name>Gi1</name>)
  • leaf-list → Múltiples elementos con mismo nombre, valores diferentes

Capas de NETCONF

NETCONF opera en capas que separan el transporte (SSH), los mensajes (RPC), y el contenido (YANG):

  1. Transport Layer: SSH (puerto 830) o TLS
  2. Messages Layer: RPC request/reply
  3. Operations Layer: <get>, <edit-config>, <copy-config>, <delete-config>
  4. Content Layer: Datos YANG codificados en XML

RESTCONF: YANG codificado en JSON

RESTCONF (RFC 8040) expone los mismos modelos YANG pero a través de HTTP REST, usando JSON como codificación por defecto. Las URLs siguen la estructura jerárquica del modelo YANG.

URL RESTCONF mapeada a modelo YANG
# Estructura de URL RESTCONF:
/restconf/data/{modelo}:{ruta-al-nodo}

# Ejemplos reales:
/restconf/data/ietf-interfaces:interfaces
/restconf/data/ietf-interfaces:interfaces/interface=GigabitEthernet1
/restconf/data/ietf-interfaces:interfaces/interface=Gi1/ipv4
GET /restconf/data/ietf-interfaces:interfaces
{
  "ietf-interfaces:interfaces": {
    "interface": [
      {
        "name": "GigabitEthernet1",
        "type": "iana-if-type:ethernetCsmacd",
        "enabled": true,
        "ietf-ip:ipv4": {
          "address": [
            {
              "ip": "192.168.1.1",
              "netmask": "255.255.255.0"
            }
          ]
        }
      }
    ]
  }
}

Diferencias Clave NETCONF vs RESTCONF

Característica NETCONF RESTCONF
Transporte SSH (puerto 830) HTTP/HTTPS
Formato XML obligatorio JSON por defecto, XML opcional
Estilo RPC (Remote Procedure Call) REST (Recursos con URLs)
Operaciones <get>, <edit-config> GET, POST, PATCH, DELETE
Modelo Ambos usan YANG (RFC 6020/7950)
💡 Identificación rápida en el examen

Si ves <rpc> o <get-config> → NETCONF (XML). Si ves /restconf/data/... o JSON con namespaces como ietf-interfaces: → RESTCONF.

Configuración vía YANG (NETCONF)

Crear interfaz Loopback via NETCONF/YANG
<rpc message-id="101">
  <edit-config>
    <target><candidate/></target>
    <config>
      <native xmlns="http://cisco.com/ns/yang/Cisco-IOS-XE-native">
        <interface>
          <Loopback>
            <name>100</name>
            <ip>
              <address>
                <primary>
                  <address>1.1.1.1</address>
                  <mask>255.255.255.255</mask>
                </primary>
              </address>
            </ip>
          </Loopback>
        </interface>
      </native>
    </config>
  </edit-config>
</rpc>

<!-- Seguido de commit -->
<rpc>
  <commit/>
</rpc>

Datastores YANG (Candidate vs Running)

YANG define operaciones sobre diferentes almacenes de datos:

  • running: Configuración activa actual
  • candidate: Configuración temporal (editar sin afectar running)
  • startup: Configuración de arranque

El patrón es: editar candidate → validar → commit → copiar a running.

Manipular YANG desde Python

Aunque no parseamos YANG directamente (es complejo), sí interpretamos las instancias de datos que obedecen al modelo. Librerías como ncclient (NETCONF) y requests (RESTCONF) manejan el modelo por nosotros.

ncclient: NETCONF con Python
from ncclient import manager

# Conectar usando modelo YANG implícito
with manager.connect(
    host="192.168.1.1",
    port=830,
    username="admin",
    password="cisco",
    hostkey_verify=False
) as m:
    
    # Get-config solicita datos del modelo candidate
    filter = '''
    <filter>
      <interfaces xmlns="urn:ietf:params:xml:ns:yang:ietf-interfaces"/>
    </filter>
    '''
    
    config = m.get_config(source='running', filter=filter)
    print(config)
requests: RESTCONF con Python
import requests
import json

url = "https://192.168.1.1/restconf/data/ietf-interfaces:interfaces"
headers = {
    "Accept": "application/yang-data+json",
    "Content-Type": "application/yang-data+json"
}

resp = requests.get(url, headers=headers, auth=("admin", "cisco"), verify=False)
data = resp.json()  # Los datos siguen estructura YANG pero en JSON

# Navegar según modelo YANG (ietf-interfaces)
for intf in data["ietf-interfaces:interfaces"]["interface"]:
    print(f"Interface: {intf['name']}")

Resumen de Statements YANG

Statement Descripción Ejemplo
module Define el namespace del modelo module cisco-ios { ... }
container Agrupación lógica de nodos container interfaces { ... }
list Secuencia de entradas con key list interface { key "name"; ... }
leaf Valor escalar simple leaf mtu { type uint16; }
leaf-list Array de valores simples leaf-list dns { type string; }
typedef Definición de tipo personalizado typedef percent { type uint8; ... }
grouping Reutilizable block (like struct) grouping address { ... }
uses Instancia un grouping uses address;
augment Extiende otro modelo augment /if:interfaces { ... }
config true=configuración, false=estado config false;

Tipos de Datos Built-in

  • Enteros: int8, int16, int32, int64, uint8, uint16, uint32, uint64
  • Decimales: decimal64
  • Strings: string (con patterns length constraints)
  • Booleanos: boolean
  • Enumeraciones: enumeration
  • Bits: bits (flags)
  • Binarios: binary (base64)
  • Referencia: leafref (apunta a otro leaf)
  • Identidad: identity (para extensibilidad)
