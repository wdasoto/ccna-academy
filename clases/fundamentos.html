CCNA 200-301 Módulo 1 · Fundamentos 🌐 La Base de Todo

📖 ¿Qué es una red y por qué existe?

Una red de computadoras es un conjunto de dispositivos interconectados que comparten recursos e información. Desde dos computadoras con un cable hasta Internet — la red más grande jamás construida — el objetivo siempre es el mismo: permitir la comunicación entre dispositivos.

💾 Compartir recursos

Impresoras, archivos, bases de datos, conexión a Internet. Una red permite que múltiples usuarios accedan a los mismos recursos sin necesidad de copias físicas o conexiones dedicadas.

💬 Comunicación

Correo electrónico, videoconferencia, mensajería instantánea, VoIP. La red digitaliza y transporta la comunicación humana a velocidades cercanas a la luz.

💰 Reducción de costos

Una impresora compartida por 50 usuarios es más económica que 50 impresoras. Un enlace WAN centralizado es más barato que 50 conexiones individuales al ISP.

🔒 Centralización de seguridad

Gestionar la seguridad en un firewall central es más eficiente que proteger individualmente cada dispositivo. Copias de seguridad, control de acceso y monitorización centralizados.

🏛️ Modelo OSI — El marco de referencia universal

El modelo OSI (Open Systems Interconnection, ISO/IEC 7498-1, 1984) es el marco de referencia conceptual que divide la comunicación en red en 7 capas, cada una con responsabilidades claramente definidas. No es un protocolo — es un modelo conceptual que ayuda a entender cómo funciona la comunicación en redes y facilita el troubleshooting.

El 3-way handshake TCP — Establecimiento de conexión

📌 SYN → SYN-ACK → ACK

  1. SYN: el cliente envía un segmento con el flag SYN activo al servidor. Indica que quiere iniciar una conexión y anuncia su número de secuencia inicial (ISN).
  2. SYN-ACK: el servidor responde con SYN y ACK activos. Confirma el SYN del cliente (ACK = ISN_cliente + 1) y anuncia su propio ISN.
  3. ACK: el cliente confirma el SYN del servidor (ACK = ISN_servidor + 1). La conexión está establecida — comienza la transferencia de datos.

🌐 El modelo TCP/IP — El que realmente usa Internet

El modelo TCP/IP (también llamado modelo de Internet o DoD) es el modelo de referencia real que usa Internet y las redes modernas. Surgió antes que OSI — el ARPANET en los años 70 ya usaba TCP/IP. Mientras OSI es teórico y de referencia, TCP/IP es práctico y real.

🔑 Protocolos TCP/IP más importantes

Capa de Internet — IP y ICMP

📌 IP (Internet Protocol) — El protocolo de red

IP es el protocolo de enrutamiento que proporciona direccionamiento lógico y determina la ruta de los paquetes. Características clave:

  • Sin conexión: cada paquete es independiente, puede tomar rutas diferentes
  • Best-effort: no garantiza entrega, orden ni integridad (eso lo hace TCP en L4)
  • Fragmentación: puede dividir paquetes grandes en fragmentos más pequeños (en IPv4)
  • TTL (Time To Live): cada router decrementa el TTL en 1. Si llega a 0 → el paquete se descarta (previene bucles infinitos)

📌 ICMP — Internet Control Message Protocol

ICMP (IP protocolo número 1) es el protocolo de diagnóstico de red. Genera mensajes de error y de información:

Mensaje ICMPTipoCuándo se genera
Echo Request / Reply8 / 0Ping — verifica conectividad
Destination Unreachable3El destino no es alcanzable (red, host, puerto)
Time Exceeded11TTL expiró — usado por traceroute
Redirect5El router conoce una ruta mejor
Fragmentation Needed3/4El paquete es demasiado grande y el DF bit está activo

ARP — Address Resolution Protocol

📌 ARP — Resuelve IP → MAC en la misma subred

Cuando un dispositivo conoce la IP destino pero necesita la MAC para construir el frame Ethernet, usa ARP:

  1. El host envía un ARP Request en broadcast (FF:FF:FF:FF:FF:FF) preguntando "¿Quién tiene la IP X.X.X.X?"
  2. El host con esa IP responde con un ARP Reply en unicast con su MAC address
  3. El solicitante guarda la relación IP↔MAC en su tabla ARP (caché ARP) para futuros usos

La tabla ARP tiene un tiempo de vida (timeout). En Windows: arp -a. En Cisco IOS: show ip arp.

📦 Encapsulación — Cómo viajan los datos en la red

La encapsulación es el proceso de añadir cabeceras (y a veces tráilers) en cada capa del modelo al enviar datos. Al recibir, el proceso inverso (desencapsulación) quita las cabeceras capa por capa. Cada capa solo puede leer su propia cabecera — no ve las cabeceras de otras capas.

📌 El proceso completo de encapsulación — De arriba a abajo

Aplicación genera:          [ DATA ]
↓ Capa 4 Transport añade:   [ TCP/UDP Header | DATA ]          = SEGMENTO
↓ Capa 3 Network añade:     [ IP Header | TCP Header | DATA ]  = PAQUETE
↓ Capa 2 Data Link añade:   [ ETH Header | IP | TCP | DATA | ETH Trailer ] = FRAME
↓ Capa 1 Physical convierte: 010110110101001110100...             = BITS

🔄 ¿Qué cambia en cada salto de router?

⚠️ Pregunta crítica del CCNA: ¿Qué cambia al pasar por un router?

Al cruzar un router de una subred a otra, el paquete se desencapsula hasta la capa 3, el router toma la decisión de routing basándose en la IP destino, y luego re-encapsula con las MACs del siguiente salto.

Campo¿Cambia al cruzar un router?¿Por qué?
MAC origen✅ SÍ cambiaSe pone la MAC de la interfaz del router que envía
MAC destino✅ SÍ cambiaSe pone la MAC del siguiente salto (next-hop)
IP origen❌ NO cambia (sin NAT)La IP origen siempre es la del host que generó el paquete
IP destino❌ NO cambia (sin NAT)La IP destino siempre es la del host final
TTL / Hop Limit✅ SÍ cambiaEl router decrementa el TTL en 1 por cada salto
Puerto origen/destino❌ NO cambiaLos puertos TCP/UDP son de extremo a extremo

🔢 MTU y Fragmentación

📌 MTU — Maximum Transmission Unit

El MTU es el tamaño máximo de un frame que puede enviar una interfaz. El MTU estándar de Ethernet es 1500 bytes (de datos IP — la cabecera Ethernet va aparte). Si un paquete IP es mayor que el MTU del enlace:

  • IPv4: el router fragmenta el paquete en fragmentos más pequeños. El host destino los reensambla. Si el bit DF (Don't Fragment) está activo, el router descarta el paquete y envía ICMP "Fragmentation Needed".
  • IPv6: los routers NUNCA fragmentan. Solo el host origen puede fragmentar. Si el paquete es demasiado grande, el router envía ICMPv6 "Packet Too Big" y el host reduce su tamaño.

Path MTU Discovery (PMTUD): mecanismo que permite a los hosts descubrir el MTU mínimo en toda la ruta hacia el destino.

🔌 Ethernet — El estándar dominante de LAN

Ethernet (IEEE 802.3) es la tecnología de red de área local más usada del mundo. Desde su creación en los años 70 por Xerox, Intel y DEC hasta los 400 Gbps actuales, Ethernet ha evolucionado pero mantiene compatibilidad retroactiva. 🔢 MAC Address — La identidad de capa 2

Una MAC address (Media Access Control) es un identificador de 48 bits (6 bytes) grabado en la tarjeta de red (NIC). Escrita en notación hexadecimal: 00:1A:2B:3C:4D:5E o en formato Cisco: 001a.2b3c.4d5e.

Tipo de MACPatrónDescripciónEjemplo
UnicastBit LSB del primer byte = 0Identifica un dispositivo específico. El switch envía solo a ese puerto.00:1A:2B:3C:4D:5E
MulticastBit LSB del primer byte = 1Identifica un grupo de dispositivos que comparten un grupo multicast.01:00:5E:xx:xx:xx (IPv4 multicast)
BroadcastTodos los bits = 1Llega a todos los dispositivos del segmento. Todos los switches la inundan.FF:FF:FF:FF:FF:FF

📌 OUI — Organizationally Unique Identifier

Los primeros 3 bytes (24 bits) de la MAC = OUI, asignado por el IEEE al fabricante. Los últimos 3 bytes los asigna el fabricante al dispositivo. Cisco usa varios OUIs: 00:00:0C, 00:E0:1E, 00:17:59, entre muchos otros.

🔧 Medios físicos — El sustrato de la comunicación

El medio físico es el canal por el que viajan las señales de red. La elección del medio determina la velocidad, distancia, coste, inmunidad al ruido y las limitaciones de la red. Para el CCNA es fundamental conocer los tres grandes tipos.

🟤 Cable de cobre — UTP y STP

📌 UTP — Unshielded Twisted Pair (el más común)

Pares de cables trenzados sin blindaje. El trenzado reduce la interferencia electromagnética (EMI) y la diafonía (crosstalk) entre pares. Las categorías más relevantes para el CCNA:

Los switches y NICs modernos tienen Auto-MDIX (Automatic Medium-Dependent Interface Crossover) que detecta automáticamente si necesita cruzar los pares. Con Auto-MDIX, puedes usar cable directo o cruzado indistintamente — el equipo se adapta. En Cisco IOS se activa con mdix auto.

💥 Dominios de colisión y broadcast

Dominio de colisión vs Dominio de broadcast

📌 La diferencia crítica

  • Dominio de colisión: el área donde dos dispositivos pueden causar una colisión al transmitir simultáneamente. Un switch crea un dominio de colisión SEPARADO por cada puerto (full-duplex). Un hub tiene UN solo dominio de colisión para todos los puertos.
  • Dominio de broadcast: el área donde un broadcast (FF:FF:FF:FF:FF:FF) llega a todos los dispositivos. Un switch L2 tiene UN dominio de broadcast (todos los puertos, misma VLAN). Los routers separan dominios de broadcast — cada interfaz del router es un dominio de broadcast diferente.

⚠️ Los switches NO rompen dominios de broadcast — solo los routers lo hacen

Un switch con 24 puertos en la misma VLAN tiene 24 dominios de colisión (uno por puerto) pero SOLO 1 dominio de broadcast. Para separar dominios de broadcast necesitas VLANs (que crean dominios de broadcast lógicos) o un router que conecte las subredes.

🔄 CSMA/CD — Cómo Ethernet maneja las colisiones

📌 CSMA/CD — Carrier Sense Multiple Access with Collision Detection

En redes half-duplex (obsoleto hoy), CSMA/CD define el proceso para acceder al medio compartido:

  1. Carrier Sense: el dispositivo escucha el medio antes de transmitir. Si está ocupado, espera.
  2. Multiple Access: cualquier dispositivo puede transmitir cuando el medio está libre.
  3. Collision Detection: si dos dispositivos transmiten simultáneamente → colisión. Los dos detectan la colisión.
  4. Ambos emiten una jam signal para alertar a todos de la colisión.
  5. Algoritmo backoff exponencial: cada dispositivo espera un tiempo aleatorio antes de reintentar.

⚠️ CSMA/CD solo aplica en half-duplex. Con full-duplex (switches modernos) no hay colisiones — cada dispositivo tiene su canal de TX y RX dedicado. CSMA/CD está desactivado en full-duplex.

! Ver velocidad y duplex de una interfaz
SW1# show interfaces FastEthernet0/1
FastEthernet0/1 is up, line protocol is up (connected)
  Hardware is Fast Ethernet, address is 0013.1985.abcd
  MTU 1500 bytes, BW 100000 Kbit/sec, DLY 100 usec
  Full-duplex, 100Mb/s, media type is 10/100BaseTX
  ...
  Input errors: 0 CRC: 0 Frame: 0 Overrun: 0 Ignored: 0
! Full-duplex, 100Mb/s → configuración óptima
! Si aparece "Half-duplex" → posible problema de negociación

! Configurar duplex y velocidad manualmente
SW1(config)# interface FastEthernet0/1
SW1(config-if)# duplex full
SW1(config-if)# speed 100

! Mismatch auto↔manual causa duplex mismatch → muchos errores

🧪 Laboratorio Magistral — Verificación de red desde cero

📋 Topología: PC-A — SW1 — R1 — SW2 — PC-B (red simple con 2 subredes)

PC-A: 192.168.1.10/24 · GW: 192.168.1.1 · DNS: 8.8.8.8
R1 Gi0/0: 192.168.1.1/24 (inside) · R1 Gi0/1: 192.168.2.1/24 (outside)
PC-B: 192.168.2.10/24 · GW: 192.168.2.1

Fase 1 — Configuración básica del router

R1(config)# hostname R1
R1(config)# no ip domain-lookup
! Evitar que el router intente resolver comandos mal escritos como DNS
R1(config)# interface GigabitEthernet0/0
R1(config-if)# description TO-LAN-A
R1(config-if)# ip address 192.168.1.1 255.255.255.0
R1(config-if)# no shutdown
R1(config)# interface GigabitEthernet0/1
R1(config-if)# description TO-LAN-B
R1(config-if)# ip address 192.168.2.1 255.255.255.0
R1(config-if)# no shutdown
R1(config)# end
R1# copy running-config startup-config

Fase 2 — Verificación modelo OSI capa por capa

! ── Capa 1 — Física ──────────────────────────────────
R1# show interfaces GigabitEthernet0/0
GigabitEthernet0/0 is up, line protocol is up
! "up, up" = L1 OK (cable conectado) y L2 OK (protocolo activo)
! "up, down" = L1 OK pero L2 falló (problema de configuración)
! "down, down" = L1 falló (cable, velocidad, duplex)
! "administratively down" = interface apagada con shutdown

Fase 4 — Protocolo de verificación sistemático

📌 El método de troubleshooting capa por capa (bottom-up)

  1. L1 — Física: ¿El cable está conectado? ¿Las luces del puerto están activas? show interfaces → "up, up"
  2. L2 — Enlace: ¿La tabla ARP tiene la MAC? ¿Hay errores CRC? show ip arp, show mac address-table
  3. L3 — Red: ¿La tabla de routing tiene la ruta? ¿Las IPs están bien configuradas? show ip route, show ip interface brief
  4. L4 — Transporte: ¿El puerto de destino está abierto? ¿Hay ACL bloqueando? telnet IP PUERTO
  5. L7 — Aplicación: ¿El servicio está corriendo? ¿La configuración del servidor es correcta?

📋 Resumen Maestro — Fundamentos de Redes CCNA

✅ Los 30 puntos más críticos del examen

  1. LAN = red local (edificio). WAN = red de área amplia. MAN = red metropolitana.
  2. Topología estrella: todos conectados a un switch central. La más usada en LANs modernas.
  3. Modelo jerárquico Cisco: Core (velocidad máxima) + Distribution (políticas/routing) + Access (usuarios finales).
  4. OSI tiene 7 capas. TCP/IP tiene 4 capas. OSI es referencia teórica; TCP/IP es el usado en la realidad.
  5. Mnemónico OSI de abajo a arriba: Physical → Data Link → Network → Transport → Session → Presentation → Application.
  6. PDUs: bits (L1) → frame/trama (L2) → paquete (L3) → segmento/datagrama (L4) → data (L5-7).
  7. TCP: orientado a conexión, fiable, con 3-way handshake (SYN→SYN-ACK→ACK).
  8. UDP: sin conexión, no fiable, rápido. Para VoIP, streaming, DNS, DHCP.
  9. Puertos clave: HTTP=80, HTTPS=443, SSH=22, Telnet=23, DNS=53, DHCP=67/68, FTP=20/21, SMTP=25, SNMP=161/162.
  10. IP es sin conexión y best-effort. TTL previene bucles (se decrementa en 1 por router).
  11. ICMP: protocolo de diagnóstico. Ping = Echo Request/Reply (tipos 8/0). Traceroute usa Time Exceeded (tipo 11).
  12. ARP: resuelve IP → MAC en la misma subred. Broadcast request, unicast reply. Tabla ARP en routers: show ip arp.
  13. DNS: resuelve nombres → IPs. UDP 53 (consultas), TCP 53 (transferencias de zona).
  14. Registros DNS: A (IPv4), AAAA (IPv6), CNAME (alias), MX (correo), PTR (DNS inverso).
  15. Encapsulación: cada capa añade su cabecera al bajar. Desencapsulación: cada capa elimina la suya al subir.
  16. Al cruzar un router: las MACs cambian (src y dst) pero las IPs NO cambian (sin NAT). TTL se decrementa.
  17. MTU Ethernet = 1500 bytes. IPv4 puede fragmentar. IPv6 NO fragmenta en routers — usa PMTUD.
  18. Trama Ethernet: Preamble(7) + SFD(1) + DST MAC(6) + SRC MAC(6) + EtherType(2) + Data(46-1500) + FCS(4).
  19. FCS usa CRC-32 para detección de errores. El receptor recalcula y descarta el frame si difieren.
  20. MAC address = 48 bits. OUI (primeros 3 bytes) = fabricante. EtherType 0x0800=IPv4, 0x86DD=IPv6, 0x0806=ARP.
  21. Broadcast MAC = FF:FF:FF:FF:FF:FF. Multicast MAC empieza con 01:xx.
  22. Cable UTP Cat5e soporta GigabitEthernet hasta 100m. Cat6a soporta 10GBase-T hasta 100m.
  23. Cable directo (straight-through): PC-Switch, Router-Switch. Cable cruzado: Switch-Switch (legacy sin Auto-MDIX).
  24. Cable rollover (consola): PC COM/USB → Puerto consola Cisco (pines invertidos).
  25. Fibra multimodo: núcleo 50/62.5μm, corta distancia (~500m), menor costo. Monomodo: 9μm, larga distancia (km), mayor costo.
  26. Wi-Fi 6 = 802.11ax. Wi-Fi 5 = 802.11ac (5GHz). Wi-Fi 4 = 802.11n (dual-band). Wi-Fi hereda de IEEE 802.11.
  27. Hub = 1 dominio de colisión. Switch L2 = 1 dominio colisión por puerto, 1 dominio de broadcast por VLAN.
  28. Router separa dominios de broadcast. Cada interfaz = dominio de broadcast propio.
  29. CSMA/CD = half-duplex (obsoleto). Full-duplex = sin colisiones. Duplex mismatch → muchos errores CRC.
  30. Troubleshooting bottom-up: L1 (cable/luces) → L2 (ARP/tabla MAC) → L3 (routing/IP) → L4 (puerto/ACL) → L7 (servicio).

⚠️ Los 10 errores más comunes en el examen

  1. Confundir el modelo OSI (7 capas) con TCP/IP (4 capas) — saber qué protocolos van en cada capa de cada modelo
  2. Creer que los switches separan dominios de broadcast — solo los routers (o VLANs) lo hacen
  3. Confundir dirección IP (L3) con MAC (L2) — IP cambia por subred, MAC es del hardware
  4. Creer que las IPs cambian al pasar por un router (sin NAT) — solo cambian las MACs
  5. Confundir TCP (fiable, ordenado, 3-way handshake) con UDP (sin conexión, rápido)
  6. Pensar que ARP funciona entre subredes — ARP es solo dentro de la misma subred (L2)
  7. Confundir el cable directo con el cruzado — con Auto-MDIX ya no importa, pero lo pregunta el examen
  8. No recordar que el TTL se decrementa EN CADA ROUTER, no en cada switch
  9. Creer que CSMA/CD se usa en switches modernos — solo en half-duplex (obsoleto)
  10. Confundir el puerto del servidor DHCP (67) con el del cliente (68)
