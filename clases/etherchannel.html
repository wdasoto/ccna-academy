EtherChannel

Agregación de enlaces · LACP · PAgP · Port-Channel · Balanceo de Carga · Troubleshooting

🔗 ¿Qué es EtherChannel?

EtherChannel es una tecnología de agregación de enlaces que combina múltiples interfaces físicas Ethernet en un único enlace lógico llamado Port-Channel. El resultado es mayor ancho de banda, redundancia automática y balanceo de carga — todo visto como un solo enlace por STP.
✅
Sin EtherChannel, STP bloquearía los enlaces redundantes para evitar bucles, desperdiciando capacidad. EtherChannel resuelve esto presentando todos los enlaces físicos como uno solo ante STP.

🏗️ Interfaz Port-Channel

Al crear un EtherChannel se genera automáticamente una interfaz lógica Port-Channel. Esta es la interfaz que configuras para trunk, VLANs, etc. Las interfaces físicas pertenecientes al grupo deben tener configuración idéntica.
🎯
CCNA Examen: La configuración del Port-Channel (trunk, VLANs) debe ser idéntica a la de las interfaces físicas. Si hay diferencia, el EtherChannel no se forma o entra en estado de error (flag s = suspended).
⚠️
Requisito crítico: Todas las interfaces físicas del grupo deben tener: misma velocidad, mismo dúplex, mismo modo switchport (access o trunk) y mismas VLANs permitidas.

📡 Protocolos: LACP vs PAgP vs Static

Existen tres métodos para establecer un EtherChannel: dos con negociación automática y uno forzado estático. La elección depende del fabricante del equipo y del entorno de red.
ProtocoloEstándarCompatibilidadPDUsDescripción
LACP IEEE 802.3ad / 802.1AX Multi-vendor LACPDUs Link Aggregation Control Protocol — estándar abierto, recomendado siempre
PAgP Propietario Cisco Solo Cisco PAgP packets Port Aggregation Protocol — solo entre switches Cisco
Static ON N/A Universal Ninguna Sin negociación — forzado. Riesgo de bucles si hay errores de configuración
🎯
CCNA Examen: LACP = IEEE 802.3ad = multi-vendor. PAgP = Cisco propietario. No puedes mezclar LACP con PAgP en el mismo canal. Tampoco mezclar on con LACP o PAgP.

🔵 LACP — IEEE 802.3ad

LACP es el protocolo estándar abierto preferido en entornos modernos. Intercambia PDUs llamadas LACPDUs para negociar y mantener el estado del canal. Soporta hasta 16 interfaces en el grupo — 8 activas y 8 en hot-standby.
Característica LACPDetalle
Estándar IEEE802.3ad (también conocido como 802.1AX)
Interfaces activas máx.8 por canal (simultáneas)
Interfaces hot-standby8 adicionales (toman el relevo si falla una activa)
TemporizadorShort (1 segundo) o Long (30 segundos)
Prioridad de sistemaConfigurable — default: 32768

🟠 PAgP — Propietario Cisco

PAgP es el protocolo propietario de Cisco. Funciona solo entre switches Cisco. Aunque cumple la misma función que LACP, está siendo desplazado en entornos modernos por el estándar abierto.
✅
Recomendación: Usar siempre LACP sobre PAgP. Aunque estés en un entorno 100% Cisco, LACP es el estándar y garantiza mayor compatibilidad futura con otros fabricantes.

🔄 Modos de Negociación

Cada protocolo tiene sus propios modos. La combinación de modos en ambos extremos determina si el EtherChannel se forma o no. Este es el punto más evaluado en el examen CCNA.
ProtocoloModoComportamiento
LACP active Inicia la negociación LACP activamente — envía LACPDUs
LACP passive Espera que el otro extremo inicie — responde LACPDUs pero no las envía primero
PAgP desirable Inicia la negociación PAgP activamente — envía PAgP packets
PAgP auto Espera que el otro extremo inicie — responde PAgP packets pero no los envía primero
Static on Fuerza el EtherChannel sin ningún protocolo de negociación — no envía ni recibe PDUs

✅❌ Combinaciones Válidas e Inválidas

SW1SW2Protocolo¿Se forma?Motivo
active active LACP ✅ Sí Ambos inician — se negocian correctamente
active passive LACP ✅ Sí Uno inicia, el otro responde
passive passive LACP ❌ No Ambos esperan — nadie inicia la negociación
desirable desirable PAgP ✅ Sí Ambos inician — se negocian correctamente
desirable auto PAgP ✅ Sí Uno inicia, el otro responde
auto auto PAgP ❌ No Ambos esperan — nadie inicia la negociación
on on Static ✅ Sí Forzado sin negociación — ambos lados deben ser on
on active / passive — ❌ Error No se puede mezclar on con LACP ni con PAgP
🎯
Regla de oro CCNA: passive + passive = ❌ y auto + auto = ❌ porque ambos esperan que el otro inicie. Al menos un extremo debe ser active (LACP) o desirable (PAgP).

🏗️ Topología de Laboratorio

Dos switches (SW1 y SW2) conectados con 4 interfaces FastEthernet. Objetivo: crear un EtherChannel LACP con las 4 interfaces funcionando como trunk.
TOPOLOGÍA Fa0/1 ══════════════════ Fa0/1 Fa0/2 ══════════════════ Fa0/2 [SW1] Fa0/3 ══════════════════ Fa0/3 [SW2] Fa0/4 ══════════════════ Fa0/4 ↕ Port-Channel1 ↕ 4 × 100Mbps = 400 Mbps total

⌨️ Configuración LACP — SW1 (modo active)

SW1 — LACP ACTIVE SW1# configure terminal   ! Paso 1: Seleccionar el rango de interfaces SW1(config)# interface range FastEthernet0/1 - 4   ! Paso 2: Configurar encapsulación y modo trunk SW1(config-if-range)# switchport trunk encapsulation dot1q SW1(config-if-range)# switchport mode trunk   ! Paso 3: Crear EtherChannel con LACP modo active SW1(config-if-range)# channel-group 1 mode active Creating a port-channel interface Port-channel 1 SW1(config-if-range)# exit   ! Paso 4: Configurar la interfaz lógica Port-Channel SW1(config)# interface port-channel 1 SW1(config-if)# switchport trunk encapsulation dot1q SW1(config-if)# switchport mode trunk SW1(config-if)# end   SW1# copy running-config startup-config

⌨️ Configuración LACP — SW2 (modo passive)

SW2 — LACP PASSIVE SW2(config)# interface range FastEthernet0/1 - 4 SW2(config-if-range)# switchport trunk encapsulation dot1q SW2(config-if-range)# switchport mode trunk ! passive — espera que SW1 (active) inicie la negociación SW2(config-if-range)# channel-group 1 mode passive SW2(config-if-range)# exit SW2(config)# interface port-channel 1 SW2(config-if)# switchport trunk encapsulation dot1q SW2(config-if)# switchport mode trunk SW2(config-if)# end
✅
SW1 = active (inicia) + SW2 = passive (responde) = ✅ EtherChannel LACP formado correctamente. También funciona active + active.

⌨️ Configuración PAgP

PAgP DESIRABLE / AUTO ! SW1 — PAgP desirable (inicia la negociación) SW1(config)# interface range GigabitEthernet0/1 - 2 SW1(config-if-range)# switchport mode trunk SW1(config-if-range)# channel-group 1 mode desirable   ! SW2 — PAgP auto (responde la negociación) SW2(config)# interface range GigabitEthernet0/1 - 2 SW2(config-if-range)# switchport mode trunk SW2(config-if-range)# channel-group 1 mode auto

⚖️ Configurar Balanceo de Carga

El balanceo de carga define cómo se distribuye el tráfico entre los enlaces físicos. Se configura con un comando global en el switch y aplica a todos los EtherChannels.
LOAD BALANCE SW1(config)# port-channel load-balance src-dst-ip   ! Opciones disponibles: ! src-mac — Por MAC origen (default en L2) ! dst-mac — Por MAC destino ! src-dst-mac — Combinación de ambas MACs ! src-ip — Por IP origen ! dst-ip — Por IP destino ! src-dst-ip — Combinación IP origen + destino ← RECOMENDADO   ! Verificar método configurado: SW1# show etherchannel load-balance EtherChannel Load-Balancing Configuration: src-dst-ip
✅
src-dst-ip es el método más balanceado para tráfico general. Usa la combinación de IP origen e IP destino para distribuir uniformemente el tráfico entre todos los enlaces físicos del canal.

🚨 Errores Comunes — Troubleshooting

❌
Error 1: Configuración inconsistente entre interfaces
Una interfaz con VLAN 10 y otra con VLAN 20 en el mismo grupo EtherChannel.
Solución: verificar con show running-config que todas las interfaces tengan config idéntica.
❌
Error 2: passive + passive o auto + auto
Ambos extremos esperan que el otro inicie — el canal nunca se forma.
Solución: cambiar al menos un extremo a active (LACP) o desirable (PAgP).
❌
Error 3: Mezclar on con LACP/PAgP
Un extremo en on y el otro en active o desirable.
Solución: si usas on, ambos extremos deben estar en on.
❌
Error 4: Velocidades o tipos de interfaz diferentes
Mezclar FastEthernet con GigabitEthernet en el mismo grupo.
Solución: verificar con show interfaces que todas las físicas sean del mismo tipo y velocidad.
🎯
CCNA Examen evalúa: (1) diferencia LACP vs PAgP, (2) combinaciones de modos válidas, (3) requisitos de configuración idéntica en todas las interfaces, (4) interpretación de flags en show etherchannel summary.
