ACLs

Listas de Control de Acceso · Standard · Extended · Named · Aplicación · Troubleshooting

🔒 ¿Qué es una ACL?

Una ACL (Access Control List) es una lista ordenada de instrucciones que el router evalúa para permitir (permit) o denegar (deny) tráfico. Se aplica en una interfaz del router en dirección inbound (entrante) o outbound (saliente). Las ACLs son la herramienta de seguridad más fundamental en redes Cisco.
✅
Doble función: Las ACLs no solo filtran tráfico — también se usan para identificar tráfico en NAT, QoS, VPNs, route maps y distribución de rutas.
🎯
CCNA Examen: Solo se puede aplicar una ACL por interfaz, por dirección y por protocolo. Es decir, máximo una ACL IPv4 inbound y una ACL IPv4 outbound por interfaz.

⚙️ Funcionamiento — Reglas Fundamentales

El router evalúa el tráfico contra las ACEs (Access Control Entries) de arriba hacia abajo, en orden. Al encontrar la primera coincidencia (match), ejecuta la acción y detiene la evaluación. Si ninguna ACE coincide, se aplica el implicit deny all.
FLUJO DE EVALUACIÓN Paquete llega → ¿Coincide ACE 1? → Sí → PERMIT/DENY → FIN ↓ No ¿Coincide ACE 2? → Sí → PERMIT/DENY → FIN ↓ No ¿Coincide ACE 3? → Sí → PERMIT/DENY → FIN ↓ No coincidió ninguna IMPLICIT DENY ALL ← ¡Bloquea TODO lo que no fue permitido!
⚠️
Implicit Deny All: Toda ACL tiene un deny any any implícito al final que no se ve con show running-config. Si no hay ningún permit, todo el tráfico será bloqueado.
ReglaDescripción
Primera coincidenciaEl router para en la primera ACE que coincide — el orden importa muchísimo
Implicit denySi ninguna ACE coincide → el paquete es descartado silenciosamente
Una por direcciónMáximo una ACL inbound y una outbound por interfaz por protocolo
No se puede insertarLas ACLs numeradas no permiten insertar ACEs — hay que reescribir toda la lista
Named sí permite editarLas ACLs con nombre permiten eliminar o insertar ACEs individuales

📍 Dirección de Aplicación — Inbound vs Outbound

DirecciónCuándo se evalúaVentajaRecomendación
Inbound (in) Antes de que el paquete sea procesado/ruteado Más eficiente — descarta antes de consumir CPU para rutear ACLs Extended
Outbound (out) Después del proceso de routing, antes de salir por la interfaz Filtra tráfico de múltiples interfaces entrantes en un punto ACLs Standard
DIRECCIÓN DE APLICACIÓN [ROUTER] / \ PC → Gi0/0 (inbound ACL) Gi0/1 (outbound ACL) → Internet   Inbound: ACL evalúa ANTES del routing → ahorra procesamiento Outbound: ACL evalúa DESPUÉS del routing → más flexible pero menos eficiente

📊 Tipos de ACL

TipoRango numéricoFiltra porDónde aplicar
Standard 1–99 · 1300–1999 Solo IP origen Lo más cerca del destino
Extended 100–199 · 2000–2699 IP origen, IP destino, protocolo, puerto Lo más cerca del origen
Named Standard Nombre alfanumérico Solo IP origen Lo más cerca del destino
Named Extended Nombre alfanumérico IP origen, destino, protocolo, puerto Lo más cerca del origen
🎯
Regla de ubicación CCNA: Standard ACL → cerca del destino (solo filtra origen — si se pone cerca del origen podría bloquear demasiado tráfico). Extended ACL → cerca del origen (descarta antes de que el tráfico recorra la red innecesariamente).

📋 ACL Standard — Solo IP Origen

La ACL Standard filtra tráfico basándose únicamente en la IP de origen. No puede distinguir el destino, el protocolo ni el puerto. Es simple pero efectiva para control de acceso básico. Números del 1 al 99 y del 1300 al 1999.
Sintaxis:
access-list [número] {permit|deny} [IP-origen] [wildcard]
access-list [número] {permit|deny} host [IP-exacta]
access-list [número] {permit|deny} any
KeywordEquivalenteSignificado
host 10.1.1.110.1.1.1 0.0.0.0Solo ese host exacto /32
any0.0.0.0 255.255.255.255Cualquier dirección IP
10.1.1.0 0.0.0.255Subred /24Toda la red 10.1.1.0/24

⌨️ Ejemplos ACL Standard

EJEMPLO 1 — DENEGAR UN HOST ! Denegar solo el host 192.168.1.10, permitir todo lo demás Router(config)# access-list 10 deny host 192.168.1.10 Router(config)# access-list 10 permit any   ! Sin el "permit any" → implicit deny bloquearía TODO
EJEMPLO 2 — PERMITIR UNA RED ! Solo la red 10.1.1.0/24 puede pasar — todo lo demás denegado Router(config)# access-list 10 permit 10.1.1.0 0.0.0.255 ! (implicit deny any al final — no hace falta escribirlo)
EJEMPLO 3 — APLICAR A INTERFAZ Router(config)# interface GigabitEthernet0/1 ! Aplicar outbound (cerca del destino — regla Standard) Router(config-if)# ip access-group 10 out   ! Verificar Router# show ip interface GigabitEthernet0/1 Router# show access-lists
⚠️
Error clásico: Olvidar el permit any al final de una ACL que tiene deny explícitos. El implicit deny al final bloqueará TODO lo que no fue explícitamente permitido.

🔬 ACL Extended — Control Total

La ACL Extended puede filtrar por IP origen, IP destino, protocolo (IP, TCP, UDP, ICMP) y número de puerto. Es mucho más granular y poderosa que la Standard. Números del 100 al 199 y del 2000 al 2699. Se coloca cerca del origen.
Sintaxis:
access-list [número] {permit|deny} [protocolo] [src] [wildcard-src] [dst] [wildcard-dst] [operador puerto]
ParámetroOpcionesEjemplo
Protocoloip · tcp · udp · icmp · ospf · eigrptcp
IP origenhost X · any · red + wildcardhost 192.168.1.10
IP destinohost X · any · red + wildcardhost 10.0.0.1
Operador puertoeq · neq · lt · gt · rangeeq 80

PuertoProtocoloServicio
20 / 21TCPFTP (datos / control)
22TCPSSH

53UDP/TCPDNS

🎯
CCNA Examen: Memoriza: SSH=22, Telnet=23, DNS=53, HTTP=80, HTTPS=443, FTP=21. También puedes usar nombres: eq www en vez de eq 80, eq ssh en vez de eq 22.

⌨️ Ejemplos ACL Extended

EJEMPLO 1 — BLOQUEAR HTTP ! Denegar tráfico HTTP de la red 192.168.1.0/24 hacia cualquier destino Router(config)# access-list 100 deny tcp 192.168.1.0 0.0.0.255 any eq 80 Router(config)# access-list 100 permit ip any any
EJEMPLO 2 — PERMITIR SOLO SSH Y HTTPS ! Solo tráfico SSH y HTTPS hacia el servidor 10.0.0.100 Router(config)# access-list 100 permit tcp any host 10.0.0.100 eq 22 Router(config)# access-list 100 permit tcp any host 10.0.0.100 eq 443 Router(config)# access-list 100 deny ip any host 10.0.0.100 Router(config)# access-list 100 permit ip any any
EJEMPLO 3 — BLOQUEAR ICMP (PING) ! Denegar ping desde la LAN hacia Internet, permitir el resto Router(config)# access-list 101 deny icmp 192.168.1.0 0.0.0.255 any Router(config)# access-list 101 permit ip any any   ! Aplicar inbound en Gi0/0 (cerca del origen — regla Extended) Router(config)# interface GigabitEthernet0/0 Router(config-if)# ip access-group 101 in
EJEMPLO 4 — RANGO DE PUERTOS ! Permitir puertos bien conocidos (1024 y menores) de TCP Router(config)# access-list 100 permit tcp any any range 1 1023   ! Permitir solo FTP (puertos 20 y 21) Router(config)# access-list 100 permit tcp any any range 20 21

🏷️ Named ACL — La Forma Moderna

Las Named ACLs usan un nombre alfanumérico en lugar de un número. Ofrecen dos ventajas críticas sobre las numeradas: se pueden eliminar ACEs individuales con no y se pueden insertar ACEs en posiciones específicas con números de secuencia. Son la forma recomendada en entornos modernos.
✅
Con Named ACLs puedes editar sin reescribir toda la lista. Esto es fundamental en producción donde borrar y reescribir una ACL puede causar un corte de servicio temporal.

⌨️ Sintaxis Named ACL

NAMED STANDARD ACL ! Crear ACL standard con nombre Router(config)# ip access-list standard BLOCK-PC1 Router(config-std-nacl)# deny host 192.168.1.10 Router(config-std-nacl)# permit any Router(config-std-nacl)# exit   ! Aplicar en interfaz Router(config)# interface GigabitEthernet0/1 Router(config-if)# ip access-group BLOCK-PC1 out
NAMED EXTENDED ACL Router(config)# ip access-list extended FILTER-WEB Router(config-ext-nacl)# permit tcp 192.168.1.0 0.0.0.255 any eq 80 Router(config-ext-nacl)# permit tcp 192.168.1.0 0.0.0.255 any eq 443 Router(config-ext-nacl)# deny ip 192.168.1.0 0.0.0.255 any Router(config-ext-nacl)# permit ip any any

✏️ Editar ACEs con Números de Secuencia

Las Named ACLs asignan números de secuencia automáticamente (10, 20, 30...) a cada ACE. Puedes insertar una nueva ACE entre dos existentes o eliminar una ACE específica sin tocar el resto.
INSERTAR Y ELIMINAR ACEs ! Ver la ACL con sus números de secuencia Router# show ip access-lists FILTER-WEB Extended IP access list FILTER-WEB 10 permit tcp 192.168.1.0 0.0.0.255 any eq www 20 permit tcp 192.168.1.0 0.0.0.255 any eq 443 30 deny ip 192.168.1.0 0.0.0.255 any 40 permit ip any any   ! Insertar nueva ACE entre la 10 y la 20 (secuencia 15) Router(config)# ip access-list extended FILTER-WEB Router(config-ext-nacl)# 15 permit tcp 192.168.1.0 0.0.0.255 any eq 8080   ! Eliminar solo la ACE de secuencia 30 Router(config-ext-nacl)# no 30   ! Eliminar toda la ACL Router(config)# no ip access-list extended FILTER-WEB
🎯
CCNA Examen: Para eliminar una entrada de una ACL numerada completa hay que eliminar toda la lista y recriarla. Para las Named ACLs se puede eliminar una ACE específica con no [número-secuencia].

🏗️ Topología de Laboratorio

Red corporativa con zona LAN, servidores y acceso a Internet. Objetivo: implementar políticas de seguridad con ACLs para proteger los servidores y controlar el acceso a Internet.
TOPOLOGÍA LAN-Usuarios: 192.168.1.0/24 ← PC1, PC2, PC3 | [R1] Gi0/0: 192.168.1.1/24 ← LAN usuarios Gi0/1: 10.0.0.1/30 ← enlace a DMZ Gi0/2: 203.0.113.1/30 ← enlace ISP (Internet) | DMZ: 10.0.0.0/30 [Servidor Web] 10.0.0.2 ← HTTP/HTTPS [Servidor SSH] 10.0.0.3 ← SSH admin

⌨️ Política 1 — Proteger Servidor Web

Solo permitir tráfico HTTP/HTTPS hacia el servidor web. Bloquear todo lo demás. Aplicar cerca del origen (Extended ACL en Gi0/0 inbound).
ACL EXTENDED — PROTEGER SERVIDOR WEB R1(config)# ip access-list extended PROTECT-WEB ! Permitir HTTP y HTTPS hacia el servidor web R1(config-ext-nacl)# permit tcp 192.168.1.0 0.0.0.255 host 10.0.0.2 eq 80 R1(config-ext-nacl)# permit tcp 192.168.1.0 0.0.0.255 host 10.0.0.2 eq 443 ! Permitir SSH solo desde administrador R1(config-ext-nacl)# permit tcp host 192.168.1.100 host 10.0.0.3 eq 22 ! Bloquear cualquier otro acceso a la DMZ R1(config-ext-nacl)# deny ip any 10.0.0.0 0.0.0.3 ! Permitir el resto del tráfico (hacia Internet) R1(config-ext-nacl)# permit ip any any R1(config-ext-nacl)# exit   ! Aplicar en Gi0/0 inbound (cerca del origen) R1(config)# interface GigabitEthernet0/0 R1(config-if)# ip access-group PROTECT-WEB in

⌨️ Política 2 — Bloquear Acceso a Internet a un Departamento

El departamento de contabilidad (192.168.1.128/26) no debe tener acceso a Internet. El resto de la LAN sí puede navegar. Usar Standard ACL cerca del destino (Gi0/2 outbound).
ACL STANDARD — BLOQUEAR DEPARTAMENTO R1(config)# access-list 10 deny 192.168.1.128 0.0.0.63 ! /26 R1(config)# access-list 10 permit any   ! Aplicar outbound en Gi0/2 (interfaz hacia ISP — cerca del destino) R1(config)# interface GigabitEthernet0/2 R1(config-if)# ip access-group 10 out

⌨️ ACL para proteger el Router (VTY)

Caso especial: controlar quién puede acceder al router vía Telnet o SSH. Se aplica con access-class en las líneas VTY, no con ip access-group.
PROTEGER ACCESO VTY (TELNET/SSH) ! Solo el administrador 192.168.1.100 puede conectarse por SSH R1(config)# access-list 5 permit host 192.168.1.100   ! Aplicar en líneas VTY con access-class (no access-group) R1(config)# line vty 0 4 R1(config-line)# access-class 5 in R1(config-line)# transport input ssh
🎯
CCNA Examen: Para interfaces físicas se usa ip access-group. Para líneas VTY (Telnet/SSH) se usa access-class. Esta distinción aparece frecuentemente en el examen.

🚨 Errores Comunes en ACLs

❌
Error 1: Olvidar permit any al final
Una ACL con solo deny explícitos + implicit deny bloquea absolutamente todo.
Solución: siempre agregar permit ip any any al final si quieres permitir el tráfico no especificado.
❌
Error 2: Orden incorrecto de ACEs
Poner un permit any antes de un deny específico — el deny nunca se evalúa.
Regla: siempre poner las ACEs más específicas primero y las más generales al final.
❌
Error 3: Standard ACL aplicada cerca del origen
Una Standard ACL solo filtra por IP origen — si se pone cerca del origen puede bloquear tráfico válido hacia múltiples destinos.
Solución: Standard ACL → cerca del destino. Extended ACL → cerca del origen.
❌
Error 4: Wildcard mask incorrecta
Usar máscara de subred en vez de wildcard mask: deny 192.168.1.0 255.255.255.0 es incorrecto.
Solución: la wildcard es la inversa: deny 192.168.1.0 0.0.0.255
❌
Error 5: Dirección aplicada al revés
Aplicar outbound cuando debería ser inbound o viceversa.
Verificar con show ip interface [interfaz] qué ACL está en qué dirección.
✅
Método sistemático: 1) show access-lists → ver contadores 2) show ip interface → verificar dónde está aplicada 3) Revisar orden de ACEs 4) Verificar wildcards 5) Probar con ping/traceroute.
