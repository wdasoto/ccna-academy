MERAKI DASHBOARD — Cloud Networking API

La API REST que gobierna la infraestructura cloud de Cisco. Gestiona MX, MS, MR, MV y MT desde un único endpoint: organizaciones, redes, VLANs, clientes wireless, Auto VPN, webhooks y assurance. Autenticación por API Key, rate limiting inteligente y SDK oficial para Python.

☁️ Cloud-Managed 🎯 CCNA Domain 6.5 ⏱ ~90 min lectura 💻 api.meraki.com 🐍 SDK + requests

¿Qué es Meraki y por qué es "cloud-first"?

Cisco Meraki es la familia de productos de red gestionados 100% desde la nube. A diferencia de los controladores on-premise (DNA Center, WLC tradicional), Meraki elimina la necesidad de controladores locales: cada dispositivo se conecta directamente a los servidores cloud de Cisco vía HTTPS, descarga su configuración automáticamente y reporta telemetría en tiempo real.

La Dashboard API v1 expone esta plataforma como una API REST completa. Con una simple API Key, puedes crear redes, configurar VLANs, desplegar SSIDs, analizar clientes y monitorizar la salud de miles de sitios desde un script Python ejecutándose en cualquier parte del mundo.

💡 Meraki vs. DNA Center — Cuándo usar cada uno
  • Meraki: Ideal para sucursales distribuidas, retail, educación y healthcare que necesitan despliegue rápido sin ingenieros locales. Gestión 100% cloud. Zero-touch provisioning.
  • DNA Center: Ideal para campus empresariales grandes con requerimientos de SD-Access, microsegmentación y control on-premise. Integración profunda con ISE y WLAN controllers físicos.
  • La realidad: Muchas empresas híbridas usan ambos. Meraki para el edge (sucursales), DNA Center para el core/campus.

Autenticación: API Key y OAuth 2.0

Meraki soporta dos modelos de autenticación. El examen CCNA se enfoca principalmente en la API Key, pero un ingeniero de automatización completo debe entender ambos flujos.

🔑

API Key

Una clave secreta larga asociada a tu cuenta de administrador. Se envía en el header X-Cisco-Meraki-API-Key en cada petición. Ideal para scripts internos y automatización propia.

🌐

OAuth 2.0

Flujo authorization_code para aplicaciones de terceros (Meraki Marketplace). El usuario autoriza tu app sin compartir su contraseña ni API Key. Usa client_id, client_secret y scopes.

API Key en Python requests

auth_meraki.py — Patrón base con requests
import requests
import os

# Nunca hardcodees la key. Usa variables de entorno.
API_KEY = os.getenv("MERAKI_API_KEY", "tu-api-key-aqui")
BASE_URL = "https://api.meraki.com/api/v1"

headers = {
    "X-Cisco-Meraki-API-Key": API_KEY,
    "Content-Type": "application/json",
    "Accept": "application/json"
}

def get_organizations():
    r = requests.get(f"{BASE_URL}/organizations", headers=headers)
    r.raise_for_status()
    return r.json()

orgs = get_organizations()
for org in orgs:
    print(f"🏢 {org['name']} — ID: {org['id']}")

OAuth 2.0 para aplicaciones de terceros

Si desarrollas una app para el Meraki Marketplace, no puedes pedir la API Key del cliente. En su lugar, usas OAuth 2.0 con el flujo de código de autorización:

  1. Registras tu app en Meraki Developer Hub. Obtienes client_id y client_secret.
  2. El usuario accede a https://api.meraki.com/oauth/authorize?client_id=XXX&redirect_uri=YYY&scope=read-networks y autoriza.
  3. Meraki redirige a tu redirect_uri con un code temporal.
  4. Tu backend intercambia ese code por un access_token vía POST a /oauth/token.
  5. Usas el access_token en el header Authorization: Bearer <token>.
oauth_callback.py — Intercambio de código por token
def exchange_code_for_token(code, client_id, client_secret, redirect_uri):
    url = "https://api.meraki.com/oauth/token"
    payload = {
        "grant_type": "authorization_code",
        "code": code,
        "client_id": client_id,
        "client_secret": client_secret,
        "redirect_uri": redirect_uri
    }
    r = requests.post(url, data=payload)
    data = r.json()
    return data["access_token"], data.get("refresh_token")

Rate Limiting: 5 peticiones por segundo

Meraki impone un límite estricto de 5 llamadas por segundo por organización. Si lo excedes, recibes HTTP 429 Too Many Requests. La respuesta incluye el header Retry-After con los segundos a esperar.

rate_limit_handler.py — Backoff exponencial y throttling
import time

class MerakiRateLimiter:
    def __init__(self, max_rate=5):
        self.min_interval = 1.0 / max_rate
        self.last_call = 0

    def wait(self):
        elapsed = time.time() - self.last_call
        if elapsed < self.min_interval:
            time.sleep(self.min_interval - elapsed)
        self.last_call = time.time()

def meraki_request(limiter, method, url, headers, **kwargs):
    limiter.wait()
    r = requests.request(method, url, headers=headers, **kwargs)
    if r.status_code == 429:
        retry = int(r.headers.get("Retry-After", 1))
        print(f"⏳ Rate limit. Esperando {retry}s...")
        time.sleep(retry)
        return meraki_request(limiter, method, url, headers, **kwargs)
    r.raise_for_status()
    return r
⚡ Regla de oro de producción

Si gestionas múltiples organizaciones, el rate limit es por organización, no global. Sin embargo, un mismo script que itere 100 redes en una org debe respetar los 5 req/s. El SDK oficial maneja esto automáticamente.

Jerarquía: Organizations y Networks

En Meraki, todo se organiza en una estructura de dos niveles:

  • Organization: El contenedor de nivel más alto. Representa a la empresa o división. Contiene múltiples networks, admins, configuraciones de branding y licencias.
  • Network: Una red lógica dentro de una organización. Puede ser de tipo wireless, appliance (MX), switch (MS), camera (MV), sensor (MT), o combined (varios tipos juntos).
GET/organizations
Lista todas las organizaciones donde tu API Key tiene permisos.
GET/organizations/{orgId}/networks
Lista las networks dentro de una organización específica.
POST/organizations/{orgId}/networks
Crea una nueva network. Requiere name, type y opcionalmente tags y timeZone.

Config Templates (Plantillas de configuración)

Meraki permite crear templates para aplicar configuraciones idénticas a múltiples networks. Cambias el template y todas las networks vinculadas heredan el cambio automáticamente. Esto es crítico para operaciones de retail o franquicias con cientos de sucursales idénticas.

GET/organizations/{orgId}/configTemplates
Lista las plantillas de configuración disponibles en la organización.
POST/organizations/{orgId}/configTemplates/{templateId}/bind
Vincula una network existente a un template. La network hereda toda la configuración del template.
💡 Tip para escala masiva

Para desplegar 50 tiendas idénticas, la arquitectura recomendada es: 1) Crear un template con VLANs, SSIDs y firewall rules base; 2) Crear cada network como combined; 3) Vincular al template con autoBind: true; 4) Hacer override solo en parámetros locales como direcciones IP estáticas.

Inventario y gestión de dispositivos

Los dispositivos Meraki se identifican por su número de serie (ej. Q2XX-XXXX-XXXX). Una vez que reclamas un dispositivo en tu organización (claim), puedes gestionarlo completamente por API.

GET/organizations/{orgId}/devices
Inventario completo de la organización con modelo, serial, MAC, red asignada y estado.
GET/networks/{netId}/devices
Dispositivos dentro de una network específica.
GET/networks/{netId}/devices/{serial}
Detalle de un dispositivo: nombre, dirección, lat/lng, notas, tags, firmware.
POST/networks/{netId}/devices/claim
Reclama dispositivos nuevos (por serial o order number) y los añade a la network.

Inventario organizacional con paginación

inventory_devices.py — Listar todo con paginación automática
def get_all_org_devices(org_id):
    """Obtiene todos los dispositivos respetando paginación (perPage)."""
    devices = []
    url = f"{BASE_URL}/organizations/{org_id}/devices"
    params = {"perPage": 1000}

    while url:
        r = requests.get(url, headers=headers, params=params)
        r.raise_for_status()
        data = r.json()
        devices.extend(data)

        # Meraki usa Link header para paginación (next)
        link_header = r.headers.get("Link", "")
        url = None
        if 'rel="next"' in link_header:
            import re
            match = re.search('<(.*?)>; rel="next"', link_header)
            if match:
                url = match.group(1)
                params = None

    return devices

⚡ Claim vs. Assign

Claim mueve el dispositivo del inventario "sin dueño" de Cisco a tu organización. Assign (o mover entre networks) lo ubica dentro de una network específica para que herede configuración. Si haces claim pero no lo añades a una network, el dispositivo aparecerá en el inventario pero no tendrá configuración aplicada.

VLANs, DHCP, Firewall y Auto VPN

Los dispositivos MX (appliance) actúan como L3 gateway, firewall y concentrador VPN. La API permite gestionar VLANs, rutas estáticas, DHCP scopes, NAT, port forwarding y reglas de firewall de capa 3 y 7.

GET/networks/{netId}/appliance/vlans
Lista VLANs configuradas en el MX de la network.

PUT/networks/{netId}/appliance/vlans/{vlanId}
Actualiza parámetros de una VLAN existente: DNS, DHCP, reserved IP ranges.

Auto VPN y SD-WAN

El Auto VPN de Meraki permite levantar túneles IPsec site-to-site entre MXs automáticamente con solo un clic. La API te permite configurar los hubs, los spokes, las subnets participantes y las políticas SD-WAN para balanceo de uplinks.

GET/networks/{netId}/appliance/vpn/siteToSiteVpn
Configuración actual de Auto VPN: hubs, mode (spoke/hub), subnets locales.
PUT/networks/{netId}/appliance/vpn/siteToSiteVpn
Actualiza el rol (hub/spoke) y los hubs a los que conectarse.
💡 Trampa frecuente del examen

El examen CCNA puede confundirte entre SD-WAN y Auto VPN. Recuerda: Meraki MX soporta ambos. Auto VPN es la tecnología de túneles automáticos; SD-WAN es la política de selección de path (VPN vs. Internet directo, balanceo por rendimiento). La API los gestiona por separado.

def find_client_by_mac(net_id, mac):

Monitoreo, logs y alertas vía Webhook

Meraki ofrece dos mecanismos de monitorización: pull (consultas periódicas a la API) y push (webhooks que envían eventos en tiempo real a tu servidor). La combinación de ambos es el estándar en arquitecturas cloud-modernas.

📊

Organization-wide Logs

Eventos de cambio de configuración, conexión/disconexión de VPN, cambios de rol en admins. Audit trail completo.

🔔

Alerting Channels

Email, SMS, webhook, Push API. Las alertas cubren caída de uplink, VPN down, rogue AP, cámara offline, sensor threshold.

🌡️

Sensor Telemetry

Los MT reportan temperatura, humedad, puerta abierta/cerrada, agua detectada. Datos históricos vía API.

Endpoints de monitoreo

GET/organizations/{orgId}/configurationChanges
Audit log de cambios en la configuración de la organización. Útil para compliance y troubleshooting.
GET/networks/{netId}/events/eventTypes
Tipos de eventos disponibles en la network (client connection, dhcp, vpn, etc.).

def create_http_server(net_id, name, url, secret):
    """Registra un webhook HTTP server en la network."""
    endpoint = f"{BASE_URL}/networks/{net_id}/webhooks/httpServers"
    payload = {

                "type": "uplinkStatusChange",
                "enabled": True

Receptor de alertas con Flask

webhook_receiver.py — Servidor de alertas Meraki
from flask import Flask, request
import hmac, hashlib

app = Flask("__main__")
SHARED_SECRET = "mi-secreto-compartido"

@app.route("/webhook/meraki", methods=["POST"])
def meraki_alert():
    payload = request.get_data()
    sig = request.headers.get("X-Meraki-Signature", "")

    # Validar firma HMAC-SHA256 para integridad
    expected = hmac.new(
        SHARED_SECRET.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()

    if not hmac.compare_digest(sig, expected):
        return "Unauthorized", 401

    data = request.get_json()
    print(f"🚨 ALERTA: {data.get('alertType')}")
    print(f"   Dispositivo: {data.get('deviceName')}")
    print(f"   Timestamp: {data.get('occurredAt')}")

    # Acciones automatizadas: abrir ticket, enviar SMS, escalar
    return "OK", 200

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)
⚡ Validación de webhooks es obligatoria

Meraki envía el header X-Meraki-Signature con un HMAC-SHA256 del payload usando tu sharedSecret. Si no validas esto, cualquier actor puede enviar POSTs falsos a tu endpoint y generar falsas alarmas o bloqueos. Nunca confíes en la IP de origen como único mecanismo de seguridad.

SDK oficial vs. requests puro vs. Postman

Meraki ofrece un SDK Python oficial (meraki) que abstrae la paginación, el rate limiting, el re-intento ante errores y la serialización de payloads. Es la opción recomendada para producción, pero entender requests puro es esencial para el examen y para debuguear.

🐍

SDK meraki

Clase DashboardAPI. Auto-retry, logging integrado, manejo de 429, soporte async. Ideal para scripts largos y orquestación.

📡

requests puro

Control total sobre headers, paginación y errores. Útil para entender los fundamentos y para lenguajes donde no hay SDK.

🧪

Postman / Insomnia

Colecciones oficiales en Postman Workspace. Ideal para explorar endpoints, guardar ejemplos y compartir con el equipo.

Colección oficial de Postman

Cisco Meraki mantiene una colección oficial en el Meraki Developer Hub. Pasos para importar:

  1. Descarga la colección JSON desde https://developer.cisco.com/meraki/api-v1/
  2. En Postman: File > Import y selecciona el JSON.
  3. Crea un Environment con variables: baseUrl, apiKey, orgId, netId.
  4. En la pestaña Authorization de la colección, selecciona tipo API Key:
    • Key: X-Cisco-Meraki-API-Key
    • Value: {{apiKey}}
    • Add to: Header
💡 ¿Cuándo usar cada herramienta?
  • Postman/Insomnia: Descubrimiento de API, pruebas rápidas, documentación para equipos no-dev.
  • requests puro: Scripts simples, entornos sin permisos para instalar librerías, o cuando necesitas interceptar/modificar headers exóticos.
  • SDK meraki: Cualquier proyecto que supere 50 líneas de código. El manejo automático de rate limits y paginación ahorra horas de debugging.

🚀 Proyecto Final — NOC Dashboard Meraki

Este proyecto integra todo: autenticación con el SDK, paginación automática, rate limiting, inventario multi-org, monitoreo de uplinks, exportación CSV y un receptor de alertas webhook. Es un centro de operaciones de red (NOC) en Python que puede ejecutarse en un contenedor Docker o una Raspberry Pi en la oficina.

💡 Escenario de producción

Gestionas 120 tiendas retail con Meraki MX+MS+MR. Cada mañana necesitas validar que ningún uplink está caído, exportar el inventario completo para finanzas (depreciación de activos), y recibir en Slack cualquier caída de VPN entre sucursales y la central. Este script lo automatiza.

noc_dashboard.py — Orquestador Meraki completo
#!/usr/bin/env python3
"""
noc_dashboard.py
CCNA Academy M8 T5 — NOC Dashboard para Meraki Dashboard API
─────────────────────────────────────────────────────────────
- Multi-org inventory con export CSV
- Uplink health check (loss & latency)
- Alert webhook server con validación HMAC
- Logging profesional y manejo de rate limits via SDK
"""

import os
import csv
import sys
import json
import hmac
import hashlib
import logging
from datetime import datetime
from flask import Flask, request

try:
    import meraki
except ImportError:
    print("pip install meraki flask")
    sys.exit(1)

# ══════════════════════════════════════════════════════
# CONFIGURACIÓN
# ══════════════════════════════════════════════════════
API_KEY        = os.getenv("MERAKI_API_KEY")
ORG_IDS        = os.getenv("MERAKI_ORG_IDS", "").split(",")
WEBHOOK_SECRET = os.getenv("WEBHOOK_SECRET", "cambia-esto")
REPORT_DIR     = "./meraki_reports"

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%H:%M:%S"
)

# ══════════════════════════════════════════════════════
# CLIENTE MERAKI
# ══════════════════════════════════════════════════════
class MerakiNOC:
    def __init__(self):
        self.dashboard = meraki.DashboardAPI(
            api_key=API_KEY,
            wait_on_rate_limit=True,
            output_log=False,
            print_console=False
        )

    def export_inventory(self):
        """Exporta CSV de dispositivos de todas las orgs configuradas."""
        os.makedirs(REPORT_DIR, exist_ok=True)
        ts = datetime.now().strftime("%Y%m%d_%H%M%S")
        fname = f"{REPORT_DIR}/inventory_{ts}.csv"

        with open(fname, "w", newline="") as f:
            writer = csv.writer(f)
            writer.writerow(["Org", "Network", "Name", "Serial", "Model", "MAC", "Firmware", "Status"])

            for org_id in ORG_IDS:
                if not org_id:
                    continue
                try:
                    org = self.dashboard.organizations.getOrganization(org_id)
                    devices = self.dashboard.organizations.getOrganizationDevices(
                        organizationId=org_id,
                        total_pages='all'
                    )
                    statuses = self.dashboard.organizations.getOrganizationDevicesStatuses(
                        organizationId=org_id,
                        total_pages='all'
                    )
                    status_map = {s["serial"]: s["status"] for s in statuses}

                    for d in devices:
                        writer.writerow([
                            org["name"],
                            d.get("networkId", "N/A"),
                            d.get("name", "N/A"),
                            d["serial"],
                            d["model"],
                            d["mac"],
                            d.get("firmware", "N/A"),
                            status_map.get(d["serial"], "unknown")
                        ])
                    logging.info(f"[{org['name']}] {len(devices)} dispositivos exportados")
                except Exception as e:
                    logging.error(f"Error en org {org_id}: {e}")

        logging.info(f"✅ Reporte guardado: {fname}")
        return fname

    def check_uplink_health(self):
        """Verifica pérdida y latencia de uplinks por org."""
        for org_id in ORG_IDS:
            if not org_id:
                continue
            try:
                uplinks = self.dashboard.organizations.getOrganizationDevicesUplinksLossAndLatency(
                    organizationId=org_id,
                    total_pages='all'
                )
                for dev in uplinks:
                    for uplink in dev.get("uplinks", []):
                        loss = uplink.get("lossPercent", 0)
                        lat  = uplink.get("latencyMs", 0)
                        iface = uplink.get("interface", "?")
                        if loss > 5 or lat > 200:
                            logging.warning(
                                f"⚠️  {dev['serial']} [{iface}] — Loss: {loss}% | Lat: {lat}ms"
                            )
            except Exception as e:
                logging.error(f"Error uplinks org {org_id}: {e}")

# ══════════════════════════════════════════════════════
# WEBHOOK SERVER
# ══════════════════════════════════════════════════════
app = Flask("__main__")
noc = MerakiNOC()

@app.route("/webhook/meraki", methods=["POST"])
def receive_alert():
    payload = request.get_data()
    sig = request.headers.get("X-Meraki-Signature", "")
    expected = hmac.new(WEBHOOK_SECRET.encode(), payload, hashlib.sha256).hexdigest()

    if not hmac.compare_digest(sig, expected):
        return "Unauthorized", 401

    data = request.get_json()
    logging.warning(
        f"🚨 [{data.get('alertType')}] {data.get('deviceName')} — {data.get('occurredAt')}"
    )
    return "OK", 200

# ══════════════════════════════════════════════════════
# EJECUCIÓN PRINCIPAL
# ══════════════════════════════════════════════════════
def main():
    print("\n☁️  Meraki NOC Dashboard — CCNA Academy M8 T5\n")
    noc.export_inventory()
    noc.check_uplink_health()
    logging.info("🏁 NOC check completado.")

if __name__ == "__main__":
    main()
